## 📚 日志体系设计

### 1. 日志级别与使用场景

```java
// ❌ 错误示例：日志级别混乱
logger.info("用户登录失败: " + e.getMessage()); // 应该用 WARN 或 ERROR
logger.debug("订单创建成功"); // 应该用 INFO
logger.error("查询用户列表"); // 不是错误,不该用 ERROR

// ✅ 正确示例：日志级别清晰
logger.trace("进入 getUserById 方法,参数: userId={}", userId);  // 最详细,生产环境关闭
logger.debug("SQL执行: {}, 耗时: {}ms", sql, duration);          // 开发调试用
logger.info("用户 {} 登录成功", username);                        // 关键业务流程
logger.warn("Redis连接失败,降级到数据库查询", e);                // 可恢复的异常
logger.error("订单支付失败: orderId={}", orderId, e);            // 严重错误
```

### 2. 日志规范设计

#### 日志格式标准
```
[时间] [日志级别] [线程名] [类名] [TraceId] - 日志内容

示例:
2024-11-29 14:30:15.123 INFO  [http-nio-8080-exec-1] c.s.service.UserService [abc123def456] - 用户登录成功: username=zhangsan, ip=192.168.1.100
```

#### 关键字段说明
```java
@Slf4j
@RestController
public class UserController {
    
    @GetMapping("/user/{id}")
    public User getUser(@PathVariable Long id) {
        // ✅ 包含关键信息的日志
        log.info("查询用户: userId={}, ip={}, userAgent={}", 
            id, 
            HttpContextUtil.getClientIP(), 
            HttpContextUtil.getUserAgent()
        );
        
        // ✅ 性能日志
        long startTime = System.currentTimeMillis();
        User user = userService.getUserById(id);
        long duration = System.currentTimeMillis() - startTime;
        
        log.info("查询用户完成: userId={}, duration={}ms", id, duration);
        
        return user;
    }
}
```

## 🔍 Docker 日志高级搜索

### 1. 基础搜索技巧

```bash
# ==================== 按时间范围搜索 ====================
# 查看最近1小时的日志
docker logs --since 1h archat-backend

# 查看最近30分钟的日志
docker logs --since 30m archat-backend

# 查看指定时间段的日志
docker logs --since "2024-11-29T14:00:00" --until "2024-11-29T15:00:00" archat-backend

# ==================== 按行数搜索 ====================
# 最新100行
docker logs --tail 100 archat-backend

# 最新1000行并实时滚动
docker logs --tail 1000 -f archat-backend

# ==================== 按关键字搜索 ====================
# 查找ERROR级别日志
docker logs archat-backend 2>&1 | grep "ERROR"

# 查找包含特定用户的日志
docker logs archat-backend 2>&1 | grep "userId=1001"

# 查找多个关键字(OR关系)
docker logs archat-backend 2>&1 | grep -E "ERROR|WARN|Exception"

# 查找同时包含多个关键字(AND关系)
docker logs archat-backend 2>&1 | grep "userId=1001" | grep "订单"
```

### 2. 高级过滤技巧

```bash
# ==================== 提取特定字段 ====================
# 只显示时间和错误信息
docker logs archat-backend 2>&1 | grep "ERROR" | awk '{print $1, $2, $NF}'

# 统计每个错误类型出现次数
docker logs archat-backend 2>&1 | grep "Exception" | awk -F: '{print $1}' | sort | uniq -c | sort -rn

# ==================== 按TraceId追踪完整请求链路 ====================
TRACE_ID="abc123def456"
docker logs archat-backend 2>&1 | grep "$TRACE_ID" | less

# 导出到文件方便分析
docker logs archat-backend 2>&1 | grep "$TRACE_ID" > /tmp/trace_${TRACE_ID}.log

# ==================== 正则表达式搜索 ====================
# 查找所有慢SQL(耗时>1000ms)
docker logs archat-backend 2>&1 | grep -E "SQL.*duration=[0-9]{4,}ms"

# 查找特定IP的访问日志
docker logs archat-backend 2>&1 | grep -E "ip=192\.168\.1\.[0-9]+"

# 查找所有5xx错误
docker logs archat-backend 2>&1 | grep -E "status=5[0-9]{2}"
```

### 3. 性能分析技巧

```bash
# ==================== 统计QPS ====================
# 统计每秒请求数
docker logs archat-backend 2>&1 | grep "请求完成" | awk '{print $1}' | uniq -c

# ==================== 找出最慢的10个接口 ====================
docker logs archat-backend 2>&1 | grep "duration=" | \
  awk -F'duration=' '{print $2}' | \
  awk '{print $1}' | \
  sed 's/ms//' | \
  sort -rn | \
  head -10

# ==================== 分析错误趋势 ====================
# 按小时统计错误数
docker logs archat-backend 2>&1 | grep "ERROR" | \
  awk '{print $1 " " substr($2,1,2)}' | \
  sort | uniq -c

# 输出示例:
#  15 2024-11-29 14  (14点有15个错误)
#  42 2024-11-29 15  (15点有42个错误)
```

## 📊 日志分析实战案例

### 案例1: 找出频繁失败的接口

#### 场景
监控告警: "接口错误率突增到10%"

#### 分析步骤

**第1步: 统计错误分布**
```bash
# 统计每个接口的错误次数
docker logs archat-backend 2>&1 | \
  grep "ERROR" | \
  grep -oP 'uri=/api/\S+' | \
  sort | uniq -c | sort -rn

# 输出示例:
#  156 uri=/api/order/create
#   45 uri=/api/user/profile
#   12 uri=/api/payment/callback
```

**第2步: 分析具体错误原因**
```bash
# 查看订单接口的所有错误
docker logs archat-backend 2>&1 | \
  grep "ERROR" | \
  grep "uri=/api/order/create" | \
  grep -oP 'Exception: \K.*' | \
  sort | uniq -c

# 输出示例:
#  120 Connection refused (数据库连接失败)
#   30 NullPointerException (空指针)
#    6 TimeoutException (超时)
```

**第3步: 找到典型错误的完整堆栈**
```bash
# 找到第一个NullPointerException的完整日志
docker logs archat-backend 2>&1 | \
  grep -A 20 "NullPointerException" | \
  head -25 > /tmp/npe_stacktrace.log

# 查看
cat /tmp/npe_stacktrace.log
```

**结论**: 156个错误中,120个是数据库连接问题 → 检查数据库连接池配置

### 案例3: 性能瓶颈分析

#### 场景
Grafana显示: 接口P99延迟从200ms增加到5s

#### 分析步骤

**第1步: 找出最慢的请求**
```bash
# 提取所有请求的耗时
docker logs archat-backend 2>&1 | \
  grep "请求完成" | \
  awk -F'duration=' '{print $2}' | \
  awk '{print $1}' | \
  sed 's/ms//' | \
  awk '$1 > 3000 {print $1}' | \
  wc -l

# 结果: 45个请求耗时超过3秒
```

**第2步: 分析慢在哪个环节**
```bash
# 找到一个慢请求的TraceId
docker logs archat-backend 2>&1 | \
  grep "duration=" | \
  grep -oP 'duration=[3-9][0-9]{3}ms.*?\[.*?\]' | \
  head -1

# 输出: duration=5200ms [def789]

# 追踪这个请求
TRACE_ID="def789"
docker logs archat-backend 2>&1 | grep "\\[$TRACE_ID\\]"

# 输出(关注每步的时间戳):
# 14:35:10.100 INFO  [def789] 接收请求
# 14:35:10.150 DEBUG [def789] 查询缓存 (耗时: 50ms)
# 14:35:10.200 DEBUG [def789] 缓存未命中,查询数据库
# 14:35:15.200 DEBUG [def789] 数据库查询完成 (耗时: 5000ms ← 这里慢了!)
# 14:35:15.300 INFO  [def789] 请求完成
```

**结论**: 数据库查询耗时5秒 → 检查慢查询

## 🛠️ 日志聚合方案(进阶)

### 方案1: ELK Stack (Elasticsearch + Logstash + Kibana)

#### 适用场景
- 日志量大(GB/天级别)
- 需要全文搜索
- 多个服务的日志需要统一管理

#### 快速部署
```yaml
# docker-compose.elk.yml
version: '3.8'

services:
  elasticsearch:
    image: elasticsearch:8.11.0
    environment:
      - discovery.type=single-node
      - "ES_JAVA_OPTS=-Xms512m -Xmx512m"
      - xpack.security.enabled=false
    ports:
      - "9200:9200"
    volumes:
      - es-data:/usr/share/elasticsearch/data

  logstash:
    image: logstash:8.11.0
    volumes:
      - ./logstash/pipeline:/usr/share/logstash/pipeline
    ports:
      - "5044:5044"
    depends_on:
      - elasticsearch

  kibana:
    image: kibana:8.11.0
    ports:
      - "5601:5601"
    environment:
      - ELASTICSEARCH_HOSTS=http://elasticsearch:9200
    depends_on:
      - elasticsearch

volumes:
  es-data:
```

#### Logstash 配置
```ruby
# logstash/pipeline/logstash.conf
input {
  tcp {
    port => 5044
    codec => json_lines
  }
}

filter {
  # 解析日志
  grok {
    match => { "message" => "%{TIMESTAMP_ISO8601:timestamp} %{LOGLEVEL:level} %{DATA:thread} %{DATA:class} \[%{DATA:traceId}\] - %{GREEDYDATA:msg}" }
  }
  
  # 转换时间格式
  date {
    match => [ "timestamp", "ISO8601" ]
  }
}

output {
  elasticsearch {
    hosts => ["elasticsearch:9200"]
    index => "archat-logs-%{+YYYY.MM.dd}"
  }
}
```

#### 应用配置(Logback)
```xml
<!-- logback-spring.xml -->
<configuration>
    <!-- ELK输出 -->
    <appender name="LOGSTASH" class="net.logstash.logback.appender.LogstashTcpSocketAppender">
        <destination>localhost:5044</destination>
        <encoder class="net.logstash.logback.encoder.LogstashEncoder"/>
    </appender>
    
    <root level="INFO">
        <appender-ref ref="LOGSTASH" />
    </root>
</configuration>
```

## 🐛 线上Debug技巧

### 1. 安全的线上调试方法

```
✅ 允许的方法:
```

### 2. Arthas 动态诊断

#### 安装Arthas

#### 常用命令

**查看方法调用情况**

### 1. 日志分析脚本
```bash
#!/bin/bash
# analyze_logs.sh - 自动分析日志

echo "=== 日志分析报告 ==="

  awk -F: '{print $NF}' | sort | uniq -c | sort -rn | head -10

echo ""
echo "3. 最慢的10个请求"

  grep -oP 'duration=\K[0-9]+' | sort -rn | head -10 | \
  awk '{print $1 "ms"}'

echo ""

docker logs --since 1h archat-backend 2>&1 | grep "请求完成" | wc -l | \
```

### 2. 日志告警脚本
```bash
#!/bin/bash
# alert_on_errors.sh - 错误日志告警

THRESHOLD=10  # 阈值: 1分钟内错误超过10次

ERROR_COUNT=$(docker logs --since 1m archat-backend 2>&1 | grep -c "ERROR")

if [ $ERROR_COUNT -gt $THRESHOLD ]; then
    echo "告警: 最近1分钟内发生 $ERROR_COUNT 个错误(阈值: $THRESHOLD)"
    
    # 发送钉钉通知
    curl -X POST https://oapi.dingtalk.com/robot/send?access_token=YOUR_TOKEN \
      -H 'Content-Type: application/json' \
      -d "{
        \"msgtype\": \"text\",
        \"text\": {
          \"content\": \"【告警】应用错误率过高: ${ERROR_COUNT}次/分钟\"
        }
      }"
fi
```
