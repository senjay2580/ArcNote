# 大数据集群 及其安装

### 1. **HDFS（Hadoop Distributed File System）**

- **作用**：分布式文件系统，专为大数据存储设计。
- **特点**：
  - 高容错（副本机制）
  - 面向流式数据访问
  - 适合存储大文件（如 TB、PB 级）

------

### 2. **MapReduce**

- **作用**：Hadoop 最早期的计算框架，适合批处理任务。
- **执行方式**：分为两个阶段：
  - Map 阶段：分发任务并生成中间键值对
  - Reduce 阶段：汇总处理结果
- **缺点**：计算速度慢（尤其是中间结果需要写入磁盘）

------

### 3. **YARN（Yet Another Resource Negotiator）**

- **作用**：Hadoop 的资源管理系统，负责管理集群的资源并调度任务。
- **工作方式**：
  - ResourceManager 管控全局资源
  - NodeManager 负责每台机器的任务运行情况
  - 应用程序（如 MapReduce、Spark、Flink）通过 YARN 提交任务

------

### 4. **Spark**

- **作用**：内存计算框架，替代 MapReduce 的更快的大数据处理工具。
- **特点**：
  - 支持批处理、流处理、机器学习等
  - 高效利用内存（RDD/DAG 调度机制）
  - 可以运行在 YARN 上、Kubernetes、独立集群等

------

### 5. **Flink**

- **作用**：现代化的大数据流处理框架，适合实时计算。
- **特点**：
  - 原生流处理（即使是批处理，也是流模拟出来的）
  - 精确一次语义（exactly-once）
  - 高吞吐、低延迟，适用于实时监控、报警系统

------

## 二、它们之间的关系

| 系统组件      | 类别         | 依赖/关系                                     |
| ------------- | ------------ | --------------------------------------------- |
| **HDFS**      | 分布式存储   | MapReduce、Spark、Flink 都可用它作为底层存储  |
| **MapReduce** | 批处理引擎   | 运行在 YARN 上，读写 HDFS                     |
| **YARN**      | 资源管理器   | 管理集群资源，调度 Spark/Flink/MapReduce      |
| **Spark**     | 通用计算引擎 | 可运行在 YARN、独立模式、K8s，支持批/流处理   |
| **Flink**     | 实时计算引擎 | 也可运行在 YARN，擅长事件流处理和复杂状态计算 |

| 技术          | 当前状态                      | 是否推荐继续学？                          |
| ------------- | ----------------------------- | ----------------------------------------- |
| **HDFS**      | ✅ 仍是主流存储                | 推荐（除非你使用云原生替代，如 S3）       |
| **YARN**      | ✅ 仍在使用，但有 K8s 替代趋势 | 推荐了解                                  |
| **MapReduce** | ❌ **已过时**，新项目极少用    | ❗**不推荐用于项目，作为历史知识了解即可** |
| **Spark**     | ✅ 企业主流批处理框架          | 强烈推荐                                  |
| **Flink**     | ✅ 实时计算的主流工具          | 推荐进阶学习（尤其是大厂/实时需求场景）   |

| 应用目标 | 数据分析、报表、离线模型训练等       | 实时监控、风控、告警、在线推荐等      |

**Standby NameNode** 是为了在 Active NameNode 宕机时接管服务的（热）备用节点，确保 HDFS 不停服务。

每个block块要备份到不同结点里面 （与kafka类似）

NameNode（负责**管理**整个 HDFS 文件系统的元数据） 存储**元数据**

| 元数据内容                      | 说明                                           |
| ------------------------------- | ---------------------------------------------- |
| 文件和目录的命名空间            | 类似 Linux 的 `/user/hadoop/data.txt` 路径结构 |
| 每个文件被切分成哪些块（Block） | 默认 128MB 一块，每个文件可能由多个块组成      |
| 每个块存储在哪些 DataNode 上    | 比如：Block1 -> DN1, DN3, DN5                  |

![image-20250720112830788](C:/Users/33813/AppData/Roaming/Typora/typora-user-images/image-20250720112830788.png)

从节点（DataNode）会通过心跳机制定期上报 汇报的信息包括 这个结点存了哪些的block、结点的健康状态



因为 NameNode 要快速响应客户端的读写请求，所有操作都依赖这些元数据：

- 查找文件 → 查找文件的 block → 查找 block 在哪些 DataNode
- 如果这些信息存在磁盘，每次都读取，会非常慢
   ✅ 所以 **存内存中，速度快、延迟低**

## HDFS高可用

NameNode **负责维护元数据（Metadata）**，例如**文件目录结构**、文件到块的映射、块到 DataNode 的位置。

所有这些信息都保存在 **内存中**！

> HDFS 联邦机制是将一个大规模的 HDFS 文件系统拆分成多个独立的命名空间（Namespace），每个命名空间由一个独立的 NameNode 管理，多个 **NameNode 并行工作，共享一组 DataNode，实现 NameNode 水平扩展。**

**命名空间（Namespace）**是**文件系统中所有路径（目录和文件）组成的整体集合**，也可以理解为文件系统的**完整目录树结构和文件命名的总和**。

它包含了目录结构，也包含了文件在这个目录结构中的位置。





联邦解决的是**扩展性问题**；

HA 解决的是**单点故障问题**。
