## **Caffeine** （也可与 Spring Cache 集成）

JDK内置的Map可作为缓存的一种实现方式，然而严格意义来讲，其不能算作缓存的范畴。

~~~java
@Configuration
@EnableCaching //开启缓存
public class CaffeineConfig {
    public static final int DEFAULT_MAXSIZE = 10000;
    public static final int DEFAULT_TTL = 600;
    /**
     * 定义cache名称、超时时长（秒）、最大容量
     * 每个cache缺省：10秒超时、最多缓存50000条数据，需要修改可以在构造方法的参数中指定。
     */
    public enum Caches{
        getUserById(600),          //有效期600秒
        listCustomers(7200,1000),  //有效期2个小时 , 最大容量1000
        ;
        Caches() {
        }
        Caches(int ttl) {
            this.ttl = ttl;
        }
        Caches(int ttl, int maxSize) {
            this.ttl = ttl;
            this.maxSize = maxSize;
        }
        private int maxSize=DEFAULT_MAXSIZE;	//最大數量
        private int ttl=DEFAULT_TTL;		//过期时间（秒）
        public int getMaxSize() {
            return maxSize;
        }
        public int getTtl() {
            return ttl;
        }
    }

    /**
     * 创建基于Caffeine的Cache Manager
     * @return
     */
    @Bean
    @Primary
    public CacheManager caffeineCacheManager() {
        SimpleCacheManager cacheManager = new SimpleCacheManager();
        ArrayList<CaffeineCache> caches = new ArrayList<CaffeineCache>();
        for(Caches c : Caches.values()){
            caches.add(new CaffeineCache(c.name(),
                    Caffeine.newBuilder().recordStats()
                            .expireAfterWrite(c.getTtl(), TimeUnit.SECONDS)
                            .maximumSize(c.getMaxSize())
                            .build())
            );
        }
        cacheManager.setCaches(caches);
        return cacheManager;
    }
}
~~~

# 浏览器缓存

**常见注意问题：**

| 后端控制的 HTTP 缓存 | localStorage                                    |                                            |
| -------------------- | ----------------------------------------------- | ------------------------------------------ |
| **控制方**           | 后端（服务器通过 HTTP 头控制）                  | 前端（JavaScript 直接操作）                |
| **存储位置**         | 浏览器的缓存目录（磁盘 / 内存，用户通常不可见） | 浏览器的本地数据库（可通过开发者工具查看） |
| **生命周期**         | 由服务器指定（如 `max-age` 控制过期时间）       | 永久有效（除非手动删除）                   |
| **访问范围**         | 浏览器自动用于 HTTP 请求，服务器可间接感知      | 仅前端 JS 可读写，服务器无法直接访问       |
| **存储内容**         | 主要是 HTTP 响应资源（文件、接口数据等）        | 字符串格式的键值对（需手动序列化对象）     |
| **核心目的**         | 减少网络请求，提升资源加载效率                  | 前端本地数据持久化和共享                   |

对一个网站而言，CSS、javascript、logo、图标这些静态资源文件更新的频率都比较低，而这些文件又几乎是每次 http 请求都需要的，如果将这些文件缓存在浏览器中，可以极好的改善性能。

缓存可以说是性能优化中简单高效的一种优化方式了。一个优秀的缓存策略可以缩短网页请求资源的距离，减少延迟，并且由于缓存文件可以重复利用，还可以减少带宽，降低网络负荷。

对于一个数据请求来说，可以分为发起网络请求、后端处理、浏览器响应三个步骤。浏览器缓存可以帮助我们在第一和第三步骤中优化性能，比如说直接使用缓存而不发起请求，或者发起了请求但后端存储的数据和前端一致，那么就没有必要再将数据回传回来，这样就减少了响应数据。

通过设置 http 头中的 **cache-control 和 expires 的属性和 HTML Meta 标签，**可**设定浏览器缓存**(配置)，缓存时间可以是数天，甚至是几个月。

web 资源的缓存策略一般由服务器来指定，可以分为两种，分别是**强缓存策略和协商缓存策略。**

## 协商缓存（思想要理解）

### `Last-Modified`的局限性

`Last-Modified`基于资源的修改时间判断是否更新，但存在以下缺陷：

- **时间精度不足**：只能精确到秒级（部分服务器可能精确到毫秒，但 HTTP 协议头中时间格式为秒级）。如果资源在 1 秒内被多次修改（比如 1 秒内编辑并保存了 2 次），`Last-Modified`会认为资源未变，导致客户端复用旧缓存，出现数据不一致。
- **内容未变但时间变了**：比如手动修改了文件但未改变内容（如误触保存），`Last-Modified`会更新（时间变了），但资源实际内容没变。此时服务器会误判为 “资源已更新”，返回完整资源，浪费带宽。
- **部分资源无修改时间**：比如动态生成的资源（如服务器实时计算的 JSON），可能没有明确的 “最后修改时间”，`Last-Modified`无法有效生成。

### 2. `ETag`的优势：基于内容的唯一标识

`ETag`是服务器对资源内容计算的唯一标识（通常是哈希值或版本号），内容不变则标识不变，完美解决了`Last-Modified`的上述问题：

- **精度更高**：哪怕资源内容只有 1 个字符的变化，哈希值也会完全不同，能识别秒级内的修改。
- **无视无关时间变化**：只要内容没变，无论文件的修改时间如何变动，`ETag`都保持一致，避免 “假更新” 导致的无效传输。
- **适配动态资源**：对于动态生成的资源（如数据库查询结果），服务器可根据内容计算`ETag`，无需依赖文件系统的修改时间。

### 3. 实际应用：两者结合使用

HTTP 协议允许服务器同时返回`Last-Modified`和`ETag`，客户端会同时携带`If-Modified-Since`和`If-None-Match`到服务器。此时：

- 服务器会**优先校验`ETag`**（因为精度更高），若`ETag`匹配，则直接返回 304；
- 若`ETag`不匹配，再校验`Last-Modified`（作为补充判断），最终决定是否返回新资源。

这种组合既利用了`Last-Modified`的低成本（读取文件时间比计算哈希更简单），又通过`ETag`保证了缓存判断的准确性，是更可靠的协商缓存方案。

简单说：`Last-Modified`是 “看时间猜变化”，`ETag`是 “看内容定变化”，后者更精准，两者结合能覆盖更多场景。

使用Spring Cache的好处：

    1，提供基本的Cache抽象，方便切换各种底层Cache；

    4，支持比较复杂的缓存逻辑；



**Redis 序列化问题**

如果不配置 `RedisTemplate`，直接用 Redis 原生 API 操作数据会很繁琐（需要自己处理连接、序列化等）。配置后：

- **统一序列化方式**：避免默认序列化导致的乱码问题（比如键用 `StringRedisSerializer` 存为字符串，值用 `Jackson2JsonRedisSerializer` 存为 JSON）。
- **简化操作**：通过 `redisTemplate.opsForValue().set(key, value)`、`redisTemplate.opsForHash().put(...)` 等方法，轻松操作 Redis 的各种数据类型（字符串、哈希、列表等）。

~~~java
@Configuration
@EnableCaching
public class RedisConfig {
 
/**
     * 自定义key规则
     * @return
*/
@Bean
public KeyGenerator keyGenerator() {
return new KeyGenerator() {
@Override
public Object generate(Object target, Method method, Object... params) {
                StringBuilder sb = new StringBuilder();
                sb.append(target.getClass().getName());
                sb.append(method.getName());
                for (Object obj : params) {
                    sb.append(obj.toString());
                }
                return sb.toString();
            }
        };
    }
 
/**
     * 设置RedisTemplate规则
     * @param redisConnectionFactory
* @return
*/
@Bean
public RedisTemplate<Object, Object> redisTemplate(RedisConnectionFactory redisConnectionFactory) {
        RedisTemplate<Object, Object> redisTemplate = new RedisTemplate<>();
        redisTemplate.setConnectionFactory(redisConnectionFactory);
        Jackson2JsonRedisSerializer jackson2JsonRedisSerializer = new Jackson2JsonRedisSerializer(Object.class);
 
//解决查询缓存转换异常的问题
ObjectMapper om = new ObjectMapper();
// 指定要序列化的域，field,get和set,以及修饰符范围，ANY是都有包括private和public
om.setVisibility(PropertyAccessor.ALL, JsonAutoDetect.Visibility.ANY);
// 指定序列化输入的类型，类必须是非final修饰的，final修饰的类，比如String,Integer等会跑出异常
om.enableDefaultTyping(ObjectMapper.DefaultTyping.NON_FINAL);
        jackson2JsonRedisSerializer.setObjectMapper(om);
 
//序列号key value
redisTemplate.setKeySerializer(new StringRedisSerializer());
        redisTemplate.setValueSerializer(jackson2JsonRedisSerializer);
        redisTemplate.setHashKeySerializer(new StringRedisSerializer());
        redisTemplate.setHashValueSerializer(jackson2JsonRedisSerializer);
 
        redisTemplate.afterPropertiesSet();
return redisTemplate;
    }
 
/**
     * 设置CacheManager缓存规则
     * @param factory
* @return
*/
@Bean
public CacheManager cacheManager(RedisConnectionFactory factory) {
        RedisSerializer<String> redisSerializer = new StringRedisSerializer();
        Jackson2JsonRedisSerializer jackson2JsonRedisSerializer = new Jackson2JsonRedisSerializer(Object.class);
 
//解决查询缓存转换异常的问题
ObjectMapper om = new ObjectMapper();
        om.setVisibility(PropertyAccessor.ALL, JsonAutoDetect.Visibility.ANY);
        om.enableDefaultTyping(ObjectMapper.DefaultTyping.NON_FINAL);
        jackson2JsonRedisSerializer.setObjectMapper(om);
 
// 配置序列化（解决乱码的问题）,过期时间600秒
RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofSeconds(600))
                .serializeKeysWith(RedisSerializationContext.SerializationPair.fromSerializer(redisSerializer))
                .serializeValuesWith(RedisSerializationContext.SerializationPair.fromSerializer(jackson2JsonRedisSerializer))
                .disableCachingNullValues();
 
        RedisCacheManager cacheManager = RedisCacheManager.builder(factory)
                .cacheDefaults(config)
                .build();
return cacheManager;
    }
}
~~~

当 Spring Cache 底层使用 Redis 作为缓存介质时，最终存储在 Redis 中的键是 **“缓存分区（value）+ 自定义逻辑键” 的组合 **，格式通常为：

```plaintext
缓存分区::自定义逻辑键  
```

#### 举例说明：

假设使用如下缓存注解：

```java
@Cacheable(value = "userCache", keyGenerator = "keyGenerator")  
public User getUserById(Long id) { ... }  
```

调用 `getUserById(1)` 时：

1. 自定义KeyGenerator 

   生成的逻辑键为：

   ```
   com.example.service.UserServicegetUserById1
   ```

   （类名 + 方法名 + 参数）。

2. Redis 中实际存储的键为：

   ```
   userCache::com.example.service.UserServicegetUserById1
   ```

其中：

- `userCache` 是 `@Cacheable` 中 `value` 指定的缓存分区（逻辑分组）。
- `::` 是 Spring 默认的分隔符（可通过配置修改）。
- 后面的部分是自定义 `KeyGenerator` 生成的逻辑键。

| 注解           | 作用                                                         | 典型场景                                           |
| -------------- | ------------------------------------------------------------ | -------------------------------------------------- |
| `@Cacheable`   | **查缓存**：先查缓存，未命中则执行方法并将结果存入缓存       | 商品详情查询、用户信息查询（高频读场景）           |
| `@CachePut`    | **更缓存**：执行方法后，将结果更新到缓存                     | 商品信息修改后同步更新缓存                         |
| `@CacheEvict`  | **删缓存**：执行方法后，删除指定缓存                         | 商品删除后移除缓存，避免脏数据                     |
| `@Caching`     | **组合操作**：同时配置多个缓存注解（如查 + 删）              | 复杂场景（如更新订单后，既删列表缓存又更详情缓存） |
| `@CacheConfig` | **类级别配置**：统一指定类中缓存的 `value`（缓存名）、`keyGenerator` 等 | 减少重复配置，简化代码                             |



`@CacheConfig` 是 SpringCache 提供的**类级别缓存配置注解**，作用是**将当前类中所有缓存注解（如 `@Cacheable`、`@CachePut` 等）的公共配置抽取到类上统一声明**，从而减少重复代码，简化配置。

#  缓存注意事项

- **缓存穿透防护**：

对null值进行适当缓存（使用unless = "#result == null"）
考虑使用Bloom过滤器

- **缓存雪崩防护：**

为不同缓存设置不同的过期时间
添加**随机抖动因子**到过期时间

- **缓存一致性：**

重要数据建议配合数据库事务
考虑使用@CachePut更新策略

- **内存管理：**

合理设置maximumSize防止OOM
对大对象考虑使用weakValues()或softValues()

- **分布式环境：**

本地缓存需要配合**消息总线**实现**多节点同步**
或考虑使用多级缓存（本地+Redis）


# 缓存实战基础到进阶

|缓存层级|位置|核心作用|特点|
|---|---|---|---|
|浏览器缓存|用户浏览器本地|减少前端重复请求|离用户最近，性能最好，可控性低|
|客户端缓存|前端应用内存 / 本地存储|缓存前端常用数据（如用户信息）|前端自主控制，仅当前页面 / 会话有效|
|中间件缓存|Nginx/CDN 等反向代理|缓存静态资源 / 通用接口数据|介于前端和后端之间，减轻后端压力|
|本地缓存|后端服务器进程内|缓存高频热点数据|读写极快，分布式场景下不一致|
|Redis 缓存|分布式缓存服务器|缓存后端通用数据|读写快，支持分布式，可控性高|
对于刚入门的开发者，**优先落地「浏览器 HTTP 缓存 + 后端 Redis 缓存」** 即可满足 80% 的小型项目需求，无需一上来全量配置所有层级。

- 理由：这两层覆盖了「减少前端请求」和「减少后端数据库压力」的核心目标，配置简单、易验证、问题少。
- 进阶方向：后续再添加「中间件缓存（Nginx）」和「后端本地缓存」。


前后端分服务器时，前端静态资源（CSS/JS/ 图片）部署在**前端服务器（Nginx/CDN）**，后端接口部署在**后端服务器（Node.js/Java）**：

- 前端静态资源的请求直接发给前端服务器 / CDN，后端服务器根本接收不到这些请求，自然无法配置响应头；
- 只有「直接返回资源的服务器」才能给响应加 `Cache-Control` 头 —— 也就是前端服务器 / CDN。

### 1. 第一层：浏览器 HTTP 缓存（前端 / 后端配置）

浏览器缓存是最外层缓存，核心通过**HTTP 响应头**控制，新手只需掌握 2 类缓存规则：

#### （1）强缓存（优先命中，不发请求）

- 配置方式：后端返回 `Cache-Control` 响应头（主流），比如设置静态资源缓存 1 小时。

#### （2）协商缓存（发请求，验证是否过期）

适用于动态数据（如用户信息接口），避免强缓存导致数据过期。

- 配置方式：后端返回 `ETag`（资源唯一标识）或 `Last-Modified`（资源最后修改时间）。

具体内容见上

---

###  第二层：后端 Redis 缓存（核心）

Redis 是分布式缓存，新手只需掌握「缓存读取 - 回写 - 删除」的基础逻辑，配合过期时间兜底。

### 后端本地缓存

本地缓存（如 Node.js 的`lru-cache`、Java 的 Caffeine）是服务器进程内的缓存，读写比 Redis 更快，但分布式部署时会有一致性问题，适合缓存「非核心热点数据」。

### 中间件缓存

Nginx 缓存介于浏览器和后端之间，适合缓存静态资源或通用接口


**后端 Redis + 数据库的一致性是核心难题**（因为涉及并发、分布式），前端缓存一致性解决起来更简单


```ad-tip
### 一、先理解：前端缓存一致性的核心矛盾
缓存的本质是「用空间换时间」，但会带来「数据过期」问题。前端缓存一致性的核心目标是：
- 缓存未过期时，优先用缓存（保证性能）；
- 缓存过期/后端数据更新时，能精准更新缓存（保证数据最新）；
- 避免「全量禁用缓存」（牺牲性能）或「全量用缓存」（数据不一致）的极端情况。

### 二、前端缓存一致性的具体解决方案
前端缓存主要分两类：**HTTP缓存**（浏览器层面）、**前端本地缓存**（LocalStorage/SessionStorage/IndexedDB），针对不同缓存类型有不同解决方案：

#### 方案1：HTTP缓存（浏览器缓存）—— 最基础也最常用
HTTP缓存是浏览器自动处理的缓存（如静态资源：JS/CSS/图片），解决一致性的核心是「精准控制缓存过期策略」：

| 策略                | 实现方式                                                                 | 适用场景                     | 一致性保障                          |
|---------------------|--------------------------------------------------------------------------|------------------------------|-----------------------------------|
| 强缓存 + 缓存指纹   | 1. 给静态资源加哈希后缀（如 `app.abc123.js`）<br>2. 配置 `Cache-Control: max-age=31536000` | 不常变的静态资源（如框架库） | 资源更新时，哈希值变化 → 浏览器请求新资源 |
| 协商缓存（对比缓存） | 1. 配置 `Cache-Control: no-cache` + `ETag/Last-Modified`<br>2. 每次请求先问后端「资源是否更新」 | 频繁更新的静态资源/接口数据  | 后端判断资源是否变化：<br>- 未变 → 返回304，用缓存<br>- 已变 → 返回200+新资源 |

~~~nginx
server {
    # 配置ETag（自动生成文件哈希）
    etag on;
    # 配置Last-Modified
    if_modified_since on;

    location ~ \.(js|css)$ {
        # 开启协商缓存，不直接用本地缓存
        add_header Cache-Control "no-cache";
        # 静态资源目录
        root /usr/share/nginx/html;
    }
}
~~~

#### 方案2：前端本地缓存（LocalStorage等）—— 业务数据缓存
针对接口返回的业务数据（如用户信息、列表数据），手动管理缓存时的一致性方案：

##### （1）时间戳过期策略（简单易实现）
- 缓存时，同时存储「数据 + 缓存时间戳」；
- 读取时，判断当前时间是否超过「缓存有效期」：
  - 未过期 → 用缓存；
  - 已过期 → 请求新数据，更新缓存。

~~~javascript
// 缓存数据
function setCache(key, data, expire = 3600) { // expire：有效期（秒）
  const cacheObj = {
    data,
    timestamp: Date.now(),
    expire: expire * 1000 // 转毫秒
  };
  localStorage.setItem(key, JSON.stringify(cacheObj));
}

// 读取缓存
function getCache(key) {
  const cacheStr = localStorage.getItem(key);
  if (!cacheStr) return null;

  const cacheObj = JSON.parse(cacheStr);
  // 判断是否过期
  if (Date.now() - cacheObj.timestamp > cacheObj.expire) {
    localStorage.removeItem(key); // 清除过期缓存
    return null;
  }
  return cacheObj.data;
}

// 使用示例
async function getUserInfo() {
  // 先读缓存
  const cacheData = getCache('userInfo');
  if (cacheData) return cacheData;

  // 缓存过期/无缓存，请求新数据
  const res = await fetch('/api/user/info');
  const newData = await res.json();
  // 更新缓存（有效期1小时）
  setCache('userInfo', newData, 3600);
  return newData;
}
~~~

##### （2）版本号/哈希校验策略（精准更新）
- 后端返回「业务数据 + 数据版本号」（如 `version: "v1.2"`）；
- 前端缓存时存储「数据 + 版本号」；
- 每次请求前，先问后端「当前数据版本号」：
  - 版本号一致 → 用缓存；
  - 版本号不一致 → 请求新数据，更新缓存。

~~~javascript
async function getGoodsList() {
  // 1. 先读本地缓存的版本号和数据
  const cache = JSON.parse(localStorage.getItem('goodsList')) || { version: '', data: null };
  
  // 2. 请求后端版本号
  const versionRes = await fetch('/api/goods/version');
  const latestVersion = await versionRes.text();
  
  // 3. 版本一致 → 用缓存
  if (cache.version === latestVersion && cache.data) {
    return cache.data;
  }
  
  // 4. 版本不一致 → 请求新数据
  const dataRes = await fetch('/api/goods/list');
  const newData = await dataRes.json();
  
  // 5. 更新缓存
  localStorage.setItem('goodsList', JSON.stringify({
    version: latestVersion,
    data: newData
  }));
  return newData;
}
~~~

##### （3）主动更新策略（实时一致性）
适用于「数据更新后需要立即同步」的场景（如用户修改个人信息）：
- 执行「数据修改操作」后，**主动删除/更新对应缓存**；
- 下次读取时，自动请求新数据。

~~~javascript
// 修改用户信息后，主动清除缓存
async function updateUserInfo(newInfo) {
  await fetch('/api/user/update', {
    method: 'POST',
    body: JSON.stringify(newInfo)
  });
  // 清除缓存，确保下次读取最新数据
  localStorage.removeItem('userInfo');
}
~~~

#### 方案3：服务端推送（实时一致性）
适用于高实时性场景（如聊天、秒杀、实时榜单）：
- 后端数据更新时，主动推送给前端（WebSocket/SSE）；
- 前端收到推送后，立即更新缓存 + 页面渲染。

~~~javascript
// 建立WebSocket连接
const ws = new WebSocket('ws://your-server.com/ws');

ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  // 后端推送「数据更新」消息
  if (data.type === 'goods_update') {
    // 更新本地缓存
    localStorage.setItem('goodsList', JSON.stringify({
      version: data.version,
      data: data.list
    }));
    // 更新页面
    renderGoodsList(data.list);
  }
};
~~~

### 三、最佳实践总结
| 场景                          | 推荐方案                                  |
|-------------------------------|-------------------------------------------|
| 静态资源（JS/CSS/图片）       | 强缓存 + 哈希指纹 + 协商缓存              |
| 低频更新的业务数据（用户信息） | 时间戳过期策略 + 主动更新                 |
| 高频更新的业务数据（商品列表） | 版本号校验策略                            |
| 实时数据（聊天/秒杀）         | WebSocket推送 + 本地缓存实时更新          |

### 总结
1. **核心原则**：缓存一致性的关键是「精准控制缓存有效期」，而非禁用缓存；
2. **分层策略**：HTTP缓存（浏览器）用「哈希+协商缓存」，本地缓存（业务数据）用「时间戳/版本号+主动更新」；
3. **实时场景**：高实时性需求优先用WebSocket/SSE推送，保证数据即时一致。

简单来说，解决缓存一致性的核心思路是：「能复用的缓存尽量复用，该更新的缓存精准更新」，既兼顾性能，又保证数据最新。
```

# 多级缓存 
```ad-info
多级缓存的工作流程
以 Web 应用为例，用户请求的数据访问流程如下：

读取 L1 本地缓存：优先从本地内存（如 Caffeine）查询数据，命中则直接返回。
查询 L2 分布式缓存：若本地缓存未命中，则访问 Redis 等分布式缓存。
回填本地缓存：若 Redis 命中数据，将结果写入本地缓存（避免后续重复访问 Redis）。
访问数据库：若所有缓存均未命中，查询数据库并将结果回填到 Redis 和本地缓存。
示例场景：

用户 A 访问商品详情页，本地缓存未命中 → 查询 Redis → 未命中 → 查询数据库 → 返回结果并回填 Redis 和本地缓存。
用户 B 访问同一商品时，直接从本地缓存获取数据，无需访问 Redis 或数据库。
二、为什么需要多级缓存？
（以下内容为之前回答的简要总结，完整版可参考上文）

减少访问延迟：热点数据存储在更快的层级（如本地内存）。
降低后端压力：通过缓存拦截减少数据库访问。
提升扩展性：本地缓存解决单机热点，分布式缓存解决全局共享。
```


###  常见方案-方案的决策也是要看场景决定的 ！！

#### 方案一、Cache-Aside（缓存旁路模式）

以数据库为基准，缓存仅作为“旁路”辅助，不主动参与数据更新，适用于**读多写少**的场景。

- **读操作**
    1. 先查 L1 本地缓存，命中则返回
    2. 未命中则查询 L2 分布式缓存，命中则返回并同步到 L1
    3. 均未命中则查询数据库，返回数据并更新 L2 和 L1。
- **写操作**
    1. 先更新数据库
    2. 再删除 L2 和 L1 缓存（非更新）

**优势：** 避免缓存更新时的并发冲突问题（如两个写操作同时更新缓存导致数据不一致）

**注意事项：** 删除缓存可能失败，需配合”重试机制”（如消息队列异步重试）或“设置缓存过期时间“兜底。


#### 方案二、Write-Through（写透模式）

写操作先更新缓存，再由[缓存同步](https://so.csdn.net/so/search?q=%E7%BC%93%E5%AD%98%E5%90%8C%E6%AD%A5&spm=1001.2101.3001.7020)更新数据库，确保缓存与数据库同时修改。

- **流程：**
    1. 写操作直接更新 L1 缓存
    2. L1 缓存更新后，立即同步更新 L2 缓存
    3. L2 缓存更新后，再同步更新数据库
    4. 所有层级更新成功后，再返回更新成功。

**优势：** 缓存与数据库强一致性，适合对**一致性要求极高**的场景。

**缺点：** 写性能差（多了更新缓存的步骤），且任意一个环节失败会导致整个写操作失败。

#### 方案三、Write-Behind（写回模式）

写操作先更新缓存，数据库异步更新（如批量延迟写入），牺牲一部分一致性换取高性能。

- **流程**
    1. 写操作直接更新 L1 和 L2 缓存，标记为“脏数据”
    2. 缓存系统定期（或达到阈值）异步将“脏数据”批量写入数据库
    3. 数据库写入成功后，清除”脏数据“标记。

**优势：** 写性能极高（减少数据库 IO 次数），适合高并发写的场景（如日志收集）

**风险管理：** 若缓存宕机，会导致未及时同步到数据库的“脏数据”丢失，需要持久化缓存（如 Redis 的 RDB/AOF）降低风险。

#### 方案四、基于消息队列的异步一致性

通过消息队列解耦数据库与缓存的更新，确保更新操作不会丢失，适合分布式系统。

- **流程**
    - 写操作先更新数据库，成功后发送“数据更新消息”到消息队列
    - 消息队列消费者监听消息，依次删除/更新 L2 和 L1 缓存
    - 若缓存更新失败，消息队列重试（如 RabbitMQ 的死信队列机制）。

**优势：** 解决“[数据库更新](https://so.csdn.net/so/search?q=%E6%95%B0%E6%8D%AE%E5%BA%93%E6%9B%B4%E6%96%B0&spm=1001.2101.3001.7020)成功但缓存更新失败“的问题，保证最终一致性。

**注意：** 需处理“消息重复消费”（如给消息加唯一 ID 去重）。


### 4. 特殊场景补充策略

#### 4.1 缓存与数据库并发更新冲突

当多个线程同时更新同一数据时，可能出现“旧数据覆盖新数据”的问题。

- **解决方案**
    - 给缓存增加**版本号**：每次更新版本号 + 1，更新缓存时检查版本号是否大于当前缓存版本，避免旧数据覆盖
    - 分布式锁：更新缓存前获取锁（如 Redis 的 SETNX），确保同一时间只有一个线程更新。

#### 4.2 热点 Key 一致性

热点 Key（如秒杀商品库存）被高频访问，若缓存失效可能引发数据库雪崩。

**解决方案**

- **互斥锁：** 缓存失效时，只允许一个线程查询数据库并更新缓存，其他线程等待
- **热点 Key 永不过期：** 不设置过期时间，通过消息队列异步更新缓存（如库存变化时主动推送更新）
- **本地缓存兜底：** L1 本地缓存保留热点 Key 的“兜底值”，即使 L2 缓存失效，也能返回较新数据。

#### 4.3 跨层级缓存同步延迟

L1 本地缓存为进程内缓存，多节点部署时，某节点更新数据后，其他节点的 L1 缓存可能仍为旧值。

**解决方案**

- **缓存更新通知：** 使用 发布 - 订阅 模式（如 Redis Pub/Sub），某节点更新缓存后，向其他节点发送“缓存失效通知“，触发其他节点删除本地 L1 缓存
- **缩短 L1 缓存过期时间：** 让 L1 缓存的过期时间远短于 L2（如 L1 10 秒，L2 1 小时），通过快速过期减少不一致窗口。

### 5. 一致性与性能的平衡原则

- **不追求“强一致性”：** 多级缓存的本质是”性能优化工具“，强一致性会牺牲性能，多数场景下保证“最终一致性”即可（如允许毫秒级/秒级的不一致）
- **按业务场景选方案**
    - 金融交易：优先使用 Write - Through + 分布式锁，保证强一致性
    - 电商商品详情：用 Cache-Aside + 消息队列，允许短时间不一致；
    - 日志 / 统计数据：用 Write-Behind，优先保证写入性能。
- **兜底机制：** 所有方案都需要配合“缓存过期时间”，即使更新失败，过期后也能从数据库加载新数据，避免脏数据永久留存。
