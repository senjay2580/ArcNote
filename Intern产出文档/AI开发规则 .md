## 🎯 **AI开发规则 - 完整版**

### 📋 **核心原则**

#### **1. 理解优先原则**
- 开发代码前必须确保理解项目代码架构、技术栈和业务处理逻辑
- 分析整体业务逻辑，理解重要的业务类和业务方法
- 掌握全链路流程和数据流转

#### **2. 复用性原则**
- 实现功能前必须先检查现有代码是否可复用
- 如果复用会造成问题，提出选择方案供用户决定
- 优先扩展现有组件而非重新创建

#### **3. 最佳实践原则**
- 参照业界优秀项目的最佳实践
- 遵循设计模式和架构原则
- 代码质量和可维护性优先

### 🎨 **特征化分支规则**



#### 前端开发时：  

组件复用优先  
检查现有项目组件是否可复用或扩展，优先根据项目实际技术栈（如Vue 3/React/Angular/Svelte等）选型适配的主流组件库：  

- Vue 3生态：推荐UnoCSS tailwindcss或者行业中使用最流行最稳定最好用扩展性最强的（禁止使用Element UI）；  
- React生态：推荐Ant Design、Material UI、Chakra UI、ShadCN UI、UnoCSS 、tailwindcss；  
- 需选择社区活跃、维护迭代及时、扩展性强且支持按需引入的组件库，复用组件时评估适配性，若现有组件可通过props扩展、插槽/插槽属性定制满足需求则优先复用，若复用导致逻辑耦合或性能损耗需明确说明并提供独立实现方案  

遵循对应框架最佳实践  
- Vue 3：严格遵循Composition API规范，按业务逻辑拆分Composables（如`useUser`、`useRequest`），合理使用`ref`/`reactive`/`computed`，避免选项式API与组合式API混用；  
- React：遵循Hooks规则，抽离自定义Hook复用逻辑，使用函数组件+React.memo/useMemo/useCallback优化渲染；  

状态管理  
- Vue 3：使用Pinia，遵循单一数据源原则，按业务域划分store模块（如用户、订单、权限），避免跨模块状态冗余；  
- React：优先使用React Context+useReducer或Zustand/Jotai（轻量场景），复杂场景用Redux Toolkit，按切片（Slice）拆分状态；  

样式规范  
使用CSS Module（React/Vue）、Scoped CSS（Vue）或CSS-in-JS（如Styled Components/Emotion，React）隔离样式作用域；遵循BEM命名规范（或CSS Modules默认命名规则）提升样式可读性；**响应式设计优先**，结合组件库内置响应式能力或CSS Grid/Flex/媒体查询实现多端适配，优先使用CSS变量管理主题样式  

组件设计  
通用UI组件（按钮、输入框、表格等）优先复用组件库，业务组件（如订单卡片、用户列表）按“原子化”拆分（页面→模块→基础组件），通过props传递配置、插槽扩展内容，确保组件低耦合、高内聚

### 🔧 **代码质量标准**

#### **命名规范**
- 类名：大驼峰 (`UserService`)
- 方法名：小驼峰 (`getUserById`)
- 常量：全大写下划线 (`MAX_RETRY_COUNT`)
- 包名：全小写 (`com.senjay.archat`)

#### **注释规范**
```java
/**
 * 用户服务接口
 * 
 * @author senjay
 * @since 1.0.0
 */
public interface UserService {
    /**
     * 根据用户ID获取用户信息
     * 
     * @param userId 用户ID
     * @return 用户信息
     * @throws UserException 用户不存在时抛出
     */
    User getUserById(Long userId);
}
```

### ⚡ **快速决策模板**

当需要实现新功能时：

```
1. 现有代码是否可复用？
   ├── 是 → 扩展现有代码
   └── 否 → 继续评估

2. 是否符合现有架构？
   ├── 是 → 按架构规范实现
   └── 否 → 提出架构调整建议

3. 是否遵循最佳实践？
   ├── 是 → 开始实现
   └── 否 → 调整设计方案
```
