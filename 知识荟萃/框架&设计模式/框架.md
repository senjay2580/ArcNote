## 反射

<span style="color:#CC0000;">**作用：**</span>

反射（Reflection）是程序在**运行时**动态获取、操作类信息的能力，获取类后还可以动态创建这个类的实例和增删改这个实例的字段属性（实现灵活的逻辑控制）

1. **突破编译时限制**
   - 运行时动态加载类（如通过类名字符串），无需提前在代码中显式引入类。
2. **访问私有成员**
   - 通过反射可**绕过访问权限检查**（如调用`private`方法、修改`final`字段）。
3. **动态操作对象**
   - 运行时**创建对象、调用方法、修改属性，实现灵活的逻辑控制**。
4. **框架设计的基石**
   - Spring、MyBatis等框架依赖反射实现依赖注入、动态代理等核心功能。

---

**运行时动态获取类的信息”是指在程序运行过程中，通过某种机制(如Java反射)来获取类的结构和属性，而不是在程序编译时就已经确定这些信息。这种方式使得程序可以在<span style="color:#FF0000;">运行时根据需要加载类、创建对象、调用方法等，提供了极大的灵活性和动态性</span>**

<span style="color:#FF0000;">**使用：**</span>



**class对象：**

类对象(Class Object)是一个特殊的对象，它代表了**Java中的一个类或接口的运行时信息**。每个类或接口在Java虚拟机(JVM)中都有一个唯一的类对象，这个类对象是 java.lang.Class 类的实例。**类对象提供了关于类或接口的元数据信息，这些信息包括类的名称、字段、方法、构造函数、父类、实现的接口等。**

**获取class对象**

~~~java
// 方式1：通过类名（编译时已知）
Class<?> clazz = User.class;

// 方式2：通过对象实例
User user = new User();
Class<?> clazz = user.getClass();

// 方式3：通过全类名字符串（运行时动态加载）
Class<?> clazz = Class.forName("com.example.User");

// Class.forName 或
// ClassLoader.loadClass方法可以动态加载类，并获取其类对象
~~~

**创建对象实例**

~~~java
// 无参构造
Object obj = clazz.newInstance();

// 有参构造（需先获取Constructor）
Constructor<?> constructor = clazz.getConstructor(String.class, int.class);
Object obj = constructor.newInstance("Alice", 25);
~~~

~~~java
import java.lang.reflect.Field;
import java.lang.reflect.Method;

public class ReflectMultipleDemo {
    public static void main(String[] args) throws Exception {
        // 1. 获取类对象
        Class<?> clazz = Class.forName("Person");
        Object obj = clazz.getDeclaredConstructor().newInstance();

        // 2. 获取所有字段（包括私有字段）
        System.out.println("—— 所有字段 ——");
        Field[] fields = clazz.getDeclaredFields();
        for (Field field : fields) {
            field.setAccessible(true); // 访问私有字段
            System.out.println("字段名: " + field.getName() + "，类型: " + field.getType().getSimpleName());
            // 设置默认值
            if (field.getType() == String.class) {
                field.set(obj, "默认名");
            } else if (field.getType() == int.class) {
                field.set(obj, 18);
            }
            System.out.println("字段值: " + field.get(obj));
        }

        // 3. 获取所有方法（public，包括继承）
        System.out.println("\n—— 所有 public 方法（含继承）——");
        Method[] methods = clazz.getMethods();
        for (Method method : methods) {
            System.out.println("方法名: " + method.getName());
        }

        // 4. 获取所有声明的方法（包括 private）
        System.out.println("\n—— 所有声明的方法（包括 private）——");
        Method[] declaredMethods = clazz.getDeclaredMethods();
        for (Method method : declaredMethods) {
            System.out.println("方法名: " + method.getName());
            method.setAccessible(true);
            if (method.getParameterCount() == 0) {
                method.invoke(obj); // 只调用无参方法
            }
        }
    }
}

~~~

## 静态代理/动态代理

[联系类/方法增强](##Bean后置处理器（after）)

**主要是类型安全和性能开销上的问题**

---



**静态代理(Static Proxy)**
1.定义
静态代理是指**在编译时就已经确定了代理类和被代理类的关系**。代理类和被代理类在编译时就已经定义好，代理类实现了和被代理类相同的接口或继承了被代理类。
2.特点
**类型安全:由于代理类和被代理类在编译时就已经确定，编译器可以在编译阶段检查类型错误，因此类型安全。**
**性能优势:静态代理在运行时不需要进行额外的动态操作(不需要反射这个动态操作)，性能较高。**



！！！

**缺乏灵活性:代理类和被代理类的关系在编译时就已经确定，难以在运行时动态改变，**

---



**动态代理(Dynamic Proxy)**
1.定义
动态代理是指在**运行时动态生成代理类。代理类的生成和代理逻辑的实现都是在运行时完成的**，而不是在编译时。动态代理通常通过Java的 java.lang.reflect.Proxy 类和InvocationHandler 接来实现。

2. 特点
**灵活性高:**可以在运行时动态生成代理类，代理类和被代理类的关系可以在运行时改变。
**性能开销:**由于需要在运行时动态生成代理类和处理方法调用，动态代理会有一定的性能开销。
**类型不安全:动态代理操作在运行时进行，编译器无法检查类型错误，可能导致运行时错误。**

# Spring

**IoC（Inversion of Control:控制反转）** 是一种设计思想，而不是一个具体的技术实现。IoC 的思想就是将原本在程序中手动创建对象的控制权，交由 Spring 框架来管理。

- **控制**：指的是对象创建（实例化、管理）的权力
- **反转**：控制权交给外部环境（Spring 框架、IoC 容器）



Bean 代指的就是那些被 IoC 容器所管理的对象。 IoC 容器帮助我们**管理哪些对象，这个是通过配置元数据来定义的。配置元数据可以是 XML 文件、注解或者 Java 配置类。**

### [@Component 和 @Bean 的区别是什么？](#component-和-bean-的区别是什么)

- `@Component` 注解作用于类，而`@Bean`注解作用于方法。
- `@Component`通常是通过类路径扫描来自动侦测以及自动装配到 Spring 容器中（我们可以使用 `@ComponentScan` 注解定义要扫描的路径从中找出标识了需要装配的类自动装配到 Spring 的 bean 容器中）。`@Bean` 注解通常是我们在标有该注解的方法中定义产生这个 bean,`@Bean`告诉了 Spring 这是某个类的实例，当我需要用它的时候还给我。
- `@Bean` 注解比 `@Component` 注解的自定义性更强，而且很多地方我们只能通过 `@Bean` 注解来注册 bean。比如当我们引用第三方库中的类需要装配到 `Spring`容器时，则只能通过 `@Bean`来实现

**springAOP:  批量对Spring容器中bean的方法做增强，并且这种增强不会与原来方法中的代码耦合。**

-  Joinpoint（连接点）：所谓连接点是指那些可以被增强到的点。在spring中,**这些点指的是方法**，因为spring只支持方法类型的连接点

-  **Pointcut（切入点）：所谓切入点是指被增强的连接点（方法）**

-  **Advice（通知/ 增强）：所谓通知是指具体增强的代码**

-  Target（目标对象）：被增强的对象就是目标对象

-  **Aspect（切面）：是切入点和通知（引介）的结合**

-  Proxy （代理）：一个类被 AOP 增强后，就产生一个结果代理类（简单来说这个类被AOP增强后就不是原本的类了）





~~~java
@Target({ElementType.METHOD})//该注解可以加在方法上
@Retention(RetentionPolicy.RUNTIME)
public @interface InvokeLog {
}

// ---------------------------------------------
@Service
public class PhoneService {

    @InvokeLog  
    public void deleteAll(){
        System.out.println("PhoneService中deleteAll的核心代码");
    }
}



// ---------------------------------------------
@Component
@Aspect
public class MyAspect {

//    用Pointcut注解中的属性来指定对哪些方法进行增强
    @Pointcut("@annotation(com.sangeng.aspect.InvokeLog)")
    //  @Pointcut("execution(* com.sangeng.service..*.*(..))") 也可以定义切点表达式
    public void pt(){}

    /*
        用@Before注解来指定该方法中是增强的代码，并且是在被增强方法执行前执行的
        @Before的属性写上加了@Pointcut注解的方法: 方法名()
    */
    @Before("pt()")
    public void methodbefore(){
        System.out.println("方法被调用了");
    }
}

~~~

**环绕通知非常特殊，它可以对目标方法进行全方位的增强。**

我们可以在除了环绕通知外的所有通知方法中增加一个**JoinPoint类型**的参数。这个参数封装了被增强方法的相关信息。<span style="color:#FF0000;">**我们可以通过这个参数获取到除了异常对象和返回值之外的所有信息。**</span>

`ProceedingJoinPoint` 是 `JoinPoint` 的子接口，专门用于**环绕通知（Around Advice）**

<span style="color:#FF0000;">**`JoinPoint` 是整体上下文，`Signature` 是其中的“方法签名”细节**</span>



```java
@Before("pt()")
public void methodbefore(JoinPoint jp){
    Object[] args = jp.getArgs();//方法调用时传入的参数 
    Object target = jp.getTarget();//被代理对象
    MethodSignature signature = (MethodSignature) jp.getSignature();//获取被被增强方法签名封装的对象
    // 一般使用signature 的子接口MethodSignature 
    System.out.println("Before方法被调用了");
}

//----------------------------------------------------

  	@Around("pt()")
    public void around(ProceedingJoinPoint pjp){
        System.out.println("目标方法前"); // 增强语句
        try {
            pjp.proceed();//目标方法执行 被增强的方法
            System.out.println("目标方法后");
        } catch (Throwable throwable) {
            throwable.printStackTrace();
            System.out.println("目标方法出现异常");
        }finally {
            System.out.println("finally中进行增强");
        }
    }


//----------------------------------------------------


```

#### 🌟**环绕通知 vs 其他通知的区别：**

| 特性                   | 环绕通知（@Around）                    | 其他通知（@Before、@After等）               |
| ---------------------- | -------------------------------------- | ------------------------------------------- |
| 是否能控制方法执行时机 | ✅ 可以手动控制是否、何时执行目标方法   | ❌ 不行，方法执行由框架控制                  |
| 是否能获取返回值       | ✅ 可以获取并修改返回值                 | ❌ 除非是 After Returning，但也只能读不可改  |
| 是否能捕获异常         | ✅ 可以 try-catch 捕获、处理或抛出异常  | ❌ After Throwing 虽能感知异常，但不能阻止它 |
| 功能强度               | 最强                                   | 辅助型                                      |
| 是否能修改参数、结果等 | ✅ 可修改参数（通过反射）、可修改返回值 | ❌ 通常只是读，不建议改参数或结果            |

**总结**：如果需要**完全控制方法调用的过程（甚至不执行它）**，用 `@Around`；如果只是插入日志、权限检查等逻辑，其他通知可能更**轻量。**

**2.挂起外界事务**

如果当前线程已经存在一个事务（外层事务），它会 **挂起当前事务**；

然后开启一个**新的事务**；

新事务**独立提交或回滚**，不会影响被挂起的事务；

新事务结束后，恢复外层事务。



---



**3.嵌套事务**

如果当前有事务，则在 **当前事务内部开启一个嵌套事务（Savepoint）**；

嵌套事务失败时，可以 **单独回滚到 savepoint**，不会导致整个事务回滚；

最终是否提交，还依赖**外部事务是否提交**；

---



一定要考虑事务传播行为的情况 1️⃣ **服务 A 调用服务 B，B 也有事务要求**

 **Spring 提供的 7 种事务传播行为**

**是** ***被调用的方法**（callee）* **对于** ***调用它的方法**（caller）* **来说的。**

**也就是内部对于外务而言！**

| Propagation 常量               | 含义                                                         |
| ------------------------------ | ------------------------------------------------------------ |
| **REQUIRED** （默认）          | 如果当前有事务，就融入；如果没有，就新建一个事务。           |
| **REQUIRES_NEW**               | 挂起当前事务，重新开启一个新事务。原事务等着它完成后再恢复。 |
| **NESTED** 嵌套                | 如果当前有事务，就在这个事务中嵌套一个子事务（依赖保存点）；没有事务时，等同于 REQUIRED。 |
| **SUPPORTS**                   | 有事务就融入；没有事务也可以运行（非事务方式运行）。         |
| **NOT_SUPPORTED**              | **不支持事务**， 当前有事务，就挂起它。自己非事务方式运行。  |
| **NEVER**                      | 不允许事务，如果当前存在事务就抛出异常。                     |
| **MANDATORY** 强制性的；义务的 | 必须在事务中运行，如果没有事务就抛异常。                     |

~~~JAVA
@Component  // 标注为 Spring 组件，自动被扫描注册为 BeanPostProcessor
public class JdkProxyBeanPostProcessor implements BeanPostProcessor {

    /**
     * 该方法在 Spring 容器初始化完成后，对每个 Bean 进行后置处理。
     * 如果符合条件，会返回代理对象替换原始 Bean，从而实现增强。
     *
     * @param bean     当前被初始化的 Bean 实例
     * @param beanName Bean 的名称
     * @return 可能是原始 Bean，也可能是代理 Bean
     * @throws BeansException
     */
    @Override
    public Object postProcessAfterInitialization(final Object bean, String beanName) throws BeansException {
        // 判断当前 Bean 是否是 UserService 类型（或其子类），只对 UserService 做代理增强
        if (bean instanceof UserService) {
            // 使用 JDK 动态代理创建代理对象，参数说明如下：
            return Proxy.newProxyInstance(
                // 1. 类加载器，使用当前 Bean 的类加载器
                bean.getClass().getClassLoader(),
                // 2. 代理需要实现的接口列表，JDK 动态代理只能代理接口，这里用 Bean 实现的接口
                bean.getClass().getInterfaces(),
                // 3. 调用处理器，定义代理方法被调用时的行为
                new InvocationHandler() {
                    /**
                     * 代理对象调用方法时会执行该方法
                     * @param proxy  代理对象
                     * @param method 被调用的方法对象
                     * @param args   方法参数
                     * @return       方法返回值
                     * @throws Throwable
                     */
                    @Override
                    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
                        // 方法执行前增强逻辑：打印日志
                        System.out.println("👉 JDK代理：方法开始：" + method.getName());

                        // 执行目标对象（即原始 Bean）的方法，并传入参数
                        Object result = method.invoke(bean, args);

                        // 方法执行后增强逻辑：打印日志
                        System.out.println("👈 JDK代理：方法结束：" + method.getName());

                        // 返回目标方法执行结果
                        return result;
                    }
                }
            );
        }
        // 如果不是 UserService 类型，则不做处理，直接返回原始 Bean
        return bean;
    }
}
~~~

~~~java
@Component  // 标注为 Spring 组件，自动注册为 BeanPostProcessor
public class CglibProxyBeanPostProcessor implements BeanPostProcessor {

    /**
     * Spring 容器初始化每个 Bean 后都会调用此方法。
     * 如果 Bean 类型符合条件，则返回代理对象实现增强；否则返回原 Bean。
     *
     * @param bean     当前被初始化的 Bean 实例
     * @param beanName Bean 名称
     * @return 代理对象或原始 Bean
     * @throws BeansException
     */
    @Override
    public Object postProcessAfterInitialization(final Object bean, String beanName) throws BeansException {
        // 判断 Bean 是否是 OrderService 类型，只有这个类才增强
        if (bean instanceof OrderService) {
            // 创建 CGLIB 的 Enhancer 对象，用于生成代理类
            Enhancer enhancer = new Enhancer();
            // 设置代理的父类为目标 Bean 的类（CGLIB 是继承方式实现代理）
            enhancer.setSuperclass(bean.getClass());

            // 设置方法拦截器（MethodInterceptor），拦截目标方法执行
            enhancer.setCallback(new MethodInterceptor() {
                /**
                 * 代理对象调用方法时，都会进入该方法
                 * 
                 * @param obj    代理对象本身
                 * @param method 被调用的方法对象
                 * @param args   方法参数
                 * @param proxy  方法代理，用于调用父类（目标类）方法
                 * @return       方法返回值
                 * @throws Throwable
                 */
                @Override
                public Object intercept(Object obj, Method method, Object[] args, MethodProxy proxy) throws Throwable {
                    // 方法执行前增强逻辑，比如打印日志
                    System.out.println("👉 CGLIB代理：方法开始：" + method.getName());

                    // 调用目标 Bean 的方法（用 proxy.invoke 调用，传入目标 bean）
                    // 注意：这里调用了 proxy.invoke(bean, args)，是调用目标对象方法
                    Object result = proxy.invoke(bean, args);

                    // 方法执行后增强逻辑，比如打印日志
                    System.out.println("👈 CGLIB代理：方法结束：" + method.getName());

                    return result;
                }
            });

            // 生成代理对象，替代原 Bean 返回
            return enhancer.create();
        }
        // 如果不是目标类型，直接返回原始 Bean
        return bean;
    }
}
~~~

### **处理器适配器的作用**

无论处理器是哪种类型（如基于注解的 `@Controller`、传统的 `Controller` 接口实现类、`HttpRequestHandler` 等），`HandlerAdapter` 都能让 `DispatcherServlet` 以相同的方式调用它们。

#### **参数解析过程**

1. `DispatcherServlet` 调用 `RequestMappingHandlerAdapter`。
2. `RequestMappingHandlerAdapter` 遍历 `HandlerMethodArgumentResolver` 列表，找到支持当前参数类型的解析器。
3. 解析器从 `HttpServletRequest` 中**提取数据并绑定到方法参数。**

#### **常见参数解析器**

| 参数类型             | 解析器                                 | 说明                          |
| :------------------- | :------------------------------------- | :---------------------------- |
| `@RequestParam`      | `RequestParamMethodArgumentResolver`   | 处理 `?name=value` 查询参数   |
| `@PathVariable`      | `PathVariableMethodArgumentResolver`   | 处理 `/user/{id}` 路径变量    |
| `@RequestBody`       | `RequestResponseBodyMethodProcessor`   | 解析 JSON/XML 请求体          |
| `HttpServletRequest` | `ServletRequestMethodArgumentResolver` | 直接注入 `HttpServletRequest` |
| `@ModelAttribute`    | `ModelAttributeMethodProcessor`        | 绑定表单数据或 POJO           |

---



`HandlerAdapter` 使用 **`HandlerMethodReturnValueHandler`（返回值处理器）** 来处理方法的返回值，例如：

- 返回 `String`（视图名称）、`ModelAndView`、`@ResponseBody`、`ResponseEntity` 等。

#### **常见返回值处理器**

| 返回值类型         | 处理器                                 | 说明                                          |
| :----------------- | :------------------------------------- | :-------------------------------------------- |
| `String`（视图名） | `ViewNameMethodReturnValueHandler`     | 返回逻辑视图名，由视图解析器渲染              |
| `@ResponseBody`    | `RequestResponseBodyMethodProcessor`   | 使用 `HttpMessageConverter` 序列化为 JSON/XML |
| `ModelAndView`     | `ModelAndViewMethodReturnValueHandler` | 直接返回模型和视图                            |
| `ResponseEntity`   | `HttpEntityMethodProcessor`            | 完全控制 HTTP 响应（状态码、头、体）          |
| `void`             | `ModelMethodProcessor`                 | 通常用于 REST API，不返回数据                 |

## 自动配置“配置”的是啥？

### 1. **组件 Bean 的创建和初始化**

比如：

- 数据库连接池（`DataSource`）
- Redis 操作模板（`RedisTemplate`）
- 消息队列客户端
- Web MVC 配置（拦截器、视图解析器）
- 安全模块的配置（`SecurityFilterChain`）

------

### 2. **依赖资源的连接和属性设置**

比如：

- 读取 `application.properties` 或 `application.yml` 里的配置，设置连接地址、端口、账号密码
- 根据配置文件里的信息，创建并初始化连接工厂（比如 RedisConnectionFactory、DataSource）

------

### 3. **根据环境和条件灵活启用配置**

通过注解条件判断：

- **类是否存在**：`@ConditionalOnClass`（只有项目依赖了某个库，才配置相关 Bean）
- **Bean 是否已存在**：`@ConditionalOnMissingBean`（如果你没写，框架帮你写）
- **配置属性是否开启**：`@ConditionalOnProperty`（配置里开启了才生效）

------

**注册**是指**将自动配置类告诉 Spring Boot 框架的过程**，让框架知道“有这么些自动配置类需要在启动时加载”。

注册的是：**你自己写的自动配置类的全限定类名（`com.xxx.YourAutoConfiguration`）**

```
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.example.mylogger.MyLoggerAutoConfiguration
```



Spring Boot 在项目启动时会**扫描你项目里所有的 jar 包中的资源文件**，并从里面读取这些配置类，**自动实例化 Bean**。

但 Spring Boot 本身**不会主动去找你的配置类**，所以：

 你必须 **告诉 Spring Boot：自动配置类在哪！**

## Spring 框架（Spring、Springboot、Springmvc）常见的注解

![image-20250515210447097](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250515210447097.png)

一、`@Autowired` 和 `@Resource`

### 1. `@Autowired`（Spring 提供的依赖注入注解）

- 属于 Spring 框架（`org.springframework.beans.factory.annotation.Autowired`）
- 按**类型（byType）**自动注入
- 可搭配 `@Qualifier` 实现按名称注入
- 支持构造器、字段、Setter 注入



> 若有多个同类型 Bean，必须用 `@Qualifier` 指定名称，否则报错。

2. `@Resource`（Java 提供的依赖注入注解，属于 JSR250）

- 属于 JDK 标准（`javax.annotation.Resource`）
- 按**名称（byName）**优先注入，如果找不到再按类型
- 不支持 `@Qualifier`（用属性名作为默认注入名）

---



`@Import`

- 用于将其他配置类或 Bean 注册到 Spring 容器
- 常用于模块拆分、AOP、MyBatis 配置等

```java
@Configuration
@Import(MyConfig.class)
public class AppConfig {}
```

> 也可导入 `ImportSelector` 或 `ImportBeanDefinitionRegistrar` 实现类，用于动态注册 Bean。

`@Repository`

- 用于标注持久层组件（DAO），是 `@Component` 的一种特化
- 表示这是一个操作数据库的类，Spring 会自动识别并注册 **Spring 容器管理**
- 与 `@Component` 的区别在于语义清晰

```java
@Repository
public class UserDao {}
```



**在 Spring 框架中，只有被 Spring 容器管理的类，才能享受到依赖注入（DI）、Spring事务控制、AOP 增强等功能。**

**MappedStatement ：定义了标签中的一些信息**

executor执行器 去读取 MappedStatement 

然后根据 输入的数据 去操作数据库 返回输出信息



**输入**-->将java对象类型转化为sql可操作类型-->**输出**-->将sql可操作类型转化为java对象类型



![image-20250515211345963](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250515211345963.png)



`Executor` 是 MyBatis 的内部接口，定义了数据库操作的**实际执行行为**。主要作用是：

- **<span style="color:#FF0000;">执行具体 SQL 命令</span>**（MappedStatement）
- **处理缓存**（一级缓存、二级缓存）
- **维护 Statement 生命周期**

`SqlSession` 是 MyBatis 的门面类（Facade），用于应用程序和 MyBatis 之间的交互，提供了：

- <span style="color:#FF0000;">执行 SQL（`selectOne`、`selectList`、`insert`、`update`、`delete`）</span>
- <span style="color:#FF0000;">获取 Mapper 接口的代理对</span>象
- **管理事务（提交、回滚）**
- 关闭连接等

<span style="color:#FF0000;">**开发者通常只直接使用 SqlSession。**</span>

```java
SqlSession session = sqlSessionFactory.openSession();
User user = session.selectOne("com.example.mapper.UserMapper.selectUser", 1);
session.close();
```
