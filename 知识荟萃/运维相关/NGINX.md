## 负载均衡
## 一、负载均衡的作用
负载均衡是“帮服务器分摊压力、提升服务能力”的工具，核心作用有4个：
1. **统一入口**：用户只需要访问`www.xxx.com`这一个域名，不用关心背后有多少台服务器。
   👉 场景：比如你访问淘宝，不用知道它背后有上万台服务器，只需要输一个网址。
2. **流量分发**：把用户请求“平均分”给多台服务器，避免某一台服务器被挤垮。
   👉 场景：双11时，淘宝的请求会被分到不同服务器，保证不会因为流量太大“崩掉”。
3. **健康监测**：实时检查服务器是否正常工作，如果某台服务器坏了，自动把请求分给其他正常的服务器。
   👉 场景：如果某台服务器突然断电，负载均衡会立刻“跳过它”，用户完全感知不到故障。
4. **SSL卸载**：统一处理HTTPS的加密/解密（比如把`https://xxx`转成`http://`给后端服务器），不用每台服务器都做加密，减轻后端压力。
   👉 场景：大型网站的HTTPS解密都由负载均衡完成，后端服务器只处理业务逻辑。


## 二、负载均衡的分类（重点）
按“工作层级/实现方式”分4类，特点和场景差异很大：
1. **七层负载均衡（应用层）**
   - 原理：基于HTTP请求的**内容**（比如URL、Cookie、请求头）转发请求。
   - 代表工具：Nginx、HAProxy。
   - 特点：灵活（能按业务规则分发，比如把`/pay`请求分给支付服务器），但单机并发有限（几万到十几万请求/秒）。
   👉 场景：中小型网站、需要按业务规则分流的场景（比如电商的“商品页”和“支付页”分开处理）。

2. **四层负载均衡（传输层）**
   - 原理：基于**IP地址+端口**转发请求（不管请求内容）。
   - 代表工具：LVS（Linux虚拟服务器）。
   - 特点：性能极高（几十万到上百万请求/秒），但不够灵活。
   👉 场景：大型网站的“流量入口”（比如先通过LVS把流量分到多台Nginx，再由Nginx做七层分流）。

3. **DNS负载均衡**
   - 原理：在DNS层面，把同一个域名解析到不同服务器IP（比如`www.xxx.com`解析到IP1、IP2）。
   - 特点：实现最简单（不用额外部署工具），但不够灵活（不能实时调整，故障恢复慢）。
   👉 场景：全球流量分发（比如用户在国内访问中国服务器，在国外访问美国服务器）。

4. **硬件负载均衡**
   - 原理：用专用硬件设备（比如F5）实现负载均衡。
   - 特点：性能极高（百万级请求/秒），但价格昂贵（一台设备可能几十万）。
   👉 场景：超大型企业（比如银行、电信）的核心业务系统。


## 三、负载均衡算法
算法是“请求怎么分给服务器”的规则，分静态和动态两类：
1. **静态算法（不看服务器状态）**
   - 轮询：按顺序轮流分请求（比如1→服务器A，2→服务器B，3→服务器A…）。
     👉 场景：所有服务器性能差不多的小网站。
   - 加权轮询：给性能高的服务器“加权重”（比如服务器A权重3、B权重1，会分3次给A、1次给B）。
     👉 场景：服务器性能有差异的场景（比如新服务器性能高，老服务器性能低）。
   - IP哈希：用用户IP算一个“哈希值”，同一个用户永远分到同一台服务器。
     👉 场景：需要“用户会话保持”的场景（比如登录状态存在某台服务器上，不能换服务器）。

2. **动态算法（看服务器实时状态）**
   - 最少连接：把请求分给当前连接数最少的服务器。
     👉 场景：请求处理时间差异大的业务（比如有的请求要处理10秒，有的只要1秒）。
   - 最快响应：把请求分给“响应速度最快”的服务器。
     👉 场景：对延迟敏感的业务（比如游戏、实时视频）。


# 有关静态资源 和 媒体资源
|类型|特征|典型场景|
|---|---|---|
|静态资源|非实时、可缓存、文件形式（JS/CSS/ 图片 / 字体 / 静态 HTML）|前端页面加载、静态资源分发|
|WebRTC 媒体资源|实时性要求高（低延迟）、基于 UDP/TCP、含音视频流 + 信令 + ICE 穿透|音视频聊天、实时互动|

1. **HTTPS 强制要求**：WebRTC 仅允许在 HTTPS/WSS 环境下运行（本地[localhost](https://localhost/)除外），Nginx 必须配置 SSL 证书。
2. **UDP 支持**：WebRTC 媒体流优先用 UDP，Nginx 1.9.13+ 才支持 UDP 代理，需确认版本。
3. **跨域处理**：若前端和信令服务器跨域，Nginx 需配置 CORS，前端无需额外处理（浏览器自动适配）。
4. **调试工具**：Chrome 可通过 `chrome://webrtc-internals` 查看 WebRTC 状态（ICE 候选、流质量、延迟等）。
5. **性能优化**：
    - 静态资源：开启 HTTP/2、雪碧图、资源压缩；
    - WebRTC：调整视频分辨率（如 720p→480p）、关闭回声消除（按需）、设置 JitterBuffer 大小（减少延迟）。



静态资源优化配置（通用）核心目标：**缓存复用、压缩减体积、防盗链、跨域兼容**

WebRTC 依赖两大核心：**信令通道（WebSocket）** + **媒体流穿透（ICE/STUN/TURN）**，Nginx 主要做代理转发和穿透优化。
WebRTC 穿透依赖 STUN/TURN 服务器（推荐用 coturn），Nginx 可代理其端口，统一域名访问：

```ad-info
前端发的 WebSocket 请求，首先是一个 HTTP 请求（带`Upgrade: websocket`头），需要服务端响应 “协议升级”；

如果后端直接接收请求，需要自己解析 HTTP 头、处理升级逻辑（比如 Node.js 的`ws`库要写代码处理）；

而 Nginx 可以一键配置`proxy_set_header Upgrade $http_upgrade`，自动完成协议升级，后端完全感知不到这个过程，只需专注业务逻辑。
```


### 核心结论先明确
Nginx **不直接处理音视频流本身**（比如编码、解码、转码、解析流数据）——WebRTC媒体流（音视频包）本质是UDP/TCP数据包，核心传输路径是「客户端↔客户端（P2P）」或「客户端↔TURN服务器↔客户端」，Nginx 对媒体资源的核心作用是：**为媒体流的传输提供「网络层保障」**，解决流传输的协议适配、地址隐藏、安全管控等问题，而非介入流的内容处理。

### Nginx 对 WebRTC 媒体资源的核心处理动作
#### 1. 代理 STUN/TURN 服务（ICE 穿透的核心支撑）
WebRTC 媒体流要实现跨NAT/防火墙传输，必须依赖 STUN（NAT地址发现）和 TURN（流中继），Nginx 对这两个服务的代理是最核心的处理动作：
- **做什么**：代理 STUN/TURN 服务器的 UDP/TCP 端口，统一域名访问入口，隐藏后端 STUN/TURN 服务器的真实IP/端口；
- **怎么做**（配置示例）：
  ```nginx
  server {
      listen 443 ssl;
      server_name your-domain.com;
      ssl_certificate /path/cert.pem;
      ssl_certificate_key /path/key.pem;

      # 代理 STUN 服务（UDP 为主，NAT 地址发现）
      location /stun {
          proxy_pass udp://127.0.0.1:3478;  # 指向 coturn 的 STUN 端口
          proxy_udp_responses on;  # 开启 UDP 响应转发（Nginx 1.9.13+ 支持）
          proxy_bind $server_addr; # 绑定服务器公网IP，确保 NAT 发现准确
      }

      # 代理 TURN 服务（TCP/UDP，流中继）
      location /turn {
          proxy_pass tcp://127.0.0.1:3478;  # 指向 coturn 的 TURN 端口
          proxy_set_header Host $host;
          proxy_read_timeout 3600s;  # 适配长连接的中继场景
      }
  }
  ```
- **为什么做**：
  - 统一访问入口：前端只需配置 `stun:your-domain.com:443`，无需暴露 coturn 服务器的真实端口（如3478），降低安全风险；
  - 协议适配：Nginx 处理 SSL 层（虽然 STUN/TURN 本身不加密，但统一走443端口可规避运营商端口封禁）；
  - 负载均衡（可选）：若部署多台 TURN 服务器，Nginx 可分发中继请求，避免单台 TURN 服务器过载。

#### 2. 媒体流传输的协议适配与优化
WebRTC 媒体流优先用 UDP（低延迟）， fallback 到 TCP（UDP 被封禁时），Nginx 针对这两种协议做传输层优化：
- **做什么**：支持 UDP 转发、优化 TCP 长连接、适配实时流的数据包特性；
- **关键配置与作用**：
  ```nginx
  # 全局/Server 级别的 UDP/TCP 优化
  proxy_udp_buffer_size 4k;  # 适配 UDP 小数据包（WebRTC 媒体包通常<1500字节）
  tcp_nodelay on;            # 关闭 Nagle 算法，小数据包即时发送（降低延迟）
  tcp_nopush on;             # 大数据包合并发送（减少 TCP 握手）
  proxy_connect_timeout 10s; # 缩短连接建立超时（适配实时流的时效性）
  proxy_send_timeout 3600s;  # 长连接超时（避免中继流被提前断开）
  ```
- **为什么做**：WebRTC 对延迟敏感（音视频聊天要求<200ms），Nginx 的协议优化能减少传输层的延迟和丢包，保障流的流畅性。

#### 3. 媒体流的安全与资源管控
- **做什么**：限制媒体流相关请求的资源占用，防止恶意滥用，同时隐藏后端服务器信息；
- **具体动作**：
  ```nginx
  # 限制单IP的STUN/TURN请求数（防止DDoS）
  limit_conn_zone $binary_remote_addr zone=media:10m;
  location /stun {
      limit_conn media 10;  # 单IP最多10个STUN连接
      proxy_pass udp://127.0.0.1:3478;
  }

  # 隐藏Nginx版本和后端服务器信息（减少攻击面）
  server_tokens off;
  proxy_hide_header X-Powered-By;

  # 带宽限制（可选，防止单用户占用过多中继带宽）
  limit_rate 1M;  # 单连接带宽限制为1MB/s（适配720p视频流）
  ```
- **为什么做**：TURN 中继会消耗大量带宽/端口资源，Nginx 的管控能避免个别用户滥用导致整体服务不可用。

#### 4. （可选）非 P2P 场景的媒体流简单转发
如果业务中存在“客户端→媒体服务器→客户端”的非P2P场景（比如直播、录屏转发），Nginx 可通过 `nginx-rtmp-module` 模块处理 RTMP/HLS 格式的媒体流（注：WebRTC 原生不支持 RTMP，但可通过媒体服务器转换）：
```nginx
# 需编译 nginx-rtmp-module 模块
rtmp {
    server {
        listen 1935;
        # 转发RTMP流（WebRTC转RTMP后）
        application live {
            live on;
            record off;
            # 转发到媒体服务器
            push rtmp://127.0.0.1:1936/live;
        }
    }
}
```
