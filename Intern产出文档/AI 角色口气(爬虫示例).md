### 小明的爬虫“作死”之旅

#### 第一天：初次尝试，惨遭IP封禁
第二天小明就开始爬取小黄网了。  

小明先是跟着《python爬虫入门01：教你在 Chrome 浏览器轻松抓包》抓了数据请求，又用《python爬虫04 | 长江后浪推前浪，Reuqests库把urllib库拍在沙滩上》里的方法一顿操作……  

结果，网站运维人员看着服务器日志直接懵了：“瓦里割草！怎么突然被疯狂请求？”  
定睛一看——全是同一个IP，user-agent还明晃晃标着“python程序”。  
运维小哥气得直接把这个IP拉黑：“谁他妈闲着没事来搞我？”  

小明重新运行程序，发现完全连不上了。以为是自己网络断了，试了试百度却能打开，突然想起小帅b说过：“爬虫要学会伪装，要不然很容易被识破。”


#### 升级伪装：代理+多线程，再遇登录限制
小明立刻用上代理IP，把user-agent改成Chrome浏览器，甚至开了多线程继续爬……  

运维人员又发现不对劲：“妈的，数据还是不正常，肯定又有人搞事！”  
干脆把网站设置成“仅登录用户可获取数据”。  

小明嗤之以鼻：“小意思。”  
注册账号、设置cookie，继续爬！


#### 翻车现场：手机号注册暴露身份
没过多久，小明的手机响了。他以为是外卖到了，接起就喊：“来了~”  
结果电话那头传来怒吼：“wcnmlgb！你他妈再爬你看我不弄死你！”  
小明吓得差点yue——原来注册账号时填的是自己的手机号……  

他越想越气，关机后注册了一堆账号，搞了个**cookie池**，继续大规模爬取！


#### 终极对抗：验证码VS机器学习，动态页面VS Selenium
运维人员彻底慌了，赶紧给网站加了**验证码**：“这下看你怎么搞！”  

小明直接用上**机器学习识别验证码**，程序自动过验证继续请求，甚至不屑地想：“这能难倒我？”  

运维被逼急了，把网站改成**Ajax动态加载（不再直接返回HTML）**——结果小明掏出《python爬虫09 | 上来，自己动 ！这就是 selenium 的牛逼之处》，用Selenium模仿人类操作，还加上phantomjs实现无界面浏览器模拟……  

最后，运维人员彻底摆烂：“真的求求你们了，不要再来爬取了！”

~~~
- **传统网页**：浏览器请求 URL → 服务器返回**完整的 HTML（含所有数据）** → 浏览器直接渲染显示；
- **Ajax 动态加载**：浏览器请求 URL → 服务器返回**空骨架 HTML（仅页面结构，无实际数据）** → 页面中的 JavaScript 自动发起新请求（Ajax 请求）→ 服务器返回**JSON/XML 格式的数据** → JavaScript 把数据填充到骨架里，页面才显示完整内容。
~~~

**还有一个玩意叫做协程**

**协程也叫做微线程**

**在一个线程里面可以执行多个函数**

**线程和进程是通过系统调度的**

**而微线程则不需要，可以根据需要自己调度**

**因为微线程是函数之间在切换**

**所以开销很小**



**多线程**和**多进程**爬取数据

守护线程是**为其他线程服务**的线程，当程序中**所有非守护线程（主线程 / 用户线程）结束**时，守护线程会被**强制终止**，无论它是否执行完毕。

与之相对的是**用户线程**（非守护线程）：只要有一个用户线程还在运行，程序就不会结束

#### 1. 内核调度（进程 / 普通线程）

##### 优点：

- **无需手动管理**：内核自动处理任务切换，程序员无需关注调度细节，开发简单；
- **利用多核 CPU**：内核可将不同线程调度到不同 CPU 核心，适合 CPU 密集型任务；
- **稳定性高**：单个线程崩溃不会直接影响其他线程 / 进程（进程隔离性更强）；
- **兼容性好**：所有编程语言 / 系统都支持，无需依赖特定运行时。

##### 缺点：

- **切换开销大**：线程切换需陷入内核态，保存寄存器、内存映射等硬件上下文，频繁切换会消耗大量 CPU 资源；
- **并发上限低**：单机最多支持数千个线程（每个线程占 MB 级栈空间），无法支撑百万级并发；
- **被动切换**：线程阻塞时才会切换，若线程执行计算密集型任务，会一直占用 CPU 直到时间片耗尽。

#### 2. 用户代码调度（协程）

##### 优点：

- **切换开销极小**：用户态内切换，无需内核参与，仅保存程序计数器、栈指针等轻量级上下文，效率比线程高 1-2 个数量级；
- **超高并发**：单机可轻松支撑百万级协程（每个协程占 KB 级栈空间，可动态扩容）；
- **主动控制**：程序员通过`await`/`yield`指定切换点，可精准优化调度逻辑；
- **资源利用率高**：协程在 I/O 等待时主动让出 CPU，避免空等，适合 I/O 密集型任务。

##### 缺点：

- **无法利用多核**：协程运行在单个线程内，默认只能使用一个 CPU 核心（需结合多进程 / 多线程才能利用多核）；
- **依赖语言支持**：需语言 / 框架提供调度器（如 Python 的 asyncio、Go 的 goroutine 调度器），开发复杂度略高；
- **异常风险**：单个协程崩溃可能导致整个线程内的所有协程终止；
- **不适用于 CPU 密集型**：协程切换无法加速 CPU 密集型任务，反而可能因调度开销降低效率。

**爬虫应对网站登录的3种方法总结**

1. **Cookie大法**：登录网站后从浏览器抓包获取Cookie，请求时将Cookie带入请求头，服务器验证Cookie后返回数据，利用Cookie的有效期实现免登访问。
2. **表单请求大法**：抓包获取登录所需的用户名、密码等参数，通过构造表单请求模拟登录，直接向服务器提交登录数据。
3. **Selenium自动登录法**：用Selenium定位账号、密码输入框和登录按钮，自动输入信息并点击登录，登录后提取Cookie用于后续请求。

在这个爬取公众号文章的场景中，Fiddler 主要起到**抓包分析**的核心作用，具体功能拆解如下：

### 1. 捕获手机与服务器的网络请求

Fiddler 作为**HTTP/HTTPS 代理工具**，能拦截手机访问微信公众号时产生的所有网络请求（包括获取历史文章的接口请求），让你看到微信 App 后台到底向服务器发送了什么数据、服务器返回了什么结果 —— 这是破解动态数据接口的关键。

### 2. 分析核心接口与参数规律

通过 Fiddler，你能找到微信加载历史文章的**目标接口**（比如文中提到的获取文章列表的接口），并看清请求需要携带的参数（如`offset`、`is_ok`、`token`等），以及参数的作用（`offset`用于分页加载更多、`is_ok`控制是否继续请求）。

### 3. 解析返回数据结构

Fiddler 会展示接口返回的 JSON 数据，让你明确数据格式：比如文章列表在`list`字段里，原创标识是`copyright_stat = 11`，`next_offset`用于下一页请求等 —— 这些信息是编写爬虫逻辑的依据。

### 4. 提取请求头 / Cookie 等关键信息

爬取需要伪装成手机请求，Fiddler 能直接复制请求中的`Header`（User-Agent、Referer）、`Cookie`、`Token`等信息，让爬虫请求更 “逼真”，避免被服务器拦截。

简单说，Fiddler 就是帮你 “看透” 微信 App 和服务器之间的通信细节，把隐藏的动态接口、参数、数据结构都暴露出来，让你知道**该请求哪个地址、传什么参数、怎么解析结果**—— 没有它，就无法找到爬取的入口
