序号 / 标识（Sequence Number / Identifier）
干什么用的
给每个数据单元打上"编号"，用于**排序、去重、确认、重组、唯一标识。**



差错控制 = **差错检测 + 差错纠正 / 重传**



**中间节点**（如路由器或交换机）收到报文后，会根据报文首部（比如 IP 头部的目的地址）查询本节点的**转发表**。

转发表记录了“目的地址”到“下一跳节点”的映射关系。

根据转发表，中间节点把报文转发给下一个合适的节点，直到报文到达最终目的地。



- **传统最短路径算法（如 Dijkstra）确实需要全网拓扑信息，才能计算出最短路径。**
- 这意味着如果每个节点都自己单独计算，理论上需要每个节点都掌握整个网络拓扑（所有节点和链路状态）。

<span style="color:#FF0000; font-size:1.4em;">**各种路由协议的不同设计理念**</span>

<span style="color:#FF0000; font-size:1.4em;">链路状态路由协议（Link State Protocol） —— 如 OSPF、IS-IS</span>

- 每个节点通过交换“链路状态广播包”（LSA），把自己知道的邻居信息和链路状态告诉所有节点。
- 因此，**每个节点都拥有整个网络的拓扑图**。
- 每个节点利用 Dijkstra 算法基于这个拓扑图计算自己到各个目的地的最短路径。
- 优点：路径最优，收敛快。
- 缺点：每个节点存储拓扑信息，计算复杂度较高。

> **所以在链路状态协议中，确实存在每个节点存储相同拓扑信息的情况。**

---



**<span style="color:#FF0000; font-size:1.4em;">距离向量路由协议（Distance Vector Protocol） —— 如 RIP</span>**

- 节点只知道自己邻居的距离信息。
- 路由器周期性地向邻居发送自己到各个目的地的距离向量。
- 节点根据邻居传来的距离和自身到邻居的代价计算自己到目的地的距离，逐步更新自己的路由表。
- 这种方式不需要全网拓扑信息，只需要邻居的距离信息。
- 优点：实现简单，开销低。
- 缺点：收敛慢，有环路问题。

> **距离向量协议下，节点不用存整个网络拓扑，只维护距离表，避免了拓扑信息冗余。**

**<span style="color:#FF0000; font-size:1.4em;">混合协议</span>**

- 现代网络中常用的协议（如 EIGRP）结合两者优点，平衡存储和收敛速度。

<span style="color:#FF0000; font-size:1.4em;"> **冗余与现实权衡**</span>

- **链路状态协议确实在每个节点存储了同样的网络拓扑，存在一定冗余。**
   但这样做可以保证每个节点的路径计算一致，避免路由环路，收敛速度快，网络状态透明。
- **距离向量协议减少了冗余，但路由环路和收敛问题较多。**
- **所以网络设计会根据需求选择合适的协议和算法。**

**虚电路交换**将实现可靠性的策略交给了 核心部分这样反而会导致核心部分处理的效率问题 

而交给边缘部分 也就是 客户端 利用客户端的高性能进行处理才比较好

---



**虚电路的问题**

- 虚电路要求**所有分组按序走同一条路径**，**核心交换机要维护虚电路表、处理顺序、重传等可靠性逻辑。**
- 核心交换机本来就很忙，还要处理这么多复杂的控制逻辑，效率会下降，容易成为瓶颈。

 **互联网（数据报方式）的思路**

- 把可靠性交给**边缘主机（客户端）**，比如 TCP 协议，由**发送方和接收方自己处理序号、重传、差错控制。**
- 核心路由器只负责**简单的转发**，不用管顺序、不用管重传，效率极高，这就是互联网能支撑海量流量的原因。

**核心原理：木桶效应**

主机 A 给主机 B 发送数据的最高理论速率，由**整条链路中最慢的环节决定**，也就是取三者的最小值：

- 主机 A 的网卡速率：1Gbps（发送方上限）
- 链路带宽：100Mbps（传输通道上限）
- 主机 B 的网卡速率：10Mbps（接收方上限）

三者中，最慢的是**主机 B 的 10Mbps 网卡**，所以 A 发给 B 的最高理论速率只能是 10Mbps。

![image-20250810110610053](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250810110610053.png)

还需要注意 路由器的WAN口最高速率

购买路由器时，一定要注意查看 WAN 口的速率。**WAN 口的速率至少要与你家的宽带速率相等或更高**，否则你将无法享受到宽带服务商提供的全部速度。如果你家办理了 2000Mbps 的宽带，那么一个带有 2.5G WAN 口的 Wi-Fi 6 路由器就是更好的选择。

**波长**是波在**空间维度**上，两个相邻的、相同相位点之间的距离。

在一个波的完整周期中，任何一个特定的点都可以用相位来描述。通常，我们用一个角度（比如从 0° 到 360°）来表示相位，或者用弧度（从 0 到 2π）来表示。

每种网络技术（如以太网、Wi-Fi）都是一个**完整的协议族**(区分协议族和协议)，它通常包含了对物理层（如何传输比特流）和数据链路层（如何封装成帧、进行寻址和控制）功能的具体实现。这些功能最终由网络设备（如网卡）来完成。

**有线技术**：以太网（不同速率如 10Mbps、100Mbps、1Gbps、10Gbps）、光纤（不同标准如 FDDI、SONET）等。

**无线技术**：Wi-Fi（不同协议如 802.11a/b/g/n/ac/ax）、蓝牙、蜂窝网络（如 4G/5G）等。

**物理介质**：双绞线、同轴电缆、光缆、无线电波等。

在OSI模型中，每一层都有**非常严格**的定义和边界。

- **物理层**：只处理比特流。
- **数据链路层**：只处理帧，以及MAC地址、差错控制等。
- **网络层**：只处理分组，以及IP地址、路由等。

在OSI的严格分层体系下，数据链路层**没有权利和义务**去向上层（网络层）提供一个“通用”的接口。它的职责是**专一**的，它只知道如何处理帧，并与它的上层（网络层）和下层（物理层）进行“点对点”的通信。

数据链路层向**上**提供**直接的服务接口**给网络层（在实际实现中，这个接口往往是固定的“帧格式 + 传输机制”）。



~~~
在 OSI 的严格分层体系中，数据链路层会向上层（网络层）提供服务，但这种服务是固定且单一职责的层间服务，而不是面向各种上层协议的“通用业务接口”。

数据链路层的职责仅限于：

成帧
MAC 寻址
差错检测
在同一链路上传输数据帧

因此，它只需要向网络层提供：

“把数据封装成帧并发送”

这样的基础传输能力。

它并不会关心：

数据是不是 IP
里面是不是 TCP
是否属于 HTTP、视频流或其他应用协议

也就是说：

数据链路层只负责“如何传帧”，不负责“理解上层业务”。

所以这里所谓：

“不提供通用接口”

真正的意思是：

不提供能够适配和理解各种上层协议逻辑的万能接口。
~~~



---

**TCP/IP**

**对网络层而言**，它只需要知道把数据包交给网络接口层，而不用关心底层是光纤、双绞线、还是无线电波。网络层只与这个**抽象的接口**打交道。

**对网络接口层而言**，它需要**适配**不同的底层技术。它会根据实际连接的物理介质，自动选择对应的帧封装格式（比如以太网帧、Wi-Fi帧）和物理传输方式（比如电信号、光信号、无线电波）。



一台设备（无论是路由器还是普通电脑）通常有多个网络接口：

- **有线网卡**：`eth0`、`eth1` 等。
- **无线网卡**：`wlan0`。
- **回环接口**：`lo`（用于本机通信）。
- **虚拟接口**：`tun0` 等。



**回环接口是什么？**

- **虚拟接口**：它没有实际的电缆连接，也没有物理的MAC地址。
- **特殊IP地址**：回环接口通常被分配一个特殊的IP地址，最常用的是**127.0.0.1**，这个地址被称为**回环地址（Loopback Address）**。在IPv6中，对应的地址是 `::1`。
- **数据流向**：发送到 `127.0.0.1` 的任何数据包都不会离开本机。数据包会通过网络协议栈的完整处理流程，但当它到达网络层后，会被立即**重定向回传输层**，而不是发送到物理网络上。

两个不同的应用程序，一个作为客户端，一个作为服务器，可以通过 `127.0.0.1` 地址进行数据交换。

这是一种非常便捷的进程间通信方式，因为它可以利用标准化的网络套接字（Socket）接口，而无需编写专门的IPC代码

**<span style="color:#FF0000; font-size:1.4em;">“数据报”是一个通用的术语，根据网络分层的上下文，可能指代不同层的协议数据单元（PDU）</span>**

**应用层报文 → 传输层报文段/数据报**：

> 一个应用层报文（Message） → 在传输层可能被拆分成多个报文段（Segment，例如TCP协议根据MSS分割），或作为一个数据报（Datagram，例如UDP协议）处理。TCP通过MSS（最大报文段长度）控制数据量，避免报文段过大；UDP通常不分割，但若报文过大，可能在网络层分片。

**报文段/数据报 → 网络层分组**：

> 一个传输层报文段（Segment，例如TCP）或数据报（Datagram，例如UDP） → 在网络层封装成一个IP分组（Packet/IP数据报）。通常为1对1关系，但若报文段大小超过MTU（最大传输单元）减去IP头部长度，IP层可能将其分片为多个IP分组（仅在IPv4中常见，IPv6通常要求端到端处理分片）。

**分组 → 数据链路层帧**：

> 一个网络层分组（Packet/IP数据报） → 在数据链路层封装成一个帧（Frame）。通常为1对1关系。如果分组大小超过数据链路层的MTU，网络层（IP层）会先将分组分片为多个较小的IP分片，每个分片封装为一个独立的帧。

**码元（Symbol）**

- 在**数字通信**里，码元是**在传输介质上一次性发出的最小信号单元**。
- 每个码元可以表示 1 位、2 位、3 位……甚至更多比特，具体取决于调制方式。
- **例子**：
  - **二进制调制**（如 NRZ）：1 个码元 = 1 位（0 或 1）。
  - **4 电平调制**（PAM-4）：1 个码元 = 2 位（00、01、10、11）。
  - **8 相位调制**（8-PSK）：1 个码元 = 3 位（000~111）。



**码元宽度（Symbol Duration / Symbol Interval）**

码元宽度指**每个码元占用的时间长度**（秒/码元），是时间上的度量。

## 信道的极限容量

![](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250811002036518.png)

**信道的极限容量**：信道在给定条件下，**无差错传输数据的最高速率上限**。

它由两个核心定理决定：

- 无噪声场景：奈奎斯特定理（极限波特率）
- 有噪声场景：香农定理（极限比特率）

因为如果波特率超过 2W，相邻码元的信号波形会在信道中发生**拖尾、重叠**，导致接收方无法区分前后两个码元，也就是图里说的**码间串扰**。

![image-20250811003945406](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250811003945406.png)

真实信道的极限比特率，必须同时满足两个条件：

1. 不能超过奈奎斯特的物理上限：2W×log2K
2. 不能超过香农的信息上限：Wlog2(1+S/N)

所以**实际能达到的最大比特率，是两者中的较小值**。

**编码的自同步能力**

自同步能力指编码信号能够在接收端无需单独时钟信号的情况下，通过信号本身提取时序信息，正确解码数据。通常通过信号中的频繁电平转换（transitions）实现。

~~~
你发数据，接收方得知道 “什么时候一个码元开始，什么时候结束”，不然就会把数据读错。
不归零编码（NRZ）：如果连续发一串1，电平一直是高的，中间没有任何变化。接收方根本分不清这是 1 个码元还是 100 个码元，时间久了就会 “不同步”。
归零编码（RZ）：每个码元中间都会归零，产生跳变，帮助接收方 “校准时间”。
曼彻斯特 / 差分曼彻斯特编码：每个码元中间都强制跳变，同步能力最强，所以早期以太网用它。
结论：需要同步能力，就必须在信号里 “打标记”（跳变），这就是曼彻斯特这类编码的设计初衷。
~~~



**是否浪费带宽**

带宽浪费指编码方式因引入冗余信息或高频转换，导致传输相同数据需要更多带宽

~~~
不同编码方式，传输同样的数据，需要的带宽不一样。
NRZ：效率最高，1 个码元就是 1 次电平变化，带宽利用率 100%。
曼彻斯特编码：每个码元要跳变 2 次，相当于用了 2 倍的带宽来传同样的数据，效率只有 50%。
结论：同步和抗干扰能力强的编码，往往要牺牲带宽效率。设计时要在 “效率” 和 “可靠性” 之间做权衡。
~~~



**抗干扰能力**

抗干扰能力指编码信号在噪声（如加性噪声）环境下仍能被正确解码的能力。

曼彻斯特编码因频繁转换，接收端可通过多次采样确认比特，抗噪声能力较强。

NRZI在长序列0或1时，信号持续不变，易受噪声干扰，抗干扰能力弱

~~~
噪声会干扰信号，让接收方分不清 0 和 1。不同编码方式的抗干扰能力天差地别。
NRZ/NRZI：只靠电平高低或跳变来区分，很容易被噪声 “盖过去”，误判率高。
曼彻斯特编码：靠跳变方向来区分，即使噪声把电平拉低了，只要跳变的方向没变，就能正确解码，抗干扰能力强得多。
结论：抗干扰能力越强，编码方式就越复杂，信号里的跳变也越多。
~~~

**绕射(衍射)与指向性**



**无线信号一定占用一整段连续频率（频带 / 带宽），不是单个频点**，你理解的 “只占一个频率” 是误区。

1. **中心频率**

   信道的**标称编号 / 代表频率**，只是一个参考点，不是信号只在这一点传输。

   例：2.4G 信道 1 中心频点 = **2.412 GHz**

2. **信道带宽（20MHz / 40MHz）**

   信号实际**占据的频率宽度**，以中心频点为中点，向左右扩展一段范围。

   - 20MHz 带宽：以中心频点左右各延伸 10MHz
   - 40MHz 带宽：以中心频点左右各延伸 20MHz

WiFi 信号不是只在一个 “点频率” 上传输，而是会占用一小段频率范围，这就是我们说的 “带宽”，比如：

- 2.4GHz WiFi 通常用 **20MHz 或 40MHz 带宽**
- 比如一个 20MHz 带宽的信道，就是从 2.412GHz 到 2.432GHz 这样一段连续的频率。

> CRC 的基本思想是：发送方和接收方事先约定一个生成多项式 `G(x)`，它对应一个二进制除数。设生成多项式的最高次数为 `R`，则 CRC 校验位长度为 `R` 位。发送方先在原始信息位后补 `R` 个 （左移），然后用生成多项式对应的二进制数进行模 2 除法，得到的 `R` 位余数就是**校验位**。最后发送的 CRC 码等于“原始信息位 + 校验位”。接收方收到 CRC 码后，用同一个生成多项式再次进行模 2 除法：若余数为 0，表示没有检测到错误；若余数不为 0，表示检测到错误。模 2 加法和模 2 减法本质上都是异或运算，即相同为 0、不同为 1。CRC 主要用于差错检测，实际应用中通常检测到错误后丢弃该帧或请求重传，而不是直接纠错。

海明校验码（Hamming Code）有一个明确的能力限制：

- 它 **只能纠正 1 位错误**（Single Error Correction, SEC）
- 并且 **只能检测 2 位错误**（Single Error Correction, Double Error Detection, SEC-DED）
- **三位错误** 对于标准的海明码来说是“超纲”的，既不能纠正，也不一定能检测出来，甚至可能被误判成 1 位错误并错误纠正（导致“**错误传播**”）。

### GBN、SR 协议的信道利用率

**RTT = 数据从发送方传播到接收方的时间   + ACK 从接收方传播回发送方的时间**

信道真正用来发送有效数据的时间/一个完整发送周期的总时间

~~~
因为发送方从开始发数据，到可以放心发下一个数据帧，要经历：

发数据帧
数据传播到接收方
接收方发 ACK
ACK 传播回发送方
所以总时间就是：

T_D + RTT + T_A
~~~

| 名称                | 它是什么                         | 单位         | 影响什么                 |
| ------------------- | -------------------------------- | ------------ | ------------------------ |
| 滑动窗口长度 N / WT | 一次最多允许连续发送多少个数据帧 | 个/帧        | 影响能连续发几个帧       |
| 数据帧长度          | 一个数据帧有多少比特/字节        | bit / B / kb | 影响发送一个数据帧要多久 |
| 确认帧长度          | 一个 ACK 确认帧有多少比特/字节   | bit / B / kb | 影响发送一个 ACK 要多久  |

U = 实际平均数据传输速率 / 信道最大速率   (一段时间内，真正成功传过去的有效数据量 / 这段时间总耗时)

U = 实际吞吐率 / 带宽

所以本质就是 **信道的发送能力有多少比例被用来传有效数据**

## 信道划分 介质访问控制

**1. 信道划分：提前分好**

核心思想：

```
先把信道切成几份，每个用户用自己那一份
```

比如一条路分成多条车道，每辆车走自己的车道，不抢。

常见方式：

| 方式         | 怎么分       |
| ------------ | ------------ |
| 频分复用 FDM | 按频率分     |
| 时分复用 TDM | 按时间分     |
| 波分复用 WDM | 按光波波长分 |
| 码分复用 CDM | 按编码分     |

特点：

```
优点：不会冲突 缺点：有人不用时，资源可能浪费
```

------

**2. 随机访问：大家抢**

核心思想：

```
谁想发，谁就尝试发
```

如果两个人同时发，就会冲突，再想办法处理。

比如大家共用一个麦克风，谁想说话就开口。
如果两个人同时说，就听不清，需要重来。

常见协议：

| 协议    | 思想                 |
| ------- | -------------------- |
| ALOHA   | 想发就发，冲突了重传 |
| CSMA    | 先听信道，空闲再发   |
| CSMA/CD | 边发边检测冲突       |
| CSMA/CA | 尽量避免冲突         |

特点：

```
优点：灵活，信道空闲时谁都能用 缺点：可能冲突
```

------

**3. 轮询访问：排队轮流用**

核心思想：

```
按顺序轮流获得发送权
```

比如老师点名发言，点到谁谁说。

典型协议：

```
令牌传递协议
```

谁拿到令牌，谁才能发送数据。
发完把令牌传给下一个。

特点：

```
优点：不会冲突，比较公平 缺点：需要等待轮到自己，令牌管理有开销
```

<span style="color:#FF0000;"> **不同频率的电台 = 不同的中心频率**</span>

- 收音机上的 **87.5 MHz、88.0 MHz、101.3 MHz** 等，就是不同电台的**载波中心频率**。
- 这是你调台时看到的数字，决定了你听到的是哪一家电台。
- 比如：
  - 电台A：中心频率 90.1 MHz
  - 电台B：中心频率 90.3 MHz
     （这两个电台在频谱上间隔 200 kHz）

**<span style="color:#FF0000;">200 kHz 带宽 = 每个电台占据的“频率范围”</span>**

- 虽然电台的中心频率是一个点，但它的信号不是只占那一个点的频率，而是左右各延伸一部分。
- FM 调频广播为了传输立体声、辅助数据等，需要**大约 200 kHz 的频谱宽度**。
- 例如：
  - 电台 90.1 MHz：信号大致占 90.0 MHz ~ 90.2 MHz
  - 电台 90.3 MHz：信号大致占 90.2 MHz ~ 90.4 MHz
     它们刚好紧密排列，中间会留一点保护带，避免互相干扰

## 随机访问 介质访问控制

**随机访问 MAC 全景图**

随机访问的核心是：

**大家共用一条信道，谁有数据谁就尝试发送；如果发生冲突，再按规则处理。**

它和“信道划分”不同：
信道划分是提前分资源；随机访问是不提前固定分配，靠竞争使用信道。

![image-20250812172939496](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250812172939496.png)



![image-20250812173343968](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250812173343968.png)



![image-20250812173517582](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250812173517582.png)



![image-20250812173531573](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250812173531573.png)

**纯 ALOHA** 的规则最简单：节点一旦有数据帧准备好，就立刻发送到共享信道上，不监听信道，也不等待特定时间。如果发送后能收到 ACK，说明发送成功，继续等待下一个数据帧；如果 ACK 超时，说明可能发生冲突或丢失，就随机等待一段时间后重传。纯 ALOHA 的核心是“想发就发”，实现简单，但只要两个帧在时间上有任何重叠，就会冲突，所以冲突概率高，信道利用率低。

**时隙 ALOHA** 是对纯 ALOHA 的改进。它把时间划分成一个个等长时隙，且每个时隙长度等于发送一个最长数据帧所需的时间。节点即使有数据准备好，也不能立刻随便发，而是要等到最近的下一个时隙开始时再发送。如果收到 ACK，则成功；如果 ACK 超时，则随机等待若干时隙后重传。它的核心是“只能在时隙开始发送”，这样冲突只会发生在多个节点选择同一个时隙发送时，不会出现帧只重叠一部分的情况，因此比纯 ALOHA 冲突少、效率更高。

两者共同点是：都属于随机访问协议，都不提前分配信道，都靠 ACK 判断是否成功，ACK 超时后随机等待再重传。

| 为什么叫 1-坚持？ | 因为一旦监听到信道空闲，它以概率 1 立即发送。                |
| 核心意思          | 只要空闲，我一定发。                                         |
| 流程              | 监听信道。 如果空闲，立即发送。 如果忙，就一直监听。 一旦变空闲，马上发送。 |
| 优点              | 信道空闲时能马上利用，延迟小。                               |
| 缺点              | 如果多个节点都在等信道空闲，一旦空闲，它们会一起发，容易冲突。 |
| 记忆              | 1-坚持 = 空闲必发。                                          |

| 模块             | 内容                                                         |
| ---------------- | ------------------------------------------------------------ |
| 非坚持 CSMA      | 英文：Non-persistent CSMA                                    |
| 为什么叫非坚持？ | 因为它不一直死盯着信道，也不在空闲后立刻抢。                 |
| 流程             | 监听信道。 如果空闲，发送。 如果忙，不持续监听，而是随机等待一段时间。 等完后再重新监听。 |
| 优点             | 多个节点不会一直卡在信道口，一空闲就同时冲上去，所以冲突概率较低。 |
| 缺点             | 即使信道后来已经空闲，节点也可能还在等待，所以信道可能空闲一段时间，利用率下降，延迟变大。 |
| 记忆             | 非坚持 = 忙就退开，随机等再来。                              |

| 模块              | 内容                                                         |
| ----------------- | ------------------------------------------------------------ |
| p-坚持 CSMA       | 英文：p-persistent CSMA                                      |
| 为什么叫 p-坚持？ | 因为信道空闲时，不是一定发送，而是以概率 p 发送。            |
| 适用              | 通常用于时隙信道。                                           |
| 流程              | 监听信道。 如果信道忙，继续监听。 如果信道空闲，在一个时隙开始时： 以概率 p 发送； 以概率 1-p 推迟到下一个时隙。 如果下一个时隙仍空闲，再重复这个概率选择。 |
| 优点              | 在 1-坚持和非坚持之间折中。 通过概率降低多个节点同时发送的可能。 |
| 缺点              | 需要时隙同步，参数 p 也要设置合适。                          |
| 记忆              | p-坚持 = 空闲时按概率发。                                    |

| 3. CSMA/CD 是什么            | CSMA/CD 全称：Carrier Sense Multiple Access with Collision Detection |
| 拆开                         | Carrier Sense：发送前监听载波。 Multiple Access：多节点共享信道。 Collision Detection：碰撞检测。 |
| 中文名                       | 载波监听多路访问/碰撞检测。                                  |
| 它是在 CSMA 基础上进一步改进 | CSMA 只是：发前先听。                                        |
| CSMA/CD 是                   | 发前先听，发送时也继续听；如果检测到冲突，马上停止。         |
| 流程                         | 先监听信道。 信道空闲就发送。 发送过程中继续检测是否发生碰撞。 一旦检测到碰撞，立即停止发送。 发送一个干扰信号，让其他节点都知道冲突了。 随机退避后重传。 |
| 常用于                       | 早期共享总线式有线以太网。                                   |
| 为什么主要用于有线？         | 因为有线网络中，发送设备可以检测线路上的电压/信号变化，从而判断有没有碰撞。 |
| 记忆                         | CSMA/CD = 先听再发，边发边听，撞了就停。                     |

CSMA/CD 的设计原则是：

**必须保证发送方在发送完最短帧之前，一定能检测到最远端的冲突。**

这就是为什么以太网有 **最小帧长 64 字节**。

原因是：

- 最坏情况：A 和 D 在最远两端。
- A 刚开始发，信号还没传到 D。
- D 也开始发，冲突产生。
- 冲突信号再传回 A，需要一个往返传播时间。

所以 A 必须持续发送足够久，至少覆盖这个往返传播时间。

如果帧太短，A 可能已经发完并认为成功了，但冲突还没传回来。
这就无法检测冲突。

因此以太网规定：

```
帧不能太短 发送时间必须 ≥ 最坏情况下的往返传播时间
```

这就是最小帧长的本质。

假设 <strong style="color:#c0392b;">A 节点</strong>刚开始监听到这个信道空闲，  
于是它会把自己的信号（或者说自己的数据）<strong>发送到这个信道上</strong>。  
我们知道，这些信号的传播是有速度的，对吧？  

比如在某一个时刻，<strong style="color:#2980b9;">A 节点发出的信号</strong>已经传播到某个位置。  

在这种情况下，此时 <strong style="color:#c0392b;">B 节点</strong>尝试监听信道，  

它会发现<strong style="background-color:#f9e79f;">信道是空闲的</strong>，  

因为来自 A 节点的信号还没有到达 B 节点的位置。  

于是 <strong style="color:#c0392b;">B 节点</strong>会误以为信道空闲，  

并且<strong>开始发送自己的数据</strong>。  

随着时间推移，<strong style="color:#2980b9;">A 节点的信号</strong>最终到达 B 节点，  

这时 <strong>B 节点</strong>在发出一小段数据后，  

会<strong style="background-color:#f5b7b1;">检测到冲突并立即停止发送</strong>。  

那么，<strong>B 节点</strong>发出的这小段数据会怎样呢？  

这些信号并不会停在原地，  

它们会<strong>沿着信道继续向前传播</strong>。  

当这部分信号传播到 <strong style="color:#c0392b;">A 节点</strong>位置时，  

A 节点也能<strong style="background-color:#f9e79f;">检测到冲突</strong>，  

并且<strong style="color:#e74c3c;">立即停止发送，随机等待一段时间后重发</strong>。

AP 是一种**无线网络设备**，用来让无线终端（比如手机、笔记本）通过 Wi-Fi 接入有线网络。
 它相当于无线客户端和有线局域网之间的“桥梁”。

**工作方式**

- **下行**：AP 把有线网络的信号转换成无线信号，让无线设备接入
- **上行**：AP 把无线设备发送的数据转回有线网络
- 使用 IEEE 802.11 标准（Wi-Fi）进行无线通信



AP 就类似家用无线路由器的天线

无线网卡收发是通过同一个射频链路完成的

**边发边听**几乎不可能，因为**自己发出的信号功率远大于接收到的信号**，会把接收机“淹没”

就像你在用大喇叭喊话，根本听不到别人说什么





在无线网络中，两个终端可能都能与 AP 通信，但它们之间信号互相听不到

这种情况下，即使两台设备同时发送数据，自己都没听到别人说话，也会产生碰撞

但碰撞发生在 AP 那里，发送方**没法直接检测到**

###  有线以太网（CSMA/CD 适用）

- 网线里是 **双绞线对**，收发是分开的：
  - 一对线专门发送（TX）
  - 一对线专门接收（RX）
- 这样可以**边发边听**，检测碰撞（CD）



### 无线网卡（WLAN）

- **发射机（Transmitter）** 和 **接收机（Receiver）** 通过**同一个射频链路（RF Chain）**连接到天线
- 天线和射频电路是共用的，要么处于发送模式，要么处于接收模式**（半双工）**
- 当你发送信号时，自己的发射功率比接收到的信号**强百万倍以上**，接收机根本没法同时“听”外面的信号（会被自己发射的信号“淹没”）

**当电脑网卡接收到帧后的 数据流通和处理过程？？？**

**操作系统网络栈**，也叫 **TCP/IP 栈** 或 **协议栈**，是操作系统内核中负责处理网络通信的 **一套协议实现和数据处理机制**。

它的核心作用是：

1. **封装与解封装数据**

   - 把应用程序的数据封装成协议格式发送
   - 把收到的网络数据解封装并交给应用程序

2. **协议处理**

   - 负责 IP、TCP/UDP、ARP、ICMP 等协议的解析、校验、路由和流量控制

3. **接口提供**

   - 提供 Socket、Netlink 等接口，让应用程序能够读写网络数据而不用关心物理细节

   

**串行传输（Serial）**：比特一个接一个发送，例如网线上的电信号就是串行比特流。

**并行处理（Parallel）**：计算机内部处理数据是按字节/字（8、16、32、64 位）一次处理多比特



**帧缓冲** = 网卡内存或 DMA 区域，用来临时存储接收到或待发送的数据帧



## 以太网和 IEEE 802.3



**"以太网帧 = MAC 帧"是总称,它有两种格式:Ethernet V2 和 IEEE 802.3。两者帧头帧尾一模一样,唯一本质区别是源地址后那 2 字节——V2 当"类型"用、直接标上层协议;802.3 当"长度"用、得借 LLC/SNAP 标上层。接收方用 ≥1536 还是 ≤1500 来区分二者。实际网络里通用的是 Ethernet V2。**

LLC层负责告诉数据链路层“上面来的数据属于哪个网络层协议”。

~~~
IP数据报
↓
ARP报文
↓
IPX报文

到了数据链路层以后，LLC负责区分：

这是IP的
这是ARP的
这是别的协议的

然后交给正确的上层协议处理。
~~~

## 以太网交换机

交换机本质就是多端口的网桥

![image-20250813220915992](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250813220915992.png)

结点之间彼此是知道对方的 MAC 地址的

 

  ![image-20250813222632597](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250813222632597.png)



![image-20250813223053122](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250813223053122.png)



- 以太网是**无连接广播介质**，刚接入时，交换机并不知道某个 MAC 地址对应哪个端口。
- 如果不知道，就只能**泛洪（Flooding）**：把数据帧复制到除入口端口外的所有端口发出去，这样效率低，还可能造成环路广播风暴。
- 有了自学习，交换机能像“快递分拣员”一样，记住每个设备在哪个口，精准转发。

<span style="font-size:1.4em; color:#FF0000;">**核心机制：边转发边学习**</span>

以太网帧结构中，前面有**源 MAC 地址**和**目的 MAC 地址**字段。交换机利用这两个字段做两件事：

**<span style="font-size:1.4em; color:#FF0000;">步骤 1：学习源 MAC 地址</span>**

1. 当交换机**收到一帧数据**时，它先读取该帧的**源 MAC 地址**（Source MAC）和接收到的**物理端口号**。
2. 在 MAC 表中查找：
   - **如果表中没有这个源 MAC**：在 MAC 表中增加一条记录：
      `源 MAC 地址 -> 入口端口`
   - **如果表中已有这个 MAC，但端口不同**：说明设备换了口或拓扑变化，更新对应的端口。
3. 这一步叫**MAC 地址学习**（Learning）。

<span style="font-size:1.4em; color:#FF0000;">**步骤 2：查表决定转发方式**</span>

然后交换机根据**目的 MAC 地址**（Destination MAC）来决定如何处理：

- **如果 MAC 表中有记录**：直接把帧转发到对应端口（单播，Unicast）。
- **如果没有记录**：说明没学过这个 MAC，交换机会对**除了入口口外的所有端口广播**（Flooding）。

当 DF = 1 时，如果路由器遇到 MTU 太小的数据包（也就是当一个 IP 数据报的 **长度超过经过的链路 MTU** ），会丢弃并发送 ICMP “需要分片但 DF=1” 消息



![image-20250814010511560](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250814010511560.png)

**标识（Identification）**

- **位置**：IPv4 头部 16 位字段（除了标志和片偏移前的字段）
- **作用**：给每个 IP 数据报分配一个 **唯一标识符**。
- **用途**：当一个大数据报被分片后，接收端可以根据 **标识字段**将属于同一个原始数据报的分片重新组合。
- **特点**：每发送一个新的 IP 数据报，标识号通常递增

**标志（Flags）**

- **位置**：3 位字段
- **作用**：控制 IP 分片行为
- **各位含义**：

| 位          | 名称                 | 含义                               |
| ----------- | -------------------- | ---------------------------------- |
| 0（最高位） | 保留位               | 必须为 0                           |
| 1（次高位） | DF（Don't Fragment） | 1 → 不允许分片，0 → 可分片         |
| 2（最低位） | MF（More Fragments） | 1 → 后面还有分片，0 → 最后一个分片 |

**片偏移（Fragment Offset）**

- **位置**：13 位字段
- **单位**：**8 字节**（每个单位 = 8 B）
- **作用**：表示 **该分片在原始数据报中的起始位置**
- **举例**：

假设原始数据报长度 4000 B，经过 MTU 1500 B 的链路分片：

1. 第一个分片：长度 1500 B → 片偏移 = 0
2. 第二个分片：长度 1500 B → 片偏移 = 1500 / 8 = 187
3. 第三个分片：长度 1000 B → 片偏移 = (1500 + 1500) / 8 = 375

- **接收端**根据 **标识 + 片偏移 + MF 位** 来正确重组原始数据报。

## ==IP地址== 重要！！！

技术发展：**IPv4 子网划分 CIDR NAT**

~~~
IPv4 地址总数太少，早期分类 ABC 类地址浪费极其严重

 子网划分 出现原因 & 解决什么
起因：ABC 类地址块太大，一个网段闲置大量 IP，地址严重浪费，且无法按部门 / 端口做网络隔离。
解决：把一个大网络拆成多个小局域网，节省 IP、能划分不同部门网段、方便管理隔离。

CIDR 无类别域间路由 出现原因 & 解决什么
起因：传统 ABC 类死板固定掩码，不够灵活；互联网路由表爆炸、路由条目太多路由器扛不住。
解决：打破 ABC 类限制，任意划网段；支持路由聚合（超网），大幅缩减全球路由表，同时更灵活分配 IP、减少浪费。

NAT 网络地址转换 出现原因 & 解决什么
起因：就算划分子网 + CIDR，公网 IPv4 还是彻底不够用，没法给每台设备都分配公网 IP。
解决：内网用私有 IP（192.168/10/172）共用一个公网 IP 上网，一台公网 IP 带几十上百台设备，从根源缓解 IPv4 地址枯竭。
~~~

~~~

以太网**硬件本身只认识 MAC 帧**，不认识 IP。

网线、交换机、网卡，底层电路设计**天生只看 MAC 地址**，根本没有解析 IP 的能力。

有！**路由器每一个网口，都自带一个独立唯一的 MAC 地址**。

**IP（三层）**：管**全球 / 跨网段定位**

作用：告诉我，这个数据包**最终要去往哪个网络**。

**MAC（二层）**：管**当前这一段网线里，下一跳给谁**

作用：告诉我，在你眼前这个局域网里，**具体扔给哪台设备的哪个网口**。



![](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250814125551666.png)

ARP（Address Resolution Protocol，地址解析协议）是**局域网**里的一座**翻译桥**，作用是：

> **把 IP 地址翻译成 MAC 地址**，让数据包能在物理网络上传输

在网络层（IP 层），我们用 **IP 地址**来标识设备，比如 `192.168.1.10`。

但在数据链路层（以太网），数据帧要靠 **MAC 地址**来投递，比如 `3C-A6-F6-12-34-56`。

网卡和交换机只认识 MAC 地址，不认识 IP 地址。

所以，当一个设备知道了对方的 **IP 地址**，但不知道 **MAC 地址** 时，就需要 ARP 来查出来



---



**路由器的每个物理接口（端口）** 相当于一个独立的网络节点。

每个接口在 **数据链路层** 都需要一个唯一的 MAC 地址，用于在所连接的局域网中通信。

因为不同接口连接的是 **不同的网络（网段）**，所以它们的 MAC 地址必须不同





**IP 地址（源 IP、目的 IP）**

- 在整个 **端到端** 传输过程中 **不变**（除非经过 NAT）。
- 作用是标识网络层端到端的通信双方。

**MAC 地址（源 MAC、目的 MAC）**

- 每经过一个链路跳（hop），都会变化。

 

当主机发送数据报时，如果目的 IP **不在本地子网**，主机就会把数据报交给默认网关，由它继续转发。



**对于主机号不能全0和全1**

## 子网划分和子网掩码

将IP地址和子网掩码都转换成二进制 对应位做**按位与（AND）运算**

任何数与 1 & 都是本身 

任何数 与 0 & 都是 0

 子网掩码：255.255.255.0 → `11111111.11111111.11111111.00000000`

MASK: 11111111.11111111.11111111.00000000

AND:  11000000.10101000.00000001.00000000  

~~~

三、H1 → H7 完整流程（跨外网、跨互联网）
H1：166.1.0.1 H7：200.1.1.2掩码：255.255.128.0
第一步：H1 两次与运算判断
H1 自身 IP & 掩码 = 166.1.0.0
H7 目标 IP & 掩码 ≠ 166.1.0.0
❌ 不在同一网段 → 发给网关 166.1.0.5
第二步：发帧到学校路由器

二层 MAC：H1MAC → 路由器 B3MAC
第三步：学校路由器

第四步：公司路由器

第五步：最终到 H7
每过一个路由器，二层 MAC 重新封装，三层 IP 永远不变。
~~~

## 路由聚合

**把一堆地址连续、前缀相同的小网络，用它们共同的更短前缀表示成一个大网络。**

与 CIDR 配套的 知识点 

把**很多条小路由条目**，合并成**一条大的路由条目**，**减少全球路由器的路由表数量**，减轻路由器负担、加快转发。

![image-20250817230218505](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250817230218505.png)

>
> 
>
> 1. 优先走**更精细、范围更小**的路由，不走笼统大网段；
> 2. 避免路由冲突，保证数据包**走最准确的路径**；
> 3. 配合 CIDR、路由聚合，让大小网段共存还不混乱。

## 地址解析协议 ARP


ARP 表的本质是：

**一张“局域网内 IP → MAC”的临时缓存表。**

电脑都有它，是因为电脑发以太网帧时必须知道下一跳的 MAC 地址。

路由器每次从某个以太网接口转发 IP 分组时，也要封装成新的以太网帧。
要封装帧，就需要下一跳 MAC。

所以路由器也维护 ARP 表。

| 表     | 谁维护                   | 内容       | 用途                 |
| ------ | ------------------------ | ---------- | -------------------- |
| ARP 表 | 主机、路由器、三层交换机 | IP → MAC   | 知道下一跳 MAC       |
| MAC 表 | 二层交换机               | MAC → 端口 | 知道帧从哪个端口转发 |



**ARP**：根据 IP 地址查询对应的 MAC 地址，实现同一局域网内设备的二层寻址。

![image-20250818003054118](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250818003054118.png)

![image-20260514113717937](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20260514113717937.png)

![image-20250818003226342](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250818003226342.png)



~~~
1. 链路层：解决「局部线路噪声」
网线、无线、链路容易突发干扰、比特出错、丢帧。二层当场发现、当场重传，最快修复，不用跑到高层。
2. 传输层：解决「全网路由复杂丢包、乱序」
互联网路由复杂：
走的路径可能不一样
包到达会乱序
路由器会限流丢包
二层管不了全程乱序、全程丢包
只能靠 TCP 端到端统一兜底。
~~~

**谁有 ARP 表？**

**只有拥有「网络层（IP 协议）」的设备，才有 ARP 表**

**A ---> B**

你需要注意的是，发送方和接收方，他们的这个设备呢？**最高层到哪里**

<span style="color:#FF0000;">**也就是我发送的时候，从最高层开始考虑一下每一层的协议，每一层协议他们应该有对应一种数据结构，比如路由器，它网络层就对应着转发表以及ARP表，交换机就对应着Mac表，一层一层进行分析处理，不要混用，不要混！！！！**</span>

发送方：**从上到下逐层协议封装**，最终转为比特流，经物理链路传给下一跳设备。

接收方：**从下到上逐层解封装**，逐层解析协议头部，剥离首部后把数据上交对应上层处理。



**下一跳 IP 不是天生就在包里的，**

**是路由器靠「路由算法」算出路由表，**

**再由路由表生成转发表，**

**路由器再根据目的 IP 查表得出下一跳。** 

**详细见路由算法** 

![image-20250818003543587](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250818003543587.png)



![](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250818004913896.png)、

交换机工作在 **数据链路层**，它只看 **MAC 地址**，不关心 IP。
 所以：

- A 发 ARP 广播，交换机会泛洪（flood）到所有端口。
- B 收到后回复单播给 A。
- 交换机学习到 A 和 B 的 MAC 地址位置，下次就不会乱泛洪了。

---

**如果经过路由器（跨网段）**

重点来了：**ARP 只在局域网里用，跨网段不行。**
 例如：

- A（192.168.1.10/24）要访问 C（192.168.2.10/24），它发现 C 不在同一子网。
- A 的 IP 层会把包交给 **默认网关（路由器 R，192.168.1.1）**。
- A 必须先对网关 R 做 ARP，得到 R 的 MAC 地址，然后把以太网帧的目的 MAC 设置为 R。
- R 收到后，在 192.168.2.0 网段再做一次 ARP，找到 C 的 MAC，替换以太网头继续转发。

也就是说：

- **多交换机 = 还是一个二层广播域，ARP 广播能穿透。**
- **有路由器 = 广播不能跨越，ARP 在每个子网都要重新解析**

 👉 普通进程不应该乱占用 67，否则整个局域网就可能被“山寨 DHCP”干扰（典型攻击手法：**伪造 DHCP 服务器**）



`53/UDP` 就表示“端口 53 用 UDP 协议”

---



为什么第三步的 链路层和 网络层都要广播呢

有时一个网段里可能配置了不止一个 DHCP 服务器（主备或者负载）。

客户端用广播是为了告诉所有 DHCP 服务器：“我决定用某某服务器的 IP 了，其他服务器请回收你们给我的地址。”

如果只单播给选中的那一台，其他服务器就不知道，容易出现地址冲突



 **在第 4 步 DHCP ACK 时，客户端才真正拿到这个 IP 地址**。

- 在第 2 步 **Offer** 中，服务器只是“保留”了一个 IP 地址（租约未确认）。
- 在第 3 步 **Request** 中，客户端表态说“我选中你给我的这个地址”。
- 在第 4 步 **Ack** 时，服务器才把租约正式写入自己的数据库，并告诉客户端“你可以用了”。

## ICMP 网际控制报文协议

**用来探测网络通不通、报错、查路由路径**，ping、tracert 全靠它。

**核心作用**

1. 网络**报错**：主机不可达、端口不可达、超时等发给源主机；
2. **连通性检测**：ping 命令就是用 ICMP；
3. **路由追踪**：tracert 靠 ICMP 查经过哪些路由器。



分层封装是：有上层就封装，没有上层就不用硬凑，不是每包都必须凑齐「应用层→传输层→网络层」全套！

### 1. **ICMP（Internet Control Message Protocol，因特网控制报文协议）**

- **协议层级**：运行在 **网络层**，属于 IP 的**伴随协议**。
- **用途**：
  - 不是用来传输应用数据，而是用来传输 **控制信息** 和 **差错信息**。
  - 当 IP 数据包在传输过程中**出现问题（无法交付、TTL 耗尽、主机/端口不可达等）**，路由器或主机会生成 ICMP 报文，通知源主机。
- **典型场景**：
  - **Ping**：通过 ICMP Echo Request / Echo Reply 来检测主机是否可达。
  - **Traceroute**：利用 ICMP Time Exceeded 消息，逐跳探测路径。
- **价值类比**：相当于 IP 网络里的“异常报警与诊断工具”，告诉你网络有没有通，但不负责传送业务数据。

------

### 2. **IGMP（Internet Group Management Protocol，因特网组管理协议）**

**互联网组管理协议**，作用：**路由器管理局域网里哪些设备要接收组播流量**。

IGMP 负责：电脑告诉路由器**我要加入哪个直播组**，路由器只把对应组播流发给有需要的设备，不瞎广播浪费带宽。



- **协议层级**：同样运行在 **网络层**，也是 IP 的**伴随协议**，但只适用于 **多播（Multicast）**。
- **用途**：
  - 用来**管理主机**与 **多播组**（Multicast Group）的关系。
  - 主机通过 IGMP 报文告诉路由器：“我想加入某个多播组（例如 224.0.0.x）”，路由器据此决定是否转发该组的多播流量到这个链路。
- **典型场景**：
  - IPTV、在线视频直播、股票行情推送等一对多场景。
  - 如果没有 IGMP，路由器就无法知道哪些用户想要收多播流量，会浪费带宽。
- **价值类比**：相当于“群聊的订阅机制”，你加入群组后，才会收到群里的消息

~~~
在 408/计算机网络常规题里，很多路由算法默认把网络抽象成带权无向图，因为普通链路通常是双向的，比如：

A —— B，代价 3

表示：

A 到 B 代价 3；

B 到 A 代价也 3。

但真实网络里更复杂，很多时候要按有向图理解。比如：

A → B 代价 10；

B → A 代价 2。

这可能是因为两边带宽不同、拥塞不同、策略不同，或者运营商路由策略不对称。

## 分层次的路由协议

**分层路由协议**：将网络划分为多层区域，域内用简单路由、域间用骨干路由，缩减路由表、减少全网泛洪，提升转发效率。



**简单路由（域内路由）**

同一个小片区内部走的路由，**就近转发、规则简单**，只管本地圈子。

**骨干路由（域间路由）**

片区和片区之间连通的**主干大动脉**，负责跨大区远距离传数据。

把巨大互联网切成无数个**自治系统 AS**（分层根基）

**AS 内部**：跑**域内路由 IGP**，搞定内部所有主机互通

内网一堆细碎子网，用**CIDR 聚合**成**一条汇总路由**

AS**边界路由器**只把**聚合后的大网段**对外发

**AS 与 AS 之间**：跑**域间路由 BGP**，只传汇总路由，不传内网细节

## TCP 连接管理

## 一、三次握手：哪些能 / 不能带数据？

表格







| 阶段  | 报文类型                     | 能否携带数据       | 核心原因                                      | 考研考点                                                   |
| ----- | ---------------------------- | ------------------ | --------------------------------------------- | ---------------------------------------------------------- |
| 握手① | 客户端→服务器 `SYN=1`        | ❌ **不能携带数据** | 此时连接未建立，服务器会丢弃携带数据的 SYN 包 | SYN 包会消耗 1 个序号，即使无数据                          |
| 握手② | 服务器→客户端 `SYN=1, ACK=1` | ❌ **不能携带数据** | 同样处于连接建立阶段，不允许带数据            | SYN 包会消耗 1 个序号                                      |
| 握手③ | 客户端→服务器 `ACK=1`        | ✅ **可以携带数据** | 此时连接已完成双向同步，通道可用              | 若携带 N 字节数据，ack 确认号会 +N；不携带则不额外消耗序号 |

> 补充说明：
>
> 握手①②虽然不带数据，但`SYN`标志位本身会消耗**1 个序号**，所以后续数据的起始序号是 `ISN+1`。
>
> 握手③如果带数据，这部分数据会直接被服务器接收并缓存，不用再额外发一次报文。

------

## 二、四次挥手：哪些能 / 不能带数据？



| 阶段  | 报文类型              | 能否携带数据         | 核心原因                                                     | 考研考点                          |
| ----- | --------------------- | -------------------- | ------------------------------------------------------------ | --------------------------------- |
| 挥手① | 客户端→服务器 `FIN=1` | ❌ **不建议携带数据** | 客户端发送 FIN 后，就进入`FIN_WAIT_1`状态，不能再发送新数据了 | FIN 包会消耗 1 个序号，即使无数据 |
| 挥手② | 服务器→客户端 `ACK=1` | ✅ **可以携带数据**   | 服务器确认收到 FIN 后，仍可以继续发送剩余数据（此时服务器到客户端的通道还开着） | 这是 TCP 半关闭状态的核心体现     |
| 挥手③ | 服务器→客户端 `FIN=1` | ❌ **不建议携带数据** | 服务器发送 FIN，代表数据已经全部发完，不会再发送数据了       | FIN 包会消耗 1 个序号             |
| 挥手④ | 客户端→服务器 `ACK=1` | ❌ **不能携带数据**   | 客户端发送 ACK 后，进入`TIME_WAIT`状态，不能再发送数据       | 此 ACK 不消耗序号                 |

> 补充说明：
>
> 挥手①/③的`FIN`标志位，和`SYN`一样，即使不带数据，也会消耗**1 个序号**。
>
> 挥手②的 ACK 报文，是服务器唯一还能发数据的机会，这就是 “四次挥手” 而不是三次的原因 —— 服务器需要时间发完剩余数据，再回复 FIN。

------

## 三、一句话极简总结

- **三次握手**：只有**第三次握手（ACK）** 可以携带数据，前两次 SYN 包不行。
- **四次挥手**：只有**第二次挥手（ACK）** 可以携带数据，其余 FIN/ACK 报文都不行。

### 释放连接

![image-20250818165230924](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250818165230924.png)

![image-20250818170150859](https://raw.githubusercontent.com/kasahuki/os_test/main/img/image-20250818170150859.png)

**挥手1（FIN=1, Client→Server）**
 客户端说：“我不再给你发数据了（应用层 write 关闭），但是我还能收数据。”
 👉 意味着：**客户端到服务端的方向被关闭**。

**挥手2（ACK, Server→Client）**
 服务端确认：“好，我知道你不会再给我发数据了。”
 👉 现在状态：**单向关闭**（客户端→服务端通道关闭；服务端→客户端通道仍然开放）。

**挥手3（FIN=1, Server→Client）**
 服务端说：“我这边也发完数据了，我也不再发了。”
 👉 意味着：**服务端到客户端的方向也要关闭**。

**挥手4（ACK, Client→Server）**
 客户端确认：“好的，我知道你不发了。”
 👉 双方都不再发送数据，**连接彻底关闭**



> 挥手2之后，客户端不是已经不能发数据了吗？为什么还能在挥手4中给服务端发 ACK 呢？



这就要区分 **TCP 数据报文（Data Segment）** 和 **TCP 控制报文（Control Segment）**：

- **数据传输通道关闭**：
   挥手1时，客户端发 FIN，相当于关闭了“数据写入”，客户端不能再发 **应用层数据** 给服务端。
- **控制==信令==仍然可用**：
   即使数据方向关闭，TCP 协议栈 **仍然可以发送必要的控制报文**（ACK、FIN、RST 等），这些报文不算“应用层数据”，只是 TCP 协议本身需要的“管理指令”。

所以挥手4时，客户端发送的 ACK 是 **协议控制报文**，不是应用层数据，因此完全合理

注意 **<u>方向的变化</u>** 意味着 角色 的**对位转变**

**DNS 解析过程：**



**浏览器缓存**：先看自己有没有记忆（最近解析过同样的域名）。

**操作系统缓存（DNS 缓存）**：如果浏览器没有，就问操作系统的 DNS 缓存。

**Hosts 文件**：还会看 `hosts` 文件里有没有硬编码的映射。



你的电脑一般会配置一个 **本地递归 DNS 服务器**（比如运营商的 DNS，或 `8.8.8.8` Google DNS）。
 浏览器把请求丢给它，它来帮你问遍全世界。



假设本地域名服务器也没有缓存，它会按层级去问：

1. **==根==域名服务器（Root Server）**
   - 这是互联网的“导航员”。
   - 它本身不知道 `www.example.com` 的 IP，但知道 **`.com` 顶级域名服务器**的地址。
   - 所以回答：“去问 `.com` 服务器吧。”
2. **==顶级==域名服务器（TLD Server）**
   - `.com` 的权威服务器接到请求。
   - 它也不知道 `www.example.com` 的 IP，但知道 `example.com` 的权威服务器地址。
   - 所以回答：“去问 `example.com` 的权威服务器。”
3. **==权威（权限）==域名服务器（Authoritative Name Server）**
   - 这是 `example.com` 自己的“管家”。
   - 它知道 `www.example.com` 具体 IP（比如 192.0.2.123）。
   - 返回这个结果给递归 DNS





**权威域名服务器是什么**

它就是某个域名的 **最终解释者**。

- 它保存了这个域名的 **DNS 记录**（比如 A 记录、MX 记录、CNAME 记录）。
- 当别人一路问下来（根 → 顶级域名 → 二级域名……），最后就是问到它。
- 它的回答就是“标准答案”，不会再去问别人。

举个例子：

- 你要访问 `tieba.baidu.com`
- 根告诉你：去问 `.com`
- `.com` 服务器告诉你：去问 `baidu.com` 的权威服务器
- `baidu.com` 的权威服务器回答：“`tieba.baidu.com` 的 IP 是 220.181.38.148”

这个回答就是权威的，别人不能改口



~~~
baidu.com                 （二级域名）
├─ www.baidu.com           （三级域名，主站）
├─ tieba.baidu.com         （三级域名，贴吧）
│   └─ student.tieba.baidu.com  （四级域名，学生贴吧）
├─ map.baidu.com           （三级域名，地图）
└─ pan.baidu.com           （三级域名，网盘）

~~~

FTP 是一个**应用层协议**，用于在网络上传输文件。它基于 TCP/IP 协议栈，典型工作方式是**控制连接 + 数据连接**分离

**控制连接（Control Connection）**

- 用于发送命令（如登录、切换目录、获取文件列表）。
- 建立一次连接即可，长时间保持。
- 默认 TCP 端口：**21**（服务端监听，客户端发起连接）。

**数据连接（Data Connection）**

- 用于实际的文件传输（上传/下载）。
- 可以是临时的连接，每次传输建立一次。
- TCP 端口：视模式而定（主动或被动）。



---



**FTP 传输模式：主动模式 vs 被动模式**

FTP 的核心复杂点在于数据连接建立方式不同。

### 主动模式（Active Mode）

**流程**：

1. 客户端连接服务器 **21 端口**（控制连接）。
2. 客户端告诉服务器自己监听的端口（通过 `PORT` 命令）。
3. 服务器从 **20 端口** 发起连接到客户端提供的端口（数据连接）。
4. 文件传输完成后，数据连接关闭。

**特点**：

- 服务器主动建立数据连接到客户端。
- 对防火墙不友好：客户端通常在 NAT/防火墙后，外部服务器难以连接客户端端口。

**端口总结**：

| 连接类型 | 客户端端口               | 服务端端口 |
| -------- | ------------------------ | ---------- |
| 控制连接 | 随机高端口               | 21         |
| 数据连接 | 随机高端口（客户端监听） | 20         |

------

###  被动模式（Passive Mode）

**流程**：

1. 客户端连接服务器 **21 端口**（控制连接）。
2. 客户端发送 `PASV` 命令，服务器返回一个随机端口号。
3. 客户端主动连接服务器指定端口建立数据连接。
4. 文件传输完成后，数据连接关闭。

**特点**：

- 客户端主动连接服务器，避免防火墙阻塞。
- 企业级服务器通常推荐被动模式。

**端口总结**：

| 连接类型 | 客户端端口 | 服务端端口             |
| -------- | ---------- | ---------------------- |
| 控制连接 | 随机高端口 | 21                     |
| 数据连接 | 随机高端口 | 随机高端口（服务器返回 |

HTTP 流水线是 **在同一个 TCP 连接上，客户端可以连续发送多个 HTTP 请求，而不必等待前一个响应返回**，服务器按顺序返回响应

非流水线模式是 **客户端每发送一个请求就等待服务器返回响应后，才能发送下一个请求**

| 特性     | 流水线（Pipelining）         | 非流水线（Non-Pipelining） |
| -------- | ---------------------------- | -------------------------- |
| 请求发送 | 连续发送，无需等待前一个响应 | 发送一个请求等待响应       |
| 响应顺序 | 必须保持顺序                 | 自然顺序                   |
| 性能     | 高（减少 RTT 等待）          | 低（串行阻塞）             |
| 适用版本 | HTTP/1.1                     | HTTP/1.0/1.1               |
| 缺点     | Head-of-Line 阻塞、兼容性差  | 延迟高                     |

*CRLF*是Carriage-Return Line-Feed的缩写，意思是回车换行，就是回车
