## 🚨 部署阶段避坑

### 坑1: Docker网络不通

**现象**:
```
Prometheus targets 显示 DOWN
错误: Get "http://mysql-exporter:9104/metrics": dial tcp: lookup mysql-exporter: no such host
```

**根本原因**: 容器不在同一网络

**排查步骤**:
```bash
# 1. 查看所有网络
docker network ls

# 2. 查看容器所属网络
docker inspect archat_prometheus | grep NetworkMode

# 3. 查看网络详情
docker network inspect monitoring
```

**解决方案**:
```yaml
# docker-compose.monitoring.yml
services:
  prometheus:
    networks:
      - monitoring
      - default  # 添加原有网络
  
  mysql-exporter:
    networks:
      - monitoring
      - default

networks:
  monitoring:
    driver: bridge
  default:
    external: true
    name: archat_default  # 修改为实际网络名
```

### 坑2: 端口冲突

**现象**:
```
Error starting userland proxy: listen tcp 0.0.0.0:3000: bind: address already in use
```

**排查命令**:
```bash
# Windows PowerShell
netstat -ano | findstr "3000"

# 输出示例:
# TCP    0.0.0.0:3000    0.0.0.0:0    LISTENING    12345
```

**解决方案A**: 停止占用端口的程序
```bash
# 根据 PID 终止进程
taskkill /PID 12345 /F
```

**解决方案B**: 修改映射端口
```yaml
services:
  grafana:
    ports:
      - "3001:3000"  # 映射到3001端口
```

### 坑3: Windows 下 Node Exporter 指标不完整

**现象**:
```
缺少 filesystem、diskstats 等指标
```

**根本原因**: Windows Docker 运行在 WSL2,路径映射问题

**解决方案**:
```yaml
# 方案1: 使用 windows_exporter(推荐)
# 下载: https://github.com/prometheus-community/windows_exporter/releases
# 启动: windows_exporter-amd64.exe --collectors.enabled="cpu,cs,logical_disk,net,os,system"

# 方案2: 使用 WSL2 内的路径
node-exporter:
  volumes:
    - /proc:/host/proc:ro
    - /sys:/host/sys:ro
    - /:/rootfs:ro
  command:
    - '--path.procfs=/host/proc'
    - '--path.sysfs=/host/sys'
    - '--path.rootfs=/rootfs'
```

### 坑4: MySQL Exporter 连接失败

**现象**:
```
level=error msg="Error pinging mysqld" err="Error 1045: Access denied for user 'root'@'172.18.0.5' (using password: YES)"
```

**常见原因**:
1. 密码错误
2. MySQL 未授权远程连接
3. 容器网络不通

**排查步骤**:
```bash
# 1. 测试数据库连接
docker exec -it mysql mysql -uroot -proot

# 2. 检查用户权限
SHOW GRANTS FOR 'root'@'%';

# 3. 授权(如果需要)
GRANT ALL PRIVILEGES ON *.* TO 'root'@'%' IDENTIFIED BY 'root' WITH GRANT OPTION;
FLUSH PRIVILEGES;
```

**最佳实践**: 创建专用监控用户
```sql
-- 创建只读用户
CREATE USER 'exporter'@'%' IDENTIFIED BY 'exporter_password';
GRANT PROCESS, REPLICATION CLIENT, SELECT ON *.* TO 'exporter'@'%';
FLUSH PRIVILEGES;
```

```yaml
# 使用专用用户
mysql-exporter:
  environment:
    - DATA_SOURCE_NAME=exporter:exporter_password@(mysql:3306)/
```

### 坑5: Prometheus 数据盘满

**现象**:
```
level=error msg="Error on ingesting samples that are too old or are too far into the future" num_dropped=1000
```

**排查磁盘占用**:
```bash
# 查看 Docker 磁盘占用
docker system df

# 查看 Prometheus 数据目录
docker exec archat_prometheus du -sh /prometheus
```

**解决方案**:
```yaml
# 1. 调整数据保留时间
prometheus:
  command:
    - '--storage.tsdb.retention.time=7d'  # 从15天改为7天
    - '--storage.tsdb.retention.size=10GB'  # 限制大小

# 2. 清理旧数据
docker exec archat_prometheus rm -rf /prometheus/wal

# 3. 定期清理 Docker 垃圾
docker system prune -a --volumes
```

## ⚡ 性能优化避坑

### 坑6: PromQL 查询超时

**现象**:
```
Grafana 仪表盘加载慢或超时
错误: "Query timeout exceeded"
```

**常见原因**: 查询范围过大、无标签过滤

**❌ 低效查询**:
```promql
# 全表扫描
sum(rate(http_requests_total[1h]))

# 时间范围过长
increase(mysql_global_status_questions[7d])
```

**✅ 优化后**:
```promql
# 添加标签过滤
sum(rate(http_requests_total{job="api-server"}[5m]))

# 使用聚合减少数据点
sum by(instance) (rate(http_requests_total[5m]))

# 缩短时间范围
increase(mysql_global_status_questions[1h])
```

**Grafana 查询优化**:
```json
{
  "targets": [
    {
      "expr": "sum(rate(http_requests_total[5m]))",
      "interval": "30s",  // 降低查询精度
      "maxDataPoints": 1000  // 限制数据点数量
    }
  ]
}
```

### 坑7: 指标爆炸(High Cardinality)

**现象**:
```
Prometheus 内存占用暴涨
查询变慢
```

**根本原因**: 标签值过多,产生大量时间序列

**❌ 错误示例**:
```java
// 用户 ID 作为标签(1000万用户 = 1000万时间序列)
Counter.builder("user.login")
    .tag("user_id", String.valueOf(userId))  // ❌ 错误!
    .register(meterRegistry)
    .increment();
```

**✅ 正确做法**:
```java
// 使用分类标签
Counter.builder("user.login")
    .tag("user_type", userType)  // 只有几种类型
    .tag("region", region)       // 有限的区域
    .register(meterRegistry)
    .increment();
```

**排查高基数指标**:
```promql
# 查询时间序列数量
count({__name__=~".+"}) by (__name__)

# 查看占用最多的指标
topk(10, count by (__name__)({__name__=~".+"}))
```

**告警规则**:
```yaml
- alert: HighCardinality
  expr: scrape_samples_scraped > 10000  # 单次采集超过1万指标
  for: 10m
  labels:
    severity: warning
  annotations:
    summary: "{{ $labels.job }} 指标数量过多"
```

### 坑8: Grafana 仪表盘刷新导致 Prometheus 压力大

**现象**:
```
多个仪表盘同时打开,Prometheus CPU 飙升
```

**原因**: 刷新间隔过短、查询复杂

**优化方案**:
```
1. 延长刷新间隔
   Settings → Time Range → Refresh: 30s → 1m

2. 使用变量缓存
   Variables → Refresh: On Dashboard Load (不要用 On Time Range Change)

3. 减少 Panel 数量
   合并相似图表,使用 Row 折叠

4. 使用查询缓存
   Data Source Settings → Cache Timeout: 60s
```

## 🔒 安全避坑

### 坑9: 监控端点未鉴权

**风险**: `/actuator` 端点暴露敏感信息

**❌ 危险配置**:
```yaml
management:
  endpoints:
    web:
      exposure:
        include: '*'  # 暴露所有端点
```

**✅ 安全配置**:
```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,prometheus  # 只暴露必要端点
      base-path: /internal/actuator  # 修改路径
  
  # 开启安全认证
  security:
    enabled: true
```

**Nginx 反向代理保护**:
```nginx
location /actuator {
    # 仅允许内网访问
    allow 10.0.0.0/8;
    allow 172.16.0.0/12;
    deny all;
    
    proxy_pass http://backend:8080;
}
```

### 坑10: Grafana 默认密码未修改

**风险**: 默认 `admin/admin` 被暴力破解

**解决方案**:
```yaml
# docker-compose.monitoring.yml
grafana:
  environment:
    - GF_SECURITY_ADMIN_PASSWORD=YourStrongPassword123!  # 强密码
    - GF_USERS_ALLOW_SIGN_UP=false  # 禁止注册
    - GF_AUTH_ANONYMOUS_ENABLED=false  # 禁止匿名访问
```

**首次登录强制修改密码**:
```bash
# 进入容器
docker exec -it archat_grafana bash

# 修改密码
grafana-cli admin reset-admin-password NewPassword123!
```

## 📊 告警避坑

### 坑11: 告警风暴

**现象**:
```
一个故障触发几百条告警
钉钉群被刷爆
```

**原因**: 未配置告警聚合、抑制规则

**解决方案(AlertManager)**:
```yaml
# alertmanager.yml
route:
  group_by: ['alertname', 'instance']
  group_wait: 30s        # 等待30秒聚合
  group_interval: 5m     # 5分钟发送一次
  repeat_interval: 1h    # 1小时重复一次

  routes:
    # 严重告警立即通知
    - match:
        severity: critical
      receiver: 'on-call'
      group_wait: 10s
    
    # 警告告警延迟通知
    - match:
        severity: warning
      receiver: 'team'
      group_wait: 5m

# 抑制规则(节点Down时不发送其他告警)
inhibit_rules:
  - source_match:
      alertname: 'NodeDown'
    target_match:
      alertname: 'HighCPU|HighMemory'
    equal: ['instance']
```

### 坑12: 告警阈值设置不合理

**❌ 错误示例**:
```yaml
# CPU 超过50%就告警(过于敏感)
- alert: HighCPU
  expr: cpu_usage > 50
  for: 1m  # 持续时间过短
```

**✅ 合理阈值**:
```yaml
groups:
  - name: cpu_alerts
    rules:
      # 三级告警
      - alert: HighCPU_Warning
        expr: cpu_usage > 70
        for: 10m  # 持续10分钟
        labels:
          severity: warning
      
      - alert: HighCPU_Critical
        expr: cpu_usage > 85
        for: 5m
        labels:
          severity: critical
      
      - alert: HighCPU_Emergency
        expr: cpu_usage > 95
        for: 2m
        labels:
          severity: emergency
```

**阈值参考表**:
| 指标 | 警告阈值 | 严重阈值 | 持续时间 |
|-----|---------|---------|---------|
| CPU | 70% | 85% | 5-10min |
| 内存 | 80% | 90% | 5min |
| 磁盘 | 80% | 90% | 30min |
| 慢查询 | 10个/秒 | 50个/秒 | 5min |
| 接口错误率 | 5% | 10% | 5min |
| 接口P99延迟 | 1s | 3s | 5min |

## 🛠️ 运维避坑

### 坑13: 数据未备份

**风险**: Prometheus/Grafana 数据丢失

**备份方案**:
```bash
#!/bin/bash
# backup_monitoring.sh

BACKUP_DIR="/backup/monitoring/$(date +%Y%m%d)"
mkdir -p $BACKUP_DIR

# 备份 Prometheus 数据
docker run --rm \
  -v archat_prometheus-data:/data \
  -v $BACKUP_DIR:/backup \
  alpine tar czf /backup/prometheus-data.tar.gz /data

# 备份 Grafana 数据
docker run --rm \
  -v archat_grafana-data:/data \
  -v $BACKUP_DIR:/backup \
  alpine tar czf /backup/grafana-data.tar.gz /data

# 备份配置文件
tar czf $BACKUP_DIR/config.tar.gz monitoring/

# 清理7天前的备份
find /backup/monitoring/ -mtime +7 -delete
```

**定时任务**(cron):
```bash
# 每天凌晨2点备份
0 2 * * * /path/to/backup_monitoring.sh
```

### 坑15: 监控系统本身无监控

**风险**: Prometheus/Grafana 挂了无人知晓

**解决方案**: 外部监控
```yaml
# prometheus.yml
scrape_configs:
  # 监控 Prometheus 自身
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

# 告警规则
groups:
  - name: monitoring_system_alerts
    rules:
      - alert: PrometheusDown
        expr: up{job="prometheus"} == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Prometheus 服务不可用!"
      
      - alert: GrafanaDown
        expr: up{job="grafana"} == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Grafana 服务不可用!"
```

**使用外部健康检查服务**(推荐):
- UptimeRobot (免费)
- Pingdom
- 云厂商监控

## 🚀 性能优化最佳实践

### 1. Prometheus 优化

```yaml
prometheus:
  command:
    # 数据保留
    - '--storage.tsdb.retention.time=15d'
    - '--storage.tsdb.retention.size=50GB'
    
    # 查询优化
    - '--query.max-concurrency=20'
    - '--query.timeout=2m'
    
    # 内存优化
    - '--storage.tsdb.max-block-duration=2h'
    - '--storage.tsdb.min-block-duration=2h'
  
  deploy:
    resources:
      limits:
        memory: 4G
        cpus: '2.0'
```

### 2. Grafana 优化

```yaml
grafana:
  environment:
    # 查询缓存
    - GF_DATAPROXY_TIMEOUT=60
    - GF_DATAPROXY_KEEP_ALIVE_SECONDS=30
    
    # 渲染优化
    - GF_RENDERING_SERVER_URL=http://renderer:8081/render
    - GF_RENDERING_CALLBACK_URL=http://grafana:3000/
```

### 3. 应用优化

```yaml
# application.yml
management:
  metrics:
    # 禁用不需要的指标
    enable:
      jvm: true
      system: true
      tomcat: true
      logback: false  # 禁用日志指标(高开销)
    
    # 指标过滤
    export:
      prometheus:
        enabled: true
        pushgateway:
          enabled: false
```

## 📚 问题排查流程

### 通用排查步骤

```
1. 确定现象
   ├─ 监控数据缺失?
   ├─ 查询报错?
   └─ 告警异常?

2. 查看日志
   ├─ docker logs archat_prometheus
   ├─ docker logs archat_grafana
   └─ docker logs archat-backend

3. 检查配置
   ├─ prometheus.yml 语法
   ├─ targets 状态
   └─ 网络连通性

4. 验证数据
   ├─ curl /actuator/prometheus
   ├─ Prometheus 查询
   └─ Grafana 测试

5. 逐步排除
   ├─ 单独测试组件
   ├─ 简化配置
```
