## 📖 告警体系设计

### 1. 告警分级

| 级别 | 严重性 | 响应时间 | 通知方式 | 示例 |
|-----|-------|---------|---------|-----|
| **P0 - Critical** | 核心服务不可用 | 立即(5分钟内) | 电话+短信+群消息 | 数据库宕机、服务全量Down |
| **P1 - High** | 影响业务功能 | 30分钟内 | 短信+群消息 | 接口错误率>10%、内存OOM |
| **P2 - Warning** | 性能下降 | 2小时内 | 群消息 | CPU>80%、慢查询增多 |
| **P3 - Info** | 潜在风险 | 工作时间处理 | 邮件 | 磁盘使用>70%、缓存命中率低 |

### 2. 告警生命周期

```
触发(Firing) → 持续(Pending) → 确认(Acknowledged) → 解决(Resolved)
     ↓              ↓                    ↓                  ↓
  发送通知      等待阈值时间         人工介入              自动恢复通知
```

### 3. 告警最佳实践

**DO(应该做)**:
- ✅ 设置合理的持续时间(`for: 5m`),避免抖动
- ✅ 告警信息包含上下文(实例名、当前值、阈值)
- ✅ 按业务影响分级,不是所有告警都是 Critical
- ✅ 提供快速定位链接(Grafana 仪表盘、日志查询)

**DON'T(不应该做)**:
- ❌ 阈值设置过于敏感(如 CPU>50% 就告警)
- ❌ 所有告警都发短信/电话(会麻木)
- ❌ 告警信息模糊(如"服务异常",不知道哪个服务)
- ❌ 没有告警降噪策略(同一问题重复告警)

## ⚙️ Prometheus 告警规则详解

### 1. 规则文件结构

```yaml
groups:                              # 告警组
  - name: example_alerts             # 组名
    interval: 30s                    # 评估间隔(可选,默认使用global)
    rules:                           # 规则列表
      - alert: AlertName             # 告警名称(必填)
        expr: prometheus_query       # PromQL 表达式(必填)
        for: 5m                      # 持续时间(可选,默认立即触发)
        labels:                      # 自定义标签(可选)
          severity: warning
          team: backend
        annotations:                 # 告警描述(可选)
          summary: "简短摘要"
          description: "详细描述"
```

### 2. 核心字段说明

#### `expr` - 告警条件
```yaml
# 返回值 > 0 触发告警
expr: cpu_usage > 80

# 复杂条件
expr: |
  (avg by(instance) (irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
  and
  node_load5 > 4
```

#### `for` - 持续时间
```yaml
# 立即触发(不推荐,容易误报)
for: 0s

# 持续5分钟后触发(推荐)
for: 5m

# 不同告警级别的建议
for: 1m   # Critical 严重告警
for: 5m   # Warning 警告
for: 10m  # Info 信息
```

#### `labels` - 标签
```yaml
labels:
  severity: critical       # 严重程度
  team: backend           # 责任团队
  service: mysql          # 服务名称
  env: production         # 环境
```

#### `annotations` - 描述(支持模板变量)
```yaml
annotations:
  summary: "{{ $labels.instance }} CPU使用率过高"
  description: "当前值: {{ $value }}%, 阈值: 80%"
  dashboard: "http://grafana:3000/d/node-overview"
  runbook: "https://wiki.company.com/runbook/high-cpu"
```

**可用变量**:
- `{{ $labels.标签名 }}`: 标签值
- `{{ $value }}`: 查询结果值
- `{{ $externalLabels }}`: 全局标签

## 📝 完整告警规则示例

### 1. 系统资源告警

创建文件: `monitoring/prometheus/rules/system_alerts.yml`
```yaml
groups:
  - name: system_alerts
    interval: 30s
    rules:
      # ==================== CPU 告警 ====================
      - alert: HighCPUUsage
        expr: |
          100 - (avg by(instance) (irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
        for: 5m
        labels:
          severity: warning
          category: system
        annotations:
          summary: "{{ $labels.instance }} CPU使用率过高"
          description: |
            实例: {{ $labels.instance }}
            当前值: {{ $value | humanize }}%
            阈值: 80%
            持续时间: 5分钟

      - alert: CriticalCPUUsage
        expr: |
          100 - (avg by(instance) (irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 90
        for: 2m
        labels:
          severity: critical
          category: system
        annotations:
          summary: "{{ $labels.instance }} CPU使用率严重过高"
          description: "当前CPU使用率 {{ $value | humanize }}%, 需立即处理"

      # ==================== 内存告警 ====================
      - alert: HighMemoryUsage
        expr: |
          (1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100 > 85
        for: 5m
        labels:
          severity: warning
          category: system
        annotations:
          summary: "{{ $labels.instance }} 内存使用率过高"
          description: "当前使用率: {{ $value | humanize }}%"

      - alert: MemoryOOM
        expr: |
          (1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100 > 95
        for: 1m
        labels:
          severity: critical
          category: system
        annotations:
          summary: "{{ $labels.instance }} 内存即将耗尽"
          description: "当前可用内存: {{ $value | humanize }}%, 可能OOM"

      # ==================== 磁盘告警 ====================
      - alert: HighDiskUsage
        expr: |
          (1 - (node_filesystem_avail_bytes{fstype!="tmpfs"} / node_filesystem_size_bytes{fstype!="tmpfs"})) * 100 > 80
        for: 10m
        labels:
          severity: warning
          category: system
        annotations:
          summary: "{{ $labels.instance }} 磁盘使用率过高"
          description: |
            实例: {{ $labels.instance }}
            挂载点: {{ $labels.mountpoint }}
            当前使用率: {{ $value | humanize }}%

      - alert: DiskWillFull
        expr: |
          predict_linear(node_filesystem_avail_bytes{fstype!="tmpfs"}[1h], 4*3600) < 0
        for: 30m
        labels:
          severity: warning
          category: system
        annotations:
          summary: "{{ $labels.instance }} 磁盘预计4小时后满"
          description: "挂载点: {{ $labels.mountpoint }}, 请提前清理"

      # ==================== 网络告警 ====================
      - alert: HighNetworkTraffic
        expr: |
          rate(node_network_receive_bytes_total[5m]) / 1024 / 1024 > 100
        for: 5m
        labels:
          severity: warning
          category: system
        annotations:
          summary: "{{ $labels.instance }} 入站流量过高"
          description: "当前入站: {{ $value | humanize }} MB/s"
```

### 2. 应用告警规则

创建文件: `monitoring/prometheus/rules/application_alerts.yml`
```yaml
groups:
  - name: application_alerts
    rules:
      # ==================== 服务可用性 ====================
      - alert: ServiceDown
        expr: up{job="spring-boot"} == 0
        for: 1m
        labels:
          severity: critical
          category: application
        annotations:
          summary: "应用 {{ $labels.application }} 不可用"
          description: "实例 {{ $labels.instance }} 已停止响应超过1分钟"
          action: "立即检查日志: docker logs {{ $labels.container }}"

      # ==================== JVM 告警 ====================
      - alert: HighJVMMemory
        expr: |
          (jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"}) * 100 > 85
        for: 5m
        labels:
          severity: warning
          category: jvm
        annotations:
          summary: "{{ $labels.application }} JVM堆内存使用过高"
          description: "当前使用率: {{ $value | humanize }}%"

      - alert: HighGCTime
        expr: |
          rate(jvm_gc_pause_seconds_sum[5m]) / rate(jvm_gc_pause_seconds_count[5m]) > 1
        for: 5m
        labels:
          severity: warning
          category: jvm
        annotations:
          summary: "{{ $labels.application }} GC平均耗时过长"
          description: "平均GC耗时: {{ $value | humanize }}s, 影响性能"

      - alert: HighGCRate
        expr: |
          rate(jvm_gc_pause_seconds_count[5m]) > 10
        for: 5m
        labels:
          severity: warning
          category: jvm
        annotations:
          summary: "{{ $labels.application }} GC频率过高"
          description: "每秒GC次数: {{ $value | humanize }}"

      # ==================== 接口告警 ====================
      - alert: HighErrorRate
        expr: |
          sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m])) by (application, uri) /
          sum(rate(http_server_requests_seconds_count[5m])) by (application, uri) * 100 > 5
        for: 3m
        labels:
          severity: critical
          category: api
        annotations:
          summary: "{{ $labels.application }} 接口 {{ $labels.uri }} 错误率过高"
          description: "当前错误率: {{ $value | humanize }}%"

      - alert: HighLatency
        expr: |
          histogram_quantile(0.99, 
            sum(rate(http_server_requests_seconds_bucket[5m])) by (le, application, uri)
          ) > 3
        for: 5m
        labels:
          severity: warning
          category: api
        annotations:
          summary: "{{ $labels.application }} 接口 {{ $labels.uri }} 响应慢"
          description: "P99延迟: {{ $value | humanize }}s"

      - alert: LowThroughput
        expr: |
          sum(rate(http_server_requests_seconds_count[5m])) by (application) < 10
        for: 10m
        labels:
          severity: warning
          category: api
        annotations:
          summary: "{{ $labels.application }} 请求量异常低"
          description: "当前QPS: {{ $value | humanize }}, 可能服务异常"

      # ==================== 数据库连接池 ====================
      - alert: DatabaseConnectionPoolExhausted
        expr: |
          hikaricp_connections_active / hikaricp_connections >= 0.9
        for: 3m
        labels:
          severity: critical
          category: database
        annotations:
          summary: "{{ $labels.application }} 数据库连接池即将耗尽"
          description: |
            活跃连接: {{ $value | humanize }}%
            建议: 检查慢查询或增加连接数

      - alert: DatabaseConnectionTimeout
        expr: |
          rate(hikaricp_connections_timeout_total[5m]) > 0
        for: 1m
        labels:
          severity: critical
          category: database
        annotations:
          summary: "{{ $labels.application }} 数据库连接超时"
          description: "每秒超时次数: {{ $value | humanize }}"
```

### 3. 业务告警规则

创建文件: `monitoring/prometheus/rules/business_alerts.yml`
```yaml
groups:
  - name: business_alerts
    rules:
      # 支付成功率过低
      - alert: LowPaymentSuccessRate
        expr: |
          sum(rate(business_payment_success_total[5m])) /
          (sum(rate(business_payment_success_total[5m])) + 
           sum(rate(business_payment_fail_total[5m]))) * 100 < 95
        for: 5m
        labels:
          severity: critical
          category: business
        annotations:
          summary: "支付成功率过低"
          description: "当前成功率: {{ $value | humanize }}%, 需立即排查"

      # AI对话耗时过长
      - alert: SlowAIResponse
        expr: |
          histogram_quantile(0.95, 
            rate(business_ai_chat_duration_seconds_bucket[5m])
          ) > 10
        for: 5m
        labels:
          severity: warning
          category: business
        annotations:
          summary: "AI对话响应慢"
          description: "P95延迟: {{ $value | humanize }}s"
```

## 🔔 AlertManager 配置

### 1. 部署 AlertManager

更新 `docker-compose.monitoring.yml`:
```yaml
services:
  alertmanager:
    image: prom/alertmanager:v0.26.0
    container_name: archat_alertmanager
    restart: unless-stopped
    ports:
      - "9093:9093"
    volumes:
      - ./monitoring/alertmanager:/etc/alertmanager
      - alertmanager-data:/alertmanager
    command:
      - '--config.file=/etc/alertmanager/alertmanager.yml'
      - '--storage.path=/alertmanager'
    networks:
      - monitoring

volumes:
  alertmanager-data:
```

### 2. AlertManager 配置文件

创建文件: `monitoring/alertmanager/alertmanager.yml`
```yaml
global:
  resolve_timeout: 5m  # 告警恢复超时

# 路由规则
route:
  receiver: 'default'  # 默认接收器
  group_by: ['alertname', 'instance']  # 按告警名和实例分组
  group_wait: 30s      # 分组等待时间
  group_interval: 5m   # 分组发送间隔
  repeat_interval: 4h  # 重复发送间隔

  routes:
    # Critical 告警立即发送
    - match:
        severity: critical
      receiver: 'critical-alerts'
      group_wait: 10s
      repeat_interval: 1h
    
    # Warning 告警延迟发送
    - match:
        severity: warning
      receiver: 'warning-alerts'
      group_wait: 5m
      repeat_interval: 12h
    
    # 业务告警单独处理
    - match:
        category: business
      receiver: 'business-team'

# 抑制规则(避免告警风暴)
inhibit_rules:
  # 节点Down时,抑制该节点的其他告警
  - source_match:
      alertname: 'ServiceDown'
    target_match_re:
      alertname: 'High.*|Slow.*'
    equal: ['instance']
  
  # Critical 告警触发时,抑制相关 Warning
  - source_match:
      severity: 'critical'
    target_match:
      severity: 'warning'
    equal: ['alertname', 'instance']

# 接收器配置
receivers:
  - name: 'default'
    webhook_configs:
      - url: 'http://alertmanager-webhook:8080/webhook'

  - name: 'critical-alerts'
    webhook_configs:
      - url: 'http://dingtalk-webhook:8080/dingtalk/critical'
    # email_configs:
    #   - to: 'oncall@company.com'

  - name: 'warning-alerts'
    webhook_configs:
      - url: 'http://dingtalk-webhook:8080/dingtalk/warning'

  - name: 'business-team'
    webhook_configs:
      - url: 'http://dingtalk-webhook:8080/dingtalk/business'
```

### 3. Prometheus 关联 AlertManager

更新 `monitoring/prometheus/prometheus.yml`:
```yaml
# 告警管理器配置
alerting:
  alertmanagers:
    - static_configs:
        - targets:
            - 'alertmanager:9093'
```

## 📱 钉钉/企业微信通知

### 方式一: Prometheus Webhook Dingtalk(推荐)

**部署**:
```yaml
# docker-compose.monitoring.yml
services:
  dingtalk-webhook:
    image: timonwong/prometheus-webhook-dingtalk:v2.1.0
    container_name: archat_dingtalk_webhook
    restart: unless-stopped
    ports:
      - "8060:8060"
    volumes:
      - ./monitoring/dingtalk:/etc/prometheus-webhook-dingtalk
    command:
      - '--config.file=/etc/prometheus-webhook-dingtalk/config.yml'
    networks:
      - monitoring
```

**配置文件**: `monitoring/dingtalk/config.yml`
```yaml
targets:
  critical:
    url: https://oapi.dingtalk.com/robot/send?access_token=YOUR_TOKEN_HERE
    secret: YOUR_SECRET_HERE
    message:
      title: '【严重告警】'
      text: '{{ template "ding.link.content" . }}'
  
  warning:
    url: https://oapi.dingtalk.com/robot/send?access_token=YOUR_TOKEN_HERE
    message:
      title: '【警告】'

templates:
  - /etc/prometheus-webhook-dingtalk/template.tmpl
```

**消息模板**: `monitoring/dingtalk/template.tmpl`
```go
{{ define "ding.link.content" }}
{{ range .Alerts }}
---
**告警级别**: {{ .Labels.severity }}
**告警名称**: {{ .Labels.alertname }}
**实例**: {{ .Labels.instance }}
**告警摘要**: {{ .Annotations.summary }}
**告警详情**: {{ .Annotations.description }}
**触发时间**: {{ .StartsAt.Format "2006-01-02 15:04:05" }}
{{ end }}
{{ end }}
```

## 🎯 告警优化策略

### 1. 告警降噪

**问题**: 同一个问题产生多条告警

**解决方案**:
```yaml
# AlertManager 分组
route:
  group_by: ['alertname', 'instance', 'severity']
  group_wait: 30s
  group_interval: 5m
```

### 2. 告警抑制

**问题**: 上游故障导致下游大量告警

**解决方案**:
```yaml
inhibit_rules:
  - source_match:
      alertname: 'DatabaseDown'
    target_match:
      category: 'database'
    equal: ['instance']
```

### 3. 静默规则

**使用场景**: 计划维护、已知问题

**操作**:
1. 访问 AlertManager UI (http://localhost:9093)
2. 点击 **Silences** → **New Silence**
3. 设置匹配条件和持续时间
