## 为什么需要监控？
- **客流量（并发请求）** 高峰期有多少人？桌子够不够？（QPS、TPS）
- **顾客体验（响应时间）** 从点菜到上菜要等多久？（接口延迟）
- **异常情况（系统错误）** 有没有菜品做坏了？冰箱坏了没人知道？（异常告警）

**没有监控 = 闭着眼睛开餐厅**，等顾客投诉（用户反馈）才知道出问题，此时已经造成损失。

## 📊 监控指标全景

### 1️⃣ 系统资源层（基础设施）
**类比**：餐厅的水电气、空调、冰箱等基础设施

| 指标分类 | 监控内容 | 工具 | 为什么重要 |
|---------|---------|------|-----------|
| **CPU** | 使用率、负载、上下文切换 | Node Exporter | CPU 100% = 厨师累瘫了，系统卡死 |
| **内存** | 使用率、Swap、内存泄漏 | Node Exporter | 内存溢出 = 冰箱塞爆了，系统崩溃 |
| **磁盘** | I/O、使用率、IOPS | Node Exporter | 磁盘满了 = 仓库没空间，无法存储日志/数据 |
| **网络** | 带宽、丢包率、连接数 | Node Exporter | 网络拥塞 = 送菜的路堵了，请求超时 |

### 2️⃣ 应用服务层（业务核心）
**类比**：餐厅的厨师、服务员、点餐系统

| 指标分类 | 监控内容 | 工具 | 为什么重要 |
|---------|---------|------|-----------|
| **JVM** | 堆内存、GC 次数/耗时、线程数 | Spring Boot Actuator | GC频繁 = 厨师不停清理垃圾，影响做菜速度 |
| **接口** | QPS、响应时间、错误率 | Micrometer | 慢接口 = 某道菜做得特别慢，影响翻台率 |
| **线程池** | 活跃线程、队列长度、拒绝次数 | Micrometer | 线程池满了 = 厨师忙不过来，新订单被拒绝 |
| **异常** | 异常类型、频率、堆栈 | 日志采集 + Loki | 异常频发 = 厨房频繁出错，需要排查 |

### 3️⃣ 数据库层（数据核心）
**类比**：餐厅的食材仓库管理

| 指标分类 | 监控内容 | 工具 | 为什么重要 |
|---------|---------|------|-----------|
| **MySQL** | 慢查询、连接数、锁等待、缓存命中率 | MySQL Exporter | 慢查询 = 找食材找半天，拖慢整体速度 |
| **Redis** | 内存使用、命中率、Key数量、慢日志 | Redis Exporter | 命中率低 = 缓存形同虚设，数据库压力大 |
| **连接池** | 活跃连接、空闲连接、等待时间 | Druid/HikariCP | 连接池耗尽 = 数据库门被堵死，请求超时 |

### 4️⃣ 中间件层（辅助服务）
**类比**：餐厅的传菜系统、叫号器

| 指标分类 | 监控内容 | 工具 | 为什么重要 |
|---------|---------|------|-----------|
| **RabbitMQ** | 队列长度、消费速率、消息堆积 | RabbitMQ Exporter | 消息堆积 = 传菜跟不上，订单积压 |
| **Nginx** | 请求数、错误率、上游响应时间 | Nginx Exporter | 反向代理慢 = 前台点菜系统卡顿 |

## 🎯 监控目标与价值

### 核心价值三角
```
        ┌──────────┐
        │快速发现问题│
        └─────┬────┘
              │
      ┌───────┴───────┐
      │               │
┌─────▼─────┐   ┌────▼──────┐
│精准定位根因 │   │ 预防性运维 │
└───────────┘   └───────────┘
```

### 1. **快速发现问题**（从「用户投诉」到「系统自动预警」）
- ❌ **没有监控**：用户反馈"系统卡了"，你才知道出问题（滞后 30 分钟+）
- ✅ **有监控**：CPU 超 80% 立即告警，提前 5 分钟发现问题

### 2. **精准定位根因**（从「盲猜」到「数据说话」）
**场景**：用户反馈"提交订单很慢"
- ❌ **没有监控的排查过程**：
  ```
  开发：是不是数据库慢？ → 查日志 → 没发现
  DBA：是不是 Redis 挂了？ → 登录服务器查 → 正常
  运维：是不是网络问题？ → ping 一下 → 也正常
  最后发现：某个定时任务在跑，CPU 被占满了（浪费 2 小时）
  ```

- ✅ **有监控的排查过程**：
  ```
  1. 打开 Grafana 仪表盘
  2. 看到 CPU 100%，点击 CPU 图表查看进程
  3. 发现是定时任务进程，直接定位到代码
  4. 5 分钟解决问题
  ```

### 3. **预防性运维**（从「救火」到「防火」）
- **趋势分析**：内存每天增长 50MB，预测 20 天后溢出 → 提前优化
- **容量规划**：双11前看 QPS 峰值，提前扩容
- **性能优化**：发现某接口 P99 耗时 3 秒，针对性优化

## 🛠️ 工具选型逻辑

### 为什么选择 Prometheus + Grafana？（对比其他方案）

| 对比维度 | Prometheus + Grafana | ELK Stack | Zabbix | 云厂商监控 |
|---------|---------------------|-----------|--------|-----------|
| **学习成本** | ⭐⭐⭐ 中等（配置简单） | ⭐⭐ 高（需学ES） | ⭐⭐⭐⭐ 较低（UI友好） | ⭐⭐⭐⭐⭐ 最低（开箱即用） |
| **部署成本** | ⭐⭐⭐⭐ 低（Docker一键部署） | ⭐⭐ 高（ES集群复杂） | ⭐⭐⭐ 中等 | ⭐⭐⭐⭐⭐ 免费/低价 |
| **适用场景** | 指标监控（时序数据） | 日志分析为主 | 传统运维监控 | 小型项目 |
| **社区生态** | ⭐⭐⭐⭐⭐ 最活跃（CNCF项目） | ⭐⭐⭐⭐ 成熟 | ⭐⭐⭐ 传统 | ⭐⭐ 封闭 |
| **灵活性** | ⭐⭐⭐⭐⭐ 高度定制化 | ⭐⭐⭐⭐ 灵活 | ⭐⭐⭐ 中等 | ⭐⭐ 受限于厂商 |
| **成本** | 免费开源 | 免费开源 | 免费开源 | 按量收费（大型项目贵） |

**✅ 最终选择**：**Prometheus + Grafana**
- ✅ 开源免费，社区活跃
- ✅ 专为微服务/容器化设计（与你的 Docker 环境天然契合）
- ✅ 配置简单，5 分钟跑起来
- ✅ Spring Boot 原生支持（Micrometer）
- ✅ 丰富的 Exporter（MySQL、Redis、Node 都有现成方案）
- ✅ Grafana 仪表盘颜值高、交互强
