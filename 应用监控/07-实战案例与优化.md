## 📚 案例1: 接口响应慢排查

### 问题现象
```
用户反馈: "提交订单按钮点击后,转圈很久才成功"
时间: 2024-11-29 14:30
影响: 订单转化率下降20%
```

### 排查过程

#### 第1步: 确认问题(Grafana 仪表盘)
```promql
# 查看接口P99延迟
histogram_quantile(0.99, 
  rate(http_server_requests_seconds_bucket{uri="/api/order/create"}[5m])
)
# 结果: 5.2s (正常应该<1s)
```

**结论**: 确实存在性能问题

#### 第2步: 定位慢在哪里(分层排查)
```promql
# 1. 数据库是否慢?
rate(mysql_global_status_slow_queries[5m])
# 结果: 15 queries/s (异常高)

# 2. Redis是否慢?
redis_slowlog_length
# 结果: 0 (正常)

# 3. 外部API是否慢?
# (查看应用日志)
```

**初步定位**: 数据库慢查询

#### 第3步: 分析慢查询(MySQL Exporter + 日志)
```sql
-- 查看当前慢查询
SHOW PROCESSLIST;

-- 分析慢查询日志
SELECT * FROM mysql.slow_log ORDER BY query_time DESC LIMIT 10;
```

**发现问题**: 
```sql
SELECT * FROM orders WHERE user_id = ? ORDER BY create_time DESC;
-- 耗时: 4.8s
-- 扫描行数: 500万行
```

**根因**: `user_id` 字段未建索引

#### 第4步: 解决问题
```sql
-- 添加索引
ALTER TABLE orders ADD INDEX idx_user_id_create_time (user_id, create_time);

-- 验证效果
EXPLAIN SELECT * FROM orders WHERE user_id = ? ORDER BY create_time DESC;
-- 优化后: type=ref, rows=10 (原来是 type=ALL, rows=5000000)
```

#### 第5步: 验证恢复
```promql
# 接口P99延迟
histogram_quantile(0.99, 
  rate(http_server_requests_seconds_bucket{uri="/api/order/create"}[5m])
)
# 结果: 0.3s (正常)
```

### 经验总结

**监控指标链路**:
```
接口延迟增加 → 慢查询增多 → 找到具体SQL → 优化索引 → 验证恢复
```

**关键指标**:
- HTTP P99延迟
- 数据库慢查询数
- SQL执行时间
- 表扫描行数

**避免方法**:
- ✅ 开发阶段就添加索引
- ✅ 定期审查慢查询日志
- ✅ 压测验证性能

## 📚 案例2: 内存溢出(OOM)排查

### 问题现象
```
告警: "archat-backend JVM堆内存使用率 > 95%"
时间: 2024-11-29 02:30 (凌晨)
影响: 应用重启,部分请求失败
```

### 排查过程

#### 第1步: 查看内存趋势(Grafana)
```promql
# 堆内存使用率
jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} * 100
```

**发现**: 内存持续增长,每小时增长约10%

#### 第2步: GC分析
```promql
# GC次数趋势
rate(jvm_gc_pause_seconds_count[5m])
# 结果: Full GC频率从 0.1/s 增长到 5/s

# GC耗时占比
rate(jvm_gc_pause_seconds_sum[5m]) * 100
# 结果: 60% (严重影响性能)
```

**结论**: 频繁Full GC,疑似内存泄漏

#### 第3步: Heap Dump分析
```bash
# 1. 触发Heap Dump
docker exec archat-backend jmap -dump:format=b,file=/app/heapdump.hprof <PID>

# 2. 复制到本地
docker cp archat-backend:/app/heapdump.hprof ./

# 3. 使用 MAT 或 VisualVM 分析
```

**发现**:
- `ArrayList` 对象占用 800MB
- 持有者: `UserSessionCache`
- 大小: 50万个 `UserSession` 对象

#### 第4步: 代码审查
```java
// 问题代码
@Component
public class UserSessionCache {
    // ❌ 无限增长的缓存
    private Map<String, UserSession> cache = new HashMap<>();
    
    public void addSession(String token, UserSession session) {
        cache.put(token, session);  // 只添加,从不清理
    }
}
```

**根因**: 用户 Session 未设置过期时间,持续累积

#### 第5步: 修复代码
```java
// 修复后
@Component
public class UserSessionCache {
    // ✅ 使用带过期时间的缓存
    private Cache<String, UserSession> cache = CacheBuilder.newBuilder()
        .expireAfterWrite(30, TimeUnit.MINUTES)  // 30分钟过期
        .maximumSize(10000)  // 最多1万条
        .build();
    
    public void addSession(String token, UserSession session) {
        cache.put(token, session);
    }
}
```

#### 第6步: 验证修复
```promql
# 堆内存使用率
jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} * 100
# 结果: 稳定在 60% 左右,不再增长
```

### 经验总结

**内存泄漏排查步骤**:
```
1. 监控发现内存持续增长
2. GC分析确认泄漏
3. Heap Dump找到泄漏对象
4. 代码审查找到根因
5. 修复验证
```

**预防措施**:
- ✅ 所有缓存设置过期时间
- ✅ 使用 WeakHashMap 或 Guava Cache
- ✅ 定期 Code Review
- ✅ 压测验证长时间运行

## 📚 案例3: 数据库连接池耗尽

### 问题现象
```
告警: "hikaricp_connections_active / hikaricp_connections >= 0.9"
应用日志: "Connection pool exhausted"
影响: 50%接口超时
```

### 排查过程

#### 第1步: 查看连接池状态
```promql
# 活跃连接数
hikaricp_connections_active
# 结果: 49 / 50 (几乎满了)

# 等待连接的线程数
hikaricp_connections_pending
# 结果: 20 (大量线程等待)
```

#### 第2步: 分析连接去哪了
```promql
# 数据库当前连接数
mysql_global_status_threads_connected
# 结果: 48

# 查看长时间持有的连接
SHOW PROCESSLIST;
```

**发现**: 有30个连接处于 `Sleep` 状态,持续时间 > 10分钟

#### 第3步: 代码审查
```java
// 问题代码
@Service
public class UserService {
    
    @Autowired
    private UserMapper userMapper;
    
    public void syncUsers() {
        // ❌ 事务过大,长时间占用连接
        List<User> users = userMapper.selectAll(); // 100万条数据
        for (User user : users) {
            // 复杂业务处理,耗时30秒
            processUser(user);
        }
    }
}
```

**根因**: 
1. 事务过大,长时间占用连接
2. 连接池配置不合理(最大50个,实际需要更多)

#### 第4步: 优化方案

**方案1: 批量处理**
```java
@Service
public class UserService {
    
    public void syncUsers() {
        int pageSize = 1000;
        int pageNum = 0;
        
        while (true) {
            // ✅ 分批查询,减少单次事务大小
            List<User> users = userMapper.selectByPage(pageNum, pageSize);
            if (users.isEmpty()) break;
            
            for (User user : users) {
                processUser(user);
            }
            
            pageNum++;
        }
    }
}
```

**方案2: 异步处理**
```java
@Service
public class UserService {
    
    @Async
    public void syncUsers() {
        // ✅ 异步处理,不占用主线程连接
        // ...
    }
}
```

**方案3: 调整连接池配置**
```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 100  # 从50增加到100
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
      minimum-idle: 10
```

### 经验总结

**连接池优化原则**:
- 避免长事务
- 分批处理大数据量
- 合理配置连接池大小(公式: `核心数 * 2 + 磁盘数`)
- 监控连接池指标

## 📚 案例4: Redis缓存穿透

### 问题现象
```
告警: "redis_keyspace_misses增长异常"
数据库: CPU 100%, 慢查询暴增
影响: 接口响应时间从50ms增加到2s
```

### 排查过程

#### 第1步: 查看Redis命中率
```promql
# 缓存命中率
rate(redis_keyspace_hits_total[5m]) / 
(rate(redis_keyspace_hits_total[5m]) + rate(redis_keyspace_misses_total[5m])) * 100
# 结果: 15% (正常应该 > 80%)
```

#### 第2步: 分析未命中原因
```bash
# 查看 Redis 慢日志
redis-cli SLOWLOG GET 10

# 监控 Redis 命令
redis-cli MONITOR
```

**发现**: 大量查询不存在的Key
```
GET user:999999999  # 不存在
GET user:888888888  # 不存在
...
```

**根因**: 恶意攻击或业务逻辑错误,查询大量不存在的用户

#### 第3步: 防御方案

**方案1: 布隆过滤器**
```java
@Component
public class UserCacheService {
    
    // 布隆过滤器
    private BloomFilter<Long> userIdBloomFilter;
    
    @PostConstruct
    public void init() {
        // 初始化布隆过滤器(1000万用户,误判率0.01%)
        userIdBloomFilter = BloomFilter.create(
            Funnels.longFunnel(),
            10_000_000,
            0.0001
        );
        
        // 加载所有用户ID
        List<Long> userIds = userMapper.selectAllIds();
        userIds.forEach(userIdBloomFilter::put);
    }
    
    public User getUserById(Long userId) {
        // ✅ 先检查布隆过滤器
        if (!userIdBloomFilter.mightContain(userId)) {
            return null;  // 直接返回,不查数据库
        }
        
        // 查缓存
        User user = redisTemplate.opsForValue().get("user:" + userId);
        if (user != null) return user;
        
        // 查数据库
        user = userMapper.selectById(userId);
        if (user != null) {
            redisTemplate.opsForValue().set("user:" + userId, user, 1, TimeUnit.HOURS);
        }
        
        return user;
    }
}
```

**方案2: 缓存空值**
```java
public User getUserById(Long userId) {
    String cacheKey = "user:" + userId;
    
    // 查缓存
    User user = redisTemplate.opsForValue().get(cacheKey);
    if (user != null) {
        return user.getId() == null ? null : user;  // 区分空对象
    }
    
    // 查数据库
    user = userMapper.selectById(userId);
    
    if (user != null) {
        // ✅ 缓存正常对象(1小时)
        redisTemplate.opsForValue().set(cacheKey, user, 1, TimeUnit.HOURS);
    } else {
        // ✅ 缓存空对象(5分钟,防止穿透)
        User emptyUser = new User();
        redisTemplate.opsForValue().set(cacheKey, emptyUser, 5, TimeUnit.MINUTES);
    }
    
    return user;
}
```

**方案3: 限流**
```java
@Component
public class RateLimiterAspect {
    
    private LoadingCache<String, AtomicInteger> requestCountCache = CacheBuilder.newBuilder()
        .expireAfterWrite(1, TimeUnit.SECONDS)
        .build(new CacheLoader<String, AtomicInteger>() {
            @Override
            public AtomicInteger load(String key) {
                return new AtomicInteger(0);
            }
        });
    
    @Around("@annotation(com.senjay.common.annotation.RateLimit)")
    public Object rateLimit(ProceedingJoinPoint pjp) throws Throwable {
        String key = getClientIP();
        AtomicInteger count = requestCountCache.get(key);
        
        // ✅ 限制每IP每秒最多100次请求
        if (count.incrementAndGet() > 100) {
            throw new TooManyRequestsException("请求过于频繁");
        }
        
        return pjp.proceed();
    }
}
```

### 经验总结

**缓存穿透防御三板斧**:
1. 布隆过滤器(最彻底)
2. 缓存空值(简单有效)
3. 限流(兜底方案)

## 📚 案例5: 定时任务导致CPU飙升

### 问题现象
```
时间: 每天凌晨2点
告警: "CPU使用率 > 90%"
影响: 接口响应变慢,部分超时
```

### 排查过程

#### 第1步: 确认时间规律
```promql
# CPU使用率(最近24小时)
100 - (avg(irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)
```

**发现**: 每天凌晨2点准时飙升

#### 第2步: 查看定时任务
```bash
# 查看 Spring @Scheduled 任务
docker logs archat-backend | grep "Scheduled"

# 结果:
# 2024-11-29 02:00:00 - Starting scheduled task: syncAllUsers
```

#### 第3步: 分析任务代码
```java
@Component
public class UserSyncTask {
    
    // ❌ 同步执行,阻塞主线程
    @Scheduled(cron = "0 0 2 * * ?")
    public void syncAllUsers() {
        List<User> users = userMapper.selectAll();  // 100万用户
        
        for (User user : users) {
            // CPU密集型计算
            calculateUserScore(user);
            userMapper.update(user);
        }
    }
}
```

**根因**: 
1. 一次性加载100万数据
2. 同步执行,占用所有CPU
3. 频繁更新数据库

#### 第4步: 优化方案

**方案1: 异步+分批处理**
```java
@Component
public class UserSyncTask {
    
    @Async("taskExecutor")  // ✅ 异步执行
    @Scheduled(cron = "0 0 2 * * ?")
    public void syncAllUsers() {
        int pageSize = 1000;
        int pageNum = 0;
        
        while (true) {
            // ✅ 分批查询
            List<User> users = userMapper.selectByPage(pageNum, pageSize);
            if (users.isEmpty()) break;
            
            // ✅ 并行处理
            users.parallelStream().forEach(user -> {
                calculateUserScore(user);
            });
            
            // ✅ 批量更新
            userMapper.batchUpdate(users);
            
            pageNum++;
            
            // ✅ 休眠避免CPU持续满载
            Thread.sleep(100);
        }
    }
}

// 配置线程池
@Configuration
public class AsyncConfig {
    @Bean
    public ThreadPoolTaskExecutor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("sync-task-");
        executor.initialize();
        return executor;
    }
}
```

**方案2: 调整执行时间**
```java
// 改为凌晨4点执行(业务低峰期)
@Scheduled(cron = "0 0 4 * * ?")
```

**方案3: 增加监控**
```java
@Timed(value = "scheduled.task.sync.users")
public void syncAllUsers() {
    // ...
}
```

### 经验总结

**定时任务优化原则**:
- 异步执行,不阻塞主线程
- 分批处理,降低峰值负载
- 选择业务低峰期执行
- 添加监控,及时发现问题
