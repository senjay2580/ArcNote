### 一、核心可用能力：深度拆解各端权限/原生能力的差异与落地要点
“核心可用能力”本质是**不同终端操作系统/运行环境开放给应用的“系统级接口”** ——Web受限于浏览器沙箱，桌面端直达操作系统内核，移动端则受限于iOS/Android的应用沙箱+应用商店规则，三者的能力边界和使用方式差异极大，下面逐一展开：

#### 1. Web端：浏览器沙箱下的“有限能力”
Web应用运行在浏览器的**沙箱环境**中，浏览器为了安全，严格限制其访问本地系统资源，所有能力都依赖浏览器标准API，且需用户主动授权。
| 核心能力     | 具体说明                                                     | 落地注意点                                                   |
| ------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 本地存储能力 | Cookie（4KB）、LocalStorage/SessionStorage（5-10MB）、IndexedDB（无上限但慢） | ① 跨域时Cookie需配置`SameSite`/`Domain`；② 避免用LocalStorage存敏感数据（明文）；③ 大文件用IndexedDB + 分片存储 |
| 设备硬件调用 | 摄像头/麦克风（MediaDevices API）、定位（Geolocation API）、通知（Notification API） | ① 仅HTTPS环境可用；② 首次调用需弹窗请求用户授权，且授权状态会被浏览器持久化；③ 定位API在移动端Web精度低于原生App |
| 网络能力     | XMLHttpRequest/Fetch、WebSocket、Service Worker（离线缓存）  | ① 跨域请求需后端配置CORS；② WebSocket需处理断连重连；③ Service Worker仅支持HTTPS，且需兼容低版本浏览器（如Safari 11+） |
| 其他有限能力 | 剪贴板（Clipboard API）、文件上传（input[type=file]）、打印（window.print()） | ① 剪贴板读写需用户交互（如点击按钮），禁止静默操作；② 文件上传仅能读取用户选择的文件，无法主动遍历本地文件夹；③ 打印样式需单独适配（@media print） |

**典型问题&解决方案**：
- 问题：Web端想读取本地Excel文件并解析，但无法主动获取文件路径；
- 方案：通过`<input type="file" accept=".xlsx">`让用户选择文件，再用JS库（如xlsx.js）在浏览器端解析，避免服务端中转。

#### 2. 桌面端：操作系统级的“全量能力”
桌面端应用（如Electron/Tauri/原生C++）运行在操作系统内核层，几乎能调用所有系统级API，权限边界仅受操作系统的用户权限（如管理员/普通用户）限制。
| 核心能力          | 具体说明（Windows/macOS/Linux差异）                          | 落地注意点                                                   |
| ----------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 文件/目录操作     | 读写任意路径文件（如C盘、桌面）、遍历文件夹、修改文件权限    | ① Windows需处理UAC（用户账户控制）弹窗（管理员权限）；② macOS需适配“文件访问权限”（如访问桌面/下载文件夹需用户授权）；③ Linux需注意文件权限（chmod/chown） |
| 系统进程/服务管理 | 启动/杀死进程、注册系统服务、调用系统命令（如cmd/terminal）  | ① Windows需处理管理员权限才能操作系统服务；② macOS/Linux调用终端命令需注意路径（如/usr/bin）；③ 避免静默启动后台进程（易被杀毒软件拦截） |
| 系统级UI/交互     | 系统托盘（通知栏图标）、全局快捷键、窗口置顶/最小化/全屏、注册表（Windows）/偏好设置（macOS） | ① 全局快捷键需避免与系统快捷键冲突（如macOS的Cmd+C）；② 系统托盘图标需适配不同系统的样式规范（macOS无托盘文字）；③ Windows注册表修改需谨慎（误改可能导致系统异常） |
| 硬件/外设调用     | 访问USB设备、打印机、串口、显卡/声卡等硬件                   | ① USB设备需适配不同系统的驱动（如Windows的libusb、macOS的IOUSBKit）；② 打印机调用需兼容各品牌驱动（优先用系统打印API） |
| 应用签名/权限     | macOS需苹果开发者签名（否则提示“未受信任”）、Windows需代码签名（避免杀毒软件拦截） | ① macOS签名需付费开发者账号（99美元/年）；② Windows签名可购买第三方证书（如Symantec）；③ Linux无需强制签名，但需适配不同发行版（Ubuntu/CentOS） |

**典型问题&解决方案**：
- 问题：Electron应用打包后体积过大（默认100MB+），且启动后内存占用高；
- 方案：① 用Tauri替代Electron（基于Rust，打包后体积10MB以内）；② 精简依赖（剔除node_modules中无用包）；③ 禁用Electron的多渲染进程，改用单进程模式（仅适合轻量应用）。

#### 3. 移动端：设备级的“受控能力”
移动端应用（iOS/Android）运行在系统的应用沙箱中，能力边界由系统规则+应用商店审核决定，核心能力需申请权限且遵循严格的调用规范。
| 核心能力          | iOS vs Android 差异                                          | 落地注意点                                                   |
| ----------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 设备硬件调用      | 摄像头/麦克风/定位/蓝牙/NFC：<br>① iOS：所有权限需弹窗授权，且授权后可在设置中关闭；<br>② Android：13+版本权限分级（如照片权限可仅授权单张） | ① 权限申请需“场景化”（如调用摄像头前先提示“用于扫码登录”，而非仅“请求摄像头权限”）；② iOS拒绝权限后需引导用户到设置页开启；③ 定位权限区分“仅在使用期间”和“始终”（避免申请“始终”权限导致审核失败） |
| 推送能力          | ① iOS：仅支持APNs推送，需苹果开发者账号，且推送内容需审核；<br>② Android：支持FCM/厂商推送（小米/华为/OPPO） | ① 国内Android需集成多厂商推送SDK（否则推送到达率低）；② iOS推送需配置推送证书，且静默推送有频率限制；③ 推送内容需符合应用商店规则（如禁止推送广告） |
| 本地存储/文件操作 | ① iOS：仅能访问应用沙箱内文件，访问照片需授权；<br>② Android：10+需用Scoped Storage（仅能访问指定目录） | ① 避免存储大文件在应用沙箱（导致应用占用空间过大被用户卸载）；② Android适配Scoped Storage，避免直接访问SD卡根目录；③ 关键数据用加密存储（如iOS的Keychain、Android的Keystore） |
| 后台运行能力      | ① iOS：后台运行严格受限（仅音频/定位/VOIP等场景可申请）；<br>② Android：后台运行需申请“后台权限”，且部分厂商（如华为）会限制后台进程 | ① iOS避免依赖后台运行（如实时定位需用“后台定位模式”，且需说明用途）；② Android适配厂商白名单（如华为需引导用户加入“受保护应用”） |
| 支付/生物识别     | ① iOS：仅支持Apple Pay/面容ID/Touch ID；<br>② Android：支持微信/支付宝支付、指纹/人脸（各厂商差异） | ① 生物识别需接入系统API（而非第三方库）；② 支付功能需适配应用商店审核（如iOS内购需用IAP，禁止跳微信支付） |

**典型问题&解决方案**：
- 问题：移动端应用请求定位权限后，用户拒绝，导致功能无法使用；
- 方案：① 首次拒绝后，在功能页再次引导授权（如“未获取定位权限，无法显示附近的门店，是否前往设置开启？”）；② 提供“手动输入地址”的兜底方案，避免功能完全不可用。

### 二、性能关注点：各端性能基线与优化落地细节
“性能关注点”的核心是**不同终端的“用户感知性能”和“系统资源瓶颈”不同** ——Web用户最在意“打开快不快”，桌面端用户在意“卡不卡、占不占内存”，移动端用户在意“费不费流量、掉电快不快”，下面拆解各端的性能基线、核心瓶颈和可落地的优化方案：

#### 1. Web端：核心是“加载速度”和“兼容性”
Web端的性能瓶颈集中在**网络传输**和**浏览器渲染**，用户对“首屏加载时间”的感知极其敏感（行业基准：首屏≤3秒，否则流失率翻倍）。
| 性能关注点             | 基线标准（行业通用）                                         | 具体优化方案（可落地）                                       |
| ---------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 首屏加载速度           | 首屏TTI（可交互时间）≤3秒，LCP（最大内容绘制）≤2.5秒         | ① 资源压缩：JS/CSS压缩（Terser/CleanCSS）、图片压缩（WebP/AVIF格式，按需加载）；<br>② 资源拆分：路由懒加载（React/Vue的lazy+Suspense）、首屏仅加载核心JS/CSS；<br>③ 缓存策略：静态资源加CDN+强缓存（Cache-Control: max-age=31536000）、接口数据加协商缓存（ETag/Last-Modified）；<br>④ 预加载：关键资源用`<link rel="preload">`，预解析DNS（`<link rel="dns-prefetch">`） |
| 网络请求效率           | 接口响应时间≤500ms，并发请求数≤6（浏览器默认限制）           | ① 接口聚合：合并多个小请求（如同时获取用户信息+订单列表）；<br>② 异步请求：非核心接口（如统计、推荐）延迟加载；<br>③ 降级策略：网络差时返回缓存数据，避免白屏；<br>④ HTTP/2：启用多路复用，减少连接建立耗时 |
| 浏览器兼容性           | 覆盖Chrome（90+）、Firefox（88+）、Safari（14+）、Edge（90+） | ① 语法兼容：用Babel转译ES6+，用PostCSS适配CSS（如autoprefixer）；<br>② API兼容：用core-js/polyfill.io补全缺失的API（如Promise、fetch）；<br>③ 渐进式降级：不支持的功能（如WebGL）提供兜底方案（如静态图片） |
| 运行时性能（交互流畅） | FPS≥60，长列表滚动无卡顿                                     | ① 减少重排重绘：用transform/opacity做动画（避免触发layout）、批量修改DOM；<br>② 长列表优化：虚拟列表（如vue-virtual-scroller、react-window）；<br>③ 内存优化：及时清理定时器/事件监听，避免内存泄漏 |

**实战案例**：
- 问题：Web端首屏加载慢（原加载时间5秒）；
- 优化步骤：① 图片转WebP格式，压缩体积50%；② 路由懒加载，首屏JS体积从800KB降到300KB；③ 启用CDN+强缓存，静态资源加载时间从2秒降到500ms；④ 最终首屏加载时间降至1.8秒。

#### 2. 桌面端：核心是“启动速度”和“资源占用”
桌面端应用（尤其是Electron这类“网页套壳”框架）的性能瓶颈集中在**启动耗时**和**内存/CPU占用**，用户对“启动慢、卡顿、内存泄漏”的感知极强。
| 性能关注点               | 基线标准（行业通用）                                | 具体优化方案（可落地）                                       |
| ------------------------ | --------------------------------------------------- | ------------------------------------------------------------ |
| 启动速度                 | 冷启动≤3秒，热启动≤1秒（Electron应用基准）          | ① 精简启动项：仅加载核心模块，非核心模块（如设置、帮助）延迟加载；<br>② 打包优化：Electron用asar打包（减少文件IO）、剔除无用依赖（如删除node_modules中的test包）；<br>③ 预编译：将JS代码预编译为字节码（如Electron的v8-compile-cache）；<br>④ 替代框架：用Tauri替代Electron（冷启动时间从3秒降到500ms） |
| 内存占用                 | 轻量应用≤200MB，中量应用≤500MB（Electron应用基准）  | ① 进程优化：Electron禁用多渲染进程（用single-instance），主进程仅处理核心逻辑；<br>② 内存回收：主动清理渲染进程的DOM缓存、定时器，主进程及时释放大对象；<br>③ 资源卸载：关闭窗口时释放对应的资源（如图片、视频流）；<br>④ 监控：用Electron的`process.memoryUsage()`监控内存，超过阈值时提示用户重启 |
| 本地资源消耗（CPU/磁盘） | CPU占用≤10%（空闲时），磁盘IO≤1MB/s（非文件操作时） | ① 避免频繁磁盘读写：批量处理文件操作（如每10秒保存一次，而非实时保存）；<br>② 缓存本地数据：常用数据存内存，避免频繁读取本地文件；<br>③ 异步操作：文件解析、进程调用等耗时操作放在异步线程，避免阻塞主线程；<br>④ 垃圾回收：手动触发V8垃圾回收（Electron的`gc()`），减少内存碎片 |
| 跨系统性能适配           | Windows/macOS/Linux下性能差异≤20%                   | ① 针对macOS优化：适配Apple Silicon（ARM架构），避免Rosetta转译耗时；<br>② 针对Windows优化：禁用系统缩放的DPI适配（如Electron的`app.commandLine.appendSwitch('high-dpi-support', '1')`）；<br>③ 针对Linux优化：适配不同桌面环境（GNOME/KDE）的渲染差异 |

**实战案例**：
- 问题：Electron应用启动慢（冷启动4秒），内存占用300MB；
- 优化步骤：① 剔除无用依赖（如axios替换为原生fetch，减少100KB体积）；② 启用asar打包，文件IO耗时减少50%；③ 路由懒加载，启动时仅加载登录模块；④ 最终冷启动时间降至1.5秒，内存占用降至150MB。

#### 3. 移动端：核心是“流量/续航”和“低配置适配”
移动端的性能瓶颈集中在**移动网络（流量/延迟）**、**电池续航**和**低配置机型兼容**，用户对“耗流量、掉电快、低配机卡顿”的容忍度极低。
| 性能关注点     | 基线标准（行业通用）                                  | 具体优化方案（可落地）                                       |
| -------------- | ----------------------------------------------------- | ------------------------------------------------------------ |
| 流量消耗       | 单次启动流量≤5MB（不含大文件下载），图片/视频占比≤80% | ① 图片优化：根据网络切换分辨率（WiFi加载高清，4G加载标清）、用WebP/HEIF格式（压缩比提升50%）；<br>② 数据压缩：接口返回JSON压缩（gzip）、精简返回字段（如仅返回需要的字段，而非全量数据）；<br>③ 预加载策略：WiFi下预加载核心内容，4G下仅加载当前页面；<br>④ 缓存策略：用PWA/本地数据库（如Realm）缓存接口数据，重复请求直接读本地 |
| 电池续航       | 后台耗电≤1%/小时（空闲时），CPU唤醒频率≤1次/分钟      | ① 减少后台请求：后台仅保留核心推送（如聊天消息），禁止定时轮询接口；<br>② 优化定位频率：仅在使用时开启定位，且降低定位精度（如Android的`ACCESS_COARSE_LOCATION`）；<br>③ 减少唤醒：避免频繁唤醒CPU（如用AlarmManager替代Timer）；<br>④ 硬件调用优化：使用摄像头/蓝牙后及时释放资源，避免持续耗电 |
| 离线可用能力   | 核心功能（如查看历史数据）在无网络时可用              | ① PWA适配：配置Service Worker缓存核心静态资源和接口数据；<br>② 本地存储：关键数据存SQLite/SharedPreferences（Android）/UserDefaults（iOS）；<br>③ 同步策略：联网后自动同步本地修改的数据，冲突时提示用户解决 |
| 低配置机型适配 | 在骁龙660/iOS 12+机型上流畅运行（FPS≥30）             | ① 渲染优化：减少过度绘制（Overdraw）、禁用复杂动画（如渐变/阴影）；<br>② 内存优化：图片按需加载，超过屏幕尺寸的图片压缩后显示；<br>③ 代码优化：避免嵌套循环、减少反射调用（Android）；<br>④ 降级策略：低配机自动关闭非核心功能（如高清渲染、实时推荐） |

**实战案例**：
- 问题：移动端应用在4G网络下加载慢，且耗电快；
- 优化步骤：① 图片按网络切换分辨率（4G下加载720p，WiFi下加载1080p），流量消耗减少60%；② 接口数据gzip压缩，响应体积减少70%；③ 定位改为“仅在使用期间”开启，且降低精度，后台耗电减少80%；④ 最终4G下加载时间从3秒降到1秒，续航提升2小时。

### 一、核心跨端业务场景 & 关键注意事项

#### 1. 登录态管理：多端登录 / 踢下线 / 会话同步（高频且易出问题）

这是多端业务的基础，核心是 “统一的会话标识 + 端侧状态感知 + 服务端强制管控”，常见问题：踢下线仅生效某一端、多端登录状态不同步、移动端退出后桌面端仍在线。

| 业务场景                  | 注意事项                                                     | 最佳实践                                                     |
| ------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 多端同时登录              | ① 需明确支持的登录端类型（如 Web + 桌面 + 移动端同时在线，或限制仅 2 端）；② 避免 “一端登录，其他端自动下线” 的默认逻辑（需可配置） | ① 服务端维护 “用户 - 登录会话” 映射表（记录会话 ID、端类型、登录时间、设备信息）；② 前端展示 “当前登录设备列表”（如微信的 “登录设备管理”）；③ 支持 “允许多端登录” 和 “仅当前端登录” 两种模式 |
| 多端踢下线（主动 / 被动） | ① 被动踢下线（如改密码、账号冻结）需全端生效；② 主动踢下线（如手动踢掉某设备）需精准触发；③ 避免 “踢下线后，端侧仍能操作数据” | ① 服务端：踢下线时标记会话 ID 为 “失效”，所有接口校验会话有效性；② 端侧：- 实时感知：Web / 桌面端通过 WebSocket 推送 “踢下线通知”，移动端通过推送（APNs/FCM）触发状态校验；- 被动校验：每次接口请求携带会话 ID，服务端返回 “会话失效” 后，端侧立即跳转登录页；- 离线处理：移动端本地缓存 “会话失效标记”，下次启动直接校验 |
| 登录态过期 / 续期         | ① 不同端的会话过期策略需统一（如均为 7 天）；② 避免 “桌面端永不过期，移动端频繁过期” | ① 服务端统一设置会话过期时间，返回 “过期倒计时” 给端侧；② 端侧自动续期：活跃端（如桌面端操作中）触发续期接口，闲置端到期自动登出；③ 移动端后台运行时，通过 “静默请求” 续期（需适配系统后台限制） |

**实战示例：多端踢下线逻辑**

```plaintext
# 服务端表设计（user_sessions）
id: BIGSERIAL PRIMARY KEY,
user_id: UUID NOT NULL,
session_id: VARCHAR(64) UNIQUE NOT NULL, -- 唯一会话标识
device_type: VARCHAR(20) NOT NULL, -- web/desktop/ios/android
device_info: VARCHAR(255), -- 设备型号/浏览器版本
login_time: TIMESTAMPTZ NOT NULL,
expire_time: TIMESTAMPTZ NOT NULL,
is_valid: BOOLEAN DEFAULT TRUE -- 会话是否有效（踢下线时设为false）

# 核心逻辑
1. 端侧登录：生成session_id，写入user_sessions，返回给端侧缓存；
2. 踢下线操作：
   - 主动踢：前端选择某设备，服务端将对应session_id的is_valid设为false；
   - 被动踢：改密码/冻结账号，服务端将该用户所有session_id的is_valid设为false；
3. 端侧感知：
   - Web/桌面端：WebSocket监听“session_invalid”事件，收到后清空缓存跳登录；
   - 移动端：推送通知触发“校验会话”接口，失效则跳登录；
   - 所有接口请求：服务端先校验session_id的is_valid，失效则返回401，端侧统一拦截401跳转。
```

#### 2. 数据同步：多端操作后的一致性（核心业务体验）

多端系统最易出现 “数据不同步”（如 Web 端修改笔记，移动端仍显示旧内容），核心是 “统一的数据源 + 增量同步 + 冲突处理”。

| 业务场景                         | 注意事项                                                     | 最佳实践                                                     |
| -------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 实时数据同步（如聊天、协作编辑） | ① 避免 “多端同时修改同一数据导致覆盖”；② 低网速下同步失败的兜底；③ 移动端离线编辑后的数据同步 | ① 服务端用 “乐观锁 / 版本号” 控制（如每条数据加 version 字段，更新时校验版本）；② 实时同步：基于 WebSocket / 长轮询推送增量数据（如仅推送修改的字段，而非全量）；③ 离线同步：移动端本地缓存修改记录，联网后按 “时间戳 + 版本号” 合并，冲突时提示用户选择 “保留本地 / 覆盖 / 合并” |
| 非实时数据同步（如笔记、设置）   | ① 避免 “一端删除数据，其他端仍显示”；② 桌面端本地缓存与服务端的一致性 | ① 端侧启动时触发 “全量校验”（对比本地数据的最后更新时间与服务端）；② 数据操作后，服务端返回 “最新数据”，端侧立即更新本地；③ 桌面端本地缓存：仅缓存 “常用数据”，且标记 “缓存时间”，超过 1 小时自动校验 |
| 跨端数据适配                     | ① 移动端屏幕小，避免同步 “大体积数据”（如高清图片）；② 桌面端需同步 “本地文件” 到云端 | ① 服务端按端类型返回适配数据（如移动端返回压缩后的图片 URL，桌面端返回原图）；② 大文件同步：分块上传 / 下载，断点续传，避免单次同步失败；③ 敏感数据（如密码）：仅服务端存储，端侧不缓存，多端均通过接口获取（且加密传输） |

#### 3. 权限控制：多端权限的一致性与差异化

多端的权限逻辑需 “核心权限统一，端侧权限适配”，避免 “Web 端有权限操作，移动端无权限” 或 “移动端越权操作”。

| 业务场景       | 注意事项                                                     | 最佳实践                                                     |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 核心权限统一   | ① 避免 “角色权限在不同端解析不一致”（如管理员在 Web 端能删数据，移动端也需能删）；② 权限变更后多端实时生效 | ① 服务端统一校验权限（端侧仅做 “权限展示控制”，不做核心校验）；② 权限变更（如升级角色）：服务端推送 “权限更新” 事件，端侧刷新权限缓存；③ 端侧权限组件：封装统一的权限判断函数（如 hasPermission ('delete_note')），多端复用 |
| 端侧权限差异化 | ① 桌面端有 “本地文件操作权限”，Web / 移动端无；② 移动端有 “生物识别权限”，桌面端无 | ① 权限分级：- 核心权限（如删除数据）：服务端控制，所有端一致；- 端侧专属权限（如文件读写）：仅在对应端暴露功能，且做本地权限校验；② 功能隐藏：无权限的端侧直接隐藏功能入口（而非禁用），提升体验 |

#### 4. 业务流程：多端的体验一致性与适配性

同一业务流程在不同端需 “逻辑一致，操作适配”，避免 “流程断裂”（如 Web 端下单后，移动端无法查看订单进度）。

| 业务场景        | 注意事项                                                     | 最佳实践                                                     |
| --------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 交易 / 支付流程 | ① 移动端支付需适配 “生物识别 / 快捷支付”，桌面端适配 “扫码支付 / 网银”；② 避免 “多端重复下单” | ① 服务端生成 “唯一订单号”，下单前校验 “未支付订单”，避免重复创建；② 支付流程：- 桌面端：生成支付二维码，移动端扫码完成；- 移动端：直接调用原生支付 SDK；③ 支付状态同步：服务端推送 “支付成功” 事件，所有端实时更新订单状态 |
| 消息通知        | ① 多端通知的一致性（如 Web 端未读消息，移动端也需标记未读）；② 避免 “重复通知” | ① 服务端维护 “用户 - 未读消息” 表，标记 “已读 / 未读”，多端同步未读状态；② 通知触达策略：- 活跃端（如桌面端正在使用）：仅桌面端弹窗通知；- 非活跃端：移动端推送、桌面端系统托盘通知；③ 已读同步：一端标记已读，服务端更新状态，其他端同步清除未读标记 |
| 操作记录 / 日志 | ① 多端操作需统一记录（如 “移动端修改密码”“桌面端删除文件”）；② 日志需区分端类型 | ① 服务端日志字段：user_id + 操作内容 + 端类型 + 设备信息 + 时间；② 前端展示操作日志时，标注 “操作端”（如 “2025-12-15 桌面端（Windows）修改了昵称”）；③ 敏感操作（如改密码、踢下线）：强制记录并支持用户查看 |

#### 5. 异常处理：多端的兜底逻辑统一

多端系统的异常场景（如网络中断、接口报错、权限不足）需 “统一的兜底策略，适配的展示方式”。

| 异常类型 | 注意事项                                                     | 最佳实践                                                     |
| -------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 网络异常 | ① 移动端弱网 / 离线，桌面端断网；② 避免 “网络恢复后数据不同步” | ① 统一的网络状态监听：端侧封装网络检测工具，断网时显示统一的 “无网络” 提示；② 离线操作：- Web 端：仅支持查看缓存数据，禁用修改；- 桌面端：支持本地编辑，缓存操作记录，联网后自动同步；- 移动端：核心功能（如查看笔记）离线可用，修改操作缓存；③ 重试机制：接口请求失败后，按 “指数退避” 重试（如 1s→2s→4s），移动端后台重试需适配系统限制 |
| 接口报错 | ① 避免 “同一报错在不同端展示不同提示”；② 敏感报错（如会话失效）需统一处理 | ① 服务端返回统一的错误码 + 错误信息（如 401 = 会话失效，403 = 权限不足）；② 端侧封装统一的错误拦截器：- 401：跳转登录页；- 403：提示 “无权限操作”；- 500：提示 “服务器繁忙，请稍后重试”；③ 错误日志：端侧上报错误时，携带 “端类型 + 设备信息”，方便服务端排查 |

### 二、多端业务逻辑的通用注意事项

#### 1. 避免 “端侧硬编码业务逻辑”

- 核心逻辑（如权限判断、数据校验、订单状态流转）必须放在服务端，端侧仅做 “展示 + 调用接口”；
- 反例：移动端本地判断 “是否有权限删除数据”—— 攻击者可篡改本地代码绕过权限；
- 正例：端侧点击 “删除” 后，调用服务端删除接口，服务端校验权限后返回结果，端侧仅展示结果。

#### 2. 统一的业务规则引擎

- 多端的核心业务规则（如积分计算、会员等级、限流规则）需由服务端统一提供，避免 “Web 端积分 + 10，移动端 + 5”；
- 示例：会员等级升级规则（如消费满 1000 升级），服务端封装为接口，所有端调用该接口获取升级状态，而非本地计算。

#### 3. 端侧状态与服务端状态的 “最终一致性”

- 端侧可缓存 “非敏感状态”（如未读消息数、笔记列表），但必须提供 “手动刷新” 入口，且定期自动校验；
- 禁止端侧依赖本地缓存做 “核心决策”（如 “本地缓存显示会员未过期，就允许使用会员功能”），需每次调用接口校验。

#### 4. 多端的埋点与数据分析统一

- 埋点事件名称、参数需统一（如 “点击登录按钮”：event=click_login，参数 = device_type=web/desktop/ios）；
- 避免 “Web 端埋点叫 login_click，移动端叫 click_login_btn”，导致数据分析无法聚合。

### 三、典型问题的避坑指南

| 常见问题                                    | 原因分析                                                   | 解决方案                                                     |
| ------------------------------------------- | ---------------------------------------------------------- | ------------------------------------------------------------ |
| 踢下线后，移动端仍能操作数据                | 移动端未实时监听会话状态，仅启动时校验，且后台运行时未校验 | ① 移动端每次接口请求都校验会话；② 接收推送 “踢下线” 通知后，立即清空缓存；③ 后台运行时，每 5 分钟静默校验一次会话 |
| 多端同时修改同一笔记，内容被覆盖            | 无版本控制，后提交的覆盖先提交的                           | 服务端给每条笔记加 version 字段，更新时 SQL：`UPDATE notes SET content=?, version=version+1 WHERE id=? AND version=?`，版本不匹配则返回冲突 |
| 桌面端登录后，Web 端显示 “未登录”           | 会话 ID 未统一，桌面端和 Web 端用不同的 session_key        | 所有端登录后获取统一的 session_id，且缓存到各自的存储（Web：LocalStorage，桌面端：本地文件，移动端：SharedPreferences） |
| 移动端支付成功后，桌面端订单仍显示 “未支付” | 支付状态未实时同步，仅页面刷新时更新                       | 服务端支付成功后，通过 WebSocket 推送 “订单状态更新” 事件，所有端实时更新订单状态 |

### 四、总结：多端业务逻辑的核心原则

1. **服务端中心化**：核心业务逻辑、权限、状态由服务端统一管控，端侧仅做 “展示层 + 操作层”；
2. **状态可校验**：多端的登录态、数据状态、权限状态均可被服务端校验，且端侧能实时感知变更；
3. **体验适配化**：同一业务逻辑在不同端的操作流程适配端侧特性（如移动端支付用指纹，桌面端用扫码），但核心规则一致；
4. **异常可兜底**：断网、接口报错、同步冲突等场景，有统一的兜底策略，避免用户操作失败且无反馈。

最终，多端业务逻辑设计的目标是：**用户在不同端操作时，感知不到 “端的差异”，只感受到 “业务的一致性和体验的适配性”**，同时从技术层面杜绝安全漏洞和数据不一致问题。
