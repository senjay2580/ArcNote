# ==通信协议选型==

## Web通信技术选型

| 技术方案                    | 连接状态     | 通信方向          | 驱动方式    | 是否支持实时  | 是否兼容HTTP             |
| ----------------------- | -------- | ------------- | ------- | ------- | -------------------- |
| HTTP（普通请求）              | 短连接      | 客户端 → 服务端     | 客户端拉取   | ❌ 否     | ✅ 是                  |
| 轮询（Polling）             | 多次短连接    | 客户端 → 服务端     | 客户端定时拉取 | ⚙️ 准实时  | ✅ 是                  |
| 长轮询（Long Polling）       | 伪长连接     | 客户端 ↔ 服务端     | 服务端被动推送 | ✅ 实时性较好 | ✅ 是                  |
| SSE（Server-Sent Events） | 单连接（长连接） | 服务端 → 客户端     | 服务端主动推送 | ✅ 高实时性  | ✅ 是                  |
| WebSocket               | 真正长连接    | 客户端 ↔ 服务端（双向） | 双向通信    | ✅ 高实时性  | ⚠️ 初始走HTTP握手，之后非HTTP |

---



### HTTP（传统短连接）

- **方向：** 单向（Client → Server）
- **机制：** 客户端发一次请求，服务端响应一次，结束后连接关闭。
- **优点：**
  - 简单、通用，浏览器原生支持。
  - RESTful API 设计成熟。
- **缺点：**
  - 无法实时通信。
  - 客户端必须主动发起。
- **使用场景：**
  - 普通页面加载。
  - 一次性操作（查询、下单、提交表单）。
  - 静态资源获取

---



### 短轮询（Short Polling）

- **方向：** 单向（Client → Server）
- **机制：** 客户端定时请求服务端，如每 5 秒查询一次最新状态。
- **优点：**
  - 实现简单。
  - 完全兼容 HTTP，适合旧系统改造。
- **缺点：**
  - 延迟由轮询间隔决定。
  - **请求频繁浪费带宽与资源**。
  - 服务端压力大。
- **使用场景：**
  - 状态变化不频繁的监控页面。
  - 对实时性要求一般的应用（如进度条、定期刷新新闻）。





---



### 长轮询（Long Polling）

- **方向：** “半双工”（Server → Client 的假推送）
- **机制：**
  - 客户端发出请求，若无新数据，服务端挂起请求。
  - 一旦有更新立即返回，客户端再重新发起下一次请求。
- **优点：**
  - 兼容 HTTP/1.1。
  - 实时性强，服务端压力比轮询小。
  - 浏览器原生支持。
- **缺点：**
  - 每次返回都需要重新建立请求头。
  - 同时维持大量挂起连接对服务端线程有负担。
- **使用场景：**
  - 实时性中等、兼容性要求高的系统：
    - 即时消息、通知提醒。
    - 在线客服。
    - 在线考试、投票结果刷新



---



### SSE（Server-Sent Events）

- **方向：** 单向（Server → Client）
- **机制：**
  - 使用 HTTP 长连接（单向推送）。
  - 客户端通过 `EventSource` 订阅，服务器持续推送事件流。
- **优点：**
  - 原生支持自动重连、事件分发（message、error、open）。
  - 只占用一个 TCP 连接，轻量。
  - 兼容 HTTP/2，可利用服务器推送机制。
- **缺点：**
  - **仅支持服务端→客户端方向。**
  - IE 不支持，移动端兼容性需测试。
  - 只可发送文本载荷，二进制流不支持！
- **使用场景：**
  - 服务端状态流式更新。
  - 监控看板、日志流、交易数据推送。
  - 实时统计图表、股票行情。
  - 替代 WebSocket 的轻量实时通知方案。

---



### WebSocket

- **方向：** 双向（Client ↔ Server）
- **机制：**
  - 先通过 HTTP 握手（Upgrade 请求）。
  - 连接升级后切换为 WebSocket 协议，支持全双工通信。
- **优点：**
  - 实时性最强，真正的双向通道。
  - 消息头极小，通信高效。
  - 适合高频消息交互。
- **缺点：**
  - 不再是标准 HTTP 请求，需专门服务支持。
  - 连接保持需要心跳检测与状态管理。
  - 对代理、防火墙环境要求更高。
- **使用场景：**
  - 实时性、高交互的系统：
    - IM 聊天、协同编辑、在线游戏。
    - 实时交易系统（如币圈/股票）。
    - Web 远程控制、IoT 实时指令。
    - 实时语音、视频信令通道。





## TCP 协议

<img src="https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20251006103519109.png" alt="image-20251006103519109" style="zoom: 67%;" />

**序列号：**在建立连接时由计算机生成的随机数作为其初始值，通过 SYN 包传给接收端主机，每发送一次数据，就「累加」一次该「数据字节数」的大小。**用来解决网络包乱序问题。**

**确认应答号：**指下一次「期望」收到的数据的序列号，发送端收到这个确认应答以后可以认为在这个序号以前的数据都已经被正常接收。**用来解决丢包的问题。**

**控制位：**

**ACK：**该位为 1 时，「确认应答」的字段变为有效，TCP 规定除了最初建立连接时的 SYN 包之外该位必须设置为 1 。
**RST：**该位为 1 时，表示 TCP 连接中出现异常必须强制断开连接。(`RST`标志是 TCP 包头中的一个控制位，当设置了该标志时，意味着当前 TCP 连接遇到了异常情况，需要立即进行重置。它的作用是强制终止一个存在问题的 TCP 连接，使得通信双方能够迅速从异常状态中恢复，避免资源的无效占用和进一步错误的发生。)
**SYN：**该位为 1 时，表示希望建立连接，并在其「序列号」的字段进行序列号初始值的设定。
**FIN：**该位为 1 时，表示今后不会再有数据发送，希望断开连接。当通信结束希望断开连接时，通信双方的主机之间就可以相互交换 FIN 位为 1 的 TCP 段。

`PSH` 的具体作用

当发送方（客户端或服务器）在 TCP 包中设置 `PSH` 标志时：

- 接收方的 TCP 层会**跳过缓冲**，直接把当前数据推送给应用层（比如 WebSocket 回调函数 `on('message')`）。
- 确保数据 “即时交付”，满足实时通信的低延迟需求。

#### 三次握手

<img src="https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20251006103738301.png" alt="image-20251006103738301" style="zoom:50%;" />第三次握手是可以携带数据的，前两次握手是不可以携带数据的



到第三次握手时：

1. 客户端知道服务端的初始序列号；
2. 服务端知道客户端的初始序列号；

双方序列号同步完成，确认号机制已经可靠建立

---



> 为什么需要三次握手？

**TCP 使用三次握手建立连接的最==主要原因==是防止「历史连接」初始化了连接。**

如果采用两次握手建立 TCP 连接的场景下，服务端在向客户端发送数据前，并没有阻止掉历史连接，导致服务端建立了一个历史连接，又白白发送了数据，妥妥地浪费了服务端的资源。





**同步双方初始序列号**

**TCP 协议的通信双方， 都必须维护一个「序列号」， 序列号是可靠传输的一个关键因素，它的作用：**

**两次握手只保证了一方的初始序列号能被对方成功接收，没办法保证双方的初始序列号都能被确认接收。**

- 接收方可以去除重复的数据；
  接收方可以根据数据包的序列号按序接收；
  可以标识发送出去的数据包中， 哪些是已经被对方收到的（通过 ACK 报文中的序列号知道）；
  可见，序列号在 TCP 连接中占据着非常重要的作用，所以当客户端发送携带「初始序列号」的 SYN 报文的时候，需要服务端回一个 ACK 应答报文，表示客户端的 SYN 报文已被服务端成功接收，那当服务端发送「初始序列号」给客户端的时候，依然也要得到客户端的应答回应，这样一来一回，才能确保双方的初始序列号能被可靠的同步。



**避免资源浪费**

如果只有「两次握手」，当客户端发生的 SYN 报文在网络中阻塞，客户端没有接收到 ACK 报文，就会重新发送 SYN ，由于没有第三次握手，服务端不清楚客户端是否收到了自己回复的 ACK 报文，所以服务端每收到一个 SYN 就只能先主动建立一个连接，这会造成什么情况呢？

如果客户端发送的 SYN 报文在网络中阻塞了，重复发送多次 SYN 报文，那么服务端在收到请求后就会建立多个冗余的无效链接，造成不必要的资源浪费。

#### 四次挥手

![image-20251006125249128](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20251006125249128.png)

**具体过程：**

客户端主动调用关闭连接的函数，于是就会发送 FIN 报文，这个 FIN 报文代表客户端不会再发送数据了，进入 FIN_WAIT_1 状态；

服务端收到了 FIN 报文，然后马上回复一个 ACK 确认报文，此时服务端进入 CLOSE_WAIT 状态。在收到 FIN 报文的时候，TCP 协议栈会为 FIN 包插入一个文件结束符 EOF 到接收缓冲区中，服务端应用程序可以通过 read 调用来感知这个 FIN 包，这个 EOF 会被放在已排队等候的其他已接收的数据之后，所以必须要得继续 read 接收缓冲区已接收的数据；

接着，当服务端在 read 数据的时候，最后自然就会读到 EOF，接着 read() 就会返回 0，这时服务端应用程序如果有数据要发送的话，就发完数据后才调用关闭连接的函数，如果服务端应用程序没有数据要发送的话，可以直接调用关闭连接的函数，这时服务端就会发一个 FIN 包，这个 FIN 报文代表服务端不会再发送数据了，之后处于 LAST_ACK 状态；

客户端接收到服务端的 FIN 包，并发送 ACK 确认包给服务端，此时客户端将进入 TIME_WAIT 状态；

服务端收到 ACK 确认包后，就进入了最后的 CLOSE 状态；

客户端经过 2MSL 时间之后，也进入 CLOSE 状态

---



**服务器收到客户端的 FIN 报文时，内核会马上回一个 ACK 应答报文，但是服务端应用程序可能还有数据要发送，所以并不能马上发送 FIN 报文，而是将发送 FIN 报文的控制权交给服务端应用程序：**

- 如果服务端应用程序有数据要发送的话，就发完数据后，才调用关闭连接的函数；
- 如果服务端应用程序没有数据要发送的话，可以直接调用关闭连接的函数，
- 从上面过程可知，是否要发送第三次挥手的控制权不在内核，而是在被动关闭方（上图的服务端）的应用程序，因为应用程序可能还有数据要发送，由应用程序决定什么时候调用关闭连接的函数，当调用了关闭连接的函数，内核就会发送 FIN 报文了，所以服务端的 ACK 和 FIN 一般都会分开发送

**当被动关闭方在 TCP 挥手过程中，「没有数据要发送」并且「开启了 TCP 延迟确认机制」，那么第二和第三次挥手就会合并传输，这样就出现了三次挥手。**

#### 套接字

**1️⃣ 服务端**

1. 调用 `socket()` → 创建一个套接字（Socket 对象）
2. `bind()` → 绑定本地 IP 和端口
3. `listen()` → 告诉操作系统，这个端口可以接收连接
4. `accept()` → 阻塞等待客户端连接，一旦有三次握手完成，返回一个新的 Socket，用于与该客户端通信

> 注意：服务端 `listen` 的 Socket 是 **被动监听端口**，`accept` 返回的 Socket 才是**每个客户端的独立连接**。

2️⃣ **客户端**

1. 调用 `socket()` → 创建套接字
2. `connect()` → 主动发起连接
   - 内核会把三次握手交给 TCP 协议栈处理
   - 三次握手完成后，`connect()` 返回成功
3. 客户端得到一个 Socket 对象，可以开始收发数据

![image-20251006151706853](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20251006151706853.png)



在 WebSocket 协议中，一个完整的“消息”可能被拆成多帧发送。

- 如果消息比较大，可以分成多帧：
  - 第一帧：`FIN=0`，`Opcode=1`（Text）
  - 中间帧：`FIN=0`，`Opcode=0`（Continuation）
  - 最后一帧：`FIN=1`，`Opcode=0`（Continuation）
- 如果消息较小或只需一帧发送：
  - `FIN=1`（最后一帧，也是唯一帧）



| Opcode | 二进制 | 含义                         |
| ------ | ------ | ---------------------------- |
| `0x0`  | `0000` | Continuation frame（延续帧） |
| `0x1`  | `0001` | Text frame（文本帧）         |
| `0x2`  | `0010` | Binary frame（二进制帧）     |
| `0x8`  | `1000` | Connection Close（关闭帧）   |
| `0x9`  | `1001` | Ping（心跳请求）             |
| `0xA`  | `1010` | Pong（心跳响应）             |
| 其他   | 保留   | 未来扩展                     |

| 值   | 含义                                   |
| ---- | -------------------------------------- |
| 0    | 没有使用掩码（通常为服务器发送的数据） |
| 1    | 使用了掩码（通常为客户端发送的数据）   |

**RSV1、RSV2、RSV3（各 1 位，共 3 位）**：这三位主要用于扩展。如果没有使用自定义的扩展，通常这三位都为 `0`



![image-20251006152727058](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20251006152727058.png)

**掩码密钥（Masking - key）**

- 当 `MASK` 为 `1` 时，会有 4 个字节的掩码密钥。客户端 -> 服务端 需要
- 客户端向服务器发送数据时，必须使用掩码密钥对负载数据进行异或操作，服务器收到后会用相同的掩码密钥进行异或操作来还原数据；
- 服务器向客户端发送数据时，`MASK` 为 `0`，不需要掩码密钥。



> **断开连接**





![image-20251006162209481](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20251006162209481.png)



1. > **客户端发送 WebSocket 关闭帧（触发关闭流程）**

- **协议层**：这是 WebSocket 应用层的 “关闭请求”，携带关闭码（如 `1000` 表示正常关闭）和原因（可选）。
- 标志含义
  - `FIN=1`：这是一个完整的关闭帧（非分片）。
  - `MASKED=1`：客户端发送的帧必须用掩码加密（WebSocket 协议规定，防止客户端数据被注入攻击）。
- **作用**：告诉服务器 “应用层已完成数据交互，请求关闭连接”。



2. > **服务器确认收到关闭帧（TCP 层 ACK）**

- **协议层**：TCP 层的确认，与 WebSocket 帧无关，仅表示 “服务器已收到客户端发送的关闭帧”。
- **关键**：此时服务器应用层可能还在处理关闭逻辑（如记录日志），并未立即关闭连接。



3. > **服务器发送 WebSocket 关闭帧（响应关闭请求）**

- **协议层**：服务器应用层回复关闭帧，携带自己的关闭码（需与客户端协商一致，如同样返回 `1000`）。
- **标志含义**：`FIN=1`（完整帧），**无 MASKED**（服务器发送的帧不需要掩码，协议规定）。
- **作用**：告诉客户端 “服务器应用层已准备好关闭，同意结束连接”。



4. > **客户端确认收到服务器关闭帧（TCP 层 ACK）**

- **协议层**：TCP 层确认 “客户端已收到服务器的关闭帧（序列号到 X'-1）”。
- **关键**：此时 WebSocket 应用层的关闭协商已完成（双方都同意关闭），**接下来进入 TCP 底层的连接关闭流程。**



5. > **TCP** **四次挥手：关闭客户端到服务器的发送方向**

- **标志含义**：`FIN` 表示 “客户端已无数据发送，关闭发送方向”；`ACK` 确认之前的服务器数据。
- **作用**：TCP 层通知服务器 “客户端不会再发数据了”。



6. > **服务器确认客户端的 FIN（第二次挥手）**

- **作用**：服务器 TCP 层确认 “已收到客户端的 FIN，客户端发送方向已关闭”。
- **状态**：此时客户端 → 服务器的方向关闭，但服务器 → 客户端的方向仍可发送数据（如果有残留）。



7. > **TCP 四次挥手：关闭服务器到客户端的发送方向**

- **标志含义**：`FIN` 表示 “服务器已无数据发送，关闭发送方向”；`ACK` 再次确认客户端的 FIN。
- **作用**：TCP 层通知客户端 “服务器不会再发数据了”。



> 8. **客户端确认服务器的 FIN（第四次挥手）**

- **作用**：客户端 TCP 层确认 “已收到服务器的 FIN，服务器发送方向已关闭”。
- **结果**：TCP 连接双向关闭，资源释放。

 # ==连接保活方案设计==

### 心跳机制

> **为什么需要心跳机制 频率如何设计**

WebSocket 建立后，连接理论上是**持久的**，但现实网络中：

- 用户换 Wi-Fi、切 4G；
- NAT 网关清理空闲连接；
- 操作系统 TCP Idle Timeout；
- 服务端防火墙闲置连接回收。

这些都可能在你**不知情的情况下**断开 TCP 连接，而浏览器和服务端**都不会立刻察觉**。

**心跳机制**就是为了解决这个问题：

- 定期发送小包（ping/pong）
- 若对方未响应，则认为连接已断，执行重连、清理、状态下线等逻辑

---



心跳频率设计的关键是：**平衡“实时性”与“资源消耗”**。

 这其实就是一个“功耗与响应”权衡问题。

根据不同的场景有着不同的频率



| 频率高（心跳间隔短）   | 频率低（心跳间隔长） |
| ---------------------- | -------------------- |
| ✅ 快速检测异常断开     | ❌ 断线检测延迟高     |
| ❌ 消耗更多网络流量     | ✅ 节省流量和能耗     |

| ❌ 服务器心跳检测压力大 | ✅ 服务器负载低       |

> 常见问题



| 问题                        | 说明                     | 解决方案                                                    |
| --------------------------- | ------------------------ | ----------------------------------------------------------- |
| 🧱 心跳被浏览器缓存/代理阻断 | 部分中间层过滤空数据     | 心跳包加上少量随机数据                                      |
| ⚰️ 长时间休眠（手机锁屏）    | 移动端会暂停 JS 定时器   | 服务端主动检测断线重连（xx秒后没接收到心跳就重连 定期检测） |

| 🔁 重连风暴                  | 同时上万客户端断开并重连 | 增加重连随机抖动 (jitter)                                   |

 服务端需要：

1. **维护每个连接的最后心跳时间（lastPingTime）**；
2. **周期性扫描**（例如每 1 分钟）所有连接；
3. 如果某个连接超过一定时间（比如 90 秒）没收到任何消息，则：
   - 主动关闭该连接；
   - 记录状态；
   - 等待客户端重新上线后再恢复会话。

### 断线重连策略（指数退避算法）

WebSocket 在真实环境下并不是“永远连着”的。断线的常见原因包括：

- 用户网络切换（Wi-Fi ⇄ 4G）
- NAT 映射超时
- 服务器重启 / 负载均衡漂移
- 客户端睡眠唤醒（挂起网络服务）（移动端）

> 指数退避算法的核心：每次失败重试的等待时间呈指数级增长，但上限有限

~~~js
// 重连配置
        let reconnectAttempts = 0;          // 当前重连尝试次数
        let maxReconnectAttempts = 10;      // 最大重连次数限制
        let reconnectDelay = 1000;          // 初始延迟时间（1秒）
        let maxReconnectDelay = 30000;      // 最大延迟时间（30秒）
        let reconnectTimer = null;          // 重连定时器引用
        let isManualDisconnect = false;     // 是否手动断开标志

    // 计算指数退避延迟时间
        function getReconnectDelay() {
            // 指数退避算法: delay = min(初始延迟 * 2^尝试次数, 最大延迟)
            const delay = Math.min(reconnectDelay * Math.pow(2, reconnectAttempts), maxReconnectDelay);
            // 添加随机抖动，避免多个客户端同时重连
            const jitter = Math.random() * 0.3 * delay; // 0-30%的随机抖动
            return Math.floor(delay + jitter);
        }

        // 重连函数
        function attemptReconnect() {
            if (isManualDisconnect) {
                log('已手动断开，不进行重连', 'system');
                return;
            }

            if (reconnectAttempts >= maxReconnectAttempts) {
                log(`已达到最大重连次数(${maxReconnectAttempts})，停止重连`, 'error');
                updateStatus('连接失败', false);
                return;
            }

            const delay = getReconnectDelay();
            reconnectAttempts++;
            
            log(`第 ${reconnectAttempts} 次重连尝试，延迟 ${(delay/1000).toFixed(1)} 秒...`, 'system');
            updateStatus(`重连中(${reconnectAttempts}/${maxReconnectAttempts})`, false);

            reconnectTimer = setTimeout(() => {
                log('执行重连...', 'system');
                createWebSocket();
            }, delay);
        }
~~~

**随机抖动的作用**:

- 防止**惊群效应**（Thundering Herd）（**重连风暴**）
- 如果多个客户端同时断开，不会同时发起重连
- 分散服务器压力

### NAT超时问题

NAT（Network Address Translation）让内网设备通过一个公网 IP 上网。
 比如你的电脑（192.168.0.10）访问服务器（8.8.8.8），路由器会生成一个映射表：

| 内网IP:Port        | 公网IP:Port   | 状态        |
| ------------------ | ------------- | ----------- |
| 192.168.0.10:52130 | 1.2.3.4:12000 | established |

**NAT 映射不是永久保存的！**

因为 NAT 设备资源有限（比如家用路由器几千条就满了），
 所以它会定期清理 **“长时间没活动的连接”**。



> **为什么它会让WebSocket断开？**

因为 WebSocket 连接在底层依赖 TCP 通道。
 如果 NAT 发现这个通道“太久没数据”，就会回收映射。
 结果是：

- 客户端以为连接还在；
- 实际上 NAT 已经把管道拆了；
- 服务器发的包“无家可归” → 超时 → 连接异常。

| 网络环境     | 心跳间隔（推荐） | 说明                    |
| ------------ | ---------------- | ----------------------- |
| Wi-Fi / 有线 | 30s              | NAT 超时一般在 1-2 分钟 |
| 4G / 5G      | 15-20s           | 移动网络NAT更积极       |
| 企业内网     | 60s              | 管理更宽松              |



### 移动网络环境下的连接保持

**移动端的==连接不稳定==主要源于：**

- 系统休眠（App后台）
- 网络切换（Wi-Fi → 4G）
- 信号波动（隧道断开重连）

| 问题     | 策略                    | 说明                     |
| -------- | ----------------------- | ------------------------ |
| 网络切换 | 自动检测网络变化，重连  | 利用 `navigator.onLine`  |
| App后台  | 关闭WebSocket，后台轮询 | 节省电量，防止系统杀进程 |
| 唤醒恢复 | 延迟重连 + 校验状态     | 避免重复登录或数据乱序   |

---

#### 1️⃣ 网络切换导致的问题

**问题：**

* 网络切换（比如从 Wi-Fi 切换到移动网络）会导致 IP 地址变化或者 NAT 映射丢失。
* **TCP 连接依赖 IP + 端口 + 双端 socket 的唯一组合，IP 或 NAT 改变后原连接直接中断。**

**策略：**

* 自动检测网络变化，触发重连。

**思路/原因：**

* **主动感知变化**：客户端可以使用浏览器/系统提供的 `navigator.onLine` 或平台网络监听 API 检测网络状态变化。
* **避免无用等待**：网络切换后继续使用旧连接会失败，及时触发重连可以保证服务可用性。
* **工程化考虑**：在移动端，网络切换非常常见（Wi-Fi ↔ 4G），必须把“网络不可用”作为常态处理。

---

#### 2️⃣ App后台

**问题：**

* 移动操作系统会在 App 进入后台时限制网络，甚至杀掉长连接（如 WebSocket）。
* 如果继续维持高频连接，会增加电量消耗和系统负担。

**策略：**

* 后台使用 **低频 HTTP 轮询**（例如 30s 一次）。
* 恢复前台时再 **延迟重连**。

**思路/原因：**

* **节省电量**：低频轮询代替 WebSocket 保持状态，避免后台无限循环尝试重连。
* **防止系统杀进程**：频繁的后台任务容易被系统认为占用资源过多而终止。
* **平滑体验**：App回到前台时，不立刻重连，而是延迟几秒，让系统网络和应用环境稳定后再重连，避免冲突或异常错误。

#### 3️⃣ 延迟重连 + 状态校验

**问题：**

* 网络恢复或 App 回到前台时，如果立即重连，可能遇到 **旧连接未释放、服务端状态未同步** 或 **连续失败重连**。

**策略：**

* 延迟几秒再重连。
* 上线前，客户端主动发送一个“心跳请求”验证通道可用。

**思路/原因：**

1. **延迟重连**：

   * 避免瞬时**网络抖动或短时不可用**导致的重复重连。
   * 给系统一个“稳定窗口”，减少失败次数，降低服务器压力。
2. **状态校验**：
   * 通过心跳或状态检查，确保客户端与服务端通道一致。
   * 避免数据乱序或丢失（例如消息或操作请求在网络抖动后未同步）。

**消息的可靠性，即消息的不丢失和不重复**



**<span style="font-size:1.4em; color:#FF0000;">场景：</span>**

![image-20251006230243772](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20251006230243772.png)

R：客户端主动发送给服务器的报文
A：服务器被动应答客户端的报文，一个A一定对应一个R
N：服务器主动发送给客户端的报文

![image-20251006230341840](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20251006230341840.png)

client-A向im-server发送一个消息请求包，即msg:R
im-server在成功处理后，回复client-A一个消息响应包，即msg:A
如果此时client-B在线，则im-server主动向client-B发送一个消息通知包，即msg:N（当然，如果client-B不在线，则消息会存储离线）



从流程图中容易看到，发送方client-A收到msg:A后，只能说明im-server成功接收到了消息，并不能说明client-B接收到了消息。在若干场景下，可能出现msg:N包丢失，且发送方client-A完全不知道，例如：

- 服务器崩溃，msg:N包未发出
- 网络抖动，msg:N包被网络设备丢弃
- client-B崩溃，msg:N包未接收

​    **结论是悲观的：接收方client-B是否有收到msg:N，发送方client-A完全不可控，那怎么办呢？**

> **解决方法**

类比现有解决方案：TCP如何做到可靠呢？**超时、重传、确认**、（去重）



<img src="https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20251006235233014.png" alt="image-20251006235233014" style="zoom:67%;" />



> **消息的超时和重传**

client-A发出了msg:R，收到了msg:A之后，在一个期待的时间内，如果没有收到ack:N，client-A会尝试将msg:R重发。可能client-A同时发出了很多消息，故client-A需要在本地维护一个等待ack队列，并配合timer超时机制，来记录哪些消息没有收到ack:N，以定时重发。

一旦收到了ack:N，说明client-B收到了“你好”消息，对应的消息将从“等待ack队列”中移除。



client --> server 发送失败 直接提示发送失败即可

 

server --> client 发送失败  

- msg:N 报文丢失：说明client-B之前压根没有收到“你好”报文，超时与重传机制十分有效
- ack:N 报文丢失：说明client-B之前已经收到了“你好”报文（只是client-A不知道而已），超时与重传机制将导致client-B收到重复的消息。



> **消息的去重**

发送方client-A生成一个消息去重的msgid，保存在“等待ack队列”里，同一条消息使用相同的msgid来重传，供client-B去重，而不影响用户体验。

> **保证离线消息的可靠传递**

![image-20251007001136619](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20251007001136619.png)



- Step 1：用户A发送一条消息给用户B；
- Step 2：服务器查看用户B的状态，发现B的状态为“offline”（即B当前不在线）；

- Step 4：服务器返回用户A“发送成功”ACK确认包（注：对于消息发送方而言，消息一旦落地存储至DB就认为是发送成功了）。

- Step 1：用户B开始拉取用户A发送给ta的离线消息；
- Step 2：服务器从DB（或对应的持久化容器）中拉取离线消息；
- Step 3：服务器从DB（或对应的持久化容器）中把离线消息删除；
- Step 4：服务器返回给用户B想要的离线消息。

![image-20251007001317103](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20251007001317103.png)



> **上述流程存在的问题以及对应的优化方案**

如果用户B有很多好友，登陆时客户端需要对所有好友进行离线消息拉取，客户端与服务器交互次数就会比较多。



**优化方案1：**

- 先拉取各个好友的离线消息数量，真正用户B进去看离线消息时，才往服务器发送拉取请求（手机端为了节省流量，经常会使用这个按需拉取的优化）。


**优化方案2：**

- 一次性拉取所有好友发送给用户B的离线消息，到客户端本地再根据sender_uid进行计算，这样的话，离线消息表的访问模式就变为->只需要按照receiver_uid来查询了。登录时与服务器的交互次数降低为了1次

# ==数据模型设计==

### 消息ID 生成策略

#### 雪花算法 UUID 主键自增

雪花算法（Snowflake）是 Twitter 设计的分布式 ID 方案，结构如下（64 位整数，兼容 Java 的 long 类型）：



```
1位符号位（固定0，确保是正数）
+---------------------+----------------+----------------+
| 41位时间戳（毫秒级） | 10位机器ID     | 12位序列号     |
+---------------------+----------------+----------------+
  表示当前时间与起始时间的差值   标识分布式节点   同一节点同一毫秒内的自增数
```

- **41 位时间戳**：可表示约 69 年（2^41 / 1000 / 60 / 60 / 24 / 365 ≈ 69），需自定义起始时间（如 2020-01-01）。
- **10 位机器 ID**：可部署 1024 个节点（2^10 = 1024），通常拆分为 “数据中心 ID（5 位）+ 节点 ID（5 位）”。
- **12 位序列号**：同一节点同一毫秒内可生成 4096 个 ID（2^12 = 4096），避免高并发下的重复。



| 传统痛点                      | 数据库自增 ID（分布式场景）                                  | UUID                   | 雪花算法                   |
| ----------------------------- | ------------------------------------------------------------ | ---------------------- | -------------------------- |
| 分布式节点 ID 冲突            | ❌ 无法解决（多库自增起始值相同会冲突）                       | ✅ 天然支持（全球唯一） | ✅ 支持（机器 ID 区分节点） |
| 数据迁移 / 分库分表后 ID 重复 | ❌ 分表后自增 ID 会重复（如分表 1 从 1 开始，分表 2 也从 1 开始） | ✅ 无影响               | ✅ 无影响                   |
| ID 有序性（利于索引查询）     | ✅ 严格递增（单库场景）                                       | ❌ 无序（字符串随机）   | ✅ 大致有序（按时间戳）     |
| ID 长度（存储 / 传输成本）    | ✅ 短（int/long 类型）                                        | ❌ 长（36 位字符串）    | ✅ 短（64 位整数）          |

**UUID问题：**

> **原因 1：UUID 无序，导致 B + 树频繁 “分裂”，写入性能暴跌**

B + 树索引的 “有序性” 不仅影响查询，更影响**写入性能**—— 因为 B + 树的每个节点都有 “容量上限”（比如一个节点能存 100 个索引键），有序数据写入时，会 “追加” 到叶子节点的末尾，只有当节点满了才会触发一次 “分裂”（把节点拆成两个，调整父节点）；而无序的 UUID 写入时，会 “随机插入” 到叶子节点的任意位置，导致节点频繁分裂。



> **原因 2：UUID 是 36 位字符串，索引体积大，查询效率低**

索引的 “体积” 直接影响查询速度 —— 因为数据库查询时，会先把索引加载到**内存缓存（Buffer Pool）** 中，如果索引体积太大，内存能缓存的索引数量就少，就需要频繁从磁盘加载索引（磁盘 IO 速度比内存慢 1000 倍以上）。



> **UUID 无序导致 “索引碎片”，长期性能劣化**

- 有序的自增 ID 写入时，数据和索引都是 “连续存储” 的，几乎不会产生碎片；
- 无序的 UUID 写入时，数据会 “随机插入” 到磁盘的任意位置（比如中间某个已删除数据的空洞），导致索引的叶子节点 “不连续”—— 查询时，数据库需要跳转到多个磁盘块读取数据，速度越来越慢。

更麻烦的是：索引碎片会随着写入次数增加而累积，最终你不得不频繁执行 “OPTIMIZE TABLE”（优化表）来整理碎片，但这个操作会锁表，影响业务可用性。

### MongoDB VS Mysql 

数据库选型的核心原则是 “**业务驱动**”—— 没有 “最好的数据库”，只有 “最适合业务的数据库”。 **（其他技术也是如此）**

| 维度         | MySQL（关系型）                                              | MongoDB（文档型）                                            |
| ------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **核心逻辑** | 数据按 “表（Table）” 分类，表内按 “行（Row）” 存储，行的结构由 “列（Column）” 严格定义（如 “用户表” 必须包含 id、name、age 列）；表与表之间通过 “外键（Foreign Key）” 关联（如 “订单表” 的 user_id 关联 “用户表” 的 id）。 | 数据按 “集合（Collection）” 分类，集合内按 “文档（Document）” 存储，文档是类似 JSON 的灵活结构（BSON 格式），无需预先定义字段（如 “用户文档” 可按需包含 name、age，也可额外加 address、hobbies，其他文档不受影响）；文档间通过 “引用（Reference）” 松散关联（非强制）。 |

- **MySQL（1995 年诞生）**：源于 “传统企业级应用” 需求（如财务系统、ERP），核心诉求是**数据一致性、结构化存储、复杂关联查询**—— 比如财务数据不能出错（一致性），数据格式固定（结构化），需要频繁查询 “某部门某员工的近 3 个月报销单”（多表关联）。因此 MySQL 的设计围绕 “ACID 事务、强 schema、SQL 查询” 展开。
- **MongoDB（2007 年诞生）**：源于 “Web 2.0 时代” 需求（如社交、电商、内容管理），核心诉求是**灵活的数据结构、高写入性能、分布式扩展**—— 比如社交 App 的 “用户动态” 可能包含文字、图片、视频链接、点赞列表，字段随时可能增加（灵活结构）；峰值时每秒需写入上万条动态（高写入）；用户量从百万涨到亿级需快速扩容（分布式）。因此 MongoDB 的设计围绕 “灵活 schema、文档存储、横向扩展” 展开。

---

| 评估维度       | MySQL（关系型）                                              | MongoDB（文档型）                                            |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **数据模型**   | 结构化（强 Schema）：表结构需预先定义，字段类型、长度固定，修改结构（如加列）需 DDL 操作，可能锁表。 | 半结构化（弱 Schema）：文档结构灵活，字段可动态增减，不同文档可有不同字段，无需预定义结构。 |
| **关联方式**   | 支持多表关联（JOIN）：通过外键强制关联，可高效查询 “跨表数据”（如用户 + 订单 + 订单详情）。 | 不支持原生 JOIN：文档间通过 “引用（Reference）” 松散关联，需手动做 “两次查询”（先查用户，再查用户的订单）。 |
| **查询语言**   | SQL：标准化查询语言，支持复杂查询（GROUP BY、HAVING、子查询、窗口函数），学习成本低（开发者普遍熟悉）。 | MQL（MongoDB Query Language）：基于 JSON 的查询语法，支持文档内嵌套查询（如查 “hobbies 包含足球的用户”），但复杂统计（如多维度分组）需结合聚合管道（Aggregation Pipeline），学习成本较高。 |
| **事务支持**   | 支持 ACID 事务：从 MySQL 5.5 开始支持完整 ACID，可保证 “多操作原子性”（如转账：扣钱 + 加钱必须同时成功或失败），支持隔离级别（Read Uncommitted ~ Serializable）。 | 支持部分 ACID：从 MongoDB 4.0 开始支持 “单文档事务”，4.2 + 支持 “多文档事务”，但事务性能低于 MySQL，且分布式事务（跨分片）支持有限，更适合 “最终一致性” 场景。 |
| **写入性能**   | 写入性能中等：受限于事务日志（redo log）、索引维护，高写入场景（如每秒 10 万条日志）需做分库分表，配置复杂。 | 写入性能高：支持 “内存映射（MMAP）”“预写日志（WAL）”，文档存储无需拆分成多表，单集合写入性能可达每秒数万条，适合高写入场景（如日志、监控数据、用户动态）。 |
| **索引能力**   | 支持多种索引：主键索引、唯一索引、普通索引、联合索引、全文索引、空间索引，但索引基于 “表字段”，无法直接索引嵌套结构。 | 支持更灵活的索引：除普通索引、唯一索引外，支持 “嵌套文档索引”（如索引 “address.city”）、“数组索引”（如索引 “hobbies” 数组中的每个元素）、地理空间索引（适合 LBS 场景），索引类型更丰富。 |
| **分布式扩展** | **横向扩展复杂：需依赖第三方工具（如 Sharding-JDBC、MyCat）实现分库分表，分片规则（如按用户 ID 哈希）需手动设计，扩容时可能需数据迁移，运维成本高。** | **原生支持分布式：内置分片集群（Sharded Cluster），支持自动分片、负载均衡、故障转移，扩容时只需加分片节点，无需手动迁移数据，运维成本低。** |
| **适用数据量** | 单表适合中小数据量（千万级以内）：超过千万级需分库分表，否则查询性能会显著下降。 | 适合海量数据（亿级 +）：原生分布式架构，支持 PB 级数据存储，单分片可存千万级文档，扩展后可支撑亿级数据查询。 |
| **运维复杂度** | 运维复杂：需关注主从复制（主从切换）、分库分表配置、索引优化、慢查询监控，升级版本（如从 5.7 到 8.0）需做兼容性测试。 | 运维较简单：内置集群管理工具（mongosh、Compass），分片集群自动维护，索引优化难度低，但需关注 “文档大小限制”（默认单文档最大 16MB）、分片键选择（影响查询效率）。 |

要实现分布式（横向扩展），都必须解决两个核心问题：

1. **数据拆分**：如何将海量数据（如 10 亿条订单）拆分成多个 “数据分片”，分散存储到不同服务器上？
2. **协同管理**：如何让多个分片 “协同工作”（如查询时自动路由到目标分片、扩容时自动迁移数据、故障时自动切换），且对应用透明？

---



**mysql问题：**

- **分片协调**：需要第三方工具（如 MyCat）作为 “中间件”，接收应用的 SQL 请求，路由到目标分片；
- **数据迁移**：扩容时（如从 2 个分片加到 4 个分片），需要手动用工具（如 ShardingSphere 的 DataDiff）迁移数据，迁移过程中可能需要停服或限流；
- **故障转移**：某个分片节点宕机后，需要手动切换到从库，无法自动完成（需结合 MHA、Keepalived 等工具，配置复杂）。



**MongoDB：**

文档结构灵活（不同文档可有不同字段），分片规则无需依赖固定字段 —— 既可以按`user_id`哈希分片，也可以按`create_time`范围分片（如按月份拆分日志数据），修改规则时只需更新 Config Server 的配置，无需重构数据结构。

MongoDB 不支持原生 JOIN，鼓励将 “关联紧密的数据” 嵌入到同一个文档中（如 “订单文档” 中嵌入 “订单详情列表”），查询时只需访问一个分片，无需跨节点通信。即使需要关联（如用户和订单），也通过 “手动两次查询”（先查用户所在分片，再查订单所在分片）实现，性能可控。

### 数据库表设计实践经验

##### 字段类型



| 业务场景                    | 推荐类型                                      | 避坑提醒                                                     |
| --------------------------- | --------------------------------------------- | ------------------------------------------------------------ |
| 主键 / 用户ID / 订单号      | **BIGINT**（雪花算法） / **CHAR(32)**（UUID） | ❗ 避免 `INT`（上限仅 21 亿）；UUID 用 `CHAR(32)`（去掉“-”），别用 `VARCHAR`。 |
| 状态字段（如订单状态）      | **TINYINT** / **SMALLINT**                    | ❗ 用数字编码，不用字符串；超 255 用 `SMALLINT`。             |
| 时间字段（创建 / 更新时间） | **DATETIME** / **TIMESTAMP**                  | ❗ 跨时区用 `TIMESTAMP`；记得加默认值与自动更新。             |
| 金额（余额 / 订单金额）     | **DECIMAL(10,2)** / **BIGINT（分单位）**      | ❗ 禁用浮点型；高并发下用整数存“分”。                         |
| 短文本（用户名 / 标题）     | **VARCHAR(50)**                               | ❗ 不用 `CHAR`；长度设为“业务上限 +10%”。                     |
| 长文本（描述 / 日志）       | **TEXT** / 独立表存储                         | ❗ 不要用大 `VARCHAR`；超 10KB 建独立表（InnoDB 页 16KB）。   |

##### 字段命名

| 字段用途              | 统一命名      | 字段类型           |
| --------------------- | ------------- | ------------------ |
| 主键（唯一标识）      | `id`          | BIGINT             |
| 创建时间              | `create_time` | DATETIME/TIMESTAMP |
| 更新时间              | `update_time` | DATETIME/TIMESTAMP |
| 软删除标识            | `is_deleted`  | TINYINT            |
| 备注 / 说明           | `remark`      | VARCHAR(512)       |
| 操作员 ID（谁创建的） | `creator_id`  | BIGINT             |
| 操作员 ID（谁更新的） | `updater_id`  | BIGINT             |

> **状态 / 布尔型字段：用 “is_/has_/can_” 开头，明确逻辑**

1. 布尔型字段（值为 0/1 或 true/false）：用`is_`开头（如`is_deleted`、`is_mute`）；
2. 表示 “拥有某属性” 的字段：用`has_`开头（如`has_vip`、`has_address`）；
3. 表示 “具备某能力” 的字段：用`can_`开头（如`can_comment`、`can_withdraw`）；



> **枚举 / 状态类字段：用 “_status/_type” 结尾**

- 表示 “流程状态” 的字段（如订单状态、支付状态）：用`_status`（如`order_status`、`pay_status`）；
- 表示 “分类 / 类型” 的字段（如商品类型、消息类型）：用`_type`（如`product_type`、`msg_type`）；





> **关联字段加 “来源表前缀”，明确关联关系**

- **规则**：当字段关联其他表时，字段名以 “关联表的核心标识” 为前缀，避免用无意义的缩写；
  - 订单表（`orders`）关联用户表（`users`）：字段名用`user_id`（而非`uid`），明确关联 “用户表的主键”；
  - 订单表关联商品表（`products`）：字段名用`product_id`（而非`pid`），明确关联 “商品表的主键”；
  - 评论表（`comments`）关联订单表：字段名用`order_no`（而非`ono`），明确关联 “订单表的订单号”



> **遵循 “业务属性 + 字段类型” 的命名逻辑**

- **规则**：字段名由 “业务相关的**核心属性”+“字段类型** / 含义” 组成。
  - `order_amount`：“order”（业务属性：订单）+“amount”（类型：金额）→ 明确是 “订单金额”；
  - `user_create_time`：“user”（业务属性：用户）+“create_time”（类型：创建时间）→ 明确是 “用户创建时间”；
  - `product_stock`：“product”（业务属性：商品）+“stock”（类型：库存）→ 明确是 “商品库存”；

##### 字段约束

| 约束类型          | 作用                                 |
| ----------------- | ------------------------------------ |
| NOT NULL（非空）  | 禁止字段为空                         |
| UNIQUE（唯一）    | 禁止字段重复                         |
| DEFAULT（默认值） | 字段未赋值时用默认值                 |
| CHECK（条件校验） | 字段值需满足条件（MySQL 8.0 + 支持） |

要依赖应用层校验（如前端判断手机号非空），必须加数据库约束 —— **因为应用层可能被绕过（如黑客直接调用 API）**，数据库约束是 “最后一道防线”。

##### 冗余字段

**冗余字段**的核心原则：**“高频查询的关联字段可冗余，低频查询的字段必须关联”**，避免 “为了冗余而冗余”。

- **合理冗余案例：**
  - 订单表（`orders`）冗余`user_name`（用户名）：查询 “用户订单列表” 时，无需关联用户表（`users`），直接从订单表获取用户名，减少 JOIN；**（so 平台设计不能频繁更新 name ？）**
  - 会话表（`im_conversation`）冗余`last_msg_content`（最后一条消息内容）：展示会话列表时，无需关联消息表，提升查询速度。
- **❌ 过度冗余案例：**
  - 订单表冗余`user_address`（用户地址）：用户地址会频繁修改（如用户改收货地址），若冗余到订单表，需同步更新所有关联订单的地址，否则数据不一致；
  - 商品表冗余`category_name`（分类名称）：分类名称修改时，需同步更新所有商品的`category_name`，维护成本极高。

**判断标准**：若冗余字段 “不常修改”（如用户名、商品名称），且 “高频查询”，则可冗余；若字段 “频繁修改”（如地址、库存），则必须关联查询（通过外键关联表）。

**对于外键**： **逻辑外键 or 物理外键**

外键的核心作用是 “保证关联数据的一致性”，但 MySQL 的物理外键有性能问题，实战中更推荐 “逻辑外键”：

| 外键类型                    | 优点                                                         | 缺点                                                         | 适用场景                                         |
| --------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------ |
| **物理外键（FOREIGN KEY）** | ❶ 数据库自动保证一致性（如删除用户时，自动删除关联订单）；❷ 无需应用层处理关联逻辑。 | ❶ 性能差（删除主表数据时，需扫描多表，高并发下卡顿）；❷ 灵活性低（如需要保留历史订单，不能随用户删除）。 | 低并发、**强一致性**场景（如财务表、权限关联表） |
| 逻辑外键（字段关联）        | ❶ 性能高（无数据库层面的约束检查）；❷ 灵活性高（如用户删除后，订单的`user_id`保留，用于历史查询）。 | ❶ 需应用层处理一致性（如删除用户前，先检查是否有关联订单）；❷ 可能出现脏数据（如`user_id=999`不存在，但订单表有该`user_id`）。 | 高并发、弱一致性场景（如订单表、商品表）         |

**实战技巧**：用 “逻辑外键 + 定期校验” 平衡性能与一致性 —— 如每天凌晨执行脚本，检查订单表的`user_id`是否在用户表存在，若不存在则标记为 “异常数据”，人工处理。

##### 索引设计

MySQL 支持多种索引类型，需根据 **“查询场景”** 选择： （**业务驱动）**

索引不是 “凭感觉加”，而是 “根据高频查询 SQL 设计”

- **收集高频 SQL**：通过 MySQL 的`slow_query_log`（慢查询日志）或监控工具（如 Prometheus），收集 Top 20 的高频查询 SQL；
- **分析 SQL 的 WHERE 条件 + 排序字段**：索引需覆盖 “WHERE 条件的过滤字段” 和 “ORDER BY 的排序字段”；
- **避免冗余索引**：若已有联合索引`(a,b)`，则无需再建`(a)`的普通索引（`(a,b)`已包含`(a)`的查询能力）。

### 索引避坑：这 5 类索引绝对不能加

1. **低频查询的字段**：如 “用户表的`last_login_time`（最后登录时间）”，若每月只查 1 次，加索引会浪费存储和写入性能；
2. **区分度低的字段**：如 “订单表的`is_deleted`（删除标志，0 = 未删除，1 = 已删除）”，区分度只有 2（**也就是数据大量重复的**），加索引后查询仍需扫描大量数据，效率不如全表扫描；
3. **更新频繁的字段**：如 “商品表的`stock`（库存）”，每秒更新 100 次，每次更新需维护索引，导致写入性能下降；
4. **冗余索引**：如已有联合索引`(a,b,c)`，再建`(a)`或`(a,b)`的索引，属于冗余，浪费存储；
5. **大文本字段**：如 “商品表的`description`（商品描述）”，加索引会占用大量空间，且查询效率低（建议用全文搜索引擎如 Elasticsearch）。’

##### 特殊场景

> **分表分库：应对千万级 + 数据**

当单表数据量超过 “千万级”（MySQL）或 “亿级”（MongoDB），查询会变慢，需分表分库：

- **分表策略**
1. 水平分表（按数据行拆分）：如订单表按`user_id`哈希分表（`orders_0`到`orders_63`），同一个用户的订单在同一个表；
  2. 垂直分表（按字段拆分）：如用户表拆分为`users`（高频字段：`user_id`/`phone`/`password`）和`user_profiles`（低频字段：`avatar`/`signature`）。

- **分库策略**：按业务模块分库（如`user_db`（用户库）、`order_db`（订单库）、`product_db`（商品库）），避免单库压力过大。

**工具推荐**：MySQL 用 Sharding-JDBC（轻量级，无中间件），MongoDB 用原生分片集群（无需手动分表）。

> **软删除：保留历史数据，避免误删**

核心业务表（如订单表、用户表）不要用 “物理删除”（`DELETE`语句），而是用 “软删除”（加`is_deleted`字段）：

- 表结构加字段：`is_deleted` TINYINT NOT NULL DEFAULT 0（0 = 未删除，1 = 已删除）；
- 查询时过滤：`SELECT * FROM orders WHERE user_id = 123 AND is_deleted = 0`；
- 删除时更新：`UPDATE orders SET is_deleted = 1 WHERE order_id = 456`。

**优势**：

- 避免误删：删除后可通过`UPDATE is_deleted = 0`恢复；
- 保留历史数据：用于数据分析（如统计 “历史订单总量”）；
- 性能高：`UPDATE`比`DELETE`快（`DELETE`会触发索引维护和事务日志，且可能导致碎片）。

---



> **历史数据归档：减轻主表压力**

对于 “只查近 3 个月数据，历史数据很少查” 的场景（如日志表、订单表），需定期归档历史数据：

- 方案：
  1. 新建归档表：如`orders_history_2023`（存储 2023 年的订单）；
  2. 定时任务：每月 1 号将上月的订单从主表（`orders`）迁移到归档表，迁移后删除主表的历史数据；
  3. 历史查询：应用层判断查询时间，若查 2023 年数据则查归档表，否则查主表。

**优势**：主表数据量保持在 “百万级”，查询和写入性能稳定，归档表可存储在低成本的存储介质（如冷备份磁盘）。

import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.Update;
import org.springframework.stereotype.Service;
import javax.annotation.Resource;
import java.util.Date;
import java.util.List;

@Service
public class ImConversationService {
    // 注入 Spring 自动配置的 MongoTemplate（无需手动创建）
    @Resource
    private MongoTemplate mongoTemplate;

    // 1. 新增会话（插入文档）
    public void addConversation(ImConversation conversation) {
        conversation.setCreateTime(new Date());
        conversation.setUpdateTime(new Date());
        // insert：插入新文档（若主键已存在会报错）；save：新增或更新（主键存在则更新）
        mongoTemplate.insert(conversation);
    }

    // 2. 查用户的会话列表（按更新时间倒序，过滤已删除）
    public List<ImConversation> getUserConversationList(Long userId) {
        // 构建查询条件：userId = 传入值，且未删除（假设加了 isDeleted 字段）
        Criteria criteria = Criteria.where("userId").is(userId)
                .and("isDeleted").is(0);
        // 构建查询：条件 + 排序（updateTime 倒序）
        Query query = Query.query(criteria)
                .with(Sort.by(Sort.Direction.DESC, "updateTime"));
        // 执行查询：返回集合列表（自动映射为 ImConversation 对象）
        return mongoTemplate.find(query, ImConversation.class);
    }

    // 3. 更新会话的未读计数（原子更新）
    public void updateUnreadCount(String conversationId, Long userId, int incCount) {
        // 条件：conversationId 和 userId 匹配（定位唯一会话）
        Query query = Query.query(Criteria.where("conversationId").is(conversationId)
                .and("userId").is(userId));
        // 更新：未读计数 += incCount（incCount 可正可负，如 +1 新增未读，-1 减少未读）
        Update update = Update.update("unreadCount", Inc.inc("unreadCount", incCount))
                .set("updateTime", new Date()); // 同步更新时间
        // 执行更新：upsert=false（不存在则不插入）
        mongoTemplate.updateFirst(query, update, ImConversation.class);
    }

    // 4. 复杂查询：聚合统计（查用户的群聊数量）
    public long countGroupConversation(Long userId) {
        // 聚合管道：1. 过滤 userId 和 type=2（群聊）；2. 统计数量
        Aggregation aggregation = Aggregation.newAggregation(
                Aggregation.match(Criteria.where("userId").is(userId)
                        .and("type").is(2)), // 条件过滤
                Aggregation.count().as("groupCount") // 统计别名
        );
        // 执行聚合：返回统计结果（映射为自定义 DTO）
        AggregationResults<CountDTO> results = mongoTemplate.aggregate(
                aggregation, "conversations", CountDTO.class);
        // 处理结果（无数据时返回 0）
        return results.getUniqueMappedResult() == null 
                ? 0 : results.getUniqueMappedResult().getGroupCount();
    }

    // 统计结果 DTO
    @Data
    public static class CountDTO {
        private long groupCount;
    }
}

**关于索引**：

| 索引                                    | 类型          | 潜在问题                                                     | 建议                                                         |
| --------------------------------------- | ------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| `{ conversation_id: 1, send_time: -1 }` | 复合索引      | 成本中等：写入消息时需更新索引（可接受）。                   | ✅ 保留（核心读路径，收益远大于成本）。                       |
| `{ conversation_id: 1, read_users: 1 }` | ⚠️复合多键索引 | ❌ **高危索引**：`read_users` 是数组，MongoDB 会为每个数组元素建多键索引，群聊中数组大（上百上千人）→ 索引爆炸。 | 🚫 建议删除。改为**拆表设计**：`message_read_status(message_id, user_id, read_time)`。 |
| `{ msg_id: 1 }`                         | 唯一索引      | 写成本低；必要性强。                                         | ✅ 保留。                                                     |

- 群聊中 `read_users` 可能包含成百上千个用户 ID，数组过大会导致索引膨胀；
- 优化方案：群聊消息的 “已读状态” 可拆分到单独集合（如 `message_read_status`），避免 `messages` 集合过大。

### ==离线消息处理策略(推拉结合)==





### 消息推送 VS 消息拉取

| 对比维度       | 消息推送（Server Push）                                      | 消息拉取（Client Pull）                                      |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **触发方**     | 服务器：消息到达后，立即推送给在线用户                       | 客户端：定期（如每 30 秒）向服务器发送请求，拉取新消息       |
| **实时性**     | 高（毫秒级）：消息到达服务器后立即推送，无延迟               | 低（取决于拉取间隔）：若间隔 30 秒，最大延迟 30 秒           |
| **服务器压力** | 高：每个在线用户需维持长连接（如 WebSocket），消息量大会导致连接 / 带宽压力 | 低：仅需处理客户端的拉取请求，无长连接维护开销               |
| **客户端压力** | 低：仅需接收消息，无需主动请求                               | 高：需定期发送请求，若拉取间隔短，会消耗客户端流量 / 电量（如手机端） |
| **网络适应性** | 依赖长连接：弱网络（如 2G）下长连接易断开，需重连机制        | 不依赖长连接：基于 HTTP 短连接，弱网络下重试成本低           |
| **技术依赖**   | 需长连接技术（如 WebSocket、MQTT、XMPP）                     | 基于短连接技术（如 HTTP/HTTPS），无需额外协议                |



### ==多端登录处理（踢下线 or ==多端同步==

- **核心诉求**
  1. 安全：防止账号被盗后，攻击者通过多设备操作窃取数据（如资金转移、敏感信息泄露）；
  2. 体验：支持用户在手机、PC、平板等设备间无缝切换（如手机发消息、PC 查历史、平板标记已读）；
  3. 一致性：多设备间的消息、会话状态（如已读、置顶）需实时同步，避免 “一端操作、另一端不同步” 的矛盾。
- **技术挑战**
  1. 会话唯一性 vs 多会话共存：踢下线需保证 “同一账号仅一个有效会话”，多端同步需管理 “多个会话的并发状态”；
  2. 状态同步效率：多端同步时，如何在毫秒级内将 “消息已读”“会话删除” 等操作同步到所有设备，且不占用过多带宽；
  3. 资源消耗控制：多端同步需维持多个设备的长连接（如 WebSocket），如何避免服务器因连接数过多导致性能瓶颈。

### ==引申问题————缓存一致性问题==



**框架自带缓存可了解！！**



读写并发问题 ：  读旧数据在修改之前



**缓存更新用 “Cache-Aside 模式”，避免数据不一致**

- 场景：商品信息更新后，如何同步缓存？
- 错误做法：先更新缓存，再更新数据库（若数据库更新失败，缓存与数据库不一致）；
- 正确做法（Cache-Aside）：
  1. 先更新数据库；
  2. 再删除缓存（而非更新缓存）；
  3. 下次查询时，自动从数据库加载新数据到缓存。
- 理由：删除缓存比更新缓存更简单，且避免 “并发更新导致的缓存脏数据”。

**并发场景：**

- 并发场景下，“查询线程” 先查缓存（miss）→ 查数据库（旧数据）→ 未写入缓存前，“更新线程” 更新数据库并删除缓存→ “查询线程” 再写入旧数据到缓存，导致脏数据。

> 
>
> 但是只有在**修改数据库操作**和**删除缓存**组合在**修改缓存之前**然后还要在**读取到脏数据之后** **才会发生数据不一致**
>
> 而修改数据库要加锁，读取数据库MVCC 所以修改耗时 故修改和删除缓存 中间的时间窗口更大

延迟双删属于“权宜之计”，**依赖经验值、无法保证严格一致性、增加系统复杂度**。

**大 Key 拆分，避免阻塞 Redis 线程**

- 场景：存储一个 100MB 的大 Key（如 `user:100:history` 存储用户所有历史订单），执行 `DEL` 或 `GET` 时，会阻塞 Redis 单线程，导致所有请求超时。
- 解决方案：
  1. 拆分大 Key：将 `user:100:history` 拆分为 `user:100:history:1`、`user:100:history:2`...（按时间或分页拆分）；
  2. 限制 Key 大小：业务层约定单个 Key 不超过 10KB，通过监控（如 Redis Insight）及时发现大 Key。



其他方案：分布式锁、乐观锁（cas 版本号/时间戳）、异步解耦

> **场景 1：单聊消息发送（异步解耦）**

- **痛点**：用户 A 给 B 发消息，需同时做 3 件事：1）存消息到数据库；2）给 B 推推送通知；3）记录 A 的发送日志。同步调用会导致响应慢。
- MQ 方案
  1. A 发送消息后，“消息服务” 只往 RabbitMQ 的 `im_single_chat` 队列发一条消息（含 senderId、receiverId、content）；
  2. 3 个下游服务（存储服务、推送服务、日志服务）分别订阅 `im_single_chat` 队列，异步消费消息；
  3. “消息服务” 无需等待下游处理，立刻返回 “发送成功”，用户体验提升 3-5 倍。





> **场景 2：群聊消息广播（灵活路由）**

- **痛点**：一个 100 人的群，发消息时需给 100 个用户推送，若用 “循环调用推送接口”，会导致服务阻塞。
- MQ 方案
  1. RabbitMQ 的 **Fanout Exchange（广播交换机）**，将群聊消息路由到 100 个 “用户专属队列”（每个用户一个队列）；
  2. 推送服务订阅所有用户队列，异步给每个用户推送消息；
  3. 无需循环调用，通过交换机自动完成 “一对多” 广播，性能提升 10 倍以上。



> **场景 3：早高峰流量削峰（保护下游）**

- **痛点**：早高峰 8:30 消息量突增到 5 万条 / 秒，数据库每秒只能处理 1 万条，直接写入会导致连接超时。
- MQ 方案
  1. MQ 先接住 5 万条 / 秒的消息，持久化到磁盘；
  2. 存储服务通过 “限流消费”（RabbitMQ 的 QoS 机制），每秒只从 MQ 拉取 1 万条消息写入数据库；
  3. 高峰过后，MQ 再逐步释放积压的消息，避免数据库崩溃。

### **RabbitMQ基础概念**

| 组件                      | 核心作用                                                     |
| ------------------------- | ------------------------------------------------------------ |
| **Exchange（交换机）**    | 接收生产者发送的消息，根据 “交换机类型” 和 “路由键”，把消息分发到对应的队列 |
| **Queue（队列）**         | 暂存消息，等待消费者消费；支持持久化（避免消息丢失）         |
| **Routing Key（路由键）** | 交换机根据路由键判断消息该分发到哪个队列                     |

**4 种交换机类型（IM 场景常用）：**

1. **Direct Exchange（直连交换机）**
   - 逻辑：路由键完全匹配才分发（如路由键 “user_123” 只能分到绑定 “user_123” 的队列）
2. **Fanout Exchange（广播交换机）**
   - 逻辑**：忽略路由键，把消息分发到所有绑定的队列；**
3. **Topic Exchange（主题交换机）**
   - 逻辑：支持通配符（`*` 匹配一个词，`#` 匹配多个词），**模糊匹配路由键；**
4. **Headers Exchange（头交换机）**
   - 逻辑：根据消息头（而非路由键）匹配，用得少；

### **消息队列应用场景**



**外卖系统业务痛点**

用户下单后，需触发 5 个操作：

- 生成订单记录（订单服务）；
- 扣减商家库存（库存服务）；
- 发送商家接单通知（通知服务）；
- 推送订单给附近骑手（骑手调度服务）；
- 记录用户消费行为（数据分析服务）。

若用**同步调用**（订单服务依次调用其他 4 个服务）：

- 响应时间 = 5 个服务耗时之和（约 1-2 秒），用户会看到 “下单转圈”；
- 任一服务故障（如骑手调度服务超时），会导致整个下单流程失败；
- 新增操作（如 “发送用户积分通知”）需修改订单服务代码，耦合严重。

**MQ 落地方案**

- **选型**：RabbitMQ（低延迟 < 50ms，支持灵活路由，满足状态流转的精准通知）；
- 交换机 / 队列设计
  - 交换机类型：Direct 交换机（`order_status_exchange`），按 “订单状态” 路由；
  - 队列设计：每个下游服务一个专属队列（`order_create_queue` 库存服务、`merchant_notify_queue` 通知服务、`rider_dispatch_queue` 调度服务）；
  - 路由规则：订单创建消息的 Routing Key = `order.created`，所有绑定该 Key 的队列都会收到消息。
- **流程拆解**
  1. 用户下单，订单服务生成订单 ID 后，往 `order_status_exchange` 发送消息（含订单 ID、状态、商家 ID 等）；
  2. 交换机按 Routing Key 把消息转发到 4 个下游队列；
  3. 各服务异步消费消息：库存服务扣库存、通知服务发推送、调度服务派骑手；
  4. 每个服务处理完成后，往同一交换机发送 “状态更新消息”（如 Routing Key = `order.stock.done`），其他服务按需消费（如订单服务监听该 Key，更新订单状态为 “待接单”）。

---

- **业务痛点**：

  外卖午高峰（11:30-13:00）订单量达 **8000 单 / 秒**，但商家系统的 “接单处理能力” 仅 **2000 单 / 秒**。若直接把 8000 单 / 秒打给商家系统：

  1. 商家系统连接池耗尽，新订单无法接收；
  2. 数据库因高频写入崩溃，订单状态更新失败；
  3. 用户看到 “下单成功” 但商家收不到单，引发投诉。

- **MQ 削峰方案**：

  1. 订单服务将所有高峰订单（8000 单 / 秒）全部发送到 RabbitMQ 的 `takeaway_order_queue`（开启持久化，避免消息丢失）；
  2. 商家系统消费者端设置 **QoS=200**（每次从 MQ 拉取 200 条消息，处理完再拉取下一批），强制将消费速率控制在 2000 单 / 秒；
  3. MQ 暂存超出处理能力的消息（队列长度临时增长，只要磁盘足够就安全），高峰过后（13:00 后）逐步释放，商家系统继续处理直至队列清空。





**外卖订单状态变更解耦**

- **业务痛点**：

  外卖订单有多个状态（下单 → 商家接单 → 骑手取餐 → 送达），每个状态变更需通知多个系统：

  - 下单：通知商家系统、骑手调度系统；

  - 送达：通知用户系统、数据分析系统、财务系统（结算佣金）。

    

    **若用同步调用（订单服务依次调用各系统）：**

  1. 新增系统（如 “售后系统” 需监听 “送达” 状态）时，需修改订单服务代码，重启服务；
  2. 任一系统故障（如财务系统宕机），会导致订单状态更新失败；
  3. 代码充斥大量 “if-else”（判断状态 → 调用对应系统），维护成本高。

- **MQ 解耦方案**：

  1. 订单服务仅负责 “更新订单状态”，状态变更后，往 RabbitMQ 的 `order_status_exchange`（Topic 交换机）发送消息，Routing Key 为 `order.status.${状态}`（如 `order.status.delivered`）；
  2. 各下游系统根据需要，将自己的队列绑定到交换机，设置对应的 Routing Key：
     - 商家系统：绑定 `order.status.created`（只接收 “下单” 状态）；
     - 财务系统：绑定 `order.status.delivered`（只接收 “送达” 状态）；
     - 新上线的售后系统：直接绑定 `order.status.delivered`，无需修改订单服务；
  3. 订单服务完全不感知下游系统，只需发送状态消息即可。

- **效果**：新增 / 删除下游系统无需修改订单服务，服务故障不影响状态更新，代码维护成本降低 60%。



### MQ消息可靠性 （联系TCP报、业务消息可靠性）

RabbitMQ 的**可靠投递机制**就是为了在第①②阶段**给你可观测反馈**

**publisher confirm （**Confirm 只确认到 **Exchange**，不保证一定进入 **Queue**。）



**publisher return**  消息已经成功到达交换机（Exchange），但**找不到任何匹配队列（Queue）**

——— 让**消息不会“默默消失”**，而是**退回给生产者（Producer）**处理。

| 环节         | 问题                       | 举例                       |
| ------------ | -------------------------- | -------------------------- |
| ① 网络层     | 消息还没到 RabbitMQ 就断网 | 程序发了消息但服务未收到   |
| ② 交换机阶段 | 没找到匹配的队列           | RoutingKey 不匹配，被丢弃  |
| ③ 队列阶段   | 队列崩溃或配置不持久       | 消息入队失败或丢失         |
| ④ 消费者阶段 | 消费端异常 / ACK 丢失      | 消息被重复消费或永远未确认 |

**Requeue**

> 目标：**让未正确消费的消息重新回到队列等待下一次投递。**

当消费者在处理消息时：

- 如果返回 `NACK(requeue=true)` 或 `Reject(requeue=true)`；
- MQ 不会丢弃消息，而是**重新放回队列头或尾部**；
- 这时，**消息依然是同一条（可能还是持久化状态）**；
- RabbitMQ 会再次尝试将它投递给同一个或下一个消费者。

⚠️ requeue 的存在与是否持久化无关。
 即便消息是持久化的，也能被 requeue，因为这只是**队列层面的重新分发操作**，不是磁盘层面的写入。

**消息的重复消费问题**：



**幂等性**的目标是解决：

> “网络重试 / MQ 重发 / 接口重复调用 / 系统异常回调”等导致的**重复操作问题**

在分布式系统中，任何一次通信都可能出现：

- **请求超时重试**
- **MQ消息重复投递**
- **支付系统多次回调**
- **事务补偿重复执行**

所以幂等性是系统的“保险机制”，
 保证**无论系统如何抖动、消息如何重复，结果始终一致**。

**消息堆积如何解决**：

当生产者发送消息的速度超过了消费者处理消息的速度，就会导致队列中的消息堆积，直到队列
存储消息达到上限。之后发送的消息就会成为死信，可能会被丢弃，这就是消息堆积问题。

> 解决消息堆积有三种种思路：

- 增加**更多消费者**，**提高消费速度**
- 在消费者内开启**线程池****加快消息处理速度**
- **扩大队列容积**，提高堆积上限



rabbitMQ 的内存模型是典型的 **内存优先** 策略：

- 普通队列在投递消息时，会**立即将消息保存在内存中**；
- 如果消息太多，内存会快速被占满，引发流控（memory alarm）甚至性能抖动。

惰性队列的设计目标是：

> ✅ 减少内存压力，支持百万级消息堆积而不崩溃。

**关键点：**

- 消息写入后**直接落盘**；
- 消费时再**按需加载到内存**；
- 支持**流控稳定**、**高消息堆积量**；
- 吞吐量略低于普通队列（因为多了磁盘 IO）。
