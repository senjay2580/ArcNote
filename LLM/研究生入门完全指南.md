## 一、核心概念科普

### Token（词元）

大模型不直接读"文字"，它把文字切成一个个小块，每个小块叫一个 Token。

```
"我爱自然语言处理" → ["我", "爱", "自然", "语言", "处理"]  （5 个 Token）
"Hello world"      → ["Hello", " world"]                    （2 个 Token）
```

- 中文大约 1 个字 ≈ 1-2 个 Token
- 英文大约 1 个单词 ≈ 1-2 个 Token
- 模型的输入输出长度都用 Token 来衡量（比如"支持 128K 上下文"就是 128000 个 Token）

### 预训练（Pre-training）

让模型读海量文本（几万亿个 Token），学会"语言是怎么回事"。

```
训练方式：给模型看一句话的前半部分，让它预测下一个词
  输入："今天天气很"
  模型猜："好"（对了！参数微调一下，记住这个模式）

重复几万亿次 → 模型学会语言规律、世界知识、推理能力
```

- 预训练需要**海量数据 + 海量算力**（几千张 GPU 跑几个月）
- 这一步一般由大公司完成（OpenAI、Meta、阿里等）
- 预训练完的模型叫"基座模型"（Base Model），能续写文本但还不会对话

### 微调（Fine-tuning）

在预训练好的模型基础上，用**少量特定数据**再训练一下，让模型学会特定任务。

```
预训练模型：什么都知道一点，但不会按你的要求回答
     ↓ 用对话数据微调
微调后模型：能按指令回答问题、像 ChatGPT 一样对话

预训练模型：通用能力
     ↓ 用医疗数据微调
微调后模型：变成医疗领域专家
```

- 微调数据量远小于预训练（几千到几万条就够）
- 微调所需算力也远小于预训练（一两张显卡就能做）
- **这是大部分硕士做的事情**

### LoRA（Low-Rank Adaptation，低秩适配）

一种**高效微调**方法。不修改原模型的全部参数，只额外加一小部分可训练的参数。

```
原始模型：70 亿参数（全都冻住，不动）
LoRA 额外加的参数：几百万（只训练这些）

好比：你不重新造一辆车，只是换了方向盘和座椅
效果接近全量微调，但成本低 10-100 倍
```

**LoRA 的 rank**：控制额外参数的多少。rank 越大 → 额外参数越多 → 学习能力越强 → 但也越吃显存。常用值：8、16、32、64。

**QLoRA**：LoRA + 量化。把模型压缩到 4-bit 精度再做 LoRA，一张消费级显卡（如 RTX 4090 24G）就能微调 7B 甚至 13B 的模型。

### 超参数（Hyperparameter）

模型训练中**需要人为设定**的参数，不是模型自己学出来的。

```
超参数（你设的）         vs    模型参数（模型自己学的）
├── 学习率 = 2e-4              ├── 神经网络的权重
├── batch size = 8             ├── 注意力矩阵
├── epoch = 3                  └── ... 几十亿个数字
├── LoRA rank = 16
└── warmup steps = 100

类比：
超参数 = 你设定烤箱的温度和时间
模型参数 = 蛋糕里面的面粉鸡蛋怎么混合（由烤箱自己完成）
```

常见超参数解释：

| 超参数 | 含义 | 通俗理解 |
|--------|------|---------|
| learning rate（学习率） | 每次更新参数的步长 | 步子太大容易翻车，太小走不动 |
| batch size | 每次训练喂多少条数据 | 一次看几道题再对答案 |
| epoch | 整个数据集过几遍 | 这本书你要读几遍 |
| warmup steps | 训练初期缓慢增大学习率 | 刚开始慢慢热身，别一上来就冲刺 |
| weight decay | 参数衰减，防止过拟合 | 防止模型"死记硬背" |
| max_length | 输入序列的最大 Token 数 | 一次最多看多长的文本 |

### 可训练参数（Trainable Parameters）

模型中**会在训练过程中被更新**的参数。

```
全量微调：所有 70 亿参数都可训练（需要大量显存）
LoRA 微调：冻结 70 亿参数，只训练额外加的几百万参数

可训练参数越多 → 学习能力越强 → 但需要的显存和时间也越多
```

### Prompt（提示词）

你输入给大模型的文本指令。

```
简单 Prompt："翻译成英文：今天天气很好"
复杂 Prompt："你是一个资深的 Python 开发者，请帮我审查以下代码，
             指出潜在的 bug 和性能问题，并给出修改建议：..."

Prompt Engineering（提示词工程）= 研究怎么写 Prompt 才能让模型给出最好的回答
```

这是**最容易上手**的大模型研究方向之一，不需要训练模型，只需要设计好输入。

### 上下文（Context）

模型在生成回答时能"看到"的所有信息。

```
┌──────────────────────────────────┐
│  系统提示（System Prompt）        │  ← "你是一个医疗助手"
│  历史对话                         │  ← 之前聊了什么
│  当前问题                         │  ← 你现在问的
│  检索到的文档（如果用了 RAG）      │  ← 额外补充的知识
└──────────────────────────────────┘
           ↑ 这些全部就是"上下文"

上下文窗口 = 模型一次能看多少 Token
GPT-4: 128K tokens ≈ 一本小说的长度
```

上下文越长 → 模型能参考的信息越多 → 但推理成本也越高。

### 其他你会频繁遇到的术语

| 术语 | 一句话解释 |
|------|-----------|
| **Inference（推理）** | 模型生成回答的过程（不是训练，是用） |
| **Base Model（基座模型）** | 预训练完、还没微调的模型 |
| **Chat Model（对话模型）** | 微调过、能对话的模型 |
| **RLHF** | 用人类反馈强化学习，让模型回答更符合人类偏好 |
| **RAG** | 检索增强生成，让模型先搜索知识库再回答，减少幻觉 |
| **Embedding（嵌入）** | 把文本转换成数字向量，方便计算相似度 |
| **Hallucination（幻觉）** | 模型一本正经地胡说八道 |
| **Benchmark（基准测试）** | 标准化的测试集，用来评估模型能力 |
| **SOTA** | State of the Art，当前最好的结果 |
| **Ablation Study（消融实验）** | 逐个去掉模块，看哪个模块真正有用 |
| **Baseline（基线）** | 对比的基础模型/方法，用来证明你的方法更好 |
| **Zero-shot / Few-shot** | 不给示例 / 给几个示例就让模型做任务 |
| **量化（Quantization）** | 压缩模型精度（32bit→4bit），牺牲一点效果换取大幅降低显存 |

## 二、需要的数学基础

### 你在哪个层次决定你需要多少数学

```
理论层（数学很重）：设计新架构、证明收敛性   ← 博士/顶会研究者
工程层（代码为主）：训练、微调、部署优化      ← 大部分硕士在这里
应用层（最好上手）：用大模型解决具体问题      ← 建议从这里开始
```

### 必须知道的（入门够用）

| 概念 | 用在哪 | 理解到什么程度 |
|------|--------|---------------|
| 矩阵乘法 | Attention 计算 | 知道两个矩阵怎么乘就行 |
| 概率 / softmax | 模型输出 | 知道"把一组数字变成概率分布，加起来等于 1"就行 |
| 损失函数（Loss） | 训练过程 | 知道"衡量模型预测和正确答案的差距，越小越好"就行 |
| 梯度下降 | 参数更新 | 知道"沿着坡往下走，找到最低点"就行 |

### 进阶（做到工程层需要）

| 概念 | 用在哪 |
|------|--------|
| 线性代数（矩阵分解、特征值） | 理解 Attention、LoRA 的低秩分解 |
| 概率论（条件概率、贝叶斯） | 理解语言模型的生成过程 |
| 信息论（交叉熵） | 理解损失函数的设计 |
| 微积分（偏导、链式法则） | 理解反向传播 |

### 学习建议

- **不要提前系统学**，遇到不懂的再查
- 推荐资源：3Blue1Brown 的线性代数/微积分视频（B站有搬运，每集 15 分钟，讲得极好）
- 公式看不懂就看代码实现，代码比公式直观

## 三、研究生三年全流程

### 时间线

```
研一上（当前）── 上课为主，修学分，读论文，确定大致研究方向
    │            参加组会，听师兄师姐汇报，学习怎么做研究
    │            【目标】搞清楚你的方向在做什么，能读懂论文
    │
研一下 ──────── 课程收尾，开题（写开题报告，确定毕业论文选题）
    │            开始做第一个小实验/项目
    │            【目标】确定选题，跑通基础实验
    │
研二上 ──────── 核心科研期，全力做实验、出结果
    │            争取投一篇论文（会议或期刊），中期检查
    │            【目标】有实验结果，最好有一篇论文在投
    │
研二下 ──────── 继续科研 + 补实验，论文返修/投稿
    │            开始考虑实习/就业（如果不读博）
    │            【目标】论文接收或在投，实验基本完成
    │
研三上 ──────── 写毕业论文（大论文，5-8 万字），秋招找工作
    │            【目标】毕业论文初稿完成，工作签约
    │
研三下 ──────── 毕业论文定稿 → 送审（盲审）→ 答辩 → 离校
                 【目标】顺利毕业
```

### 关键时间节点

| 节点 | 时间 | 说明 |
|------|------|------|
| 选导师 | 入学前/研一初 | 决定你三年跟谁，最重要的选择 |
| 修课 | 研一全年 | 修够学分即可，别花太多精力 |
| 开题 | 研一下/研二初 | 确定毕业论文选题，需要答辩通过 |
| 中期检查 | 研二 | 学校检查你的进度 |
| 投小论文 | 研二前后 | 很多学校毕业要求至少 1 篇 |
| 写大论文 | 研三上 | 毕业论文，最耗时 |
| 送审（盲审） | 研三下 3-4月 | 匿名送校外专家评审，决定能否答辩 |
| 答辩 | 研三下 5-6月 | 通过就毕业 |

### 盲审是什么

- **盲审**：论文上隐去姓名、导师、学校信息，评审专家不知道是谁写的，防止"人情分"
- **外审**：评审专家是校外的，不是你们学校自己人，更客观
- 一般送 2-3 个专家，每人独立打分 + 写评审意见

| 盲审结果 | 后果 |
|---------|------|
| 全部通过 | 正常进入答辩 |
| 有修改意见 | 改完再答辩（小问题影响不大） |
| 有专家给"不通过" | 延期毕业，改完重新送审 |

## 四、实验室日常

### 一天大概长这样

```
上午  ：读论文 / 看技术博客 / 复现别人的代码
下午  ：写代码 / 跑实验 / 调参 / 等结果
晚上  ：整理实验记录 / 准备组会 / 继续跑实验
（模型训练经常挂着跑一晚上，第二天来看结果）
```

不是每天都很忙，也不是每天都有产出，**很多时候就是在反复试错**。

### 具体在干什么

#### 1. 读论文（占比 20-30%）
- 研一读得多，研二边做边读
- 不是从头读到尾，大部分时候是**带着问题找论文**
- 比如："别人怎么解决数据不够的问题？" → 搜论文 → 找到方法 → 试试看

#### 2. 写代码 + 跑实验（占比 40-50%）

这是大头。所谓"跑模型"和"调参"，具体是这样的：

```
第 1 步：准备数据集（清洗、格式转换、划分训练/测试集）
第 2 步：写训练代码（或改别人的开源代码）
第 3 步：启动训练 → 等（几小时到几天）
第 4 步：看结果 → 不好 → 改参数/改方法 → 回到第 3 步
      ↑                                    |
      └────────── 循环很多轮 ←──────────────┘
```

#### 为什么训练要等这么久？

```
模型参数量大 → 每一步计算量巨大

比如微调一个 7B（70亿参数）的模型：
- 每条数据要经过 70 亿个参数的正向计算
- 然后反向传播更新参数
- 然后来下一条数据
- 5000 条数据 × 3 个 epoch = 要算 15000 次

即使 GPU 每秒算万亿次，70 亿参数 × 15000 次 = 依然要好几个小时

全量预训练就更夸张了：几万亿 Token × 几百亿参数 = 几千张 GPU 跑几个月
```

| 任务 | 大概耗时 |
|------|---------|
| 微调 7B 模型（LoRA） | 几小时 |
| 全量微调 7B | 几天 |
| 预训练一个小模型 | 几天到几周 |
| 跑一组对比实验（5-6 个配置） | 一两天 |

等的时候不是干坐着，一般是**同时读论文、写代码、准备下一组实验**。

#### 3. 标数据

人工给数据打标签，让模型知道"正确答案"是什么。

```
文本分类任务：
  "这家餐厅太难吃了" → 标签：负面
  "服务很好，下次还来" → 标签：正面

指令微调数据：
  输入："帮我总结这篇文章"
  输出：（人工写一个高质量的总结）
```

标数据是最枯燥但最重要的工作之一。数据质量直接决定模型效果。**研一帮师兄师姐标数据是常事，别嫌烦，这是在积累对数据的理解。**

#### 4. 跑对比实验

验证你的方法比别人的好，需要公平对比：

```
相同数据集 + 相同评价指标下：
  Baseline A（别人的方法1）：准确率 75%
  Baseline B（别人的方法2）：准确率 78%
  你的方法：              准确率 82%  ← 你要证明这个
```

论文里的"实验"章节，主要就是展示这些对比结果。

#### 5. 组会

每周或双周一次，两种形式：

- **论文分享**：轮到你就讲一篇最新论文给大家听
- **进展汇报**：讲你这周做了什么实验、结果怎样、下一步计划

组会是导师了解你进度的主要渠道。每次都认真准备，哪怕进展不大也要如实说。

### 研一的你，现在的日常应该是

1. **上课**（这学期的主要任务）
2. **参加组会**，听别人讲，学习怎么汇报
3. **读论文**，每周 1-2 篇
4. **学工具**：Python、PyTorch、Linux 命令、Git
5. **跟一个师兄师姐**，帮忙打杂也行，看他们怎么做研究

**不要急着独立做课题。先观察、模仿、打基础。**

### 为什么一定要用虚拟环境？真实坑点

不用虚拟环境（Conda）会遇到的真实灾难：

**坑 1：版本冲突，两个项目互相打架**
```
项目 A（复现 2023 年论文）需要：torch==1.13, transformers==4.28
项目 B（跑最新模型）需要：torch==2.1, transformers==4.36

你在全局装了项目 B 的版本 → 回去跑项目 A → 直接报错崩溃
你降级回项目 A 的版本 → 项目 B 又跑不了了

用 Conda：两个环境完全隔离，互不影响
```

**坑 2：系统 Python 被搞坏**
```
直接在系统 Python 里 pip install → 某个包覆盖了系统依赖
→ Ubuntu 的软件管理器崩了 / 终端打不开了 / 各种奇怪报错
→ 最终重装系统

用 Conda：搞坏了只需要删掉这个环境重建，系统完全不受影响
```

**坑 3：复现论文复现不出来**
```
作者说他的代码能跑，你一跑就报错
原因：你的 numpy 版本是 1.24，他的是 1.21，某个 API 在新版本中被移除了

用 Conda：conda create -n paper_x python=3.9
         pip install -r requirements.txt  # 严格安装作者指定的版本
```

**坑 4：毕业前代码跑不起来**
```
你研二跑出了结果，研三写论文时想补一个实验
结果一年过去了，环境被其他项目改得面目全非，代码跑不起来了
你甚至不记得当时装的是什么版本

用 Conda：每个项目一个环境，`conda env export > environment.yml` 保存配置
         随时可以精确还原
```

## 七、论文怎么读

### 推荐的第一篇论文

**Attention Is All You Need (2017)** — Transformer 架构原文，整个大模型时代的地基。

论文地址：https://arxiv.org/abs/1706.03762

### 零基础读论文的方法（不要直接硬啃原文）

```
第 1 步（1-2天）：看视频讲解
  └── 李沐《Transformer 论文逐段精读》（B站）

第 2 步（半天）：看图解博客
  └── The Illustrated Transformer（搜"图解 Transformer Jay Alammar"）

第 3 步（1-2天）：回头读原文
  └── 重点看 Section 3（模型架构）和 Section 3.2（Attention 机制）
  └── 读不懂的公式跳过，抓住主干思路

第 4 步（可选）：跑代码
  └── The Annotated Transformer（Harvard NLP 带注释的 PyTorch 实现）
```

### 需要搞懂的核心概念

1. **Self-Attention**：每个词如何"看到"句子里的其他词
2. **Multi-Head Attention**：为什么要多个注意力头
3. **Positional Encoding**：Transformer 如何知道词的顺序
4. **Encoder-Decoder 结构**：输入和输出是怎么连接的

### 后续阅读顺序

```
Transformer (2017) ← 你在这里
    ↓
GPT-1 (2018) — 只用 Decoder，自回归生成
    ↓
BERT (2018) — 只用 Encoder，预训练+微调范式
    ↓
GPT-2 / GPT-3 (2019/2020) — Scaling Law，大力出奇迹
    ↓
InstructGPT (2022) — RLHF，对齐技术
    ↓
LLaMA / Qwen 系列 — 开源大模型实践
```

### 日常读论文的方法

大部分论文不需要精读，快速浏览即可：

```
5 分钟：读 Abstract + Introduction 最后一段（贡献总结）
10 分钟：看所有图表 + 实验结果
15 分钟：扫 Method 部分的大标题和关键公式
→ 判断这篇和你有没有关系 → 有关系再精读
```

## 八、做实验避坑

### 通用大坑

**坑 1：环境配置地狱**
```
你以为：pip install torch 就能开始
实际上：CUDA 版本不对、PyTorch 版本不兼容、某个库依赖冲突……

避坑：
- 每个项目用独立的 conda 环境
- 复现别人代码时严格按 requirements.txt 装指定版本
- 记录你的环境配置：conda env export > environment.yml
```

**坑 2：实验不记录**
```
跑了 30 组实验，三天后想不起来哪组参数对应哪个结果。

避坑：
- 用 wandb 自动记录
- 最差也要建一个表格，每跑一次记一行
- "可以是无效实验，但不能是盲目实验"
```

**坑 3：不设随机种子**
```python
# 代码开头永远加这几行
import torch, numpy as np, random

def set_seed(seed=42):
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)

set_seed()
```

**坑 4：一上来就跑大模型**
```
显卡爆了 → OOM → 改参数 → 又爆了 → 一天过去了

避坑：先用小数据集 + 小模型验证思路，确认可行再上规模
```

**坑 5：盲目堆模块**
```
看论文用了注意力 → 加！看到对比学习 → 也加！结果又大又慢效果还差。

避坑：
- 每次只改一个变量
- 改之前想清楚"为什么这个模块能有用"
- 说不出理由的改动大概率是浪费时间
```

**坑 6：数据预处理不一致**
```
训练集做了归一化，测试集忘了 → 结果全是假的

避坑：训练集和测试集的预处理步骤必须完全一致
```

### 大模型方向特有的坑

**坑 7：算力不够硬选大课题**
```
避坑：
- 入学就搞清楚实验室有什么显卡、能用多少
- 算力不够就做微调、做应用、做评测
- 善用免费/低价资源：Google Colab, AutoDL, 各种云平台学生优惠
```

**坑 8：什么都想学**
```
今天 RAG，明天 Agent，后天多模态，一个月过去什么都没深入。

避坑：和导师确定一个具体方向，深扎下去。LLM 更新太快，不可能全跟。
```

**坑 9：选开源代码不看 star 和 issue**
```
找了个 GitHub 项目复现，结果一堆 bug 没人维护。

避坑：
- 优先选 star 数高、近期有更新、issue 有人回复的项目
- 没有开源代码的论文方法，自己复现很可能达不到论文效果
```

## 九、写论文避坑

### 结构与内容

| 坑 | 真实后果 | 怎么避 |
|----|---------|--------|
| 选题太大 | 泛泛而谈没深度 | 选一个小而具体的切入点 |
| 文献综述写成流水账 | 审稿人直接拒 | 要有批判性分析："A方法好在哪，但缺什么，所以我做什么" |
| 摘要和结论重复 | 审稿人觉得凑字数 | 摘要讲"做了什么+结果"，结论讲"意义+局限+未来" |
| 参考文献质量低 | 论文档次直接降一级 | 引用顶会顶刊（ACL/EMNLP/NeurIPS/AAAI），少引博客 |
| 图表不清晰 | 盲审扣分 | 图用矢量格式，表格对齐工整 |
| 讨论部分太薄 | 缺乏深度 | 分析为什么有效、局限是什么、和已有工作的对比 |

### 写作习惯

- **改好一章就发给导师**，不要全写完再发
- **先定稿小论文再写毕业论文**，否则导师改了小论文你毕业论文也得重写
- **至少提前半年开始写毕业论文**，临时抱佛脚写不出合格的东西
- 避免口语化表达，不要大量使用"我们"、"然后"
- 格式统一：字体、图表大小、参考文献格式必须一致

### 投稿相关

- 大模型方向主要的顶会：ACL、EMNLP、NAACL、NeurIPS、ICML、ICLR、AAAI
- 投稿前让导师和师兄帮忙审一遍
- 被拒了不要灰心，**大部分论文都经历过被拒**，改了再投

## 十一、时间管理

### 避免虚假勤奋

> 很多人以为自己每天学了 10 小时，实际追踪发现有效时间不到 2 小时。

- 用番茄钟（25分钟专注 + 5分钟休息）
- 每天只定 2-3 个核心任务，完成就行
- 状态不好就去运动/休息，不要坐在那里耗

### 倒推规划

```
现在就做一个粗略规划：

毕业目标：X 篇论文 + 毕业论文通过
  ← 研三上：写毕业论文
  ← 研二下：论文投稿/返修
  ← 研二上：核心实验出结果
  ← 研一下：开题 + 第一个实验
  ← 研一上（现在）：打基础 + 选方向
```

### 关键原则

- **所有事提前做**：做在前面你就有很多主动权
- **明确毕业去向再倒推**：要工作就找对口实习，要读博就积累论文
- **保持匀速前进**：不要一时冲劲猛干然后长时间懈怠
- **合理设定边界**：不要把时间全部交给导师，你有自己的规划

## 十四、2025-2026 AI 前沿研究方向（与 RAG 项目强相关）

> 适合有 RAG 知识库项目基础（如 MutiExpert 多专家系统）的同学，直接延伸到研究生课题。

### 1. GraphRAG — 图增强检索（最热，强推）

**核心思路**：把知识图谱（KG）和向量检索结合，解决传统 RAG 多跳推理弱、关系感知差的问题。

```
传统 RAG：问题 → 向量检索 → 召回文本片段 → LLM 生成
GraphRAG：问题 → 实体抽取 → 图遍历/子图检索 + 向量检索 → 融合 → LLM 生成
```

**你能做的扩展**：在现有 RAG 上加 Neo4j/NetworkX 知识图谱，实体走图检索，语义走向量，两路 RRF 融合。

**复试说法**：
> "在多专家系统中，不同专家间的知识关系用图结构建模，检索时同时走语义路径和关系路径，解决了传统 RAG 多跳推理弱的问题。"

| Awesome-GraphRAG 论文集 | https://github.com/DEEP-PolyU/Awesome-GraphRAG |

### 2. 混合检索（Hybrid Retrieval）— 改动最小，提升最快

**核心思路**：向量检索（稠密）+ BM25（稀疏）双路并行，用 RRF 融合，精度比单路高 10-20%。

```
Query
  ├── 向量检索（FAISS/Chroma）→ Top-K 语义相关结果
  └── BM25 关键词检索       → Top-K 精确匹配结果
              ↓
         RRF 融合排序
              ↓
         最终 Top-K 送给 LLM
```

**你现有项目直接可加**：在 RAG 检索层旁边加 rank_bm25，结果用 RRF 融合，改动量小，效果明显。

| 资源 | 链接 |
|------|------|
| RAGFlow（生产级混合检索） | https://github.com/infiniflow/ragflow |
| rank_bm25（Python库） | https://github.com/dorianbrown/rank_bm25 |

---

### 3. LoRA 微调 + RAG 结合 — 复试加分项

**核心思路**：用领域数据对小模型（LLaMA-3/Qwen-2.5）做 LoRA/QLoRA 微调，让模型更懂领域语言，RAG 负责实时知识注入。两者互补：微调解决领域适应性，RAG 解决知识时效性。

```
领域 QA 数据对
      ↓
  LoRA 微调（rank=16, QLoRA 4-bit，一张 4090 可跑 7B 模型）
      ↓
  领域适配小模型
      ↓
  接入 RAG 系统（向量库 + 知识图谱）
      ↓
  效果 > 纯微调 > 纯 RAG
```

**复试说法**：
> "RAG 解决知识时效性，微调解决领域适应性，两者互补。用知识库里的 QA 对做监督微调，让模型更懂领域语言，再用 RAG 补充实时知识。"

| 资源 | 链接 |
|------|------|
| LLaMA-Factory（微调框架） | https://github.com/hiyouga/LLaMA-Factory |
| unsloth（4-bit 高效微调） | https://github.com/unslothai/unsloth |

---

### 4. LLM + 知识图谱问答（LLM-KG4QA）— 减幻觉

**核心思路**：用知识图谱约束 LLM 生成，检查生成内容是否与图谱事实一致，从源头减少幻觉。

```
LLM 生成答案
      ↓
  KG 验证层（实体/关系是否存在于图谱中？）
      ↓
  不一致 → 修正或标注不确定性
```

| 资源 | 链接 |
|------|------|
| LLM-KG4QA 论文+代码 | https://github.com/machuangtao/LLM-KG4QA |

### 5. Agentic RAG（智能体 RAG）— 多专家系统天然契合

**核心思路**：让 LLM 自主决定是否检索、检索什么、要不要再检索一次，而不是每次都固定检索一遍。多专家系统天然是多工具场景，Agent 自主路由到合适专家。

```
用户问题
   ↓
 Agent 判断：是否需要检索？检索哪个专家？
   ├── 需要 → 检索 → 生成 → 是否满意？
   │              ↑_______不满意，换专家再检索
   └── 不需要 → 直接生成
```
