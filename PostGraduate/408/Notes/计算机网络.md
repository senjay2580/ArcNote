序号 / 标识（Sequence Number / Identifier）
干什么用的
给每个数据单元打上"编号"，用于**排序、去重、确认、重组、唯一标识。**



差错控制 = **差错检测 + 差错纠正 / 重传**



**中间节点**（如路由器或交换机）收到报文后，会根据报文首部（比如 IP 头部的目的地址）查询本节点的**转发表**。

转发表记录了“目的地址”到“下一跳节点”的映射关系。

根据转发表，中间节点把报文转发给下一个合适的节点，直到报文到达最终目的地。



- **传统最短路径算法（如 Dijkstra）确实需要全网拓扑信息，才能计算出最短路径。**
- 这意味着如果每个节点都自己单独计算，理论上需要每个节点都掌握整个网络拓扑（所有节点和链路状态）。

<span style="color:#FF0000; font-size:1.4em;">**各种路由协议的不同设计理念**</span>

<span style="color:#FF0000; font-size:1.4em;">链路状态路由协议（Link State Protocol） —— 如 OSPF、IS-IS</span>

- 每个节点通过交换“链路状态广播包”（LSA），把自己知道的邻居信息和链路状态告诉所有节点。
- 因此，**每个节点都拥有整个网络的拓扑图**。
- 每个节点利用 Dijkstra 算法基于这个拓扑图计算自己到各个目的地的最短路径。
- 优点：路径最优，收敛快。
- 缺点：每个节点存储拓扑信息，计算复杂度较高。

> **所以在链路状态协议中，确实存在每个节点存储相同拓扑信息的情况。**

---



**<span style="color:#FF0000; font-size:1.4em;">距离向量路由协议（Distance Vector Protocol） —— 如 RIP</span>**

- 节点只知道自己邻居的距离信息。
- 路由器周期性地向邻居发送自己到各个目的地的距离向量。
- 节点根据邻居传来的距离和自身到邻居的代价计算自己到目的地的距离，逐步更新自己的路由表。
- 这种方式不需要全网拓扑信息，只需要邻居的距离信息。
- 优点：实现简单，开销低。
- 缺点：收敛慢，有环路问题。

> **距离向量协议下，节点不用存整个网络拓扑，只维护距离表，避免了拓扑信息冗余。**

**<span style="color:#FF0000; font-size:1.4em;">混合协议</span>**

- 现代网络中常用的协议（如 EIGRP）结合两者优点，平衡存储和收敛速度。

<span style="color:#FF0000; font-size:1.4em;"> **冗余与现实权衡**</span>

- **链路状态协议确实在每个节点存储了同样的网络拓扑，存在一定冗余。**
   但这样做可以保证每个节点的路径计算一致，避免路由环路，收敛速度快，网络状态透明。
- **距离向量协议减少了冗余，但路由环路和收敛问题较多。**
- **所以网络设计会根据需求选择合适的协议和算法。**

**虚电路交换**将实现可靠性的策略交给了 核心部分这样反而会导致核心部分处理的效率问题 

而交给边缘部分 也就是 客户端 利用客户端的高性能进行处理才比较好

---



**虚电路的问题**

- 虚电路要求**所有分组按序走同一条路径**，核心交换机要维护虚电路表、处理顺序、重传等可靠性逻辑。
- 核心交换机本来就很忙，还要处理这么多复杂的控制逻辑，效率会下降，容易成为瓶颈。

 **互联网（数据报方式）的思路**

- 把可靠性交给**边缘主机（客户端）**，比如 TCP 协议，由发送方和接收方自己处理序号、重传、差错控制。
- 核心路由器只负责**简单的转发**，不用管顺序、不用管重传，效率极高，这就是互联网能支撑海量流量的原因。

**核心原理：木桶效应**

主机 A 给主机 B 发送数据的最高理论速率，由**整条链路中最慢的环节决定**，也就是取三者的最小值：

- 主机 A 的网卡速率：1Gbps（发送方上限）
- 链路带宽：100Mbps（传输通道上限）
- 主机 B 的网卡速率：10Mbps（接收方上限）

三者中，最慢的是**主机 B 的 10Mbps 网卡**，所以 A 发给 B 的最高理论速率只能是 10Mbps。

![image-20250810110610053](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250810110610053.png)

还需要注意 路由器的WAN口最高速率

购买路由器时，一定要注意查看 WAN 口的速率。**WAN 口的速率至少要与你家的宽带速率相等或更高**，否则你将无法享受到宽带服务商提供的全部速度。如果你家办理了 2000Mbps 的宽带，那么一个带有 2.5G WAN 口的 Wi-Fi 6 路由器就是更好的选择。

每种网络技术（如以太网、Wi-Fi）都是一个完整的协议族，它通常包含了对物理层（如何传输比特流）和数据链路层（如何封装成帧、进行寻址和控制）功能的具体实现。这些功能最终由网络设备（如网卡）来完成。

**有线技术**：以太网（不同速率如 10Mbps、100Mbps、1Gbps、10Gbps）、光纤（不同标准如 FDDI、SONET）等。

**无线技术**：Wi-Fi（不同协议如 802.11a/b/g/n/ac/ax）、蓝牙、蜂窝网络（如 4G/5G）等。

**物理介质**：双绞线、同轴电缆、光缆、无线电波等。

在OSI模型中，每一层都有**非常严格**的定义和边界。

- **物理层**：只处理比特流。
- **数据链路层**：只处理帧，以及MAC地址、差错控制等。
- **网络层**：只处理分组，以及IP地址、路由等。

在OSI的严格分层体系下，数据链路层**没有权利和义务**去向上层（网络层）提供一个“通用”的接口。它的职责是**专一**的，它只知道如何处理帧，并与它的上层（网络层）和下层（物理层）进行“点对点”的通信。

数据链路层向**上**提供**直接的服务接口**给网络层（在实际实现中，这个接口往往是固定的“帧格式 + 传输机制”）。





---

**TCP/IP**

**对网络层而言**，它只需要知道把数据包交给网络接口层，而不用关心底层是光纤、双绞线、还是无线电波。网络层只与这个**抽象的接口**打交道。

**对网络接口层而言**，它需要**适配**不同的底层技术。它会根据实际连接的物理介质，自动选择对应的帧封装格式（比如以太网帧、Wi-Fi帧）和物理传输方式（比如电信号、光信号、无线电波）。



一台设备（无论是路由器还是普通电脑）通常有多个网络接口：

- **有线网卡**：`eth0`、`eth1` 等。
- **无线网卡**：`wlan0`。
- **回环接口**：`lo`（用于本机通信）。
- **虚拟接口**：`tun0` 等。



**回环接口是什么？**

- **虚拟接口**：它没有实际的电缆连接，也没有物理的MAC地址。
- **特殊IP地址**：回环接口通常被分配一个特殊的IP地址，最常用的是**127.0.0.1**，这个地址被称为**回环地址（Loopback Address）**。在IPv6中，对应的地址是 `::1`。
- **数据流向**：发送到 `127.0.0.1` 的任何数据包都不会离开本机。数据包会通过网络协议栈的完整处理流程，但当它到达网络层后，会被立即**重定向回传输层**，而不是发送到物理网络上。

两个不同的应用程序，一个作为客户端，一个作为服务器，可以通过 `127.0.0.1` 地址进行数据交换。

这是一种非常便捷的进程间通信方式，因为它可以利用标准化的网络套接字（Socket）接口，而无需编写专门的IPC代码

**<span style="color:#FF0000; font-size:1.4em;">“数据报”是一个通用的术语，根据网络分层的上下文，可能指代不同层的协议数据单元（PDU）</span>**

**应用层报文 → 传输层报文段/数据报**：

> 一个应用层报文（Message） → 在传输层可能被拆分成多个报文段（Segment，例如TCP协议根据MSS分割），或作为一个数据报（Datagram，例如UDP协议）处理。TCP通过MSS（最大报文段长度）控制数据量，避免报文段过大；UDP通常不分割，但若报文过大，可能在网络层分片。

**报文段/数据报 → 网络层分组**：

> 一个传输层报文段（Segment，例如TCP）或数据报（Datagram，例如UDP） → 在网络层封装成一个IP分组（Packet/IP数据报）。通常为1对1关系，但若报文段大小超过MTU（最大传输单元）减去IP头部长度，IP层可能将其分片为多个IP分组（仅在IPv4中常见，IPv6通常要求端到端处理分片）。

**分组 → 数据链路层帧**：

> 一个网络层分组（Packet/IP数据报） → 在数据链路层封装成一个帧（Frame）。通常为1对1关系。如果分组大小超过数据链路层的MTU，网络层（IP层）会先将分组分片为多个较小的IP分片，每个分片封装为一个独立的帧。

**码元（Symbol）**

- 在**数字通信**里，码元是**在传输介质上一次性发出的最小信号单元**。
- 每个码元可以表示 1 位、2 位、3 位……甚至更多比特，具体取决于调制方式。
- **例子**：
  - **二进制调制**（如 NRZ）：1 个码元 = 1 位（0 或 1）。
  - **4 电平调制**（PAM-4）：1 个码元 = 2 位（00、01、10、11）。
  - **8 相位调制**（8-PSK）：1 个码元 = 3 位（000~111）。



**码元宽度（Symbol Duration / Symbol Interval）**

码元宽度指**每个码元占用的时间长度**（秒/码元），是时间上的度量。

信号采样和量化是数字信号处理中的两个关键步骤，把连续的模拟信号转换成离散的数字信号。简单说，就是把模拟信号“切片”成离散的时间点，然后把这些点的幅度转换成数字。

**采样频率（采样率）：**
 是指单位时间内采样的次数，通常用 Hz 表示。采样频率越高，数字信号越接近原始模拟信号。

**量化级数和量化间隔：**
 量化级数越多（比如8位、16位），量化越精细，数字信号还原质量越高。

**编码的自同步能力**

自同步能力指编码信号能够在接收端无需单独时钟信号的情况下，通过信号本身提取时序信息，正确解码数据。通常通过信号中的频繁电平转换（transitions）实现。

**是否浪费带宽**

带宽浪费指编码方式因引入冗余信息或高频转换，导致传输相同数据需要更多带宽



**抗干扰能力**

抗干扰能力指编码信号在噪声（如加性噪声）环境下仍能被正确解码的能力。

曼彻斯特编码因频繁转换，接收端可通过多次采样确认比特，抗噪声能力较强。

NRZI在长序列0或1时，信号持续不变，易受噪声干扰，抗干扰能力弱

![image-20250812170821780](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250812170821780.png)

<span style="color:#FF0000;"> **不同频率的电台 = 不同的中心频率**</span>

- 收音机上的 **87.5 MHz、88.0 MHz、101.3 MHz** 等，就是不同电台的**载波中心频率**。
- 这是你调台时看到的数字，决定了你听到的是哪一家电台。
- 比如：
  - 电台A：中心频率 90.1 MHz
  - 电台B：中心频率 90.3 MHz
     （这两个电台在频谱上间隔 200 kHz）

**<span style="color:#FF0000;">200 kHz 带宽 = 每个电台占据的“频率范围”</span>**

- 虽然电台的中心频率是一个点，但它的信号不是只占那一个点的频率，而是左右各延伸一部分。
- FM 调频广播为了传输立体声、辅助数据等，需要**大约 200 kHz 的频谱宽度**。
- 例如：
  - 电台 90.1 MHz：信号大致占 90.0 MHz ~ 90.2 MHz
  - 电台 90.3 MHz：信号大致占 90.2 MHz ~ 90.4 MHz
     它们刚好紧密排列，中间会留一点保护带，避免互相干扰

于是它会把自己的信号（或者说自己的数据）<strong>发送到这个信道上</strong>。  
我们知道，这些信号的传播是有速度的，对吧？  

比如在某一个时刻，<strong style="color:#2980b9;">A 节点发出的信号</strong>已经传播到某个位置。  

在这种情况下，此时 <strong style="color:#c0392b;">B 节点</strong>尝试监听信道，  

它会发现<strong style="background-color:#f9e79f;">信道是空闲的</strong>，  

因为来自 A 节点的信号还没有到达 B 节点的位置。  

于是 <strong style="color:#c0392b;">B 节点</strong>会误以为信道空闲，  

并且<strong>开始发送自己的数据</strong>。  

随着时间推移，<strong style="color:#2980b9;">A 节点的信号</strong>最终到达 B 节点，  

这时 <strong>B 节点</strong>在发出一小段数据后，  

会<strong style="background-color:#f5b7b1;">检测到冲突并立即停止发送</strong>。  

那么，<strong>B 节点</strong>发出的这小段数据会怎样呢？  

这些信号并不会停在原地，  

它们会<strong>沿着信道继续向前传播</strong>。  

当这部分信号传播到 <strong style="color:#c0392b;">A 节点</strong>位置时，  

A 节点也能<strong style="background-color:#f9e79f;">检测到冲突</strong>，  

并且<strong style="color:#e74c3c;">立即停止发送，随机等待一段时间后重发</strong>。

AP 就类似家用无线路由器的天线

无线网卡收发是通过同一个射频链路完成的

**边发边听**几乎不可能，因为**自己发出的信号功率远大于接收到的信号**，会把接收机“淹没”

就像你在用大喇叭喊话，根本听不到别人说什么





在无线网络中，两个终端可能都能与 AP 通信，但它们之间信号互相听不到

这种情况下，即使两台设备同时发送数据，自己都没听到别人说话，也会产生碰撞

但碰撞发生在 AP 那里，发送方**没法直接检测到**

###  有线以太网（CSMA/CD 适用）

- 网线里是 **双绞线对**，收发是分开的：
  - 一对线专门发送（TX）
  - 一对线专门接收（RX）
- 这样可以**边发边听**，检测碰撞（CD）



### 无线网卡（WLAN）

- **发射机（Transmitter）** 和 **接收机（Receiver）** 通过**同一个射频链路（RF Chain）**连接到天线
- 天线和射频电路是共用的，要么处于发送模式，要么处于接收模式**（半双工）**
- 当你发送信号时，自己的发射功率比接收到的信号**强百万倍以上**，接收机根本没法同时“听”外面的信号（会被自己发射的信号“淹没”）

**操作系统网络栈**，也叫 **TCP/IP 栈** 或 **协议栈**，是操作系统内核中负责处理网络通信的 **一套协议实现和数据处理机制**。

它的核心作用是：

1. **封装与解封装数据**

   - 把应用程序的数据封装成协议格式发送
   - 把收到的网络数据解封装并交给应用程序

2. **协议处理**

   - 负责 IP、TCP/UDP、ARP、ICMP 等协议的解析、校验、路由和流量控制

3. **接口提供**

   - 提供 Socket、Netlink 等接口，让应用程序能够读写网络数据而不用关心物理细节

   

**串行传输（Serial）**：比特一个接一个发送，例如网线上的电信号就是串行比特流。

**并行处理（Parallel）**：计算机内部处理数据是按字节/字（8、16、32、64 位）一次处理多比特



**帧缓冲** = 网卡内存或 DMA 区域，用来临时存储接收到或待发送的数据帧

## 以太网交换机

交换机本质就是多端口的网桥

![image-20250813220915992](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250813220915992.png)

结点之间彼此是知道对方的 MAC 地址的

 

  ![image-20250813222632597](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250813222632597.png)



![image-20250813223053122](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250813223053122.png)



- 以太网是**无连接广播介质**，刚接入时，交换机并不知道某个 MAC 地址对应哪个端口。
- 如果不知道，就只能**泛洪（Flooding）**：把数据帧复制到除入口端口外的所有端口发出去，这样效率低，还可能造成环路广播风暴。
- 有了自学习，交换机能像“快递分拣员”一样，记住每个设备在哪个口，精准转发。

<span style="font-size:1.4em; color:#FF0000;">**核心机制：边转发边学习**</span>

以太网帧结构中，前面有**源 MAC 地址**和**目的 MAC 地址**字段。交换机利用这两个字段做两件事：

**<span style="font-size:1.4em; color:#FF0000;">步骤 1：学习源 MAC 地址</span>**

1. 当交换机**收到一帧数据**时，它先读取该帧的**源 MAC 地址**（Source MAC）和接收到的**物理端口号**。
2. 在 MAC 表中查找：
   - **如果表中没有这个源 MAC**：在 MAC 表中增加一条记录：
      `源 MAC 地址 -> 入口端口`
   - **如果表中已有这个 MAC，但端口不同**：说明设备换了口或拓扑变化，更新对应的端口。
3. 这一步叫**MAC 地址学习**（Learning）。

<span style="font-size:1.4em; color:#FF0000;">**步骤 2：查表决定转发方式**</span>

然后交换机根据**目的 MAC 地址**（Destination MAC）来决定如何处理：

- **如果 MAC 表中有记录**：直接把帧转发到对应端口（单播，Unicast）。
- **如果没有记录**：说明没学过这个 MAC，交换机会对**除了入口口外的所有端口广播**（Flooding）。

![image-20250814010511560](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250814010511560.png)

**标识（Identification）**

- **位置**：IPv4 头部 16 位字段（除了标志和片偏移前的字段）
- **作用**：给每个 IP 数据报分配一个 **唯一标识符**。
- **用途**：当一个大数据报被分片后，接收端可以根据 **标识字段**将属于同一个原始数据报的分片重新组合。
- **特点**：每发送一个新的 IP 数据报，标识号通常递增

**标志（Flags）**

- **位置**：3 位字段
- **作用**：控制 IP 分片行为
- **各位含义**：

| 位          | 名称                 | 含义                               |
| ----------- | -------------------- | ---------------------------------- |
| 0（最高位） | 保留位               | 必须为 0                           |
| 1（次高位） | DF（Don't Fragment） | 1 → 不允许分片，0 → 可分片         |
| 2（最低位） | MF（More Fragments） | 1 → 后面还有分片，0 → 最后一个分片 |

**片偏移（Fragment Offset）**

- **位置**：13 位字段
- **单位**：**8 字节**（每个单位 = 8 B）
- **作用**：表示 **该分片在原始数据报中的起始位置**
- **举例**：

假设原始数据报长度 4000 B，经过 MTU 1500 B 的链路分片：

1. 第一个分片：长度 1500 B → 片偏移 = 0
2. 第二个分片：长度 1500 B → 片偏移 = 1500 / 8 = 187
3. 第三个分片：长度 1000 B → 片偏移 = (1500 + 1500) / 8 = 375

- **接收端**根据 **标识 + 片偏移 + MF 位** 来正确重组原始数据报。

注意**转发表**和**路由表**的区别

![](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250814125551666.png)

ARP（Address Resolution Protocol，地址解析协议）是**局域网**里的一座**翻译桥**，作用是：

> **把 IP 地址翻译成 MAC 地址**，让数据包能在物理网络上传输

在网络层（IP 层），我们用 **IP 地址**来标识设备，比如 `192.168.1.10`。

但在数据链路层（以太网），数据帧要靠 **MAC 地址**来投递，比如 `3C-A6-F6-12-34-56`。

网卡和交换机只认识 MAC 地址，不认识 IP 地址。

所以，当一个设备知道了对方的 **IP 地址**，但不知道 **MAC 地址** 时，就需要 ARP 来查出来



---



**路由器的每个物理接口（端口）** 相当于一个独立的网络节点。

每个接口在 **数据链路层** 都需要一个唯一的 MAC 地址，用于在所连接的局域网中通信。

因为不同接口连接的是 **不同的网络（网段）**，所以它们的 MAC 地址必须不同





**IP 地址（源 IP、目的 IP）**

- 在整个 **端到端** 传输过程中 **不变**（除非经过 NAT）。
- 作用是标识网络层端到端的通信双方。

**MAC 地址（源 MAC、目的 MAC）**

- 每经过一个链路跳（hop），都会变化。

 

当主机发送数据报时，如果目的 IP **不在本地子网**，主机就会把数据报交给默认网关，由它继续转发。

交换机工作在 **数据链路层**，它只看 **MAC 地址**，不关心 IP。
 所以：

- A 发 ARP 广播，交换机会泛洪（flood）到所有端口。
- B 收到后回复单播给 A。
- 交换机学习到 A 和 B 的 MAC 地址位置，下次就不会乱泛洪了。

---

**如果经过路由器（跨网段）**

重点来了：**ARP 只在局域网里用，跨网段不行。**
 例如：

- A（192.168.1.10/24）要访问 C（192.168.2.10/24），它发现 C 不在同一子网。
- A 的 IP 层会把包交给 **默认网关（路由器 R，192.168.1.1）**。
- A 必须先对网关 R 做 ARP，得到 R 的 MAC 地址，然后把以太网帧的目的 MAC 设置为 R。
- R 收到后，在 192.168.2.0 网段再做一次 ARP，找到 C 的 MAC，替换以太网头继续转发。

也就是说：

- **多交换机 = 还是一个二层广播域，ARP 广播能穿透。**
- **有路由器 = 广播不能跨越，ARP 在每个子网都要重新解析**

### 1. **ICMP（Internet Control Message Protocol，因特网控制报文协议）**

- **协议层级**：运行在 **网络层**，属于 IP 的**伴随协议**。
- **用途**：
  - 不是用来传输应用数据，而是用来传输 **控制信息** 和 **差错信息**。
  - 当 IP 数据包在传输过程中**出现问题（无法交付、TTL 耗尽、主机/端口不可达等）**，路由器或主机会生成 ICMP 报文，通知源主机。
- **典型场景**：
  - **Ping**：通过 ICMP Echo Request / Echo Reply 来检测主机是否可达。
  - **Traceroute**：利用 ICMP Time Exceeded 消息，逐跳探测路径。
- **价值类比**：相当于 IP 网络里的“异常报警与诊断工具”，告诉你网络有没有通，但不负责传送业务数据。

------

### 2. **IGMP（Internet Group Management Protocol，因特网组管理协议）**

- **协议层级**：同样运行在 **网络层**，也是 IP 的**伴随协议**，但只适用于 **多播（Multicast）**。
- **用途**：
  - 用来**管理主机**与 **多播组**（Multicast Group）的关系。
  - 主机通过 IGMP 报文告诉路由器：“我想加入某个多播组（例如 224.0.0.x）”，路由器据此决定是否转发该组的多播流量到这个链路。
- **典型场景**：
  - IPTV、在线视频直播、股票行情推送等一对多场景。
  - 如果没有 IGMP，路由器就无法知道哪些用户想要收多播流量，会浪费带宽。
- **价值类比**：相当于“群聊的订阅机制”，你加入群组后，才会收到群里的消息

- 两者互不干扰，因为 TCP 和 UDP 数据包在 IP 层有 **协议字段区分**

### 释放连接

![image-20250818165230924](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250818165230924.png)

![image-20250818170150859](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250818170150859.png)

**挥手1（FIN=1, Client→Server）**
 客户端说：“我不再给你发数据了（应用层 write 关闭），但是我还能收数据。”
 👉 意味着：**客户端到服务端的方向被关闭**。

**挥手2（ACK, Server→Client）**
 服务端确认：“好，我知道你不会再给我发数据了。”
 👉 现在状态：**单向关闭**（客户端→服务端通道关闭；服务端→客户端通道仍然开放）。

**挥手3（FIN=1, Server→Client）**
 服务端说：“我这边也发完数据了，我也不再发了。”
 👉 意味着：**服务端到客户端的方向也要关闭**。

**挥手4（ACK, Client→Server）**
 客户端确认：“好的，我知道你不发了。”
 👉 双方都不再发送数据，**连接彻底关闭**



> 挥手2之后，客户端不是已经不能发数据了吗？为什么还能在挥手4中给服务端发 ACK 呢？



这就要区分 **TCP 数据报文（Data Segment）** 和 **TCP 控制报文（Control Segment）**：

- **数据传输通道关闭**：
   挥手1时，客户端发 FIN，相当于关闭了“数据写入”，客户端不能再发 **应用层数据** 给服务端。
- **控制==信令==仍然可用**：
   即使数据方向关闭，TCP 协议栈 **仍然可以发送必要的控制报文**（ACK、FIN、RST 等），这些报文不算“应用层数据”，只是 TCP 协议本身需要的“管理指令”。

所以挥手4时，客户端发送的 ACK 是 **协议控制报文**，不是应用层数据，因此完全合理

**DNS 解析过程：**



**浏览器缓存**：先看自己有没有记忆（最近解析过同样的域名）。

**操作系统缓存（DNS 缓存）**：如果浏览器没有，就问操作系统的 DNS 缓存。

**Hosts 文件**：还会看 `hosts` 文件里有没有硬编码的映射。



你的电脑一般会配置一个 **本地递归 DNS 服务器**（比如运营商的 DNS，或 `8.8.8.8` Google DNS）。
 浏览器把请求丢给它，它来帮你问遍全世界。



假设本地域名服务器也没有缓存，它会按层级去问：

1. **==根==域名服务器（Root Server）**
   - 这是互联网的“导航员”。
   - 它本身不知道 `www.example.com` 的 IP，但知道 **`.com` 顶级域名服务器**的地址。
   - 所以回答：“去问 `.com` 服务器吧。”
2. **==顶级==域名服务器（TLD Server）**
   - `.com` 的权威服务器接到请求。
   - 它也不知道 `www.example.com` 的 IP，但知道 `example.com` 的权威服务器地址。
   - 所以回答：“去问 `example.com` 的权威服务器。”
3. **==权威（权限）==域名服务器（Authoritative Name Server）**
   - 这是 `example.com` 自己的“管家”。
   - 它知道 `www.example.com` 具体 IP（比如 192.0.2.123）。
   - 返回这个结果给递归 DNS





**权威域名服务器是什么**

它就是某个域名的 **最终解释者**。

- 它保存了这个域名的 **DNS 记录**（比如 A 记录、MX 记录、CNAME 记录）。
- 当别人一路问下来（根 → 顶级域名 → 二级域名……），最后就是问到它。
- 它的回答就是“标准答案”，不会再去问别人。

举个例子：

- 你要访问 `tieba.baidu.com`
- 根告诉你：去问 `.com`
- `.com` 服务器告诉你：去问 `baidu.com` 的权威服务器
- `baidu.com` 的权威服务器回答：“`tieba.baidu.com` 的 IP 是 220.181.38.148”

这个回答就是权威的，别人不能改口



~~~
baidu.com                 （二级域名）
├─ www.baidu.com           （三级域名，主站）
├─ tieba.baidu.com         （三级域名，贴吧）

## 文件传输（送）协议 FTP

![image-20250820224949358](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250820224949358.png)



![image-20250820225118198](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250820225118198.png)



 ![image-20250820231819528](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250820231819528.png)

![image-20250820232357479](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250820232357479.png)

FTP 是一个**应用层协议**，用于在网络上传输文件。它基于 TCP/IP 协议栈，典型工作方式是**控制连接 + 数据连接**分离

**控制连接（Control Connection）**

- 用于发送命令（如登录、切换目录、获取文件列表）。
- 建立一次连接即可，长时间保持。
- 默认 TCP 端口：**21**（服务端监听，客户端发起连接）。

**数据连接（Data Connection）**

- 用于实际的文件传输（上传/下载）。
- 可以是临时的连接，每次传输建立一次。
- TCP 端口：视模式而定（主动或被动）。



---



**FTP 传输模式：主动模式 vs 被动模式**

FTP 的核心复杂点在于数据连接建立方式不同。

### 主动模式（Active Mode）

**流程**：

1. 客户端连接服务器 **21 端口**（控制连接）。
2. 客户端告诉服务器自己监听的端口（通过 `PORT` 命令）。
3. 服务器从 **20 端口** 发起连接到客户端提供的端口（数据连接）。
4. 文件传输完成后，数据连接关闭。

**特点**：

- 服务器主动建立数据连接到客户端。
- 对防火墙不友好：客户端通常在 NAT/防火墙后，外部服务器难以连接客户端端口。

**端口总结**：

| 连接类型 | 客户端端口               | 服务端端口 |
| -------- | ------------------------ | ---------- |
| 控制连接 | 随机高端口               | 21         |
| 数据连接 | 随机高端口（客户端监听） | 20         |

------

###  被动模式（Passive Mode）

**流程**：

1. 客户端连接服务器 **21 端口**（控制连接）。
2. 客户端发送 `PASV` 命令，服务器返回一个随机端口号。
3. 客户端主动连接服务器指定端口建立数据连接。
4. 文件传输完成后，数据连接关闭。

**特点**：

- 客户端主动连接服务器，避免防火墙阻塞。
- 企业级服务器通常推荐被动模式。

**端口总结**：

| 连接类型 | 客户端端口 | 服务端端口             |
| -------- | ---------- | ---------------------- |
| 控制连接 | 随机高端口 | 21                     |
| 数据连接 | 随机高端口 | 随机高端口（服务器返回 |

HTTP 流水线是 **在同一个 TCP 连接上，客户端可以连续发送多个 HTTP 请求，而不必等待前一个响应返回**，服务器按顺序返回响应

非流水线模式是 **客户端每发送一个请求就等待服务器返回响应后，才能发送下一个请求**

| 特性     | 流水线（Pipelining）         | 非流水线（Non-Pipelining） |
| -------- | ---------------------------- | -------------------------- |
| 请求发送 | 连续发送，无需等待前一个响应 | 发送一个请求等待响应       |
| 响应顺序 | 必须保持顺序                 | 自然顺序                   |
| 性能     | 高（减少 RTT 等待）          | 低（串行阻塞）             |
| 适用版本 | HTTP/1.1                     | HTTP/1.0/1.1               |
| 缺点     | Head-of-Line 阻塞、兼容性差  | 延迟高                     |

*CRLF*是Carriage-Return Line-Feed的缩写，意思是回车换行，就是回车
