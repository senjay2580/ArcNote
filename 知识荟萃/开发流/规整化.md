## 五、注意事项

1. **动词选择原则**：

   - 新增：create > add > save
   - 删除：remove > delete
   - 修改：update > modify

2. **布尔方法**：

   - 返回boolean的方法应以is/has/can开头

   

   ```java
   boolean isUserExist(Long userId);
   boolean hasPermission(Long userId, String permission);
   ```

3. **避免的命名**：

   - 不要使用process/handle等模糊动词
   - 避免doXxx/performXxx等冗余前缀

4. **一致性原则**：

   - 整个项目保持相同业务用相同动词
   - 例如：全部使用"create"或全部使用"add"



## 程序内部结构



1. **方法分组**：
   - **将相关功能的方法分组排列（申请相关、关系管理、查询方法）**
   - **私有辅助方法放在最后**
2. **注释完善：**
   - **类级别注释说明类功能**
   - **每个方法添加完整注释，说明参数和返回值**
   - **重要代码块添加注释**
3. **代码格式**：
   - **方法之间用空行分隔**
   - **逻辑代码块之间用空行分隔**
   - 统一缩进和括号风格
4. **方法排序逻辑**：
   - 公共接口方法在前
   - **核心业务方法在前**
   - **辅助方法在后**
   - **查询类方法放在最后**

**<span style="color:#FF0000;">数据库设计</span>**

~~~markdown
数据库设计流程：

1、根据业务逻辑画出E-R图（即实体关系图）：找出业务逻辑中有多少个实体，以及他们之间的联系。

2、将**E-R图**转换为关系模式：重难点是如何体现**一对多关系**，而**多对多关系**又要如何处理。

3、将数据库规范化：根据第一、二、三范式进行进一步完善，可能需要拆分字段、拆表（增加表）、添加主键等。

附：

第一范式（N1）：数据库中的表都是二维表。这个比较抽象，举个例子：一个“电话号码”字段，如果实际要对应两个号码（家庭电话、工作电话），那就不能称为二维表了。那么就要把“电话号码”拆成“家庭电话”和“工作电话”两个字段才能满足范式。

第二范式（N2）：除满足N1外，数据库的表中不存在非关键字段对任一候选关键字段对任一候选关键字段的部分函数依赖。比如视频中的petstore的E-R图中商品与订单是多对多的关系，为什么订单表中没包含商品信息，商品表中也没有包含订单信息。因为这样就不满足第二范式了。



第三范式（N3）：除了满足N2外，非主键列必须直接依赖于主键，不存在传递依赖。视频中的例子：订单表包含了用户号，那么如果加入用户名，密码，会怎么样呢，用户号依赖于订单号，但用户名和密码是直接依赖于用户号，而间接依赖于订单号的。那么就不满足第三范式了。
~~~

**可以替代连接这种高消耗性能的操作 也可以使用视图进行优化**

![image-20250523140854211](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250523140854211.png)

<span style="color:#FF0000;">**同一条信息以多份形式存在于数据库的不同地方。或者可以通过某个字段推断出来的字段**</span>



冗余字段适合那些**更新不频繁**的字段



**快照型冗余**
特点:保存历史状态，**写入后不再变更**
例子:
订单的商品信息(名称、价格)合同签订时的用户信息物流发货时的收货地址

- 商品的名称和价格其实是存储在“商品表”中的。订单表里如果也保存商品名称、价格，就造成了数据的重复。
- **但为什么还要这样做？**
   因为商品的名称和价格可能会随着时间发生变化（价格变动、商品改名）。订单里的商品信息要**保持下单时的快照，保证订单数据的历史准确性**。
   这就产生了**有意的冗余**：复制商品名称和价格到订单中，保证订单记录不受商品表后续变动影响。

---



**统计型冗余**
特点:存储聚合计算结果，避免实时COUNT/SUM
例子:
点赞数、评论数、粉丝数
订单总金额
(SUM(price*quantity))
用户积分总额

---



**关联型冗余**
特点:避免多表 JOIN，直接存储关
联表字段
例子:
员工表 冗余 部门名称(本应查departments 表)
订单表 冗余 用户手机号(本应查users 表)



**根据 强一致 or 最终一致 选择 冗余信息一致性策略**

同步事务 （一起更新）

触发器 （自动更新）

批量异步更新（定时任务触发 ）
