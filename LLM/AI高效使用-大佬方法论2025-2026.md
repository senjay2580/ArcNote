# 核心认知：为什么大多数人用 AI 没效果

```
普通人用 AI：
  "帮我写邮件" → 复制粘贴 → 完事
  效果：节省5分钟，没有本质改变

Power User 用 AI：
  重新设计整个工作流，AI 是其中的执行层
  效果：同样的人，产出 3-10 倍
```

**PwC 2025 全球调查（5万人）的核心数据：**
- 每天用 AI 的人：92% 感受到生产力提升
- 偶尔用 AI 的人：只有 58% 有感受

- 66% 的管理者：**不会雇用没有 AI 技能的人**

差距不是工具，是**思维方式**。

# 第一章：Karpathy 的范式转变——「Agentic Engineering」

## 他自己经历的工作流革命

Karpathy 在 2025 年 12 月发推（1400万浏览）：

> *"I've never felt this much behind as a programmer. The profession is being dramatically refactored."*
> "我从未感到如此落后。这个职业正在被彻底重构。"

**他的工作流转变（2025年11月 → 12月）：**

```
2025年11月：
  80% 手动写代码 + Autocomplete
  20% AI Agent

2025年12月：
  20% 手动审查和修改
  80% AI Agent 执行

他说：这是他"20年编程生涯中最大的工作方式变化"
```

**2026年2月，他正式提出：**

> *"Vibe Coding → Agentic Engineering"*
>
> "用 LLM Agent 编程正在成为专业人士的默认工作流，
> 只不过要加上更多的监督和审查。"

---

## Karpathy 的核心洞察（逐条解析）

### 洞察1：你不是在写代码，你是在指挥代码

> *"You are not writing the code directly 99% of the time...
> you are orchestrating agents who do, and acting as oversight."*

```
旧角色：程序员 = 写代码的人
新角色：程序员 = 定义目标、审查结果的工程指挥官
```

### 洞察2：给成功标准，而不是步骤

> *"LLMs are exceptionally good at looping until they meet specific goals.
> Don't tell it what to do, give it success criteria and watch it go."*

**实践翻译：**
```
❌ 低效：
"第一步做X，第二步做Y，第三步做Z"

✅ 高效：
"目标是实现[功能]，
验收标准：
1. 所有测试通过
2. 响应时间 < 200ms
3. 边界情况[A/B/C]处理正确
达到以上标准前不要停止。"
```

### 洞察3：AI Agent 会自信地走错路

> "Models make wrong assumptions on your behalf and just run along
> with them without checking... don't manage their confusion,
> don't seek clarifications, don't surface inconsistencies."

这是最关键的警告：AI 不会说"我不确定"，它会自信地干到底。

**对策：**
- 关键决策点前加 checkpoint："在继续之前，告诉我你的方案是什么"
- 分阶段确认，而不是一次性放手
- 测试套件是你的安全网——有测试才能放心让 Agent 跑

### 洞察4：Agent 能力在 2025年12月发生了「相变」

> "LLM agent capabilities have crossed some kind of threshold
> of coherence around December 2025 and caused a **phase shift**
> in software engineering."

**这意味着：** 2023-2024 年你试过觉得"还不行"的工作流，现在值得重新试一次。

---

## Karpathy 的 CLAUDE.md 实践

他总结出一套写给 AI 的"项目规则文件"原则，现已被社区整理成模板：

```markdown
# 项目规则（CLAUDE.md）

## 项目背景
[让 AI 理解项目是什么]

## 技术栈
[具体版本号，防止 AI 用过时 API]

## 代码风格
[具体规范，不要说"保持一致"，要说"用X风格"]

## 禁止行��
- 不要修改[文件X]
- 不要使用[库Y]
- 遇到[情况Z]时停下来问我

## 验收标准模板
每个任务完成后必须：
1. 所有测试通过
2. 无 lint 错误
3. 功能在[场景]下验证通过
```

**关键原则：CLAUDE.md 控制在 100 行以内**——太长会稀释注意力。

# 第二章：Simon Willison 的「平行 Agent 生活方式」

## 核心心智模型

> *"Think of LLMs as an over-confident pair programming assistant —
> fast at looking things up and executing tedious tasks,
> but likely to make mistakes, sometimes subtle, sometimes huge."*
> "把 LLM 想象成一个过度自信的结对程序员——查东西快，但会犯错，有时是小错，有时是大错。"

**他的实测数据：** AI 让他在编码部分的工作效率提升 **2-5 倍**。

---

## Simon 的五个核心实践

### 实践1：平行 Agent 工作流

> "Embrace the parallel coding agent lifestyle —
> supervise multiple AI developers working simultaneously
> rather than being tethered to a single synchronous assistant."

```
旧工作流（串行）：
  一个问题 → 等 AI 回答 → 看结果 → 下一个问题

新工作流（并行）：
  问题A → Agent1 处理
  问题B → Agent2 处理   ← 同时进行
  问题C → Agent3 处理
  你：在三个窗口之间切换，审查结果
```

**工具实现：** Claude Code 的 `--worktree` 功能，可以同时跑 3-5 个 Agent 在不同 git 分支上工作。

### 实践2：测试套件 = Agent 的超能力

> "Having a robust test suite is like giving the agents superpowers —
> they can validate and iterate quickly when tests fail."

**逻辑：**
```
没有测试：AI 改了代码，你不知道对不对 → 必须人工检查每行
有测试：AI 改了代码 → 自动跑测试 → 绿了继续，红了自动修 → 你只看最终结果
```

### 实践3：训练截止日期意识

每次用 AI 处理某个技术库时，先问：
```
"[库名] 的最新版本是什么？
你的训练截止日期之后有哪些破坏性变更（breaking changes）？
如果不确定，直接告诉我不知道。"
```

### 实践4：记录 AI 的「能力边界日志」

维护一个文档，记录：
- ✅ AI 擅长的任务（每次发现就记）
- ❌ AI 反复失败的任务（是重要信号）
- 📅 新模型发布后，用失败清单重测

> "Every time you discover a type of task the LLM can't do,
> note it down. When a new model can do it, that's a major signal."

### 实践5：快速原型心态

他的 GitHub（simonw/tools）有 **77 个 HTML+JS 小工具**，全部 AI 生成，每周增加几个。

> **核心原则：** 先用 AI 生成可运行的原型，不要追求完美的第一版。
> 在能跑的基础上迭代，远比在脑子里设计完美方案快 10 倍。

# 第三章：Addy Osmani 的「结构化工作流」

## 关于他

Google Chrome 工程总监，25年软件工程经验，2025-2026 年最活跃的 AI 工作流布道者。

他的核心立场：
> *"As software engineers, our identity was never 'the person who can write code' —
> it was 'the person who can solve problems with software.'
> AI isn't replacing engineers. It's amplifying them."*

---

## 10 大工作流原则（2026 版）

### 原则1：规格先于代码（最重要）

```
❌ 常见错误：
"帮我实现用户登录功能"
→ AI 直接开始写代码
→ 写完发现理解不对，大量返工

✅ Addy 的做法：
第一步（只生成文档，不写代码）：
"请帮我写一个 spec.md，包含：
- 功能需求列表
- 技术架构方案
- 数据模型设计
- 接口定义（输入/输出）
- 测试策略
如有不清楚的地方，主动问我。
现在不要写任何代码。"

第二步：确认规格正确后，再开始实现
```

**为什么这样做能减少 70% 的返工：** 烂规格比烂代码更难修。

---

### 原则2：把任务拆成「提示词计划」

```markdown
# 实现计划（逐步执行）

## Step 1：数据模型
只实现 User + Session 数据库模型。完成后停下。

## Step 2：注册接口
只实现 POST /auth/register。依赖 Step 1 完成。

## Step 3：登录接口
只实现 POST /auth/login，返回 JWT。

## Step 4：中间件
只实现认证中间件。
```

**每步完成后必须：** 运行测试 → 确认通过 → 再进行下一步。

---

### 原则3：模型轮换策略

```
遇到一个模型卡住或质量差？

同一个提示词 → 试 Claude
              → 试 GPT-5
              → 试 Gemini

比较输出，选最好的，或者综合三者。
```

> "Each model has its own 'personality' and different strengths."

---

### 原则4：自动化审查形成闭环

```
AI 写代码
    ↓
自动化工具检查（测试 / Lint / CodeRabbit）
    ↓
把审查反馈直接作为新提示词喂给 AI
    ↓
AI 修复
    ↓
再次自动检查
    ↓
循环直到全绿
```

这等于拥有了「无限 QA 工程师」。

---

### 原则5：只提交能解释的代码

> *"Commit frequently, and only commit code you can explain."*

这是防止 AI 代码失控的根本安全规则。AI 生成的代码你不理解 = 定时炸弹。

---

### 原则6：测试先于实现（Test-First AI）

在让 AI 写功能代码之前，先让 AI 写测试：
```
"不要写功能代码，先帮我写[功能X]的测试用例，
覆盖：正常流程、边界情况、错误情况。"

→ 审查测试是否合理
→ 通过后再让 AI 实现，以测试为目标
```

---

### 原则7：AI 擅长的 vs 不擅长的

```
✅ 让 AI 做：
- 重复性、模板化的代码（getter/setter/CRUD）
- 陌生技术栈的快速上手
- 文档、注释、测试用例
- 代码格式转换
- Debug 的第一轮分析

❌ 人自己做：
- 核心业务逻辑架构
- 安全敏感代码（最终必须人审）
- 需要深度领域知识的算法
- 关键性能优化决策
```

# 第四章：Ethan Mollick 的「知识工作者」方法论

## 他是谁

沃顿商学院教授，《Co-Intelligence》作者，TIME 最具影响力 AI 人物，研究 AI 对工作的影响。

---

## 核心框架：像管理实习生一样用 AI

> *"Treat AI like a person, but verify like a manager."*
> "像对待人一样与 AI 交互，但要像主管一样核查结果。"

### 「快速实习生」工作法（4步）

```
Step 1：给出角色 + 目标 + 约束（入职培训）
"你是一位[角色]，帮我[任务]，
约束条件：[限制]，
受众：[谁看]，
格式：[要什么]"

Step 2：先要提纲，而不是直接要最终版
→ 提纲更容易发现方向错误，比在完整版上返工便宜得多

Step 3：用具体的编辑指令修改
❌ "更好一点"（AI 不知道什么是"好"）
✅ "把第二段改得更简洁，删掉废话，保留数据"

Step 4：你做最终的事实核查和风格审定
这是你不能外包的职责。
```

---

## 何时用 AI，何时不用（Mollick 的清单）

### ✅ 用 AI 的 15 类场景（部分）

| 场景 | 说明 |
|------|------|
| 第一稿 | 任何文字类内容的起草 |
| 头脑风暴 | 让 AI 生成 50 个想法，你挑好的 |
| 研究综合 | 把多份资料喂给 AI，要求提炼结论 |
| 反驳自己 | 让 AI 充当"魔鬼代言人"，批评你的方案 |
| 简化复杂内容 | 把技术文档翻译成给特定受众的语言 |
| 模板和框架 | 让 AI 建立可复用的工作模板 |
| 代码辅助 | 不限于专业程序员 |
| 学习新领域 | 用苏格拉底式对话快速入门 |

### ❌ 不用 AI 的 5 类场景

1. **需要真实人际连接的场景**（宣泄情绪、重要关系沟通）
2. **法律/合规禁止的场景**
3. **需要真正原创思考的场景**（AI 会均质化你的思维）
4. **你完全不了解的领域**（无法验证输出，风险极高）
5. **准确性极度关键但速度不是问题的场景**

## Mollick 的「Ask for More」原则

> "The AI doesn't get tired or resentful, so ask for a lot."

**实践：**
```
❌ "给我10个营销方案"
✅ "给我50个营销方案，然后告诉我你认为最有潜力的3个，解释理由"

❌ "帮我改善这个句子"
✅ "给我这个句子的30个不同改法，覆盖：正式/非正式/幽默/严肃/简洁/详细等不同风格"
```

然后你从海量选项中选择，效率远高于反复来回修改。

## 关于 AI Agent 的管理视角

> *"Directing an AI agent on a multi-hour task now resembles
> writing a product requirements document."*

AI Agent 任务做得好不好，取决于你的需求文档写得好不好。这是管理能力，不是技术能力。

```
好的 AI Agent 指令 = 好的 PRD（产品需求文档）
包含：
- 目标（什么算成功）
- 约束（哪些不能做）
- 资源（能用什么）
- 验收标准（如何验证完成）
```

# 第五章：Power User 的系统性工作架构

基于以上所有专家的实践，整合出一套完整的「AI 生产力系统」：

## 5.1 三层分工架构

```
你（人类）负责：
├── 战���层：决定做什么、目标是什么、成功标准是什么
├── 判断层：审查 AI 输出，决定采用/修改/拒绝
└── 责任层：最终对结果负责

AI 负责：
├── 执行层：生成代码/文字/方案
├── 迭代层：根据反馈修改
└── 搜索层：在训练知识中寻找方案
```

## 5.2 每日工作流模板

```
早晨（规划阶段）：
□ 列出今天的3个核心任务
□ 对每个任务：先想"AI能帮我做哪部分？"
□ 准备好上下文（相关文档、代码、背景）

执行阶段：
□ 任务分解：大任务拆成 AI 可以一次处理的小块
□ 并行执行：多个 Agent 同时跑不同子任务
□ 及时审查：每个输出都要过一遍，不要盲目采用

收尾阶段：
□ 验证所有 AI 生成的内容
□ 记录今天发现的 AI 能力边界
□ 更新你的提示词模板库
```

## 5.3 提示词资产库（必须建立）

把你用过的最有效的提示词存成模板：

```
📁 我的提示词库/
├── 代码相关/
│   ├── 代码审查模板.md
│   ├── 写测试用例.md
│   └── debug分析.md
├── 写作相关/
│   ├── 邮件模板.md
│   ├── 周报生成.md
│   └── 内容改写.md
└── 分析相关/
    ├── 竞品分析.md
    ├── 数据解读.md
    └── 决策评估.md
```

每次找到一个高效提示词，立刻存档。这是真正的生产力复利资产。

# 第六章：真实数据——AI 能提升多少

## 量化研究数据

| 场景 | 提升幅度 | 数据来源 |
|------|---------|---------|
| 知识工作者平均 | **10-25%** | 多项研究综合 |
| 习惯用户平均 | **66%** | 2025年行业研究 |
| 编程任务（资深工程师） | **2-5倍** | Simon Willison 实测 |
| 编程任务（GitHub Copilot 研究） | 最高 **81%** | GitHub 官方报告 |
| 用户认为"工作时间减半"的任务比例 | **41%** | 丹麦知识工作者研究 |
| 工作时间压缩（90分钟→30分钟） | **3倍** | 美国工人调查 |
| 初级/中等技能工作者 | 提升最大 **40%** | MIT Sloan |

## 反直觉数据（METR 2025 研究）

**有经验的开发者在熟悉代码库中：使用 AI 反而慢了 19%。**

原因：
- 老手的直觉比 AI 辅助更快（在熟悉领域）
- AI 代码需要审查时间，审查是瓶颈
- AI 增加了 4 倍代码克隆率，创造维护负担

**AI 真正加速的场景：**
```
✅ 陌生技术栈    ✅ 重复性代码    ✅ 文档和注释
✅ 快速原型      ✅ 测试用例      ✅ 错误信息分析
```

## 警告数据

- **48%** 的 AI 生成代码含安全漏洞
- **40%** 的 Copilot 代码被标记为不安全
- 技术债增长 **15%**（没有治理规范的团队）

结论：AI 是加速器，不是跳过审查的理由。
