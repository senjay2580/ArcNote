## 📖 故障排查方法论

### 1. 黄金排查流程 (5W2H)

```
┌─────────────────────────────────────────────────┐
│            企业级故障排查流程                      │
└─────────────────────────────────────────────────┘

第1步: 现象确认 (What)
   ↓ "到底出了什么问题?"
   - 具体报错信息
   - 影响范围(多少用户/多少请求)
   - 复现条件

第2步: 时间定位 (When)
   ↓ "什么时候开始的?"
   - 故障开始时间
   - 是否有规律(如每天凌晨2点)
   - 监控曲线拐点

第3步: 环境定位 (Where)
   ↓ "哪个环境/哪个服务?"
   - 生产/测试/开发环境
   - 哪个服务实例
   - 单点故障 or 集群故障

第4步: 原因分析 (Why)
   ↓ "为什么会出现?"
   - 代码bug
   - 配置错误
   - 资源不足
   - 外部依赖

第5步: 影响评估 (Who)
   ↓ "影响了谁?"
   - 用户影响面
   - 业务影响(订单/支付等)
   - 数据一致性

第6步: 解决方案 (How)
   ↓ "怎么解决?"
   - 快速止损(临时方案)
   - 根本修复(长期方案)
   - 验证恢复

第7步: 成本评估 (How much)
   ↓ "代价有多大?"
   - 故障时长
   - 业务损失
   - 人力成本
```

## 🌐 网络问题排查实战

### 案例1: 容器之间无法通信

#### 问题现象
```bash
# 应用日志报错
2024-11-29 10:30:15 ERROR - Connection refused: connect to mysql:3306
```

#### 排查步骤

**第1步: 确认容器是否运行**
```bash
# 查看所有容器状态
docker ps -a

# 预期: 所有容器 STATUS 都是 Up
# 如果有 Exited → 先解决容器启动问题
```

**第2步: 检查容器网络**
```bash
# 1. 查看应用容器在哪个网络
docker inspect archat-backend | grep -A 20 "Networks"

# 输出示例:
# "Networks": {
#     "archat_default": {
#         "IPAddress": "172.18.0.5"
#     }
# }

# 2. 查看 MySQL 容器在哪个网络
docker inspect mysql | grep -A 20 "Networks"

# 输出示例:
# "Networks": {
#     "archat_default": {
#         "IPAddress": "172.18.0.3"
#     }
# }
```

**关键判断**:
- ✅ 同一网络 → 应该能通信
- ❌ 不同网络 → **这就是问题根因**

**第3步: 测试网络连通性**
```bash
# 进入应用容器
docker exec -it archat-backend bash

# 测试1: Ping MySQL 容器(按容器名)
ping mysql
# 预期: 能 ping 通

# 测试2: Ping MySQL 容器(按IP)
ping 172.18.0.3
# 预期: 能 ping 通

# 测试3: 测试端口连通性
telnet mysql 3306
# 或
nc -zv mysql 3306

# 预期: Connection succeeded
```

**第4步: DNS 解析测试**
```bash
# 测试容器名能否解析
nslookup mysql
# 或
dig mysql

# 预期输出:
# Name: mysql
# Address: 172.18.0.3
```

**常见问题**:
```
问题1: ping 不通容器名
原因: 不在同一网络
解决: docker-compose 中添加 networks 配置

问题2: ping 通但端口不通
原因: 服务未启动或端口配置错误
解决: 检查服务日志和端口配置

问题3: DNS 解析失败
原因: 容器名拼写错误
解决: 使用 docker ps 确认正确的容器名
```

### 案例2: 外部无法访问容器服务

#### 问题现象
```
浏览器访问 http://localhost:8180 → ERR_CONNECTION_REFUSED
```

#### 排查步骤

**第1步: 确认容器内服务正常**
```bash
# 进入容器测试
docker exec archat-backend curl http://localhost:8080/actuator/health

# 预期输出:
# {"status":"UP"}
```

**第2步: 检查端口映射**
```bash
# 查看端口映射
docker ps | grep archat-backend

# 输出示例:
# CONTAINER ID   IMAGE             PORTS
# abc123...      archat:latest     0.0.0.0:8180->8080/tcp

# 关键: 8180(宿主机) -> 8080(容器)
```

**第3步: 检查宿主机端口监听**
```bash
# Windows (PowerShell)
netstat -ano | findstr "8180"

# Linux/Mac
netstat -tuln | grep 8180

# 预期输出:
# TCP    0.0.0.0:8180    0.0.0.0:0    LISTENING
```

**第4步: 防火墙检查**
```bash
# Windows 防火墙
# 控制面板 → Windows Defender 防火墙 → 高级设置 → 入站规则
# 查找端口 8180 是否被阻止

# Linux 防火墙(如果有)
sudo firewall-cmd --list-ports
# 或
sudo iptables -L -n | grep 8180
```

**第5步: Docker 网络模式检查**
```yaml
# docker-compose.yml
services:
  backend:
    ports:
      - "8180:8080"  # ✅ 正确: 端口映射
    # network_mode: host  # ❌ 如果用了 host 模式,ports 会失效
```

**解决方案汇总**:
```
问题1: 容器内服务未启动
→ 检查应用日志: docker logs archat-backend

问题2: 端口映射配置错误
→ 修改 docker-compose.yml 中的 ports

问题3: 宿主机端口被占用
→ netstat 找到占用进程并关闭

问题4: 防火墙拦截
→ 添加防火墙规则允许端口

问题5: 应用监听 127.0.0.1
→ 改为监听 0.0.0.0
  application.yml: server.address=0.0.0.0
```

### 案例3: 网络延迟高排查

#### 问题现象
```
Grafana 显示: HTTP 接口 P99 延迟从 200ms 突增到 5s
```

#### 排查工具箱

**1. 容器内抓包**
```bash
# 安装 tcpdump
docker exec -it archat-backend bash
apt-get update && apt-get install -y tcpdump

# 抓取 MySQL 通信包
tcpdump -i any -n port 3306 -w /tmp/mysql.pcap

# 下载到本地分析
docker cp archat-backend:/tmp/mysql.pcap ./
# 使用 Wireshark 打开分析
```

**2. 网络延迟测试**
```bash
# 测试到 MySQL 的延迟
docker exec archat-backend bash -c "time echo > /dev/tcp/mysql/3306"

# 预期: real 0m0.005s (正常)
# 异常: real 0m2.000s (网络慢或 DNS 解析慢)
```

**3. DNS 解析时间测试**
```bash
# 测试 DNS 解析耗时
docker exec archat-backend bash -c "time nslookup mysql"

# 如果耗时 > 1s → DNS 问题
# 解决: 使用 IP 地址或优化 DNS 配置
```

**4. 网络工具全家桶**
```bash
# 安装网络诊断工具
docker exec -it archat-backend bash
apt-get install -y iproute2 net-tools dnsutils curl wget

# ping - 测试连通性
ping -c 5 mysql

# traceroute - 路由追踪
traceroute mysql

# mtr - 持续监控网络
mtr -r -c 10 mysql

# netstat - 查看连接状态
netstat -antp | grep 3306
```

## 📝 日志分析实战

### 1. 日志查找策略

#### Docker 容器日志

**基础查看**
```bash
# 查看最新100行
docker logs --tail 100 archat-backend

# 实时滚动查看
docker logs -f archat-backend

# 查看特定时间段(需要时间戳)
docker logs --since "2024-11-29T10:00:00" --until "2024-11-29T11:00:00" archat-backend

# 只看错误日志
docker logs archat-backend 2>&1 | grep -i "error\|exception"
```

**高级过滤**
```bash
# 查找包含关键字的前后10行
docker logs archat-backend 2>&1 | grep -B 10 -A 10 "NullPointerException"

# 统计错误出现次数
docker logs archat-backend 2>&1 | grep -c "ERROR"

# 按时间排序找最近的错误
docker logs archat-backend 2>&1 | grep "ERROR" | tail -20

# 导出日志到文件
docker logs archat-backend > /tmp/app.log 2>&1
```

#### 日志格式化分析

**使用 jq 解析 JSON 日志**
```bash
# 假设应用输出 JSON 格式日志
docker logs archat-backend 2>&1 | tail -100 | jq '.'

# 只看 ERROR 级别
docker logs archat-backend 2>&1 | jq 'select(.level=="ERROR")'

# 提取特定字段
docker logs archat-backend 2>&1 | jq '{time: .timestamp, msg: .message, trace: .stack_trace}'
```

**使用 awk 分析文本日志**
```bash
# 统计各错误类型出现次数
docker logs archat-backend 2>&1 | grep "ERROR" | awk '{print $NF}' | sort | uniq -c

# 找出最频繁的异常
docker logs archat-backend 2>&1 | grep "Exception" | awk -F: '{print $1}' | sort | uniq -c | sort -rn | head -10
```

# 提取时间戳计算耗时
# 10秒 → 15秒 = 5秒消耗在数据库查询

#### 案例: OOM 问题定位

**日志特征**:
```bash
# 查找 OOM 相关日志
docker logs archat-backend 2>&1 | grep -i "OutOfMemoryError\|OOM\|heap space"

# 输出:
# java.lang.OutOfMemoryError: Java heap space
# at java.util.Arrays.copyOf(Arrays.java:3332)
# at java.util.ArrayList.grow(ArrayList.java:275)
# at com.senjay.service.UserService.loadAllUsers(UserService.java:123)
```

**分析**:
1. 异常类型: `OutOfMemoryError: Java heap space`
2. 堆栈指向: `UserService.loadAllUsers:123`
3. **根因**: 一次性加载了大量数据到内存

**解决方案**:
```java
// ❌ 问题代码
List<User> users = userMapper.selectAll(); // 100万数据全加载

// ✅ 修复后
int pageSize = 1000;
for (int i = 0; i < totalCount; i += pageSize) {
    List<User> users = userMapper.selectByPage(i, pageSize);
    processUsers(users);
}
```
