# 线上 Bug 排查指南

当面对复杂的线上问题时，不要慌，按照以下步骤冷静处理。

---

## 核心原则

**数据不会说谎，代码不会骗人，问题一定有迹可循。**

---

## 第一步：明确问题现象

不要急着看代码，先搞清楚：

1. **问题是什么？** 用一句话描述
   - 例：余美钦进度查询显示"已入院"，但入住记录显示"已出院"

2. **正确的应该是什么？**
   - 例：两边应该都显示"在院"

3. **影响范围有多大？**
   - 只有这一个人？还是一类人都有问题？

---

## 第二步：收集数据证据

**先查数据库，不要猜**

```sql
-- 查看问题数据的完整信息
SELECT * FROM check_in_record WHERE name = '余美钦';

-- 查看相关联的数据
SELECT * FROM wait_application WHERE id_no = 'xxx';
```

记录下关键字段的值：
- 状态字段是什么？
- 时间字段是什么？
- 和预期差在哪里？

---

## 第三步：确定时间线

**时间是破案的关键线索**

1. 问题数据的时间点
   - 例：余美钦 create_at = 2025-09-15，update_at = 2025-02-24

2. 代码修改的时间点
   ```bash
   git log --oneline -10
   git show <commit_id> --format="%ci %s" --no-patch
   ```

3. 对比判断：问题是在代码修改之前还是之后发生的？

---

## 第四步：还原执行流程

**结合代码和数据，模拟当时的执行过程**

1. 找到入口方法（哪个 Action/Controller 处理这个业务）

2. 逐步跟踪代码流程：
   - 调用了哪些方法？
   - 每个方法的查询条件是什么？
   - 查询结果会是什么？（根据当时的数据状态）
   - 走了哪个分支？
   - 最终执行了什么操作？

3. 画出流程：
   ```
   performBusiness()
       ↓
   getInRecords(idNo) → 查询条件是什么？结果是什么？
       ↓
   if 有记录 → changeCheckInRecord() → 做了什么？漏了什么？
   else → addCheckInRecord()
   ```

---

## 第五步：定位根本原因

问自己：

1. **代码逻辑有没有漏洞？**
   - 是否考虑了所有情况？
   - 有没有遗漏的字段更新？

2. **查询条件对不对？**
   - 过滤条件是否正确？
   - 会不会漏查或多查？

3. **是历史代码的问题还是新代码的问题？**
   - 用 git 查看历史版本对比

---

## 第六步：验证你的判断

**不要只靠推理，要验证**

1. 查数据库验证你的推断
   ```sql
   -- 如果我的推断正确，应该能查到xxx
   SELECT ...
   ```

2. 查表结构确认约束
   ```sql
   SHOW CREATE TABLE check_in_record;
   ```

3. 必要时查 git 历史确认代码变更
   ```bash
   git show <commit>:<file_path>
   ```

---

## 第七步：制定修复方案

**分两部分：修代码 + 修数据**

### 代码修复
- 明确要改哪个方法
- 改什么逻辑
- 会不会影响其他功能

### 数据修复
- 写好 SQL，先在测试环境验证
- 加上 WHERE 条件，避免误改
- 执行前备份或开事务

```sql
-- 修复前先查询确认
SELECT * FROM check_in_record WHERE id = 402;

-- 修复
UPDATE check_in_record 
SET status = 1, update_at = NOW()
WHERE id = 402 AND name = '余美钦';
```

---

## 第八步：复盘记录

修完之后，记录下来：

1. 问题现象
2. 根本原因
3. 修复方案
4. 影响范围
5. 如何避免类似问题

---

## 保持冷静的技巧

1. **不要急着改代码** - 先搞清楚问题再动手
2. **不要猜** - 查数据、查代码、查历史，用证据说话
3. **一次只排查一个问题** - 不要同时处理多个问题
4. **画图/写下来** - 把流程写出来，比在脑子里转清晰
5. **休息一下** - 卡住了就休息5分钟，换个思路
6. **找人讨论** - 说出来的过程本身就能帮你理清思路

---

## 常用排查命令

```bash
# 查看文件修改历史
git log --oneline -- <file_path>

# 查看某次提交的内容
git show <commit_id>

# 查看某次提交时的文件内容
git show <commit_id>:<file_path>

# 对比两个版本的差异
git diff <commit1> <commit2> -- <file_path>
```

```sql
-- 查表结构
SHOW CREATE TABLE table_name;

-- 查索引
SHOW INDEX FROM table_name;
```

## 本次案例总结

| 步骤 | 做了什么 |
|------|----------|
| 明确问题 | 余美钦进度显示"已入院"，入住记录显示"已出院" |
| 收集数据 | 查 check_in_record 表，发现 status=0，create_at=2025-09-15 |
| 确定时间线 | 问题发生在 2025-09-15，代码修改在 2025-11-16，问题是旧代码造成的 |
| 还原流程 | 旧代码 getInRecords 不过滤 status，查到已出院记录，调用 changeCheckInRecord 但没设置 status=true |
| 定位原因 | changeCheckInRecord 方法漏了 setStatus(true) |
| 验证判断 | create_at 被更新说明确实调用了 changeCheckInRecord |
| 修复方案 | 当前代码已修复（新增记录），历史数据需手动 UPDATE |

---

*记住：Bug 不可怕，可怕的是没有方法。有方法，就能解决。*
