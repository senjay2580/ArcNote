# 项目生产级部署方案

## 一、环境准备

### 1. 服务器初始化与系统选择

**操作系统选择**：
- 生产环境推荐：CentOS 7/8 或 Ubuntu 20.04 LTS（长期支持版本，稳定性有保障）
- 开发/测试环境：可与生产环境保持一致，便于环境一致性验证

**服务器初始化（以CentOS 7为例）**

```bash
# 更新系统到最新稳定版本
yum update -y

# 安装基础工具
yum install -y vim wget curl net-tools lsof firewalld chrony

# 时间同步（生产环境必须，避免日志时间混乱、证书过期问题）
systemctl start chronyd
systemctl enable chronyd
chronyc sources

# 关闭SELinux（生产环境可根据安全要求调整，初学者建议关闭避免权限问题）
sed -i 's/SELINUX=enforcing/SELINUX=disabled/g' /etc/selinux/config
setenforce 0

# 启动防火墙（生产环境必须开启，仅开放必要端口）
systemctl start firewalld
systemctl enable firewalld
```

**环境差异说明**：
- 开发环境：可关闭防火墙方便调试，无需严格时间同步
- 测试环境：防火墙按需开放，需时间同步保证测试数据时间一致性
- 生产环境：必须开启防火墙，严格时间同步，禁止直接关闭SELinux（建议配置为permissive模式）

**风险提示**：生产环境直接关闭防火墙是高危操作，可能导致服务器被攻击，必须严格配置端口策略。

### 2. 服务器配置与网络策略

| 环境类型   | 硬件配置建议          | 网络策略                                   | 权限管理                                     |
| ---------- | --------------------- | ------------------------------------------ | -------------------------------------------- |
| 开发服务器 | 4核8G，50GB SSD       | 允许开发机直接访问，开放常用端口           | 开发人员可拥有sudo权限                       |
| 测试服务器 | 8核16G，100GB SSD     | 仅允许开发服务器/办公网访问，限制公网暴露  | 测试人员仅拥有应用部署权限                   |
| 生产服务器 | 16核32G起，200GB+ SSD | 仅开放80/443端口，数据库/Redis禁止公网访问 | 仅运维人员拥有root权限，应用使用普通用户运行 |

**网络策略配置（生产环境示例）**：

```bash
# 开放SSH端口（建议修改默认22端口）
firewall-cmd --zone=public --add-port=22/tcp --permanent

# 开放HTTP/HTTPS端口
firewall-cmd --zone=public --add-port=80/tcp --permanent
firewall-cmd --zone=public --add-port=443/tcp --permanent

# 仅允许内部网络访问MySQL（假设内部网段192.168.1.0/24）
firewall-cmd --zone=public --add-rich-rule='rule family="ipv4" source address="192.168.1.0/24" port protocol="tcp" port="3306" accept' --permanent

# 仅允许本地访问Redis
firewall-cmd --zone=public --add-rich-rule='rule family="ipv4" source address="127.0.0.1" port protocol="tcp" port="6379" accept' --permanent

# 重新加载防火墙配置
firewall-cmd --reload
```

**最佳实践**：生产环境SSH应修改默认端口，禁用密码登录，仅允许密钥登录，并配置防暴力破解工具（如fail2ban）。

## 二、核心组件部署

### 1. JDK安装与配置

**版本选择依据**：
- SpringBoot 2.7.x 推荐 JDK 11（LTS版本，长期支持）
- SpringBoot 3.x 必须使用 JDK 17+

**安装步骤（以JDK 11为例）**：

```bash
# 下载JDK（可从Oracle官网或OpenJDK获取）
wget https://download.java.net/java/GA/jdk11/9/GPL/openjdk-11.0.2_linux-x64_bin.tar.gz

# 创建安装目录
mkdir -p /usr/local/java

# 解压
tar -zxvf openjdk-11.0.2_linux-x64_bin.tar.gz -C /usr/local/java/

# 配置环境变量
cat >> /etc/profile << 'EOF'
export JAVA_HOME=/usr/local/java/jdk-11.0.2
export PATH=$JAVA_HOME/bin:$PATH
export CLASSPATH=.:$JAVA_HOME/lib/dt.jar:$JAVA_HOME/lib/tools.jar
EOF

# 使环境变量生效
source /etc/profile

# 验证安装
java -version
```

**多版本切换方案**：

```bash
# 安装另一个版本JDK
# ... 安装步骤同上，假设安装到/usr/local/java/jdk-17.0.8

# 配置 alternatives（CentOS）
alternatives --install /usr/bin/java java /usr/local/java/jdk-11.0.2/bin/java 100
alternatives --install /usr/bin/java java /usr/local/java/jdk-17.0.8/bin/java 200

# 切换版本
alternatives --config java

# 同样配置javac
alternatives --install /usr/bin/javac javac /usr/local/java/jdk-11.0.2/bin/javac 100
alternatives --install /usr/bin/javac javac /usr/local/java/jdk-17.0.8/bin/javac 200
alternatives --config javac
```

**环境差异**：
- 开发环境：可安装多个JDK版本方便测试兼容性
- 生产环境：只安装应用所需的特定JDK版本，减少不必要的依赖

**常见问题**：
- 环境变量不生效：检查配置文件是否正确，是否执行`source`命令，或重新登录
- 权限问题：确保JDK目录权限正确，一般设置为`755`

### 2. MySQL安装与配置

**版本选择**：推荐MySQL 8.0（性能和安全性较5.7有提升）

**安装步骤（CentOS 7）**：

```bash
# 安装MySQL官方仓库
wget https://dev.mysql.com/get/mysql80-community-release-el7-3.noarch.rpm
rpm -ivh mysql80-community-release-el7-3.noarch.rpm

# 安装MySQL
yum install -y mysql-community-server

# 启动MySQL并设置开机自启
systemctl start mysqld
systemctl enable mysqld

# 查看初始密码
grep 'temporary password' /var/log/mysqld.log
```

**安全初始化配置**：

```bash
# 执行安全配置脚本
mysql_secure_installation

# 按照提示进行以下配置：
# 1. 输入初始密码
# 2. 设置新密码（注意密码强度，生产环境需包含大小写字母、数字和特殊字符）
# 3. 移除匿名用户（是）
# 4. 禁止root远程登录（是）
# 5. 移除test数据库（是）
# 6. 重新加载权限表（是）
```

**创建应用专用用户（重要！禁止使用root连接应用）**：

```bash
# 登录MySQL
mysql -u root -p

# 创建数据库
CREATE DATABASE app_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

# 创建应用用户（仅允许从本地连接）
CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'StrongPassword@123';

# 如果应用和数据库不在同一服务器，允许指定IP连接
CREATE USER 'app_user'@'192.168.1.100' IDENTIFIED BY 'StrongPassword@123';

# 授予权限
GRANT ALL PRIVILEGES ON app_db.* TO 'app_user'@'localhost';
GRANT ALL PRIVILEGES ON app_db.* TO 'app_user'@'192.168.1.100';

# 刷新权限
FLUSH PRIVILEGES;

# 退出
exit
```

**数据目录迁移（生产环境推荐，避免系统盘满导致数据库崩溃）**：

```bash
# 停止MySQL服务
systemctl stop mysqld

# 创建新的数据目录（假设挂载了/data分区）
mkdir -p /data/mysql
chown -R mysql:mysql /data/mysql

# 复制原有数据
cp -a /var/lib/mysql/* /data/mysql/

# 修改配置文件
vim /etc/my.cnf

# 添加或修改以下配置
[mysqld]
datadir=/data/mysql
socket=/data/mysql/mysql.sock

[client]
socket=/data/mysql/mysql.sock

# 启动MySQL
systemctl start mysqld
```

**MySQL配置优化（/etc/my.cnf）**：

```ini
[mysqld]
# 基础设置
datadir=/data/mysql
socket=/data/mysql/mysql.sock
pid-file=/var/run/mysqld/mysqld.pid
user=mysql
port=3306
default_authentication_plugin=mysql_native_password

# 性能优化（根据服务器配置调整）
max_connections=1000
wait_timeout=600
interactive_timeout=600
table_open_cache=2048
innodb_buffer_pool_size=16G  # 建议设置为服务器内存的50-70%
innodb_flush_log_at_trx_commit=1  # 生产环境建议1，确保ACID
innodb_log_buffer_size=16M
innodb_log_file_size=256M
innodb_flush_method=O_DIRECT

# 字符集设置
character-set-server=utf8mb4
collation-server=utf8mb4_unicode_ci

# 慢查询日志（生产环境建议开启，便于性能优化）
slow_query_log=1
slow_query_log_file=/data/mysql/slow.log
long_query_time=2  # 超过2秒的查询记录为慢查询

# 日志轮转
log-error=/var/log/mysqld.log
expire_logs_days=7
```

**数据库备份脚本**：

```bash
# 创建备份脚本
vim /usr/local/bin/backup_mysql.sh

# 脚本内容
#!/bin/bash
BACKUP_DIR="/data/backup/mysql"
DATE=$(date +%Y%m%d_%H%M%S)
DB_NAME="app_db"
DB_USER="app_user"
DB_PASS="StrongPassword@123"

# 创建备份目录
mkdir -p $BACKUP_DIR

# 备份数据库
mysqldump -u$DB_USER -p$DB_PASS --databases $DB_NAME > $BACKUP_DIR/$DB_NAME\_$DATE.sql

# 压缩备份文件
gzip $BACKUP_DIR/$DB_NAME\_$DATE.sql

# 删除7天前的备份
find $BACKUP_DIR -name "*.sql.gz" -mtime +7 -delete

# 添加执行权限
chmod +x /usr/local/bin/backup_mysql.sh

# 添加到定时任务，每天凌晨2点执行
crontab -e
# 添加以下内容
0 2 * * * /usr/local/bin/backup_mysql.sh
```

**环境差异**：
- 开发环境：可使用简单密码，允许远程连接方便调试
- 测试环境：密码强度适中，仅允许测试服务器访问
- 生产环境：强密码策略，严格限制访问来源，必须配置定期备份

**风险提示**：
- 使用root账号直接连接应用是高危操作，一旦应用被入侵，攻击者可获取数据库完全控制权
- 生产环境不配置数据库备份可能导致数据丢失无法恢复
- MySQL默认配置性能较差，需要根据服务器配置进行优化

### 3. Redis安装与配置（可选）

**安装步骤**：

```bash
# 安装依赖
yum install -y gcc gcc-c++ make

# 下载Redis（选择稳定版本）
wget http://download.redis.io/releases/redis-6.2.6.tar.gz

# 解压并编译
tar -zxvf redis-6.2.6.tar.gz
cd redis-6.2.6
make
make install PREFIX=/usr/local/redis

# 创建配置文件目录
mkdir -p /usr/local/redis/conf
cp redis.conf /usr/local/redis/conf/
```

**配置优化（/usr/local/redis/conf/redis.conf）**：

```ini
# 以守护进程方式运行
daemonize yes

# 绑定地址（生产环境建议仅绑定本地）
bind 127.0.0.1

# 端口设置
port 6379

# 设置密码（生产环境必须！）
requirepass StrongRedisPassword@456

# 工作目录
dir /data/redis

# 内存配置（根据服务器内存调整，避免OOM）
maxmemory 4G
maxmemory-policy allkeys-lru  # 内存满时删除最少使用的key

# 持久化配置
# RDB配置（适合数据不频繁更新场景）
save 900 1    # 900秒内有1个key变化则保存
save 300 10   # 300秒内有10个key变化则保存
save 60 10000 # 60秒内有10000个key变化则保存
dbfilename dump.rdb

# AOF配置（适合数据安全性要求高的场景）
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec  # 每秒同步一次，平衡性能和安全性

# 避免缓存穿透配置
# 开启保护模式
protected-mode yes

# 限制并发连接
maxclients 10000
```

**创建Redis系统服务**：

```bash
# 创建服务文件
vim /etc/systemd/system/redis.service

# 内容如下
[Unit]
Description=Redis Server
After=network.target

[Service]
Type=forking
ExecStart=/usr/local/redis/bin/redis-server /usr/local/redis/conf/redis.conf
ExecStop=/usr/local/redis/bin/redis-cli -a StrongRedisPassword@456 shutdown
Restart=always

[Install]
WantedBy=multi-user.target

# 启动Redis并设置开机自启
systemctl daemon-reload
systemctl start redis
systemctl enable redis
```

**环境差异**：
- 开发环境：可关闭密码，允许远程连接方便调试
- 测试环境：设置简单密码，限制内部网络访问
- 生产环境：必须设置强密码，仅允许本地或特定服务器访问，合理配置内存限制

**风险提示**：
- Redis无密码且暴露在公网是极高危操作，可能被攻击者入侵并植入挖矿程序
- 不设置maxmemory可能导致Redis耗尽服务器内存，引发系统OOM
- 持久化策略选择不当可能导致数据丢失

### 4. Nginx安装与配置

**安装步骤（CentOS 7）**：

```bash
# 安装官方仓库
rpm -Uvh http://nginx.org/packages/centos/7/noarch/RPMS/nginx-release-centos-7-0.el7.ngx.noarch.rpm

# 安装Nginx
yum install -y nginx

# 启动Nginx并设置开机自启
systemctl start nginx
systemctl enable nginx
```

**基础配置（/etc/nginx/nginx.conf）**：

```nginx
user  nginx;
worker_processes  auto;  # 自动设置为CPU核心数

error_log  /var/log/nginx/error.log warn;
pid        /var/run/nginx.pid;

events {
    worker_connections  1024;  # 每个worker进程的最大连接数
    use epoll;  # Linux下高效的事件驱动模型
}

http {
    include       /etc/nginx/mime.types;
    default_type  application/octet-stream;

    # 日志格式
    log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
                      '$status $body_bytes_sent "$http_referer" '
                      '"$http_user_agent" "$http_x_forwarded_for"';

    access_log  /var/log/nginx/access.log  main;

    # 性能优化
    sendfile        on;
    tcp_nopush      on;
    tcp_nodelay     on;
    keepalive_timeout  65;
    types_hash_max_size 2048;

    # 开启gzip压缩
    gzip on;
    gzip_min_length 1k;
    gzip_buffers 4 16k;
    gzip_http_version 1.1;
    gzip_comp_level 5;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss text/javascript;
    gzip_vary on;

    # 引入站点配置
    include /etc/nginx/conf.d/*.conf;
}
```

**前后端分离配置（/etc/nginx/conf.d/app.conf）**：

```nginx
server {
    listen 80;
    server_name example.com;  # 替换为你的域名

    # 前端静态资源配置
    root /var/www/html;
    index index.html;

    # 静态资源缓存策略
    location ~* \.(jpg|jpeg|png|gif|ico|css|js)$ {
        expires 7d;  # 图片、CSS、JS缓存7天
        add_header Cache-Control "public, max-age=604800";
    }

    # 前端路由支持（SPA应用必备）
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 后端API反向代理
    location /api/ {
        proxy_pass http://127.0.0.1:8080/api/;  # 后端服务地址
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        
        # 超时设置
        proxy_connect_timeout 300s;
        proxy_read_timeout 300s;
        proxy_send_timeout 300s;
    }

    # 健康检查接口
    location /actuator/health {
        proxy_pass http://127.0.0.1:8080/actuator/health;
        allow 127.0.0.1;  # 仅允许本地访问
        deny all;
    }
}
```

**HTTPS配置（Let's Encrypt证书）**：

```bash
# 安装Certbot
yum install -y epel-release
```

**负载均衡配置（多后端实例时）**：

```nginx
# 在http块中添加
upstream backend_servers {
    server 127.0.0.1:8080 weight=1;
    server 127.0.0.1:8081 weight=1;
    # 可添加更多后端服务器
    ip_hash;  # 会话保持，同一客户端固定访问同一服务器
    # 健康检查
    keepalive 32;
}

# 在server块中修改反向代理
location /api/ {
    proxy_pass http://backend_servers/api/;
    # 其他配置同上
}
```

**环境差异**：
- 开发环境：简化配置，关闭缓存方便调试
- 测试环境：基本与生产环境一致，但可关闭HTTPS简化配置
- 生产环境：完整配置，开启HTTPS、缓存、日志等

**常见问题**：
- 404错误：检查root路径是否正确，静态文件是否存在
- 502错误：后端服务未启动或端口错误，检查proxy_pass配置
- 504错误：后端响应超时，调整proxy_read_timeout配置

## 三、应用部署

### 1. 后端SpringBoot应用部署

**创建应用目录与用户**：

```bash
# 创建应用目录
mkdir -p /opt/apps/backend
mkdir -p /opt/apps/backend/logs
mkdir -p /opt/apps/backend/config

# 创建专用用户运行应用（避免使用root）
useradd -s /sbin/nologin appuser
chown -R appuser:appuser /opt/apps/backend
```

**上传应用文件**：

```bash
# 假设本地文件为app.jar，通过scp上传
scp app.jar root@server_ip:/opt/apps/backend/

# 上传配置文件
scp application-prod.yml root@server_ip:/opt/apps/backend/config/

# 修改权限
chown appuser:appuser /opt/apps/backend/app.jar
chown appuser:appuser /opt/apps/backend/config/application-prod.yml
```

**创建启动脚本（/opt/apps/backend/start.sh）**：

```bash
#!/bin/bash
APP_NAME=app.jar
APP_DIR=/opt/apps/backend
LOG_DIR=$APP_DIR/logs
CONFIG_FILE=$APP_DIR/config/application-prod.yml

# JVM参数优化（根据服务器内存调整）
# 8G内存服务器示例
JVM_OPTS="-Xms4g -Xmx4g -XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m"
JVM_OPTS="$JVM_OPTS -XX:+UseG1GC -XX:MaxGCPauseMillis=200"
JVM_OPTS="$JVM_OPTS -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=$LOG_DIR/heapdump.hprof"
JVM_OPTS="$JVM_OPTS -XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:$LOG_DIR/gc.log"
JVM_OPTS="$JVM_OPTS -Djava.security.egd=file:/dev/./urandom"

# 启动应用
nohup java $JVM_OPTS -jar $APP_DIR/$APP_NAME --spring.config.location=$CONFIG_FILE > $LOG_DIR/app.log 2>&1 &

# 记录PID
echo $! > $APP_DIR/app.pid
```

**创建停止脚本（/opt/apps/backend/stop.sh）**：

```bash
#!/bin/bash
APP_DIR=/opt/apps/backend

if [ -f "$APP_DIR/app.pid" ]; then
    PID=$(cat $APP_DIR/app.pid)
    echo "Stopping application with PID: $PID"
    kill $PID
    
    # 等待进程停止
    sleep 5
    
    # 检查是否停止成功
    if ps -p $PID > /dev/null; then
        echo "Force stopping application"
        kill -9 $PID
    fi
    
    rm -f $APP_DIR/app.pid
else
    echo "Application is not running"
fi
```

**创建重启脚本（/opt/apps/backend/restart.sh）**：

```bash
#!/bin/bash
APP_DIR=/opt/apps/backend

$APP_DIR/stop.sh
sleep 3
$APP_DIR/start.sh
```

**设置脚本权限**：

```bash
chmod +x /opt/apps/backend/start.sh
chmod +x /opt/apps/backend/stop.sh
chmod +x /opt/apps/backend/restart.sh
chown appuser:appuser /opt/apps/backend/*.sh
```

**创建系统服务**：


**JVM参数优化建议**：
- 内存小于4G：`-Xms2g -Xmx2g -XX:MetaspaceSize=128m -XX:MaxMetaspaceSize=256m`
- 内存8-16G：`-Xms4g -Xmx4g -XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m`
- 内存16G以上：`-Xms8g -Xmx8g -XX:MetaspaceSize=512m -XX:MaxMetaspaceSize=1g`
- 生产环境建议使用G1GC收集器：`-XX:+UseG1GC -XX:MaxGCPauseMillis=200`

**环境差异**：
- 开发环境：JVM参数可简化，开启远程调试`-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=5005`
- 测试环境：JVM参数接近生产环境，便于性能测试
- 生产环境：完整JVM参数配置，开启GC日志和OOM Dump

## 五、一键部署脚本与自动化

### 1. 一键部署脚本示例

```bash
#!/bin/bash
# 应用部署脚本
# 使用方法: ./deploy.sh [version]

# 参数校验
if [ $# -ne 1 ]; then
    echo "Usage: $0 [version]"
    exit 1
fi

VERSION=$1
APP_NAME="app"
BACKEND_DIR="/opt/apps/backend"
BACKEND_JAR="${APP_NAME}-${VERSION}.jar"
REMOTE_SERVER="user@server_ip"

# 检查本地文件是否存在
if [ ! -f "./target/${BACKEND_JAR}" ]; then
    echo "Error: ./target/${BACKEND_JAR} not found"
    exit 1
fi

echo "Starting deployment of version ${VERSION}..."

# 1. 备份远程服务器当前版本
echo "Backing up current version..."
ssh ${REMOTE_SERVER} "cd ${BACKEND_DIR} && if [ -f ${APP_NAME}.jar ]; then mv ${APP_NAME}.jar ${APP_NAME}_backup_$(date +%Y%m%d).jar; fi"

# 2. 上传新版本
echo "Uploading new version..."
scp ./target/${BACKEND_JAR} ${REMOTE_SERVER}:${BACKEND_DIR}/

# 3. 远程部署
echo "Deploying application..."
ssh ${REMOTE_SERVER} "cd ${BACKEND_DIR} && ln -sf ${BACKEND_JAR} ${APP_NAME}.jar && systemctl restart backend"

# 4. 检查部署状态
echo "Checking deployment status..."
ssh ${REMOTE_SERVER} "systemctl status backend | grep 'active (running)'"

if [ $? -eq 0 ]; then
    echo "Deployment successful!"
    exit 0
else
    echo "Deployment failed! Check logs for details."
    exit 1
fi
```

### 3. CI/CD集成（Jenkins示例）

```groovy
            }

                branch 'main'

            }
        }
    }
    
    post {

            junit '**/target/surefire-reports/*.xml'
        }
    }
}
```

## 六、企业级方案对比与最佳实践

### 1. 部署方案对比

| 方案         | 优势                     | 劣势                     | 适用场景                         |
| ------------ | ------------------------ | ------------------------ | -------------------------------- |
| 传统部署     | 简单直接，学习成本低     | 环境一致性差，部署效率低 | 小型项目，团队技术栈有限         |
| Docker容器化 | 环境一致性好，部署简单   | 需要学习Docker知识       | 中大型项目，团队有一定技术基础   |
| Kubernetes   | 自动化程度高，可扩展性强 | 复杂度高，资源消耗大     | 大型项目，微服务架构，高可用要求 |

### 2. 最佳实践总结

1.** 环境隔离 **：开发、测试、生产环境严格隔离，配置分离
2.** 安全优先 **：最小权限原则，禁止使用root运行应用，关键服务必须设置密码
3.** 自动化部署 **：尽可能实现CI/CD自动化，减少人工操作
4.** 监控告警 **：完善的监控体系，及时发现和解决问题
5.** 备份策略 **：定期备份关键数据，并测试恢复流程
6.** 文档完善 **：记录部署流程、配置说明、应急处理方案
