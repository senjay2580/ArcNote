# 单例模式

**从作用、代码实现、应用场景说明**

![image-20250516091158866](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250516091158866.png)



**单例模式的主要作用**是保证一个类在程序运行期间只有一个实例
并提供一个全局访问点。这在需要频繁创建销毁对象或要求全局唯一  （**确保一些特定类只有一个实例，从而避免不必要的对象创建和资源消耗。**）
性的场景中非常有用，比如**数据库连接池、配置管理、日志系统**。

**一个应用程序通常只需要一个日志记录器来记录日志信息。单例模式可以确保在整个应用程序中使用同一个日志记录器实例，便于统一管理和输出日志。**

在某些情况下，系统需要一个全局的配置管理器来存储和读取配置信息，单例模式可以确保配置信息在整个应用程序中的一致性和可靠性。

---



**单例模式有两种类型**
**懒汉式:**  在真正需要使用对象时才去创建该单例类对象
**饿汉式: ** 在类加载时已经创建好该单例对象，等待被程序使用



懒汉式创建单例对象
懒汉式创建对象的方法是在程序使用对象前，先判断该对象是否已经实例化（判空），若已实例化直接返回该类对象。，否则则先执行实例化操作。

![image.png](https://i-blog.csdnimg.cn/blog_migrate/86772638fb03a27cf5b1879e91b340b3.png)



~~~java

public class Singleton {
    
    private static Singleton singleton;
    
    private Singleton(){}
    
    public static Singleton getInstance() {
        if (singleton == null) {
            singleton = new Singleton();
        }
        return singleton;
    }
    

~~~



饿汉式在类加载时已经创建好该对象，在程序调用时直接返回该单例对象即可，即我们在编码时就已经指明了要马上创建这个对象，不需要等到被调用时再去创建。

关于类加载，涉及到JVM的内容，我们目前可以简单认为在程序启动时，这个单例对象就已经创建好了。

![image.png](https://i-blog.csdnimg.cn/blog_migrate/ca90e11807a700384a107eacd9939b42.png)

---



回顾懒汉式的核心方法

~~~java
public static Singleton getInstance() {
    if (singleton == null) {
        singleton = new Singleton();
    }
    return singleton;
}
~~~

这个方法其实是存在问题的，试想一下，如果两个线程同时判断singleton为空，那么它们都会去实例化一个Singleton对象，这就变成双例了。所以，我们要解决的是**线程安全问题。**

![image.png](https://i-blog.csdnimg.cn/blog_migrate/1fd02a8dbaf4c1b62a2435093094b3b1.png)

加锁:

~~~java
public static synchronized Singleton getInstance() {
    if (singleton == null) {
        singleton = new Singleton();
    }
    return singleton;
}
// 或者
public static Singleton getInstance() {
    synchronized(Singleton.class) {   
        if (singleton == null) {
            singleton = new Singleton();
        }
    }
    return singleton;
}
~~~

这样就规避了两个线程同时创建Singleton对象的风险，但是引来另外一个问题：每次去获取对象都需要先获取锁，并发性能非常地差，极端情况下，可能会出现卡顿现象。

接下来要做的就是优化性能，目标是：如果没有实例化对象则加锁创建，如果已经实例化了，则不需要加锁，直接获取实例

所以直接在方法上加锁的方式就被废掉了，因为这种方式无论如何都需要先获取锁

~~~java

public static Singleton getInstance() {
    if (singleton == null) {  // 线程A和线程B同时看到singleton = null，如果不为null，则直接返回singleton
        synchronized(Singleton.class) { // 线程A或线程B获得该锁进行初始化
            if (singleton == null) { // 其中一个线程进入该分支，另外一个线程则不会进入该分支
                singleton = new Singleton();
            }
        }
    }
    return singleton;

~~~

**指令重排问题：**



创建一个对象，在JVM中会经过三步：

（1）为singleton分配内存空间

（2）初始化singleton对象

（3）将singleton指向分配好的内存空间

![image-20250516092611388](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250516092611388.png)

# 工厂模式

**<mark style="background: #FF5582A6;">简单工厂模式（并不是设计模式，只是一种编程习惯）  工厂方法  抽象工厂</mark>**
**作用:**

**将对象的创建与使用分离**，降低耦合。

**对扩展开放，对修改关闭**（符合开闭原则）。

**屏蔽具体实现，面向接口编程**，方便后期扩展和维护。



实现方式（代码实现）：工厂模式（不属于设计模式）、**简单工厂模式 、抽象工厂模式**

使用场景：

比如有一个业务功能根据条件的不同 对应多种不同的代码逻辑

**需要根据配置或条件决定创建哪种对象**（如不同数据库连接、支付渠道、）

# 代理模式

代理，在其最广泛的形式中，是一个用作其他事物接口的类。代理是一个包装或代理对象，客户端通过它来访问后台的实际服务对象。**使用代理可以简单地将请求转发到实际对象，或者提供额外的逻辑。在代理中可以提供额外的功能，**例如，当对实际对象的操作资源消耗较大时进行缓存，或在调用实际对象的操作之前进行前置条件检查。

# 观察者模式（发布-订阅模式）


观察者模式(Observer Pattern):定义对象间的一种一对多依赖关系，使得每当一个对象状态发生改变时，其相关依赖对象皆得到通知并被自动更新

作用：

**在观察者模式中，主题对象(Subject)和观察者对象(Observer)之间通过接口或抽象类进行交互，而不是直接依赖于具体的类。这种松耦合的设计使得对象之间的依赖性降低，提高了系统的灵活性和可维护性。**

**符合开闭原则，**



---



（接口定义了一组方法和行为的契约，但不提供具体实现。具体实现由实现接口的类来完成。这种分离使得客户端代码(使用接口的代码)只需要**依赖于抽象的接口**（不依赖具体的类），而不需要知道具体的实现细节  <span style="color:#CC0000;">，通过接口交互，可以在不修改现有代码的情况下，轻松地添加新的实现</span>。只要新的实现类遵循相同的接口契约，<span style="color:#CC6600;">客户端代码就可以无缝地使用新的实现</span>。）

（依赖于接口而不是具体的类，可以显著降低类之间的耦合度。耦合度是指两个或多个类之间的依赖关系。高耦合度的系统难以维护和扩展，因为一个类的改变可能会影响多个依赖于它的类。）

# 模板方法模式

**作用：**定义算法骨架，把不变的行为搬到父类中，变化的行为交由子类实现。

**实现方法：**见下



---

**静态行为**
在编译时确定，无法在运行时改变。例如:静态方法(static 方法)静态属性(类属性)编译时方法绑定(如 final方
法)
特点:直接通过类调用，与实例无关，性能高但灵活性低。



**动态行为**
在运行时确定，可通过继承、多态或反射修改。例如:实例方法(动态绑定，依赖对象状态)
多态方法(通过虚方法表动态分
发)
特点:行为可能随运行时上下文变化，灵活但性能略低。

**继承与多态**

**设计模式** 	如**装饰器模式(动态添加职责)策略模式(动态切换算法)**

**反射(Reflection)**
直接操作类结构的关键技

**应用场景：一个业务流程的步骤固定，但每个步骤的具体实现可能因具体业务而异时，模板方法模式非常适用。例如，在一个订单处理系统中，订单处理的流程(如验证订单、处理支付、发货、通知客户等)是固定的，但每个步骤的具体实现可能因订单类型或客户类型而不同。**

# 适配器模式

适配器:就是将两个不兼容的东西兼容起来

![image-20250523124029822](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250523124029822.png)



### 1. 类适配器（使用继承）

适配器继承目标接口和被适配的类。

```java
// 目标接口
public interface Target {
    void request();
}

// 已存在的不兼容类
public class Adaptee {
    public void specificRequest() {
        System.out.println("Adaptee's specificRequest");
    }
}

// 适配器类（继承 + 实现）
public class Adapter extends Adaptee implements Target {
    @Override
    public void request() {
        specificRequest();  // 调用旧方法
    }
}
```

### 2. 对象适配器（使用组合）✅ 推荐

适配器**持有被适配对象的引用**（更加灵活）。

```java
public class Adapter implements Target {
    private Adaptee adaptee;

    public Adapter(Adaptee adaptee) {
        this.adaptee = adaptee;
    }

    @Override
    public void request() {
        adaptee.specificRequest();
    }
}
```

### 3. 接口适配器（默认空实现）

适用于接口中方法太多，子类只需部分实现的场景。

```java
public interface Listener {
    void onClick();
    void onHover();
    void onDrag();
}

// 空实现适配器
public abstract class ListenerAdapter implements Listener {
    public void onClick() {}
    public void onHover() {}
    public void onDrag() {}
}

// 子类只需重写部分方法
public class MyClickListener extends ListenerAdapter {
    public void onClick() {
        System.out.println("Clicked!");
    }
}
```

------



| 场景         | 示例                                                   |
| ------------ | ------------------------------------------------------ |
| 接口不兼容   | 老系统使用的接口和新系统不同，但要互相调用。           |
| 第三方库适配 | 你想使用第三方库，但接口不一致时，用适配器来中间转接。 |
| 多种接口标准 | 客户端希望使用统一接口访问多种不同实现。               |



# 外观模式（Facade Pattern）



结构型设计模式，**为复杂系统提供一个统一的简化接口**   缺点就是：不满足开闭原则！！！

**一般包含三个角色：**

1. **Facade（外观类）**：统一对外提供简化接口。
2. **Subsystem Classes（子系统类）**：实现复杂的具体逻辑，外部通常不直接调用它们。
3. **Client（客户端）**：通过 Facade 调用子系统功能。

#### 子系统类（复杂的功能）

```java
class Player {
    void powerOn() {
        System.out.println("打开播放器");
    }

    void loadMovie(String name) {
        System.out.println("加载电影：" + name);
    }

    void play() {
        System.out.println("开始播放电影");
    }

    void setVolume(int level) {
        System.out.println("设置音量为：" + level);
    }
}
```

#### 外观类（简化接口）

```java
class MovieFacade {
    private Player player = new Player();

    public void watchMovie(String movieName) {
        player.powerOn();
        player.loadMovie(movieName);
        player.setVolume(10);
        player.play();
    }
}
```

#### 客户端代码

```java
public class Main {
    public static void main(String[] args) {
        MovieFacade facade = new MovieFacade();
        facade.watchMovie("《流浪地球3》");
    }
}
```

✅ **客户端无需关心播放器如何开机、怎么调音量等细节，只关心 watchMovie() 就够了。**

| 场景                             | 举例                                                  |
| -------------------------------- | ----------------------------------------------------- |
| 系统很复杂，希望隐藏子系统细节   | SpringMVC 中的 `DispatcherServlet` 就是前端控制器外观 |
| 提供多个模块的统一入口           | Java EE 中的 `Facade Bean` 用来封装多个 EJB 的调用    |
| 与旧系统交互，但不想暴露复杂接口 | 把旧系统封装成简单 Facade 接口，供新系统调用          |
| 封装第三方库的使用               | 封装 SDK 的一堆配置和初始化，只暴露统一简化调用场景   |

# 状态模式

![image-20250523130238729](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250523130238729.png)

**✅ 使用卫语句（清晰利落）**  站在函数最前面把不符合要求的人拦住

```
java复制编辑public void process(User user) {
    if (user == null) return;
    if (!user.isActive()) return;
    if (!user.hasPermission("admin")) return;s

    // 执行核心逻辑
    doSomething(user);
}
```

| 作用               | 说明                                                         |
| ------------------ | ------------------------------------------------------------ |
| ✅ 组织复杂状态逻辑 | **每个状态封装到独立类中，避免冗长的 `if-else` 或 `switch`** |
| ✅ 行为与状态解耦   | 状态切换行为不再硬编码到上下文类中                           |
| ✅ 易于扩展         | 增加新状态时只需新建类，不会影响其他状态                     |

**角色构成：**
**Context（环境类）**：持有当前状态，提供接口给外部调用，内部根据状态委托行为。

**State（抽象状态类）：**定义状态接口，不同状态有不同实现。

**ConcreteState（具体状态类）：**每个状态封装自己的行为和可能的状态切换逻辑。



#### 1️⃣ 抽象状态类：

```java
interface LiftState {
    void open();     // 开门
    void close();    // 关门
    void run();      // 运行
    void stop();     // 停止
}
```

------

#### 2️⃣ 环境类：

```java
class Context {
    private LiftState currentState;

    public void setState(LiftState state) {
        this.currentState = state;
    }

    public void open() {
        currentState.open();
    }

    public void close() {
        currentState.close();
    }

    public void run() {
        currentState.run();
    }

    public void stop() {
        currentState.stop();
    }
}
```

------

#### 3️⃣ 具体状态类：

具体的状态还需要一个环境类 来切换状态!!!

```java
class OpenState implements LiftState {
    private Context context;

    public OpenState(Context context) {
        this.context = context;
    }

    public void open() {
        System.out.println("电梯门已打开");
    }

    public void close() {
        System.out.println("正在关闭电梯门...");
        context.setState(new ClosedState(context));
    }

    public void run() {
        System.out.println("不能运行，电梯门还开着！");
    }

    public void stop() {
        System.out.println("电梯已经停止");
    }
}
```

其它状态类如 `ClosedState`、`RunningState`、`StoppedState` 同理，只是行为和状态转换逻辑不同。

------

#### 4️⃣ 客户端调用：

```java
public class Main {
    public static void main(String[] args) {
        Context lift = new Context();
        lift.setState(new OpenState(lift));

        lift.open();  // 电梯门已打开
        lift.run();   // 不能运行，电梯门还开着！
        lift.close(); // 正在关闭电梯门...
    }
}
```

# 责任链模式

解耦请求发送者与接收者，实现请求的链式传递

将请求沿着一条处理链传递，直到被某个处理者处理为止。

| 作用                            | 说明                                   |
| ------------------------------- | -------------------------------------- |
| ✅ 解耦请求和处理者              | 发送者不需要知道谁最终处理请求         |
| ✅ 灵活添加/修改处理逻辑         | 增删处理节点无需影响其他节点           |
| ✅ 避免代码中大量 if-else/switch | 替代冗长判断结构，让职责单一、职责清晰 |

**通常包含以下角色：**

| 角色                          | 说明                                               |
| ----------------------------- | -------------------------------------------------- |
| Handler（抽象处理者）         | 定义处理请求的接口，同时包含指向下一个处理者的引用 |
| ConcreteHandler（具体处理者） | 实现请求处理逻辑，决定是否处理或传递给下一个处理者 |
| Client（客户端）              | 构造责任链并发起请求                               |

~~~java
abstract class Approver {
    protected Approver next;

    public void setNext(Approver next) {
        this.next = next;
    }

    public abstract void handleRequest(int days);
}

------------------------------------------
class TeamLeader extends Approver {
    public void handleRequest(int days) {
        if (days <= 2) {
            System.out.println("组长批准了" + days + "天假");
        } else if (next != null) {
            next.handleRequest(days);
        }
    }
}

class Manager extends Approver {
    public void handleRequest(int days) {
        if (days <= 5) {
            System.out.println("经理批准了" + days + "天假");
        } else if (next != null) {
            next.handleRequest(days);
        }
    }
}

class Director extends Approver {
    public void handleRequest(int days) {
        if (days <= 10) {
            System.out.println("总监批准了" + days + "天假");
        } else {
            System.out.println("请假天数过多，不予批准");
        }
    }
}

~~~



~~~java
public class Main {
    public static void main(String[] args) {
        Approver teamLeader = new TeamLeader();
        Approver manager = new Manager();
        Approver director = new Director();

        teamLeader.setNext(manager);
        manager.setNext(director);

        teamLeader.handleRequest(1);  // 组长批准
        teamLeader.handleRequest(4);  // 经理批准
        teamLeader.handleRequest(8);  // 总监批准
        teamLeader.handleRequest(15); // 不予批准
    }
}

~~~

常见场景:审批流程、拦截器、日志处理、过滤器链等

# 享元（FlyWeight）模式

通过共享相同对象，减少系统中对象的数量，从而节省内存、提高性能（对象复用，减少 GC 压力和创建时间）。

**应用：池化技术**（利用工厂统一管理享元对象）



使用享元模式要注意以下几点：

1. **有大量相同或相似对象**（如图标、字符、商品等）
2. 对象中存在可以**分离的内部状态**（共享）与**外部状态**（由外部传入）
3. **对象创建成本高**（如初始化耗时/耗资源）

# 生成器（建造者）模式



| 项目         | 不使用生成器模式                                  | 使用生成器模式                           |
| ------------ | ------------------------------------------------- | ---------------------------------------- |
| 写法         | 调用构造方法，或者逐一调用 `setXXX()` 方法        | 链式调用 `builder.xxx().yyy().build()`   |
| 构建顺序     | 依赖于构造函数或 setter 方法的调用顺序            | 不依赖顺序，链式表达                     |
| 可选参数支持 | 构造函数不支持，必须重载多个构造函数或使用 `null` | 支持天然可选参数，按需设置               |
| 可读性       | 差，容易参数错位或看不清含义                      | 好，链式结构清晰直观                     |
| 扩展性       | 差，改参数要重写构造函数或 setter                 | 好，只需要在 builder 中添加方法          |
| 适合场景     | 参数少、结构简单                                  | 参数多、结构复杂、可选项多、构建流程多样 |

# 原型模式

：**通过复制（克隆）已有的对象来创建新对象，而不是通过 new 实例化对象**。

**主要作用是：高效创建对象，避免复杂对象重复创建。**

- 当一个对象的创建成本很高（如数据库连接、大量计算等），而需要大量相似对象时，可以通过“克隆”已有对象，快速创建新实例。

- 实现“对象复制”，特别是深拷贝（含对象内部引用）和浅拷贝（复制基本字段）的控制。

- 

- | 缺点           | 描述                                                         |
  | -------------- | ------------------------------------------------------------ |
  | 克隆复杂       | 尤其是深拷贝，对象引用之间的复制不容易处理好                 |
  | 会引发安全问题 | 如果对象中含有资源句柄（如 socket、线程），克隆后可能引发资源冲突 |
  | 不利于调试     | 克隆对象是从内存复制来的，不容易跟踪对象生命周期             |

# 备忘录模式

**快照模式**，在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态，以便以后当需要时能将该对象恢复到原先保存的状态。

这个和原型模式有点相似

都需要访问**对象的私有属性**以完成状态的保存、克隆或还原

| 角色                            | 说明                                   |
| ------------------------------- | -------------------------------------- |
| **Originator（发起人/编辑器）** | 拥有内部状态，可以创建/恢复备忘录      |
| **Memento（备忘录）**           | 存储发起人的内部状态，但不暴露给外部   |
| **Caretaker（管理者）**         | 负责存储和恢复备忘录，但不能访问其内容 |

`Memento` 是 `Editor` 的内部类或包私有类



> **保护发起人（Originator）的封装性，避免其内部状态被外部随意访问或修改。**

**得到值但是修改不了，算不算破坏封装性？**

1. **得到值（只读访问）**

- 只读取对象的状态（比如通过**getter方法**或者返回一个不可变的副本）
- 不修改对象内部状态

**这通常不算破坏封装。**

> 因为封装的关键在于“**不让外部随意修改对象内部状态**”，只读访问一般是允许的。

**2. 修改内部状态（写操作）**

- 通过外部代码直接修改对象的内部字段或者状态
- 没有经过对象自己的控制

**这才算破坏封装。**

所以：

- **Memento 的状态成员变量只能由 Originator 设置和读取**（即“只对 Originator 可见”）
- **Caretaker 只能“保存”和“传递” Memento**，但不能访问里面的具体内容

# 迭代器模式

迭代器模式（**Iterator Pattern**）是设计模式中的一种**行为型模式**，用于**顺序访问一个集合对象中的元素**，**而不暴露其内部表示（结构）**。

| 角色                                | 职责说明                                     |
| ----------------------------------- | -------------------------------------------- |
| **Iterator**（迭代器接口）          | 定义遍历元素的方法，如 `hasNext()`、`next()` |
| **ConcreteIterator**（具体迭代器）  | 实现迭代器接口，负责具体遍历逻辑             |
| **Aggregate**（聚合类接口）         | 定义创建迭代器的接口                         |
| **ConcreteAggregate**（具体聚合类） | 实现聚合接口，返回一个迭代器对象             |

~~~java
// 迭代器接口
interface BookIterator {
    boolean hasNext();
    String next();
}
// 具体迭代器
 private class BookIteratorImpl implements BookIterator {
        private int index = 0;

        @Override
        public boolean hasNext() {
            return index < size;
        }

        @Override
        public String next() {
~~~

# 桥接模式

<span style="color:#FF0000;">**组合就是在一个类中包含另一个类的对象，通过这个对象来使用它的功能，而不是继承它的功能。**</span>

 桥接模式将“抽象部分”与“实现部分”分离，使它们都可以独立地变化。

它的核心思想是：

> 抽象（Abstraction）不再固定绑定某个具体实现（Implementor），而是通过组合的方式，**在运行时动态组合实现**。

**解耦抽象与实现**，避免类爆炸。

**支持多维度扩展**，比如一个“形状”类，它既可以有“颜色”维度，也有“形状”维度，**不想每种组合都生成一个子类。**

我们以“**消息系统**”为例来说明桥接模式：

- **消息类型**：普通消息、紧急消息
- **发送方式**：Email、短信、微信

如果用继承，每种组合都要一个类，非常多。桥接模式让我们把“消息的抽象” 和 “发送方式的实现” 分开

✅ 步骤一：定义实现部分接口（Implementor）

```java
// Implementor：发送消息的接口
public interface MessageSender {
    void send(String message, String to);
}
```

✅ 步骤二：具体实现类（ConcreteImplementor）

```java
public class EmailSender implements MessageSender {
    public void send(String message, String to) {
        System.out.println("发送 Email 给 " + to + ": " + message);
    }
}

public class SmsSender implements MessageSender {
    public void send(String message, String to) {
        System.out.println("发送短信给 " + to + ": " + message);
    }
}

public class WeChatSender implements MessageSender {
    public void send(String message, String to) {
        System.out.println("发送微信给 " + to + ": " + message);
    }
}
```

✅ 步骤三：定义抽象部分（Abstraction）

```java
// 抽象消息类
public abstract class Message {
    protected MessageSender sender; // 桥接点：组合而非继承

    public Message(MessageSender sender) {
        this.sender = sender;
    }

    public abstract void sendMessage(String message, String to);
}
```

✅ 步骤四：扩展抽象部分（RefinedAbstraction）

```java
// 普通消息
public class NormalMessage extends Message {
    public NormalMessage(MessageSender sender) {
        super(sender);
    }

    public void sendMessage(String message, String to) {
        sender.send("普通：" + message, to);
    }
}

// 加急消息
public class UrgentMessage extends Message {
    public UrgentMessage(MessageSender sender) {
        super(sender);
    }

    public void sendMessage(String message, String to) {
        sender.send("【加急】" + message, to);
    }
}
```

✅ 步骤五：客户端调用（动态组合）

```java
public class Client {
    public static void main(String[] args) {
        MessageSender emailSender = new EmailSender();
        MessageSender smsSender = new SmsSender();

        Message normalEmail = new NormalMessage(emailSender);
        normalEmail.sendMessage("你好，这是普通邮件", "张三");

        Message urgentSms = new UrgentMessage(smsSender);
        urgentSms.sendMessage("紧急通知：服务器宕机", "李四");
    }
}
```



# 中介者模式 Mediator Pattern 

mediate 调停 调解 协调

![image-20250526091932981](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250526091932981.png)

![image-20250526092056070](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250526092056070.png)

降低多个对象之间的耦合。

将**对象之间复杂的通信关系封装进一个中介对象**中，解耦对象间的依赖。

使系统更易维护和扩展。

| 角色名                           | 作用                                       |
| -------------------------------- | ------------------------------------------ |
| `Mediator`（抽象中介者）         | 定义统一的接口，用于通信协调               |
| `ConcreteMediator`（具体中介者） | 实现通信逻辑，协调同事对象                 |
| `Colleague`（同事类）            | 各个参与者，只与中介者通信，不直接互相调用 |

1. 抽象中介者：

```java
public interface Mediator {
    void sendMessage(String message, Colleague sender);
}
```

2. 同事抽象类：

```java
public abstract class Colleague {
    protected Mediator mediator;
    public Colleague(Mediator mediator) {
        this.mediator = mediator;
    }
    public abstract void receive(String message);
}
```

3. 具体同事类：

```java
public class User extends Colleague {
    private String name;

    public User(String name, Mediator mediator) {
        super(mediator);
        this.name = name;
    }

    public void send(String message) {
        System.out.println(this.name + " 发送消息: " + message);
        mediator.sendMessage(message, this);
    }

    @Override
    public void receive(String message) {
        System.out.println(this.name + " 收到消息: " + message);
    }
}
```

4. 具体中介者：

```java
import java.util.ArrayList;
import java.util.List;

public class ChatRoomMediator implements Mediator {
    private List<Colleague> users = new ArrayList<>();

    public void addUser(Colleague user) {
        users.add(user);
    }

    @Override
    public void sendMessage(String message, Colleague sender) {
        for (Colleague user : users) {
            if (user != sender) {
                user.receive(message);
            }
        }
    }
}
```

5. 使用中介者：

```java
public class Main {
    public static void main(String[] args) {
        ChatRoomMediator chatRoom = new ChatRoomMediator();

        User alice = new User("Alice", chatRoom);
        User bob = new User("Bob", chatRoom);
        User charlie = new User("Charlie", chatRoom);

        chatRoom.addUser(alice);
        chatRoom.addUser(bob);
        chatRoom.addUser(charlie);

        alice.send("大家好！");
        bob.send("你好，Alice！");
    }
```

# 访问者模式

在面向对象编程中**，访问者设计模式**通过将**算法与它操作的对象结构分离**，允许在不修改这些对象结构的情况下为其添加新操作。这也符合开放封闭原则(0pen/Closed Principle)。

适用场景：

当你需要在不修改对象结构的情况下，**为对象添加新功能时**，可以考虑使用访**问者模式**。这种模式特别适用于那些对象结构稳定但需要经常添加新操作的场景。通过访问者模式，你可以将操作与对象结构分离，确保对象结构的稳定性，同时允许灵活地添加新功能。

**类似装饰器模式**

**🧠 一、核心意图**

| 模式           | 核心意图                                                     |
| -------------- | ------------------------------------------------------------ |
| **访问者模式** | 在**不改变对象结构**的前提下，**为对象添加新的操作**。通常用于处理复杂对象结构（如组合结构、语法树等）。 |
| **装饰器模式** | 动态地为对象添加新的功能，**不影响其他对象**。强调功能增强，遵循开闭原则。 |



**🧩 二、使用场景**

| 模式           | 典型场景                                                     |
| -------------- | ------------------------------------------------------------ |
| **访问者模式** | 需要对一个对象结构（如树形结构）中的元素进行多种操作，但又不希望污染这些元素的类时。比如编译器的语法树遍历、财务系统的报表生成等。 |
| **装饰器模式** | 需要在运行时为一个对象动态添加、删除功能，比如为流（InputStream）添加缓冲、加密、压缩功能。 |

访问者设计模式通过引入访问者对象，允许我们在不修改对象结构的前提下，为对象添加新功能。在实际开发中，访问者模式可以帮助我们保持代码的开闭原则，提高系统的可扩展性和维护性。

# 解释器模式

解释器模式是一种设计模式，它主要用于**构建一个解释器，用于解释一种特定语法规则的语言或表达式**。

这类“语言”可以是数学表达式、搜索表达式、过滤条件等。

**主要作用：**

- **定义一种语言的语法规则**。
- **将语言中的每个规则都封装成类（表达式）**。
- **通过解释器解释并执行这些语法规则所表示的内容**。

每个表达式类负责解释一种语法规则  **涉及“传递上下文”**  流式处理

- 以**数据的结构和组织方式**为核心驱动系统设计。

# UML

- **UML是面向对象建模的标准，包含类图、用例图、状态机图等，关注对象、行为和交互。**

<span style="font-size:1.4em; color:#FF0000;">**一、结构图（Structural Diagrams）**</span>

描述系统的静态结构，展示系统组成部分及其关系：

1. <span style="color:#FF3333;">**类图（Class Diagram）**</span>
   - 最常用的图，展示类、接口、属性、方法及它们之间的关系（如继承、关联、依赖等）。
2. **对象图（Object Diagram）**
   - 类图的实例化，展示某一时刻对象的状态及其关系。
3. **组件图（Component Diagram）**
   - 描述系统的物理组件（如库、文件、模块）及其依赖关系。
4. <span style="color:#FF3333;">**部署图（Deployment Diagram）**</span>
   - 展示系统硬件节点（如服务器、设备）及软件组件在物理环境中的部署。
5. **包图（Package Diagram）**
   - 用包（Package）组织模型元素，表示模块化或命名空间。
6. **复合结构图（Composite Structure Diagram）**
   - 展示类的内部结构（如部件、端口、连接器），常用于复杂组件建模。



---



<span style="font-size:1.4em; color:#FF3333;">**二、行为图（Behavioral Diagrams）**</span>



描述系统的动态行为，包括交互和状态变化：

7. <span style="color:#FF3333;">**用例图（Use Case Diagram）**</span>

- 从用户角度描述系统功能（参与者、用例及其关系）。

1. <span style="color:#FF3333;">**活动图（Activity Diagram）**</span>
   - 类似流程图，展示业务流程或算法的步骤、分支和并行活动。
2. <span style="color:#FF3333;">**状态机图（State Machine Diagram）**</span>
   - 描述对象在其生命周期内的状态变化及触发事件（如订单状态流转）。
3. <span style="color:#FF3333;">**顺序图（Sequence Diagram）**</span>
   - 强调对象间交互的时间顺序，按时间轴展示消息传递。
4. <span style="color:#FF3333;">**通信图（Communication Diagram，原协作图）**</span>
   - 展示对象间的交互，侧重角色和连接关系而非时间顺序。
5. <span style="color:#FF3333;">**时序图（Timing Diagram）**</span>
   - **特殊顺序图，**详细描述**状态变化的时间约束**（如实时系统）。
6. **交互概览图（Interaction Overview Diagram）**
   - 结合活动图和顺序图，概括多个交互流程的高层视图。
7. **协议状态机图（Protocol State Machine Diagram）**
   - 描述协议或接口的行为规则（如状态转换的条件）。
