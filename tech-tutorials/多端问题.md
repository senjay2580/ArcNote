### 多端业务逻辑的通用注意事项

#### 1. 避免 “端侧硬编码业务逻辑”

- 核心逻辑（如权限判断、数据校验、订单状态流转）必须放在服务端，端侧仅做 “展示 + 调用接口”；
- 反例：移动端本地判断 “是否有权限删除数据”—— 攻击者可篡改本地代码绕过权限；
- 正例：端侧点击 “删除” 后，调用服务端删除接口，服务端校验权限后返回结果，端侧仅展示结果。

#### 2. 统一的业务规则引擎

- 多端的核心业务规则（如积分计算、会员等级、限流规则）需由服务端统一提供，避免 “Web 端积分 + 10，移动端 + 5”；
- 示例：会员等级升级规则（如消费满 1000 升级），服务端封装为接口，所有端调用该接口获取升级状态，而非本地计算。

#### 3. 端侧状态与服务端状态的 “最终一致性”

- 端侧可缓存 “非敏感状态”（如未读消息数、笔记列表），但必须提供 “手动刷新” 入口，且定期自动校验；
- 禁止端侧依赖本地缓存做 “核心决策”（如 “本地缓存显示会员未过期，就允许使用会员功能”），需每次调用接口校验。

#### 4. 多端的埋点与数据分析统一

- 埋点事件名称、参数需统一（如 “点击登录按钮”：event=click_login，参数 = device_type=web/desktop/ios）；
- 避免 “Web 端埋点叫 login_click，移动端叫 click_login_btn”，导致数据分析无法聚合。

```ad-info
1. **服务端中心化**：核心业务逻辑、权限、状态由服务端统一管控，端侧仅做 “展示层 + 操作层”；
    
2. **状态可校验**：多端的登录态、数据状态、权限状态均可被服务端校验，且端侧能实时感知变更；
    
3. **体验适配化**：同一业务逻辑在不同端的操作流程适配端侧特性（如移动端支付用指纹，桌面端用扫码），但核心规则一致；
    
4. **异常可兜底**：断网、接口报错、同步冲突等场景，有统一的兜底策略，避免用户操作失败且无反馈。
```
