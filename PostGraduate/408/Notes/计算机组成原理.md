> CPU 本身不认识 “这是指令还是数据”，它只认识 “现在处于哪个工作阶段”。

**CPU 靠「硬件状态机 + 节拍 / 时钟周期」来精准控制自己处于哪个阶段**，完全是硬件电路硬编码的时序逻辑

~~~
CPU 是一个同步时序电路，所有动作都由 ** 系统时钟（CLK）** 严格同步：
时钟就像一个节拍器，每来一个脉冲（时钟周期），CPU 就完成一个固定的微操作。
一条指令的完整执行（指令周期），会被拆成固定顺序的多个时钟节拍（机器周期 / 节拍），每个节拍对应一个阶段。
CPU 内部有一个状态寄存器 / 节拍计数器，记录当前走到了第几个节拍，也就知道了自己处于哪个阶段。
~~~

**指令周期 ＞ 机器周期(CPU 周期) ＞ 时钟周期**

## 计算机软件

**1、系统软件和应用软件**

计算机软件，一般分为**系统软件和应用软件**
系统软件：是一组保证计算机系统高效、正确运行的基础软件，通常作为系统资源提供给用户使用。包括操作系统，数据库管理系统，语言处理系统(比如编译器)，分布式软件系统，网络软件系统，标准库系统，服务性系统(比如连接程序)。
应用软件：是指用户为解决某个应用领域中的各类问题而编制的程序。包括各种科学计算类程序，工程设计类程序，数据统计与处理程序。
**2、三个级别的语言**

1. **机器语言。** 又称二进制代码语言，需要编程人员记忆每条指令的二进制编码。机器语言是计算机**唯一可以直接识别和执行的语言。**
2. **汇编语言。** 汇编语言用英文单词或其缩写代替二进制的指令代码，更容易为人们记忆和理解。使用汇编语言编辑的程序，必须经过一个称为**汇编程序**的系统软件的翻译，将其转换为计算机的机器语言后，才能在计算机的硬件系统上执行。
3. **高级语言**。高级语言(如 C、C++、Java 等)是为方便程序设计人员写出解决问题的处理方案和解题过程的程序。通常高级语言需要经过**编译程序编译成汇编语言程序**，然后经过**汇编操作得到机器语言程序**，或直接由**高级语言程序翻译成机器语言程序。**

**注：数据库管理系统和数据库系统是不一样的。数据库管理系统是系统软件。而数据库系统一般是由数据库，数据库管理系统，数据库管理员和应用系统构成。所以只能说它里面有系统软件，但并不能说它为系统软件。**

**机器语言是唯一可以控制 CPU 的语言，** 但是它的符号不利于人识别和书写，为了方便理解和记忆，将机器语言换一些通俗易懂的符号，这就变成了 **汇编语言**。一般来说在在编译器中高级语言先转换为汇编在转换为机器语言，也有直接转换为机器语言的情况。

**机器语言是计算机唯一可以直接执行的语言，汇编语言用助记符编写，以便记忆。**

而正则语言是编译原理中符合正则文法的语言。

**各种翻译程序的概念：**

汇编程序（汇编器）：将汇编语言程序翻译成机器语言程序。
解释程序（解释器）：将源程序中的语句按执行顺序逐条翻译成机器指令并立即执行。（Python、Shell）
编译程序（编译器）：翻译一句执行一句，边翻译边执行：由高级语言转化为汇编语言的过程称为编译，把汇编语言源程序翻译成机器语言程序的过程称为汇编。（exe，比如 C/C++、Java）
**软件和硬件的逻辑功能等价性：**

对某一个功能来说，既可以有软件实现又可以由硬件实现，从用户的角度来看，它们在功能上是等价的。这一等价性被称为**软/硬件逻辑功能等价性。**（eg：浮点数运算）

机器字长 = 寄存器字长 = 数据通路字长 

**指令字长 ≠ 机器字长**

**指令字长由 ISA 决定，不是由机器字长决定**	

- 机器字长：CPU 一次能处理多少位（硬件能力）
- 存储字长：内存一个格子存多少位（存储能力）
- 指令字长：一条指令占多少位（指令集设计）
- 字 / 字长：机器字长的通俗叫法，核心是 CPU 的处理能力
- ![image-20260407170521807](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20260407170521807.png)

吞吐量和响应时间是**一对矛盾指标**：

- 追求高吞吐量：系统会批量处理任务，单个任务的等待时间变长；
- 追求低响应时间：系统会优先处理单个请求，整体吞吐量会下降。

---



**最底层：时钟脉冲（时钟周期）**

CPU 的「心跳」，**1 个时钟周期 = 1 个脉冲 = 1 个最基础的微操作**（比如`PC→MAR`、`ALU加法`）

**中间层 1：微操作序列**

一条机器指令，会被拆解为**N 个微操作**，每个**微操作对应 1 个时钟周期**

**中间层 2：机器指令**

程序员 / 编译器能看到的「完整指令」（比如`LOAD`取数、`MUL`乘法），由**一段微操作序列**实现

**最上层：指令执行流程**

一条**机器指令**从「取指 → 分析 → 执行」的完整过程，由**多个机器周期**组成，每个机器周期包含多个时钟周期

1. 归零律（自己异或自己 = 0）
a⊕a=0
任何数和自己异或，结果一定是 0
2. 恒等律（和 0 异或不变）
a⊕0=a
任何数和 0 异或，还是它自己
3. 交换律
a⊕b=b⊕a
顺序随便换，结果不变
4. 结合律
(a⊕b)⊕c=a⊕(b⊕c)
括号随便挪，结果不变
5. 自反性（超级重要）
a⊕b⊕b=a
一个数异或同一个数两次，会变回去
用途：简单加密、还原数据
6. 二进制按位翻转（取反）
想把某一位翻转：
bit⊕1
0^1=1
1^1=0
想保持不变：
bit⊕0
7. 交换两个数（不用临时变量）
a=a⊕b
b=a⊕b
a=a⊕b
结果：a 和 b 交换了。
8. 判断两个数是否不同
a⊕b=0⇔a=b
结果为 0 → 相同
结果非 0 → 不同

**当需要给这个存储单元写入数据时：**

1. **选中单元**：给**字线加高电平**（图中①端给高），**MOS 管导通**，电容与位线连通。
2. **写入`1`**：给**位线加高电平（如 1.5V）**，电流通过导通的 MOS 管给电容充电，电容两端电压升高，存储电荷，代表`1`。
3. **写入`0`**：给**位线加低电平（如 0V）**，电容通过导通的 MOS 管放电，电荷流失，电压归零，代表`0`。
4. **完成写入**：撤除字线高电平，MOS 管截止，电容与位线断开，电荷被「锁存」在电容中，数据保持。

**读出操作（从电容取数据）**

当需要读取这个存储单元的数据时：

1. **选中单元**：给字线加高电平，MOS 管导通，电容与位线连通。

2. 电荷共享与检测

   ：电容的电荷会通过导通的 MOS 管流到位线，使位线电压发生微小变化：

   - 若电容存`1`（有电荷）：位线电压升高，检测电路识别为`1`；
   - 若电容存`0`（无电荷）：位线电压无明显变化，检测电路识别为`0`。

   

3. **破坏性读出与刷新**：⚠️ 关键特性：读出操作会**放掉电容的电荷**（破坏性读出），因此读完后必须立即「重写」（刷新），把数据写回电容，否则数据会丢失。

**数据保持与刷新（DRAM 的核心特点）**

- 电容不是理想元件，会通过 MOS 管的漏电流**缓慢漏电**，即使 MOS 管截止，电荷也会逐渐流失，因此数据无法永久保持。
- 为了防止数据丢失，必须**定期刷新**：每隔约 64ms，对所有存储单元执行一次「读出 - 重写」操作，给电容补充电荷，维持数据。
- 这也是「动态（Dynamic）」RAM 名字的由来：**数据是动态的，需要刷新才能维持**；而 SRAM（静态 RAM）用**触发器**存储，不需要刷新，但单元面积大、成本高。

以内存芯片为例，完整的读写流程里，片选是**第一步**：

1. CPU 发出地址信号，地址译码器根据地址，给**目标内存芯片**的`$\overline{CS}$`引脚发低电平（有效），选中该芯片。
2. 芯片被选中后，才会响应后续的`WE`（写使能）/`OE`（输出使能）信号，执行读写操作。
3. 读写完成后，`$\overline{CS}$`变回高电平（无效），芯片断开总线，回到休眠。

阶段 1：初始状态（最左侧，全灰色）

- **地址**：交叉 + 灰色 → 地址在跳变，不稳定、无效，RAM 不知道要读哪个单元
- ***\*CS\****：高电平 + 灰色 → 芯片未选中，不工作，处于待机
-  ***\*DOUT\****：灰色 → 数据不稳定，没有有效输出
- **状态**：读操作还没开始，CPU 准备发命令

阶段 2：地址有效（地址从灰色变白色，交叉消失）

- 地址

  ：交叉结束，变成两条平行平直线，进入白色稳定区 → 

  地址有效

  - 含义：CPU 已经给出了**确定、不变的地址**（比如 A12~A0 是固定的 0/1 组合），RAM 拿到了门牌号
  - 约束：从这一刻起，地址必须**全程保持稳定**，绝对不能中途跳变，否则读错单元

  

- ***\*CS\****：保持高电平 + 灰色 → 芯片还没被选中，不工作

- ***\*DOUT\****：保持灰色 → 数据还没开始输出

- 时间起点

  ：这是两个核心时间参数的起点：

  - tRC（读周期时间）：从「地址有效」开始计时
  - tA（存取时间）：从「地址有效」开始计时

  

阶段 3：片选有效（CS从高变低，拉低有效）

- **地址**：保持白色平直线，稳定不变 → 地址持续有效

- **CS**

  ：从高电平跳变到低电平，进入白色有效区 → 

  片选有效

  - 含义：CPU 告诉 RAM「我要读了，你开始工作吧」
  - 约束：CS必须在**地址稳定后**再拉低，不能先拉低再给地址，否则 RAM 读错
  - 时间起点：tCO（片选到输出稳定时间）从「CS有效」开始计时

  

- ***\*DOUT\****：还是灰色，但开始向白色过渡 → RAM 开始根据地址，从存储单元中读取数据，数据从无效向有效变化

- **状态**：RAM 正式启动读操作，开始寻址对应存储单元

阶段 4：数据稳定（DOUT从灰色变白色，数据有效）

- **地址**：保持白色平直线，稳定不变 → 地址持续有效

- ***\*CS\****：保持低电平白色区，持续选中 → 芯片持续工作

- **DOUT**

  ：交叉结束，变成白色平直线 → 

  数据稳定

  - 含义：RAM 已经把对应地址单元的数据，完整、正确地输出了，CPU 可以安全采样、读取数据

  - 时间终点：

    - tA（存取时间）：从「地址有效」到「数据稳定」的总时间，代表 CPU 给地址后，等多久能拿到数据
    - tCO（片选到输出稳定时间）：从「CS有效」到「数据稳定」的总时间，代表片选后等多久能拿到数据

    

  

- **核心结论**：这是读操作的**核心目标**，CPU 在这个时刻读取数据，完成读操作

阶段 5：操作结束（CS拉高，地址失效）

- ***\*CS\****：从低电平跳变到高电平，回到灰色区 → 片选失效，芯片取消选中，停止工作

- 地址

  ：从白色平直线变回交叉 + 灰色 → 

  地址失效

  - 含义：CPU 撤销地址，准备发下一个地址，本次读操作正式结束
  - 时间终点：tRC（读周期时间）：从「地址有效」到「地址失效」的总时间，代表**一次完整读操作的最小时间间隔**（两次连续读操作，必须至少间隔tRC，否则 RAM 来不及处理）

  

- ***\*DOUT\****：从白色平直线变回交叉 + 灰色 → 数据失效，撤销输出，准备下一次读操作

- **状态**：RAM 回到待机，等待下一次读命令

**指令 微指令 微操作**

**指令** 是计算机程序中的基本操作单位，它定义了计算机硬件执行的具体操作。

指令通常包括操作码（Opcode）和操作数（Operands），操作码指示了执行的操作类型（如加法、乘法、存储等），操作数则是操作所需的数据或地址。

在高级语言或汇编语言中编写的代码会被编译成一系列机器指令，这些指令直接由中央处理器（CPU）执行。

**主存（主内存，Main Memory）通常存储的是指令**

| 类型        | 内容                   | 说明                                    |
| ----------- | ---------------------- | --------------------------------------- |
| **指令**    | `ADD R1, R2, R3`       | `R1 = R2 + R3`，把 R2 和 R3 相加存到 R1 |
| **微指令 1** | `Load R2 → ALU.inputA` | 把 R2 寄存器的值送到 ALU 的 A 输入端    |
| **微指令 2** | `Load R3 → ALU.inputB` | 把 R3 的值送到 ALU 的 B 输入端          |
| **微指令 3** | `ALU.perform(ADD)`     | 执行 ALU 加法操作                       |
| **微指令 4** | `ALU.output → R1`      | 将 ALU 输出写回到寄存器 R1              |

---



**微指令** 是指一组更加底层、更加具体的控制信号序列，用于控制计算机中的微操作或硬件级别的操作。

微指令通常由控制单元（Control Unit）根据指令集架构（Instruction Set Architecture, ISA）中的指令来生成，它们直接控制硬件中的寄存器、算术逻辑单元（ALU）、存储单元等。

微指令的执行可以理解为对硬件内部具体操作的控制，如存储器的地址选择、数据传输路径的控制等。

**微指令则通常存储在控制单元内部的控制存储器（Control Memory）中，用于实现对硬件中各种微操作的精细控制。**

- **控制器（硬件）**：是工厂里的**物理流水线机器**，负责按顺序把一个个零件（指令）加工出来。它只认最底层的动作。
- **操作系统（软件）**：是工厂的**总调度厂长**，负责给机器安排任务、管理资源、处理突发状况。它指挥的是整个工厂的运转。



| 维度         | **控制器 (Control Unit)**                                    | **操作系统 (Operating System)**                              |
| ------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **本质**     | **硬件硬件硬件**是 CPU 内部的物理电路。                      | **软件软件软件**是运行在硬件上的程序集合。                   |
| **层级**     | **最底层（微架构）**直接控制门电路、触发器、总线。           | **中间层（系统架构）**介于硬件和用户应用之间。               |
| **核心任务** | 执行**一条一条**的机器指令。只做 “取指、译码、执行” 这三件事。 | 管理**整个系统**的资源。负责进程调度、内存管理、文件 IO 等。 |
| **工作粒度** | 微秒级，处理最精细的单个操作。                               | 毫秒 / 秒级，管理复杂的大型任务（如一个游戏、一个浏览器）。  |
| **主动性**   | 被动的。只有有指令给它，它才动，完全按顺序走。               | 主动的。它是 “老大”，决定什么时候运行哪个程序，什么时候响应中断。 |



虽然一个是硬件，一个是软件，但它们在工作时是**深度绑定、协同工作**的。

**1. 操作系统是 “指挥官”，控制器是 “执行者”**

当你在电脑上双击打开一个软件时，**操作系统**先去分配内存、加载程序代码。

- 当 CPU 开始运行这段代码时，**控制器**物理上把指令一条条读出来、译码并执行。
- **操作系统的核心功能（如进程切换），最终必须靠控制器的指令序列来实现。**



2. **中断是它们沟通的唯一语言**

这是两者联系最紧密的地方。

- **现象**：当你移动鼠标时，鼠标向 CPU 发出中断请求。
- **操作系统**：预先写好了 “中断处理程序”，告诉 CPU 收到这个信号该做什么。
- **控制器**：检测到中断信号，**立刻打断**当前正在执行的指令流（保存现场），然后跳转到操作系统预设的 “中断处理程序” 去执行（刷新屏幕显示鼠标移动）。

**每执行完一条指令后 cpu 都会检查是否有中断信号需要处理**



### 运算器的基本结构

![image-20250821175914753](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250821175914753.png)

CPU 的所有操作，**最终都要通过「寄存器」这个中间桥梁**完成

数据不能直接从内存跳进 ALU（运算器），太慢了。数据先从内存跑到 **通用寄存器**，再从寄存器送到 ALU 里计算。

ALU 算完的结果，先写回通用寄存器，而不是立刻写回内存。

~~~
内存是外部存储，CPU 内部总线、ALU 的输入输出，只能直接对接CPU 内部的寄存器（通用寄存器 R0~R3、累加器 ACC 等）。
内存和 CPU 之间的总线（地址总线、数据总线、控制总线），同一时间只能完成一次「读」或「写」操作，无法同时完成「从内存 A 读数据」+「向内存 B 写数据」两个动作。
要完成内存→内存的传输，必须分两步：
内存 A → CPU 寄存器（读操作，占用总线）
CPU 寄存器 → 内存 B（写操作，占用总线）
直接做「内存→内存」指令，硬件上需要额外的缓冲电路、总线仲裁，会大幅增加 CPU 复杂度，得不偿失。
~~~



![image-20250821180842372](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250821180842372.png)

**ALU 输入**：必须保证两路数据在时钟边沿同步到达（通过寄存器/时序控制实现）。

**ALU 输出**：必须加暂存器（输出寄存器），以保证结果稳定、**避免总线竞争，并支持流水线处理**

**结果未稳定就被使用**

- 假设 ALU 正在做加法 `R1 + R2 → R3`。
- ALU 内部的加法器电路传播延迟可能是 **10ns**。
- 在这 10ns 内，输出线上的信号还在“抖动”，不是稳定的结果。
- 如果这时结果直接放到总线上，其他单元可能读到的是错误值。

👉 **暂存器的作用**：在时钟沿到来时，锁存 ALU 输出，保证总线上拿到的是稳定结果




 输入端的信号不会同时到达 ALU 内部。例如：

- 输入 A 的信号先到；
- 输入 B 的信号稍微延迟几纳秒才到。
   在这段时间里，ALU 的输出就会根据“**未完成”的输入先产生一个临时结果，随着输入逐步到齐，输出会不断变化。**

**这种变化过程就是“抖动”**（Glitch）：
 在最终正确结果稳定前，输出可能会经历几次错误的高低电平变化

### **微操作是什么？**

- 微操作就是对寄存器级别的一次最小操作。
- 例如：
  - `PC → MAR`（把程序计数器送到存储器地址寄存器）
  - `MDR → IR`（把存储器数据寄存器的内容送到指令寄存器）
  - `AC ← AC + DR`（累加器加上数据寄存器内容）

> 这些都是最细粒度的操作，不能再拆分



### **控制信号是什么？**

- 微操作的实现，靠硬件控制信号来驱动。
- 比如：
  - 想执行 `PC → MAR`，硬件必须发出两个控制信号：
    - **PC 输出使能**
    - **MAR 写入使能**

### **IR（Instruction Register，指令寄存器）是什么？**

- **定义**：IR 是 CPU 内部的一个关键寄存器，用来 **存放当前正在执行的指令**。
- **来源**：指令从主存（内存）取出时，首先会进入 **MDR（存储数据寄存器）**，然后再传送到 **IR**。
- **作用**：一旦指令进入 IR，就说明这条指令已经被 CPU 取出，接下来 CPU 就会根据 IR 的内容来进行译码和执行

![image-20250821232802472](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250821232802472.png)

指令里写的地址分两种：

1. **直接地址**：框里直接是数据的真实地址，直接用；
2. **间接地址**：指令里给的不是数据地址，只是一个「地址的编号」，这个编号对应的内存单元里，**才存着真正的数据地址**。

间址周期具体做什么

1. 将指令中的形式地址送入内存；
2. 访问内存，取出里面存放的**有效地址**；
3. 把最终真实地址保存下来，给执行周期用；

### ==硬布线控制器==

控制单元采用硬布线控制器

硬布线控制器，是 CPU 里**产生控制信号的两种主流方式之一**，另一种是微程序控制器。

| 简称    | 全称                                      | 含义                                                         |
| ------- | ----------------------------------------- | ------------------------------------------------------------ |
| **FE**  | **Fetch Cycle / Instruction Fetch Cycle** | **取指周期**控制信号 `FE=1` 表示当前处于取指阶段，正在从内存中取出指令。 |
| **IND** | **Indirect Cycle**                        | **间址周期**控制信号 `IND=1` 表示当前处于间址阶段，正在计算并获取操作数的有效地址。 |
| **EX**  | **Execute Cycle**                         | **执行周期**控制信号 `EX=1` 表示当前处于执行阶段，正在执行指令的具体操作（取数、运算、存结果）。 |
| **INT** | **Interrupt Cycle**                       | **中断周期**控制信号 `INT=1` 表示当前处于中断阶段，正在处理中断请求，保存断点并准备跳转。 |

微命令是「控制信号」，微操作是「硬件动作」；微命令是因，微操作是果。

也就是每一个微命令和微操作都是一一对应的

| 概念       | 全称            | 本质                                                         | 例子                                                         |
| ---------- | --------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **微命令** | Micro Command   | 控制器发出的**控制信号**，是纯电信号（如高 / 低电平、脉冲），用来 “开 / 关” 硬件通路 | `MARin`（允许数据写入 MAR）、`MemR`（主存读信号）、`Yin`（允许数据写入 Y 寄存器） |
| **微操作** | Micro Operation | 硬件部件接收到微命令后，**执行的具体动作**，是数据通路里的一次数据传输 / 运算 | `PC→MAR`（PC 的内容送到 MAR）、`M(MAR)→MDR`（主存数据读到 MDR）、`ACC→ALU`（ACC 的数据送入 ALU） |

所以叫“硬布线（hardwired）”：控制逻辑在电路里是固定的，不依赖软件

**CPU 如何按「机器周期 + 节拍」，把一条指令拆成微操作并按时序执行**的完整流程

把一条机器指令，分解成一个个「微操作」，再按时间顺序（节拍）安排好，让 CU 按节拍发出控制信号，指挥 CPU 硬件完成指令。

是**CPU 的控制单元（CU）的设计者**，在设计硬件时就已经**固定死**的一套 “执行脚本”。

- 设计阶段：工程师要先为 CPU 的所有指令（取指、间址、执行），定义好：

  1. 这条指令要拆成哪些微操作（比如 `PC→MAR`、`M(MAR)→MDR`）
  2. 这些微操作按什么顺序执行（机器周期）
  3. 每个微操作放在哪个节拍（T₀/T₁/T₂）里

  

- 做成一张控制逻辑表 / 微程序，烧进 CU 里，让它知道：

  > 当现在是「取指周期的 T₀节拍」，就发出 `PCout` 和 `MARin` 信号。

  

  1. 时钟（节拍）：整个流程的 “指挥棒”

  - 时钟每跳一下，就切换到下一个节拍（T₀→T₁→T₂→下一条指令的 T₀）。
  - 节拍决定了 “现在 CPU 处于哪个时间阶段”，是 CU 判断该发什么信号的依据之一。

  2. 控制单元（CU）：流程的 “总指挥”

  CU 会根据条件，决定当前节拍发什么控制信号：

~~~
二进制本身一模一样，CPU 根本看不懂数字是什么意思。一串 01011001它可以是：
一条加法命令
一个内存地址
一个普通数字数据
一个负数
光看数字，完全分不清。

CPU 靠两大判断依据：
① 现在处于【取指周期】
只要是 取指阶段从内存读出来的东西一律当成：指令（命令）
硬件强制区分：
取指阶段读到的数据 → 硬件强制送入 IR 指令寄存器👉 进了 IR = 它就是命令、操作码CPU 直接当成指令解析。
② 只要是【执行周期 / 访存阶段】
从内存读出来的东西一律当成：操作数（纯数据）
硬件强制区分：
执行阶段读到的数据 → 送入 MDR → 累加器 ACC / Y 寄存器👉 进运算通路 = 它就是普通数据、数字
③ 那地址怎么区分？
凡是放进 MAR 的，一律当成地址不管二进制长什么样只要被送到 MAR（地址寄存器）硬件自动把它当成地址、发给主存、用来找存储单元。

**硬布线控制器（Hardwired Control Unit）**

直接用逻辑门电路（组合逻辑），把「当前节拍 + 指令操作码 + 状态条件」直接翻译成控制信号，不需要额外存储。

把控制信号的组合，编成一条条「微指令」，再组成「微程序」，存放在控制存储器（CM）里。执行指令时，按顺序读出微指令，发出对应的控制信号。

### ==微程序控制器的基本原理==  考的多

控制器发出的 **控制信号的组合** 就是微命令



![image-20250822104448568](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250822104448568.png)



内存是由 **存储单元阵列** 组成的（行 × 列），每个存储单元都可以看作一个小的触发器（存 bit）。

- 如果直接把 MAR 输出的二进制信号连到内存，所有存储单元都会“看到”这个地址。
- 但我们只希望 **唯一的一个单元** 被选中进行读/写。

👉 所以，需要一个 **地址译码器（Decoder）** 来做“二进制地址 → 独热码（one-hot）”的转换：

- MAR 输出 `n` 位地址（比如 16 位 → 可以寻址 2^16 = 65536 个单元）。
- 译码器把这个 16 位地址 **翻译成唯一的 1 条选通信号**，只激活一个存储单元，其余全为 0。



**微命令** → 控制产生 **微操作**

多个**微命令**打包 → 组成一条 **微指令**

多条 **微指令** 有序排列 → 组成一段 **微程序**

每一条**机器指令**，对应一段专属**微程序**

### 指令流水线

#### 基本概念和性能指标

![image-20250822172310642](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250822172310642.png)

**取指周期**：按 PC 地址从主存取指令到 IR，PC 自增。

**指令分析（译码）**：CU 识别操作码和寻址方式，决定后续步骤。

**（可选）间址周期**：如果是间接寻址，从主存中取出有效地址，更新 IR 的地址字段。

**执行周期**：根据分析结果，完成运算 / 访存 / 跳转等操作。

**（可选）中断周期**：如果有中断请求，保护断点并转中断服务程序。

#### 影响因素和分类

**IF、ID、EX、M、WB** 其实是 **处理器流水线（Pipeline）** 中对 **指令周期的进一步细分阶段**。它和你前面问的“取指周期、间址周期、执行周期、访存周期”是两种不同层次的划分：

- **取指/间址/执行/访存** → 是传统冯诺依曼体系结构里对整个 **指令周期（Instruction Cycle）** 的功能性划分。
- **IF/ID/EX/M/WB** → 是现代 CPU 在流水线技术中，把“执行一条指令”的工作拆成更细的流水阶段。

---



五级流水线阶段解释

1. **IF (Instruction Fetch, 取指阶段)**
   - 从内存（指令 Cache/主存）中取出指令。
   - 更新 PC（程序计数器）。
2. **ID (Instruction Decode, 指令译码阶段)**
   - 对指令进行译码，确定操作类型、寄存器地址、操作数位置等。
   - 读取寄存器堆（寄存器文件）中的源操作数。
3. **EX (Execute, 执行阶段)**
   - 由 ALU（算术逻辑单元）进行运算，例如加法、逻辑比较、地址计算等。
   - 如果是分支指令，会在这里计算跳转目标。
4. **M (Memory Access, 访存阶段)**
   - 如果指令需要访问数据内存（如 load/store），就在这一阶段完成。
   - 其他类型的指令可能跳过这一阶段。
5. **WB (Write Back, 回写阶段)**
   - 把运算结果或访存结果写回寄存器堆，以供后续指令使用。

---



**传统指令周期**（取指、间址、执行、访存）

### 五段式指令流水线

![image-20250824230326813](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250824230326813.png)

**IF（Instruction Fetch，取指令）**

- 从指令存储器（内存或 Cache）中取出指令。
- 使用 PC（程序计数器）提供地址，然后 PC 自增。
- 关键数据：PC、指令寄存器（IR）。

✅ 输出：指令 IR。







------

**ID（Instruction Decode，译码/读寄存器）**

- 对 IR 中的二进制指令进行译码，识别指令类型（算术、访存、跳转等）。
- 从寄存器堆中读出源操作数（rs、rt）。
- 计算分支目标地址（用于分支预测/转移）。

✅ 输出**：控制信号、源操作数、目标寄存器号。**



ALU（算术逻辑单元）执行什么操作，是由 **控制信号（Control Signals）** 决定的，而这些控制信号来源于 **指令译码阶段（ID）** 的控制单元



ID阶段的寄存器堆是指在指令解码阶段（Instruction Decode, ID）中使用的寄存器集合。寄存器堆用于存储和管理处理器中的数据，允许指令在执行时快速访问所需的操作数。



在这个阶段，处理器会解码指令并确定需要访问哪些寄存器。寄存器堆的设计和实现对处理器的性能有着重要影响，因为它直接关系到数据的读取和写入速度。



寄存器堆通常由多个寄存器组成，每个寄存器可以存储一定数量的位（如32位或64位）。在ID阶段，寄存器堆的内容会被读取，以便为后续的执行阶段提供必要的数据。

---



**EX（Execute，执行/运算）**

- ALU 执行实际操作：
  - 算术运算（加减乘除）
  - 逻辑运算（与或非异或）
  - 地址计算（基址 + 偏移）
- 如果是分支指令，此时判断条件是否满足。

✅ 输出：ALU 结果（可能是数据或地址）。

------

**MEM（Memory Access，访存）**

- 如果指令是 **load/store**，需要访问数据存储器：
  - load：取出数据
  - store：写入数据
- 如果是算术/逻辑指令，则直接跳过此阶段。

✅ 输出：内存数据 or ALU 结果。

------

**WB（Write Back，写回寄存器）**

- 将结果写回寄存器堆（如 `rd` 或 `rt`）。
- load 的数据 / ALU 的结果最终存入寄存器，供后续指令使用。

✅ 输出：更新后的寄存器堆。

**把运算结果存回寄存器堆**

- 之前 ALU 或访存阶段得到的结果，需要写回寄存器，才能被后续指令使用

**数据总线**是一组“物理导线”，这些导线本身没有固定“谁发谁收”，而是**根据操作（读 or 写）动态决定方向**

**控制总线**是一堆**专用的控制信号线**，每一根线的方向是固定的



---



地址总线的作用：**CPU 发出“我要访问的地址”**。

- 地址总线只需要从 **CPU → 存储器/外设** 单向传递。
- 比如 CPU 想读内存地址 `0x1000`，就把 `0x1000` 放到地址总线上，内存芯片看到这个地址后，才知道应该去哪个存储单元读写。
- 所以地址总线只要单向，没必要“反过来传地址”。

结论：**地址总线是单向的，从 CPU 指向外设/内存。**



数据总线是用来搬运数据的。数据既可能是：

- **CPU → 内存/外设**（写操作，比如 CPU 把一个数写入内存）。
- **内存/外设 → CPU**（读操作，比如 CPU 从内存取指令，或从硬盘读数据）。

所以数据总线必须能双向传输。

结论：**数据总线是双向的，方向取决于读/写。**



控制总线就比较有意思，它包含了各种**控制信号**，**方向上有双向的情况：**

- **CPU → 外设/内存**
  - 读信号（RD）、写信号（WR）、内存读、内存写、I/O 读、I/O 写等。
  - 中断响应信号（INTA）等。
- **外设/内存 → CPU**
  - 中断请求（INT）、DMA 请求（DRQ）、准备好信号（READY）等。

所以控制总线的方向是 **双向的，但具体一根线是有方向性的**。

- 比如 `RD` 信号一定是 CPU 发的，
- 而 `INT` 信号一定是外设发的。

整体上，控制总线是一组既有输入线也有输出线的混合集合

~~~
🔹 1. 同步总线（Synchronous）
核心机制
所有传输严格按统一的总线时钟（CLK）进行，每步动作占固定时钟周期数。


CLK    ┌┐┌┐┌┐┌┐┌┐
    ───┘└┘└┘└┘└┘└───
       T1 T2 T3 T4

固定 4 个周期完成一次传输 —— 不管对方多快多慢
特点
项	内容
时钟来源	统一 CLK
需要反馈信号	不需要
速度	最快（无握手开销）
缺点	迁就最慢设备 —— 快设备也要等
适用	片内总线、CPU-Cache、CPU-内存紧耦合


 2. 异步总线（Asynchronous）
核心机制
没有统一时钟，靠请求 Req + 应答 Ack 握手推进。


Req ───┐        ┌─────     CPU："准备好了吗？"
       └────────┘          
Ack ──────┐  ┌────────     外设："好了！"
          └──┘

每一步都要等对方确认 —— 完全自适应速度
三种互锁方式（之前讲过，这里简要回顾）
方式	特点	速度	可靠性
不互锁	谁都不等	最快	最差
半互锁	主方等	中	中
全互锁	双方都等	最慢	最高
特点
项	内容
时钟	无统一 CLK
需要反馈信号	必需（Ack 就是反馈信号）
速度	比同步慢（有握手开销）
优点	灵活，适配任意速度设备
适用	长距离、速度差异大的连接，如 CPU ↔ 慢速外设

~~~

### 总线仲裁（了解即可 不是考点）

~~~
总线仲裁（考频：中，以选择题为主）
会考的 3 种集中仲裁方式对比
方式	核心特点	优先级	线数	故障敏感度
链式查询	总线响应信号串行经过各设备	固定（离仲裁器越近越高）	最少（3 根控制线：BR/BG/BS）	最敏感（任一设备坏 → 后面全瘫）

必记三个"最"

链式查询：线最少、速度最慢、优先级最死
独立请求：线最多、速度最快、最灵活
计数器：折中
高频考点
给场景选方式
设备少、优先级固定 → 链式
要求高响应速度 → 独立请求
链式查询的致命缺点：对硬件故障敏感（前一台坏，后面全收不到 BG）
独立请求线数 = 2N（N 个设备，BR + BG 各 N 根）
分布式仲裁（自举分布式、并行竞争）基本不考计算，只需认识名词。
~~~

### 总线标准（了解即可 不是考点）

**总线标准 = 不同厂家的设备之间通信的"统一规矩"**。 规定了机械、电气、功能、时序四大特性的具体参数，**让任何遵循同一标准的设备都能互相插拔、互相通信**。

~~~
总线标准（考频：低-中，纯选择题记名词）
必记的 4 个对比维度
标准	并行/串行	用途	拓扑	热插拔
PCI	并行	系统 I/O 总线（老）	多点共享	否
PCIe	串行（点对点）	现代显卡/SSD	点对点（非共享）	是
USB	串行	外设	树形（通过 Hub）	是
SATA	串行	硬盘	点对点	是
4 个绝对会考的点
PCIe 是串行、点对点（✕ 并行、✕ 共享）—— 这是最大坑点，很多人以为"P 开头都是并行"
USB 是树形拓扑，通过 Hub 扩展，最多 127 个设备
USB 支持热插拔（几乎必考）
PCIe vs PCI：PCIe 用串行差分对传输，性能远高于并行 PCI
容易混的冷门名词（见过会选即可）
AGP：专为显卡设计，已被 PCIe 取代
ISA：上古并行总线（了解）
FireWire (IEEE 1394)：苹果系外设总线



~~~

局部总线  →  高速扩展卡         VESA / PCI / AGP / PCIe
设备总线  →  外设              RS-232 / SCSI / USB / IDE / SATA



| **串行（6 个）** | **并行（9 个）**        |
| ---------------- | ----------------------- |
| FBS / QPI        | ISA / EISA              |
| **PCIe**         | VESA                    |
| **USB**          | **PCI** / AGP           |
| **SATA**         | **IDE** / SCSI / PCMCIA |
| RS-232C          |                         |



~~~
1. PCI
并行 + 同步 + 共享总线
1992 年，局部总线
工作频率独立于 CPU 主频 ⭐
支持即插即用、突发传输

2. PCIe（PCI-E）
串行 + 点对点 + 全双工
2001 年，局部总线
支持 x1 / x4 / x8 / x16 多通道
嵌入式时钟（不需独立 CLK 线）
不需要仲裁

3. USB
串行 + 树形 + 主从结构
1996 年，设备总线
差模（差分）信号，每次 1 bit
最多 127 个设备
支持热插拔 + 即插即用
设备不能主动发起传输

4. SATA
串行 + 点对点
2001 年，替代并行 IDE
支持热插拔
连接硬盘、光驱

5. IDE / ATA
并行，1986 年
又称 Parallel ATA
连接硬盘、光驱
已被 SATA 取代

- **高速外设（比如磁盘）**：用 DMA，因为数据量大、传输快，CPU 根本忙不过来；
- **低速外设（比如打印机、键盘）**：不用 DMA，因为数据量小、速度慢，CPU 自己处理就行。

![image-20250822233419025](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250822233419025.png)

![image-20250822233638247](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250822233638247.png)

通道（I/O Channel），是一种**专门用来管理 I/O 的、有独立指令集的处理器**，也叫 “通道处理器”。

它的本质，是一个**能独立执行 I/O 指令的专用 CPU**，专门接管 CPU 的 I/O 管理工作。

![image-20250822234134876](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250822234134876.png)

**操作码**：CPU 层面的指令类型，告诉 CPU 要对 I/O 接口做什么（比如读、写、控制）。

**命令码**：接口层面的具体操作，告诉 I/O 接口要对设备做什么（比如磁盘寻道、打印机换行）。

**设备码**：指定要操作哪个 I/O 设备（比如 0 号磁盘、1 号打印机）。



I/O 接口的作用：

- 接口里有**数据缓冲寄存器**，设备把数据先存在接口里，CPU 可以先干别的事，等准备好再一次性读取，解决了速度不匹配的问题。

- **CPU 的信号是标准的数字信号（固定电压、时序），用的是 CPU 自己的总线协议。**

- **不同外设的信号五花八门：**

  - 键盘、鼠标是串行信号；
  - 磁盘是专用的 ATA/SCSI 信号；
  - 显示器是模拟 / 视频信号；
  - 打印机是并口 / USB 信号。

  

**CPU 根本没法直接识别这些乱七八糟的信号，必须靠 I/O 接口：**

- 接口负责把外设的信号转换成 CPU 能懂的标准信号，再发给 CPU；
- 也把 CPU 的指令转换成外设能懂的信号，发给设备。



**不同外设的控制逻辑完全不同：**

- 磁盘需要控制磁头定位、扇区读写、错误校验；
- 打印机需要控制字车移动、走纸、加热；
- 键盘需要处理按键抖动、扫描码转换。

如果让 CPU 直接控制这些细节，CPU 的指令集、控制逻辑会变得无比复杂，而且换个新设备就要改 CPU 设计，完全不现实。

**I/O 接口的作用：**

- 把不同设备的控制逻辑封装起来，CPU 只需要和统一的接口通信，不用管设备的具体细节；
- 比如 CPU 发一个 “读数据” 的指令，接口会自动处理磁盘的寻道、等待，准备好数据再通知 CPU。



**驱动程序**为什么是 I/O 软件？

- 驱动程序直接负责和 **具体硬件** 打交道。
- 它知道**设备**寄存器的**格式**、初始化方式、中断处理等。
- 操作系统并不关心硬件细节，而是通过调用驱动，间接控制设备。



![image-20250822234146666](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250822234146666.png)

I/O 控制方式 = CPU 「怎么控制外设读写数据」的四种干活模式 就是：CPU 到底用什么办法，把内存 和 外设（键盘 / 硬盘 / 打印机）之间的数据搬来搬去。

屏幕上正在显示的每一帧画面，都先完整存在 VRAM 里，显卡再从 VRAM 里读取数据，转换成信号输出给显示器。

> 像素点会 “消失”：无论是 CRT 还是 LCD 显示器，每个像素点的光只能维持极短的时间，必须不断刷新才能保持亮度。 
>
> 显卡需要持续输出信号：为了保证画面不消失，显卡必须每秒多次把一整帧图像数据发给显示器。 
>
> VRAM 就是 “图像仓库”：显卡把一整帧图像提前存在 VRAM 里，刷新时直接从 VRAM 读取数据，再输出给显示器，保证画面稳定。

### 多重中断

屏蔽字（中断屏蔽字）

**中断屏蔽字（Interrupt Mask / Interrupt Mask Register, IMR）** 是 CPU 或外设中用于 **控制中断响应的位集合**。它决定了哪些中断可以被允许，哪些中断被屏蔽。

**避免中断冲突**：当 CPU 正在处理重要任务或当前中断时，可以临时屏蔽其他中断。

~~~
8259A + 8086 完整中断响应流程（以 IR3 为例，带考点标注）
0. 前置条件（系统初始化阶段）
8259A 已完成初始化：设置了基向量号、优先级方式、EOI 模式等。
CPU 已执行 STI 指令，开中断（IF=1）。
外设（如串口、硬盘）已准备好数据，等待向 CPU 发送中断。

1. 外设发中断请求（外设 → 8259A）
外设完成数据传输或状态变化，通过 IR3 引脚向 8259A 发送高电平请求信号。
8259A 收到信号后，硬件自动将 IRR（中断请求寄存器）的第 3 位置 1，锁存这个请求。
考点：IRR 的作用是「锁存所有未处理的中断请求」，直到被响应或被屏蔽。

2. 中断屏蔽检查（8259A 内部）
8259A 将 IRR 的请求信号，与 IMR（中断屏蔽寄存器）逐位相与。
若 IMR[3] = 0（未屏蔽），则 IR3 的请求信号通过屏蔽检查，送入优先权电路；若为 1 则被直接丢弃。
考点：IMR 是操作系统实现「中断屏蔽 / 关中断」的硬件基础，可屏蔽中断都受它控制。

3. 优先权判优（8259A 内部）
优先权电路同时接收所有通过屏蔽检查的请求，按预设优先级（默认IR0>IR1>…>IR7）选出优先级最高的请求。
若此时没有更高优先级的请求，IR3 被选中，送入控制逻辑；若有更高优先级（如 IR1），则 IR3 请求暂存，等待更高优先级中断处理完毕。
考点：这是实现中断嵌套的关键步骤，优先级高的中断可以打断低优先级中断的服务过程。

4. 向 CPU 发中断请求（8259A → CPU）
控制逻辑确认 IR3 是当前最高优先级请求后，通过 INT 引脚向 8086 的 INTR 引脚发送高电平信号。
这个信号就是「可屏蔽中断请求信号」，告诉 CPU：“有外设需要处理，请响应我”。
考点：INTR 是可屏蔽中断请求输入，受 IF 标志位控制；NMI 不受控制，必须响应。

5. CPU 在指令周期结束时检测中断请求（CPU 内部）
CPU 执行指令时，会在每条指令的最后一个时钟周期（指令结束时刻），检测 INTR 引脚的信号。
同时检查 IF 标志位：
若 IF=1（开中断）且 INTR=1，CPU 准备响应中断；
若 IF=0（关中断），CPU 会忽略 INTR 请求，继续执行下一条指令。
考点：CPU 只能在一条指令执行完毕后响应中断，指令执行中途不会响应。

6. CPU 自动完成「中断响应」的硬件操作（无需软件干预）
CPU 同意响应中断后，会自动执行以下步骤（这是 408 大题的核心）：
关中断：自动清除 IF 标志位（IF=0），防止后续中断打断当前的断点保存过程，保证操作的原子性。
保存断点与现场关键信息：将当前程序的 FLAGS、CS、IP 依次压入堆栈保存（断点就是下一条要执行指令的 CS:IP）。
发中断响应信号 INTA：CPU 通过 INTA 引脚，向 8259A 发送两个连续的负脉冲信号。
考点：这一步是 CPU 硬件自动完成的，不需要程序员写指令，也是中断和子程序调用的核心区别之一。

7. 8259A 回送中断向量号（CPU ↔ 8259A）
8259A 收到第一个 INTA 脉冲后，将 ISR 的第 3 位置 1，标记该中断正在处理；同时清除 IRR 的第 3 位，表示请求已被响应。
收到第二个 INTA 脉冲后，8259A 将 IR3 对应的 ** 中断向量号（n）** 送到数据总线上，CPU 读取该向量号（例如0BH）。
考点：两个 INTA 脉冲的分工：第一个置位 ISR，第二个送向量号；向量号由 8259A 的基向量号 + IR 引脚号生成。

8. CPU 计算中断向量地址，跳转至中断服务程序（CPU 内部）
CPU 根据向量号 n，计算中断向量在内存中的地址：
中断向量地址=n×4
以 n=0BH 为例：0BH × 4 = 002CH，CPU 从内存 0000:002CH 处读取 4 个字节（2 字节 IP + 2 字节 CS），得到中断服务程序的入口地址。
CPU 将 CS:IP 加载为中断服务程序的入口地址，开始执行中断服务程序。
考点：中断向量表固定在内存低地址区，每个向量占 4 字节，向量地址公式必须会背。

9. 中断服务程序开头：软件保护现场（程序员写的）
中断服务程序执行的第一条指令，通常是保护通用寄存器、段寄存器等现场信息，将它们压入堆栈保存，避免破坏原程序的数据。
asm
PUSH AX   ; 保存AX寄存器
PUSH BX   ; 保存BX寄存器
PUSH CX   ; 保存CX寄存器
STI       ; 开中断，允许更高优先级的中断嵌套（可选）
考点：硬件只保存了 FLAGS、CS、IP，通用寄存器需要软件手动保护，这是大题常考的细节。

10. 执行中断服务程序的核心逻辑（处理外设事件）
针对 IR3 对应的外设，执行具体的处理操作：
若是串口中断：读取数据寄存器，处理接收 / 发送的数据；
若是硬盘中断：读取状态寄存器，确认传输完成，处理数据块；
若是时钟中断：更新系统时间，调度进程等。
处理完成后，准备退出中断服务程序。

11. 中断服务程序结尾：恢复现场与中断结束（程序员写的）
关中断（可选，根据需要）：执行 CLI 指令，防止恢复现场过程中被打断。
恢复现场：从堆栈中弹出之前保存的通用寄存器，恢复原程序的运行环境。
asm
POP CX
POP BX
POP AX
发送 EOI（中断结束）命令：向 8259A 发送 EOI 命令，清除 ISR 的第 3 位，表示该中断已处理完毕，允许同级或更低优先级的中断被响应。
asm
MOV AL, 20H   ; EOI命令字
OUT 20H, AL   ; 写入8259A的控制端口
考点：EOI 命令是用来清除 ISR 位的，不发 EOI 会导致后续同级中断无法被响应。

12. 中断返回（CPU 硬件自动完成）
中断服务程序的最后一条指令是 IRET（中断返回指令）。
CPU 执行 IRET 时，自动从堆栈中依次弹出 IP、CS、FLAGS，恢复原程序的断点和标志位，同时 IF 标志位也被恢复为开中断状态。
CPU 回到原主程序的断点处，继续执行被打断的指令。
考点：IRET 指令和普通的 RET 指令不同，它会同时恢复 FLAGS 标志位，是中断返回的专用指令。
~~~



**中断请求触发器**：外设发出中断请求，锁存请求信号

**中断屏蔽触发器**：控制是否屏蔽该中断（可屏蔽中断用）

**中断向量 / 中断向量地址**：存放中断服务程序入口

**程序状态字 PSW**：保存断点现场、中断标志位

**中断判优电路**：多个中断同时到达时，决定优先级



**步骤 1：中断请求**

外设 / 异常源完成工作或发生故障，向 CPU 发送**中断请求信号**，硬件自动置位中断请求标记。



**步骤 2：中断判优与中断屏蔽**

- 若多个中断同时到来，硬件按**优先级**排队（硬件判优 / 软件判优）；
- 若该中断被**屏蔽**，则暂时不响应；
- 不可屏蔽中断（如掉电）直接跳过屏蔽，强制响应。



**步骤 3：中断响应（CPU 自动完成，指令周期结束时刻检测）**

CPU 在**每条指令执行结束、取下一条指令前**，统一检测中断请求：

1. 关中断：自动清除中断允许标志，防止现场保存期间被新中断打断；
2. **保存断点**：将当前程序的 **PC（程序计数器）、PSW** 压入堆栈保存；
3. 识别中断源：获取**中断向量地址**；
4. 转入中断服务程序：将中断服务程序入口地址送入 PC。

> 重点：**中断响应是硬件自动操作，无需程序员干预**



**步骤 4：中断现场保护（中断服务程序开头，软件完成）**

除了硬件保存的 PC/PSW，还需要手动保存**通用寄存器、累加器**等通用现场，避免破坏原程序数据。



**步骤 5：执行中断服务程序**

针对性处理中断事件：

- I/O 中断：完成数据读写、外设状态处理；
- 异常中断：报错、回滚、缺页处理等；



**步骤 6：中断返回**

1. 恢复现场：从堆栈还原通用寄存器、累加器；
2. **开中断**：恢复中断允许，允许后续中断；
3. 中断返回指令（IRET）：从堆栈弹出断点 PC、PSW；
4. CPU 回到原主程序**断点处**，继续执行原程序。

**I/O 设备地址（I/O Address）** —— CPU 发出的地址信号，决定访问哪个设备的哪个寄存器



**初始化阶段（CPU 参与）**

- CPU **配置** DMA 控制器：
  - **起始地址寄存器** → 要写入或读出的内存地址。
  - **计数寄存器** → 传输的数据字数（多少个字）。
  - **控制寄存器** → 传输方向（I/O → 内存，还是内存 → I/O），工作方式（突发、周期窃取、块传送等）。
- CPU 发出启动命令，让 DMA 控制器接管。



**总线申请与占用阶段**

- 外设准备好数据（或需要数据），向 DMA 控制器发出 **DMA 请求（DRQ）**。
- DMA 控制器向 CPU 发出 **总线请求（BR，Bus Request）**。
- CPU 执行完当前总线周期后，响应 **总线授权（BG，Bus Grant）**，把总线控制权交给 DMA 控制器。
- 此时 CPU 让出总线，暂停对内存的直接访问



**数据传送阶段（DMA 执行）**

DMA 控制器完全接管总线，直接在 **外设 ↔ 内存** 之间搬运数据：

1. **发出内存地址**（来自起始地址寄存器）。
2. **产生读/写控制信号**：
   - 若是 I/O→内存：DMA 先发出 I/O 读，再发内存写。
   - 若是 内存→I/O：DMA 先发内存读，再发 I/O 写。
3. **数据传输完成后**：
   - 地址寄存器 +1，指向下一个字。
   - 计数寄存器 -1，剩余字数减少。
4. **重复上述步骤**，直到计数器减到 0



**传输结束与恢复**

- 当计数寄存器 = 0，DMA 控制器完成一批传输：
  - 向外设发出 **DMA 结束信号（DACK）**。
  - 向 CPU 发送一个 **中断信号**，通知传输完成。
- CPU 收到中断后，执行相应的中断服务程序，继续正常运行。
- 总线控制权归还给 CPU





 ![image-20250823173538666](https://cdn.jsdelivr.net/gh/kasahuki/os_test@main/img/image-20250823173538666.png)



**CPU 在很多时候（译码、执行阶段）是不使用主存总线的**。DMA 控制器就是利用这些「CPU 不访存的间隙」，或者「CPU 访存的短暂间隔」，去 “借” 一个主存周期来完成数据传输。
